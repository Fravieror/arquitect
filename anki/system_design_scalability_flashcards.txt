#deck:Senior Architect :: System Design & Scalability
What is Scalability in software systems?	Scalability is the ability of a system to handle more work by adding resources — without breaking or slowing down.<br><br><b>Analogy:</b> Think of a restaurant. A scalable restaurant can serve more customers by adding more tables, chefs, or even opening new locations — without the kitchen catching fire.<br><br><b>Two main types:</b><br>- <b>Vertical scaling (scale up):</b> Make the existing machine bigger (more CPU, RAM).<br>- <b>Horizontal scaling (scale out):</b> Add more machines.<br><br><b>Senior insight:</b> Vertical scaling has a ceiling (you can only buy so much RAM). Horizontal scaling is the goal for large systems, but it adds complexity (distributed state, coordination, networking).
What is Vertical Scaling (Scale Up)?	Adding more power to a single machine — more CPU cores, RAM, or faster disks.<br><br><b>Example:</b> Your server has 8 GB RAM and starts struggling. You upgrade it to 64 GB RAM.<br><br><b>Pros:</b><br>- Simple — no code changes needed.<br>- No distributed system complexity.<br><br><b>Cons:</b><br>- Has a hard limit (the biggest machine available).<br>- Expensive at high tiers.<br>- Single point of failure — if it dies, everything dies.<br><br><b>Common mistake:</b> Treating vertical scaling as a long-term solution. Use it to buy time, not as a strategy.
What is Horizontal Scaling (Scale Out)?	Adding more machines (nodes/instances) to distribute the load instead of making one machine bigger.<br><br><b>Example:</b> Instead of one server handling 10,000 requests/sec, you run 10 servers each handling 1,000 req/sec behind a load balancer.<br><br><b>Pros:</b><br>- Theoretically unlimited scale.<br>- Better fault tolerance (one node dies, others keep going).<br><br><b>Cons:</b><br>- Requires stateless services (or shared state management).<br>- More infrastructure complexity.<br>- Network latency between nodes.<br><br><b>Senior insight:</b> Horizontal scaling forces you to solve hard problems: session management, distributed caching, data consistency.
What is a Load Balancer and why is it needed?	A load balancer sits in front of your servers and distributes incoming requests across them so no single server gets overwhelmed.<br><br><b>Analogy:</b> A supermarket with 10 checkout lanes. The greeter at the door tells each customer which lane to go to — that's the load balancer.<br><br><b>Algorithms:</b><br>- <b>Round Robin:</b> Requests go to servers in order (1,2,3,1,2,3...).<br>- <b>Least Connections:</b> Goes to the server with fewest active connections.<br>- <b>IP Hash:</b> Same client always goes to same server (useful for session stickiness).<br><br><b>Trade-off:</b> The load balancer itself can become a bottleneck. Use multiple LBs or managed cloud services (AWS ALB, Azure Front Door).
What is Stateless vs Stateful architecture?	<b>Stateless:</b> Each request contains all the information needed to process it. The server doesn't remember previous requests.<br><br><b>Stateful:</b> The server keeps information about the client between requests (session data, connection state).<br><br><b>Example (stateless):</b> REST API with JWT token — every request carries the auth token. Any server can handle it.<br><br><b>Example (stateful):</b> WebSocket connection — the server keeps the socket open and remembers who you are.<br><br><b>Why it matters:</b> Stateless services scale horizontally with no friction. Stateful services need sticky sessions or shared state (Redis), which adds complexity.<br><br><b>Rule of thumb:</b> Design services to be stateless. Offload state to a shared store (database, Redis, cookie).
What is Database Sharding?	Sharding splits a large database into smaller pieces called <b>shards</b>, each stored on a different server. Each shard holds a subset of the data.<br><br><b>Analogy:</b> A phone book split alphabetically — A–F on server 1, G–M on server 2, N–Z on server 3.<br><br><b>Shard key:</b> The column/field used to decide which shard a record goes to (e.g., user ID, region).<br><br><b>Pros:</b><br>- Distributes read and write load.<br>- Allows massive datasets.<br><br><b>Cons:</b><br>- Cross-shard queries are painful and slow.<br>- Rebalancing shards when data grows unevenly is complex.<br>- Transactions across shards are hard.<br><br><b>Common mistake:</b> Choosing a bad shard key that creates "hot shards" (one shard gets 90% of traffic).
What is Database Replication?	Replication copies data from a primary (master) database to one or more replicas (secondaries/slaves).<br><br><b>Why:</b><br>- <b>Read scaling:</b> Send reads to replicas, writes to primary.<br>- <b>High availability:</b> If primary dies, promote a replica.<br><br><b>Types:</b><br>- <b>Synchronous:</b> Write confirmed only after replica also writes. Safe, but slower.<br>- <b>Asynchronous:</b> Write confirmed immediately, replica catches up later. Fast, but risk of data loss on failover.<br><br><b>Common mistake:</b> Reading from a replica and getting stale data (replication lag). Don't read your own writes from a replica right after writing.<br><br><b>Senior insight:</b> Replication helps with reads but doesn't help with write throughput — for that you need sharding or a different data model.
What is a CDN (Content Delivery Network)?	A CDN is a network of servers distributed around the world that cache and serve static content (images, CSS, JS, videos) from the server closest to the user.<br><br><b>Analogy:</b> Instead of everyone flying to Paris to visit the Louvre, the Louvre makes copies of paintings and displays them in local museums worldwide.<br><br><b>Benefits:</b><br>- Reduces latency (content served from nearby server).<br>- Reduces load on your origin server.<br>- Protects against traffic spikes.<br><br><b>Examples:</b> Cloudflare, AWS CloudFront, Akamai.<br><br><b>Senior insight:</b> CDNs can also cache API responses (edge caching) and handle DDoS mitigation. Cache invalidation strategy is critical — stale CDN cache is a common bug source.
What is Caching and what problem does it solve?	Caching stores the result of an expensive operation so the next request can get it instantly without repeating the work.<br><br><b>Analogy:</b> You look up a phone number, write it on a sticky note (cache). Next time you need it, you read the note instead of searching again.<br><br><b>What gets cached:</b> Database query results, computed values, API responses, HTML pages.<br><br><b>Common cache stores:</b> Redis, Memcached, in-memory dictionaries.<br><br><b>The hard part — Cache Invalidation:</b> How do you update or remove stale data?<br>- TTL (Time To Live): cache expires after N seconds.<br>- Event-based: invalidate when data changes.<br><br><b>Common mistake:</b> Caching mutable data without a proper invalidation strategy — users see outdated information.
What are the main Caching Strategies?	<b>1. Cache-Aside (Lazy Loading):</b> App checks cache first. On miss, fetches from DB and populates cache.<br><code>value = cache.get(key) or (db.get(key) then cache.set(key, value))</code><br><br><b>2. Write-Through:</b> Every write goes to cache AND DB simultaneously. Cache always fresh, but writes are slower.<br><br><b>3. Write-Behind (Write-Back):</b> Write to cache immediately, DB is updated asynchronously later. Fast writes, risk of data loss.<br><br><b>4. Read-Through:</b> Cache sits in front of DB and fetches automatically on miss (library handles it).<br><br><b>Trade-offs:</b><br>- Cache-aside: simple but cache miss on first read.<br>- Write-through: consistent but doubles write latency.<br>- Write-behind: fastest but risky if cache crashes before DB write.<br><br><b>Most common in practice:</b> Cache-aside with TTL.
What is a Message Queue and when should you use one?	A message queue is a buffer that holds messages (tasks/events) sent by a producer until a consumer processes them. Producer and consumer are decoupled — they don't talk directly.<br><br><b>Analogy:</b> A restaurant order ticket system. Waiters (producers) put orders on a ticket rail. Chefs (consumers) grab tickets and cook. If the kitchen is slow, tickets pile up — but the waiter keeps taking orders.<br><br><b>When to use:</b><br>- When processing is slow and you don't want the user to wait (e.g., sending emails, resizing images).<br>- When you need to smooth out traffic spikes.<br>- When services need to be decoupled.<br><br><b>Examples:</b> RabbitMQ, AWS SQS, Kafka.<br><br><b>Senior insight:</b> Queues enable async processing and resilience. If the consumer crashes, messages are still in the queue to be retried.
What is the difference between a Message Queue and an Event Streaming platform (Kafka vs RabbitMQ)?	<b>Message Queue (RabbitMQ, SQS):</b><br>- Message is consumed once and deleted.<br>- Push-based: broker pushes messages to consumers.<br>- Great for task distribution, job queues.<br>- Simple routing rules (exchanges, topics).<br><br><b>Event Streaming (Kafka, Kinesis):</b><br>- Messages are retained for a configurable period (days/weeks).<br>- Consumers pull messages and track their own offset.<br>- Multiple consumers can read the same event independently.<br>- Great for audit logs, event sourcing, real-time analytics.<br><br><b>Key question:</b> Do multiple services need to react to the same event? → Kafka. Is this a one-time task? → Queue.<br><br><b>Common mistake:</b> Using Kafka for simple job queues (overkill). Using RabbitMQ when you need message replay or multiple independent consumers.
What is the CAP Theorem?	CAP Theorem says that a distributed system can only guarantee 2 out of these 3 properties at the same time:<br><br><b>C — Consistency:</b> Every read gets the most recent write (all nodes see the same data).<br><b>A — Availability:</b> Every request gets a response (even if it might be stale).<br><b>P — Partition Tolerance:</b> The system keeps working even if network messages between nodes are lost.<br><br><b>The catch:</b> Network partitions WILL happen in distributed systems. So you must choose: CP or AP.<br><br><b>CP systems:</b> Refuse to answer if they can't guarantee consistency (e.g., HBase, Zookeeper).<br><b>AP systems:</b> Answer with potentially stale data (e.g., DynamoDB, Cassandra in some configs).<br><br><b>Senior insight:</b> Most real systems are tunable. Cassandra lets you choose consistency level per query. "Eventually Consistent" is AP — the data will be consistent eventually, just not immediately.
What is Eventual Consistency?	Eventual consistency means: if no new updates are made, all nodes in the system will eventually converge to the same value — but they might show different values in the meantime.<br><br><b>Analogy:</b> Think of Google Docs. If you and a friend edit simultaneously offline, you might see different versions. When you both reconnect, Google merges the changes and eventually everyone sees the same document.<br><br><b>Where it's used:</b> DNS, social media feeds, shopping carts on Amazon, DynamoDB default behavior.<br><br><b>Trade-off:</b> Better availability and performance, but you might read stale data.<br><br><b>Common mistake:</b> Using an eventually consistent store for financial transactions. Bank balances need strong consistency — you can't have a node serving a stale balance that lets you overdraw.
What is the difference between SQL and NoSQL databases? When to use each?	<b>SQL (Relational):</b><br>- Data in structured tables with fixed schema.<br>- ACID transactions (Atomic, Consistent, Isolated, Durable).<br>- Powerful joins and complex queries.<br>- Scales vertically (mostly); horizontal sharding is hard.<br>- Examples: PostgreSQL, MySQL, SQL Server.<br><br><b>NoSQL:</b><br>- Flexible schema (document, key-value, column, graph).<br>- Designed for horizontal scaling.<br>- Trade consistency for availability/performance.<br>- Examples: MongoDB (document), Redis (key-value), Cassandra (column), Neo4j (graph).<br><br><b>Use SQL when:</b> Data has relationships, you need transactions, schema is stable.<br><br><b>Use NoSQL when:</b> Massive scale, flexible/variable schema, specific access patterns (e.g., time series, graph traversal).<br><br><b>Common mistake:</b> Using NoSQL "because it scales" without understanding your query patterns. MongoDB joins (lookups) at scale are painful.
What is an API Gateway?	An API Gateway is a single entry point for all client requests. It sits in front of your microservices and handles: routing, authentication, rate limiting, SSL termination, request transformation, and caching.<br><br><b>Analogy:</b> A hotel reception desk. Guests (clients) don't walk directly into the kitchen or housekeeping. They ask the front desk (API Gateway), which routes their request to the right department.<br><br><b>What it does:</b><br>- Routes <code>GET /users</code> to User Service.<br>- Routes <code>POST /orders</code> to Order Service.<br>- Validates JWT token before forwarding.<br>- Rate limits to 100 req/sec per client.<br><br><b>Examples:</b> AWS API Gateway, Kong, NGINX, Apigee.<br><br><b>Senior insight:</b> The API Gateway can become a bottleneck and single point of failure. Make it stateless, horizontally scalable, and highly available.
What is Rate Limiting and why is it important?	Rate limiting restricts how many requests a client can make to your API within a time window. It protects your system from being overwhelmed.<br><br><b>Why it matters:</b><br>- Prevents abuse and DDoS attacks.<br>- Ensures fair usage across all clients.<br>- Protects downstream services from being flooded.<br><br><b>Common algorithms:</b><br>- <b>Token Bucket:</b> Tokens fill up at a rate; each request uses one. Allows bursting.<br>- <b>Leaky Bucket:</b> Requests enter a queue and are processed at a fixed rate. Smooth output.<br>- <b>Fixed Window:</b> Count requests per minute window. Simple but vulnerable to edge bursts.<br>- <b>Sliding Window:</b> More accurate, no edge-of-window bursting.<br><br><b>Implementation:</b> Usually in the API Gateway or middleware, using Redis counters.<br><br><b>Senior insight:</b> Return <code>HTTP 429 Too Many Requests</code> with a <code>Retry-After</code> header so clients know when to retry.
What is a Circuit Breaker pattern?	A Circuit Breaker prevents a failing service from being called repeatedly, giving it time to recover and stopping failures from cascading.<br><br><b>Analogy:</b> Your home circuit breaker. If there's a power surge (service overload), it trips and cuts power to prevent a fire. You reset it after the problem is fixed.<br><br><b>States:</b><br>- <b>Closed:</b> Normal operation. Requests flow through.<br>- <b>Open:</b> Too many failures detected. Requests are blocked immediately (fail fast). No calls to the failing service.<br>- <b>Half-Open:</b> After a timeout, lets a few test requests through. If they succeed, closes again. If they fail, stays open.<br><br><b>Libraries:</b> Polly (.NET), Hystrix (Java), resilience4j.<br><br><b>Senior insight:</b> Without circuit breakers, one slow downstream service can exhaust all your threads and bring down the entire application (cascading failure).
What is the Bulkhead pattern?	The Bulkhead pattern isolates components of a system so that a failure in one doesn't drain resources from others.<br><br><b>Analogy:</b> A ship's hull is divided into watertight compartments (bulkheads). If one compartment floods, the others stay dry and the ship floats.<br><br><b>In software:</b> Separate thread pools or connection pools for different services. If Service A is slow and fills its thread pool, Service B's pool is unaffected.<br><br><b>Example:</b><br><code>// Service A has its own pool of 20 threads<br>// Service B has its own pool of 10 threads<br>// Service A flood doesn't steal Service B's threads</code><br><br><b>Trade-off:</b> Uses more resources (more pools), but prevents one bad service from killing everything.<br><br><b>Senior insight:</b> Combine Bulkhead + Circuit Breaker for robust microservice resilience.
What is a Retry Pattern and what pitfalls should you avoid?	Retry means automatically trying a failed request again, hoping the failure was transient (temporary).<br><br><b>Good for:</b> Network timeouts, temporary service unavailability, rate limit errors (429), some 503s.<br><br><b>Bad for:</b> Business logic errors (400, 404), authentication errors (401) — retrying won't help.<br><br><b>Pitfalls:</b><br>- <b>Retry storm:</b> All clients retry at the same time → floods the recovering service → it fails again.<br>- <b>Solution: Exponential Backoff with Jitter.</b> Wait longer each retry, add randomness.<br><code>wait = min(base * 2^attempt + random_ms, max_wait)</code><br><br><b>Idempotency:</b> Only safe to retry if the operation is idempotent (same result if called multiple times). A <code>GET</code> is safe to retry. A <code>POST /charge-card</code> is NOT unless you use idempotency keys.<br><br><b>Senior insight:</b> Always combine retries with circuit breakers. Don't retry indefinitely — set a max attempt count.
What is the Saga Pattern for distributed transactions?	In microservices, you can't use a single database transaction across services. The Saga pattern breaks a long transaction into a sequence of local transactions, each with a compensating action if something fails.<br><br><b>Example — Order flow:</b><br>1. Create Order ✓<br>2. Reserve Inventory ✓<br>3. Charge Payment ✗ (fails!)<br>→ Compensate: Release Inventory, Cancel Order<br><br><b>Two styles:</b><br>- <b>Choreography:</b> Each service listens for events and reacts. No central coordinator. Simple but hard to track.<br>- <b>Orchestration:</b> A central saga orchestrator tells each service what to do and handles failures. More complex but easier to reason about.<br><br><b>Trade-off:</b> Sagas are eventually consistent. You must design compensating transactions for each step.<br><br><b>Senior insight:</b> Sagas are complex. Only use them when you truly need distributed transactions. Often, rethinking your data model to co-locate data in one service is a better solution.
What is CQRS (Command Query Responsibility Segregation)?	CQRS separates the <b>write model</b> (Commands) from the <b>read model</b> (Queries). Instead of one model that does both, you have two separate optimized models.<br><br><b>Command side:</b> Handles writes/mutations. Optimized for consistency and business rules.<br><b>Query side:</b> Handles reads. Optimized for fast queries, often denormalized or pre-aggregated.<br><br><b>Why:</b> Read and write workloads often have very different requirements. A normalized DB great for writes is painful for complex read dashboards.<br><br><b>Example:</b> Writes go to PostgreSQL (normalized). A read-optimized Elasticsearch index is updated asynchronously for fast full-text search.<br><br><b>Trade-off:</b> Added complexity. Data is eventually consistent between write and read models.<br><br><b>Senior insight:</b> CQRS pairs naturally with Event Sourcing. Don't use it unless you have a real read/write scaling problem — it's a significant architectural investment.
What is Event Sourcing?	Instead of storing the current state of an entity, Event Sourcing stores the full history of events that led to that state. The current state is derived by replaying events.<br><br><b>Analogy:</b> A bank account. Traditional: store balance = $500. Event Sourcing: store every transaction (deposit $100, withdraw $50, deposit $200...). Balance is always recalculated from history.<br><br><b>Benefits:</b><br>- Complete audit trail of all changes.<br>- Can replay events to rebuild state or debug.<br>- Can create new projections (read models) from history.<br>- Natural fit with CQRS.<br><br><b>Challenges:</b><br>- Events must be immutable — never change past events.<br>- Replaying many events can be slow → use snapshots.<br>- Schema evolution is hard (how do you handle changed event structure?).<br><br><b>Senior insight:</b> Event sourcing is powerful but heavy. Only adopt it for domains where audit history and temporal queries matter (finance, healthcare, compliance).
What is Idempotency and why does it matter in distributed systems?	An operation is <b>idempotent</b> if calling it multiple times has the same effect as calling it once.<br><br><b>Example:</b><br>- <code>GET /user/123</code> — idempotent. Call it 100 times, same result.<br>- <code>PUT /user/123</code> with full body — idempotent. Replaces the resource each time.<br>- <code>POST /orders</code> — NOT idempotent. Each call creates a new order.<br><br><b>Why it matters:</b> Networks are unreliable. You might retry a request because the response was lost — but the server already processed it. Without idempotency, you get duplicate charges, duplicate emails, etc.<br><br><b>Solution — Idempotency Keys:</b> Client sends a unique ID with each request. Server stores it and returns the cached response if it sees the same ID again.<br><code>POST /charge HTTP/1.1<br>Idempotency-Key: a1b2c3d4</code><br><br><b>Senior insight:</b> Stripe, PayPal, and all serious payment APIs use idempotency keys. Any API that has side effects should support them.
What is a Consistent Hash Ring?	A consistent hashing ring is a technique to distribute data across servers in a way that minimizes reshuffling when servers are added or removed.<br><br><b>Problem it solves:</b> If you have 3 servers and use <code>hash(key) % 3</code> to assign data, adding a 4th server invalidates ~75% of cache entries (because now <code>% 4</code> gives different results).<br><br><b>How it works:</b> Place servers on a virtual circle (ring) by hashing their names. Place each key on the ring by hashing it. The key belongs to the first server clockwise from it.<br><br><b>Adding a server:</b> Only keys between the new server and its predecessor need to move — typically ~1/N of all keys.<br><br><b>Virtual nodes:</b> Each physical server gets multiple positions on the ring to ensure even distribution.<br><br><b>Used in:</b> Memcached, DynamoDB, Cassandra, Redis Cluster.<br><br><b>Senior insight:</b> Consistent hashing is a foundational concept for any distributed cache or storage system.
What is a Reverse Proxy?	A reverse proxy sits between clients and your backend servers. Clients talk to the proxy; the proxy forwards requests to the appropriate backend.<br><br><b>Difference from a forward proxy:</b> A forward proxy acts on behalf of the client (VPN, corporate proxy). A reverse proxy acts on behalf of the server.<br><br><b>What a reverse proxy provides:</b><br>- <b>Load balancing:</b> Distribute traffic across multiple backend servers.<br>- <b>SSL termination:</b> Handles HTTPS so backend servers don't have to.<br>- <b>Caching:</b> Cache responses to reduce backend load.<br>- <b>Compression:</b> Compress responses before sending to clients.<br>- <b>Security:</b> Hides backend topology, can filter malicious traffic.<br><br><b>Examples:</b> NGINX, HAProxy, Traefik, AWS ALB.<br><br><b>Senior insight:</b> In Kubernetes, the Ingress controller is effectively a reverse proxy. NGINX is the most common choice.
What is Service Discovery in microservices?	Service Discovery is how services find each other's network addresses (IP + port) without hardcoding them — because in dynamic environments (containers, Kubernetes), addresses change constantly.<br><br><b>Two patterns:</b><br><b>1. Client-Side Discovery:</b> The service itself queries a service registry (e.g., Eureka, Consul) to get the address of the target service, then calls it directly.<br><br><b>2. Server-Side Discovery:</b> The client calls a load balancer or API gateway, which queries the registry and routes the request. Client doesn't know about discovery.<br><br><b>Service Registry:</b> A database of available service instances and their addresses. Services register on startup, deregister on shutdown, and send heartbeats to stay listed.<br><br><b>In Kubernetes:</b> Service Discovery is built in. Services get a stable DNS name (<code>my-service.namespace.svc.cluster.local</code>) backed by kube-proxy and CoreDNS.<br><br><b>Examples:</b> Consul, Eureka, Kubernetes Services, AWS Cloud Map.
What is a Health Check endpoint and why is it critical?	A health check endpoint is a simple API endpoint that reports whether the service is functioning correctly. Infrastructure tools (load balancers, Kubernetes, monitoring) poll it to decide if the instance should receive traffic.<br><br><b>Types:</b><br>- <b>Liveness probe:</b> "Is the app alive?" If this fails, restart the container. Check: can the process respond at all?<br>- <b>Readiness probe:</b> "Is the app ready to serve traffic?" If this fails, stop sending traffic (but don't restart). Check: is the DB connection pool ready? Are dependencies up?<br><br><b>Example response:</b><br><code>GET /health → 200 OK<br>{ "status": "healthy", "db": "ok", "cache": "ok" }</code><br><br><b>Common mistake:</b> Liveness probe that checks external dependencies (DB, Redis). If the DB is down, all instances fail liveness → all get restarted → thundering herd when DB recovers. Only use external dependency checks in readiness probes.<br><br><b>Senior insight:</b> Always implement both. The distinction between liveness and readiness saves you from catastrophic restart loops.
What is a Graceful Shutdown?	Graceful shutdown means: when a service receives a stop signal (SIGTERM in Linux), it stops accepting new requests, finishes processing in-flight requests, then exits cleanly.<br><br><b>Why it matters:</b><br>- Without it: users get connection errors mid-request during deployments.<br>- With it: ongoing requests complete normally; only new requests are rejected or routed elsewhere.<br><br><b>Sequence:</b><br>1. Receive SIGTERM.<br>2. Stop accepting new connections (mark readiness probe as unhealthy).<br>3. Wait for in-flight requests to complete (with a timeout).<br>4. Close DB connections, flush buffers, release locks.<br>5. Exit with code 0.<br><br><b>Example (C# ASP.NET Core):</b><br><code>app.Lifetime.ApplicationStopping.Register(() => {<br>&nbsp;&nbsp;// drain connections<br>});</code><br><br><b>Senior insight:</b> Kubernetes sends SIGTERM then waits <code>terminationGracePeriodSeconds</code> (default 30s) before force-killing. Always set this longer than your max request duration.
What is Fault Tolerance vs High Availability?	<b>Fault Tolerance:</b> The system continues operating normally even when a component fails — no degradation, no downtime. Achieved through full redundancy.<br>Example: A plane that can fly on 2 engines even if 1 fails.<br><br><b>High Availability (HA):</b> The system minimizes downtime through fast failover. There might be a brief interruption, but recovery is quick.<br>Example: A server cluster where if one node dies, another takes over in seconds.<br><br><b>Metrics:</b><br>- <b>99.9% availability (three nines):</b> ~8.7 hours downtime/year.<br>- <b>99.99% (four nines):</b> ~52 minutes/year.<br>- <b>99.999% (five nines):</b> ~5 minutes/year.<br><br><b>Trade-off:</b> Higher availability costs more (more redundancy, more complexity).<br><br><b>Senior insight:</b> True fault tolerance (zero downtime) is extremely expensive. Most systems target 99.9%–99.99% HA. Define your SLA first, then engineer to meet it — not the other way around.
What is RTO and RPO in disaster recovery?	These metrics define how much pain you can tolerate when disaster strikes:<br><br><b>RTO (Recovery Time Objective):</b> How long can the system be down? The max acceptable downtime after a failure.<br>Example: "We must be back online within 1 hour."<br><br><b>RPO (Recovery Point Objective):</b> How much data can we lose? The max acceptable gap between the last backup and the failure point.<br>Example: "We can afford to lose up to 15 minutes of data."<br><br><b>Trade-off:</b> Lower RTO/RPO = more expensive. Zero RPO (no data loss) requires synchronous replication. Near-zero RTO requires hot standby environments.<br><br><b>Strategies ranked by cost (cheapest → most expensive):</b><br>- Backup & Restore (high RTO/RPO)<br>- Pilot Light (minimal resources running)<br>- Warm Standby (scaled-down replica)<br>- Multi-Site Active-Active (near-zero RTO/RPO)<br><br><b>Senior insight:</b> Align RTO/RPO with business requirements. Ask "what does 1 hour of downtime cost us?" — that determines how much to spend on DR.
What is Blue-Green Deployment?	Blue-Green deployment runs two identical production environments — <b>Blue</b> (current live) and <b>Green</b> (new version). You switch traffic from Blue to Green when Green is ready.<br><br><b>Steps:</b><br>1. Green environment has the new version deployed and tested.<br>2. Switch the load balancer/DNS to point to Green.<br>3. Green is now live. Blue is idle but kept as a rollback option.<br>4. If Green has issues → switch back to Blue instantly.<br><br><b>Pros:</b><br>- Zero downtime deployment.<br>- Instant rollback (just flip the switch).<br><br><b>Cons:</b><br>- Requires double the infrastructure (two full environments).<br>- Database migrations are tricky — both Blue and Green might share the same DB.<br><br><b>Senior insight:</b> DB migrations must be backward compatible. You can't run a migration that Blue can't also handle until Blue is fully decommissioned. Use expand-contract pattern for schema changes.
What is Canary Deployment?	Canary deployment gradually rolls out a new version to a small percentage of users first. If metrics look good, you increase the percentage until 100%.<br><br><b>Analogy:</b> Coal miners used to bring canary birds into mines. If the canary died from toxic gas, miners evacuated. In software, a small group of users is the "canary" — if they experience errors, you abort the rollout.<br><br><b>Process:</b><br>1. Deploy new version alongside old.<br>2. Route 1% → 5% → 25% → 50% → 100% of traffic to new version.<br>3. Monitor error rate, latency, business metrics at each stage.<br>4. Auto-rollback if thresholds are breached.<br><br><b>Pros:</b> Real production validation with limited blast radius. Gradual confidence building.<br><br><b>Cons:</b> More complex routing logic. Two versions running simultaneously (API compatibility required).<br><br><b>Senior insight:</b> Tools: Kubernetes + Argo Rollouts, AWS CodeDeploy, Flagger, Spinnaker. Always define rollback thresholds (error rate > X%) before deploying.
What is a Distributed Tracing system?	Distributed tracing tracks a single request as it flows through multiple services in a microservices architecture. Each service adds a "span" to a "trace," creating a timeline of what happened where.<br><br><b>Analogy:</b> A package delivery tracking system. The package (request) passes through sorting centers (services). Each stop is logged with timestamps so you can see the full journey and where it got delayed.<br><br><b>Key concepts:</b><br>- <b>Trace:</b> The full end-to-end journey of one request.<br>- <b>Span:</b> A single unit of work within a trace (e.g., a DB call, an HTTP request to another service).<br>- <b>Trace ID:</b> A unique ID that links all spans for one request across all services.<br><br><b>Tools:</b> Jaeger, Zipkin, AWS X-Ray, Honeycomb, Datadog APM.<br><br><b>Standard:</b> OpenTelemetry (OTEL) — vendor-neutral instrumentation SDK.<br><br><b>Senior insight:</b> Without distributed tracing, debugging latency in microservices is nearly impossible. Instrument from day one.
What is Observability and how does it differ from Monitoring?	<b>Monitoring:</b> Watching predefined metrics and alerting when they cross thresholds. You know what questions to ask ahead of time.<br>Example: "Alert when CPU > 80%."<br><br><b>Observability:</b> The ability to understand the internal state of a system from its external outputs (logs, metrics, traces). You can ask new questions without deploying new code.<br>Example: "Why is this specific user's requests slow on Tuesdays only?"<br><br><b>The 3 Pillars of Observability:</b><br>- <b>Logs:</b> Timestamped records of discrete events.<br>- <b>Metrics:</b> Numeric measurements over time (counters, gauges, histograms).<br>- <b>Traces:</b> Request flows across services.<br><br><b>Monitoring tells you THAT something is wrong. Observability helps you understand WHY.</b><br><br><b>Senior insight:</b> Invest in structured logging, consistent correlation IDs across services, and meaningful metrics from the start. Retrofitting observability into a large system is painful.
What are Structured Logs and why use them instead of plain text logs?	<b>Plain text log:</b><br><code>2024-01-15 ERROR Failed to process order 12345 for user 67890</code><br><br><b>Structured log (JSON):</b><br><code>{"timestamp":"2024-01-15T10:30:00Z","level":"ERROR","event":"order_failed","orderId":"12345","userId":"67890","errorCode":"PAYMENT_DECLINED"}</code><br><br><b>Why structured is better:</b><br>- Machine-parseable: log aggregation tools (Elasticsearch, Splunk, CloudWatch) can query specific fields.<br>- Filtering: <code>level:ERROR AND userId:67890</code><br>- Alerting on specific fields.<br>- No more regex parsing of free-text logs.<br><br><b>Libraries:</b> Serilog (C#), Winston/Pino (Node.js), structlog (Python).<br><br><b>Best practices:</b><br>- Always include correlation/trace ID.<br>- Use log levels correctly (DEBUG, INFO, WARN, ERROR).<br>- Never log PII (passwords, credit card numbers, SSNs).<br><br><b>Senior insight:</b> Set DEBUG level off in production. DEBUG logs at scale can fill disks and cost thousands in log ingestion fees.
What are the key Metrics types in observability? (Counter, Gauge, Histogram)	<b>Counter:</b> Always goes up. Tracks cumulative events.<br>Examples: total requests served, total errors, total bytes sent.<br><code>http_requests_total{method="GET", status="200"} 14523</code><br><br><b>Gauge:</b> Can go up or down. Tracks current state.<br>Examples: current active connections, memory usage, queue depth.<br><code>queue_depth{queue="orders"} 342</code><br><br><b>Histogram:</b> Samples observations and counts them in buckets. Used for measuring distributions like latency.<br><code>http_request_duration_seconds_bucket{le="0.1"} 8923<br>http_request_duration_seconds_bucket{le="0.5"} 14012</code><br><br><b>Why histograms matter:</b> Averages lie. A 200ms average could hide that 10% of users experience 5s responses. Histograms let you calculate percentiles (p50, p95, p99).<br><br><b>Senior insight:</b> Always track p99 latency, not just average. Your worst 1% of users are telling you where the real problems are.
What are the RED and USE methods for monitoring?	Two popular frameworks for knowing WHAT to measure:<br><br><b>RED Method</b> (for services/APIs — from Weaveworks):<br>- <b>R — Rate:</b> Requests per second (how much traffic?)<br>- <b>E — Errors:</b> Failed requests per second (how often does it fail?)<br>- <b>D — Duration:</b> Latency distribution (how long does it take?)<br><br><b>USE Method</b> (for infrastructure/resources — from Brendan Gregg):<br>- <b>U — Utilization:</b> % of time the resource is busy (CPU %, disk I/O %)<br>- <b>S — Saturation:</b> Amount of work waiting (queue depth, run queue)<br>- <b>E — Errors:</b> Error events (disk errors, dropped packets)<br><br><b>When to use which:</b><br>- RED for microservice dashboards (is my API healthy?)<br>- USE for infrastructure dashboards (is my server healthy?)<br><br><b>Senior insight:</b> Every service should have a RED dashboard. Every host should have a USE dashboard. Start here before building complex custom metrics.
What is the Four Golden Signals method from Google SRE?	Google's Site Reliability Engineering book defines four signals that matter most for monitoring a service:<br><br><b>1. Latency:</b> Time to serve a request. Track both successful AND failed requests separately. A fast error is still an error.<br><br><b>2. Traffic:</b> Demand on your system. Requests per second, active users, transactions/sec.<br><br><b>3. Errors:</b> Rate of failed requests. Includes explicit errors (500s), implicit errors (wrong data), and policy errors (too slow).<br><br><b>4. Saturation:</b> How "full" is your service? CPU%, memory%, disk%, queue depth. Predicts upcoming problems before they cause errors.<br><br><b>Alerting rule:</b> Alert on these four signals. If none of the four are degraded, your users are probably fine.<br><br><b>Senior insight:</b> These map almost directly to the RED method + Saturation. The SRE book is foundational reading for any architect designing production systems.
What is an SLI, SLO, and SLA?	<b>SLI (Service Level Indicator):</b> A specific metric that measures service quality.<br>Example: "The percentage of successful HTTP requests in the last 5 minutes."<br><code>SLI = (successful_requests / total_requests) * 100</code><br><br><b>SLO (Service Level Objective):</b> A target value for an SLI. An internal goal.<br>Example: "99.9% of requests must succeed each month."<br><br><b>SLA (Service Level Agreement):</b> A contract with a customer defining the SLO. Has financial/legal consequences if breached.<br>Example: "We guarantee 99.9% uptime. If we breach it, you get service credits."<br><br><b>Error Budget:</b> How much you're allowed to fail within your SLO.<br>99.9% SLO → 0.1% error budget → ~43.8 minutes of downtime/month allowed.<br><br><b>Senior insight:</b> Error budgets are powerful. If you've used your error budget, stop deploying new features and focus on reliability. If you have budget left, move fast. This creates a healthy balance between velocity and reliability.
What is Alerting fatigue and how do you prevent it?	Alert fatigue happens when engineers receive so many alerts that they start ignoring them — including real critical ones.<br><br><b>Causes:</b><br>- Alerts on symptoms that don't impact users (CPU spike that auto-resolved).<br>- Alerts with no actionable response.<br>- Too sensitive thresholds causing constant noise.<br>- Duplicate alerts for the same root cause.<br><br><b>Prevention:</b><br>- <b>Alert on user impact, not symptoms:</b> Alert on high error rate, not high CPU. High CPU that doesn't cause errors isn't worth waking someone up.<br>- <b>Every alert needs a runbook:</b> What do I do when this fires? If there's no action, it shouldn't be an alert.<br>- <b>Use SLO-based alerting:</b> Alert when you're burning through your error budget too fast.<br>- <b>Separate pages from tickets:</b> Not everything needs to wake someone up at 3am.<br><br><b>Senior insight:</b> A good on-call rotation requires good alerts. Teams that receive 50+ alerts per night become ineffective. Regularly review and prune alerts.
What is the difference between Latency, Throughput, and Bandwidth?	<b>Latency:</b> Time for a single request to complete — from start to finish. Measured in milliseconds.<br>Analogy: How long it takes one car to drive from city A to city B.<br><br><b>Throughput:</b> Number of operations completed per unit of time. Measured in requests/sec, transactions/sec.<br>Analogy: How many cars travel from city A to city B per hour.<br><br><b>Bandwidth:</b> Maximum capacity of the channel — data transferred per second. Measured in Mbps, Gbps.<br>Analogy: The number of lanes on the highway.<br><br><b>Key relationship:</b> High bandwidth doesn't mean low latency. You can have a fat pipe (high bandwidth) that's slow (high latency) — like a satellite connection.<br><br><b>Little's Law:</b> <code>Throughput = Concurrency / Latency</code><br>If latency doubles, you need twice the concurrency to maintain throughput.<br><br><b>Senior insight:</b> When optimizing, identify whether you have a latency problem or a throughput problem — they often have different solutions.
What is a P99 (percentile) latency and why does it matter more than average?	Percentile latency measures the latency experienced by a specific percentage of requests.<br><br><b>p50 (median):</b> 50% of requests are faster than this value. The "typical" experience.<br><b>p95:</b> 95% of requests are faster. 5% are slower.<br><b>p99:</b> 99% of requests are faster. 1% are slower — the "tail latency."<br><b>p99.9:</b> The worst 0.1% of requests.<br><br><b>Why averages lie:</b> Imagine 100 requests: 99 complete in 10ms, 1 completes in 10,000ms.<br>Average = ~109ms — looks fine. But 1% of users wait 10 seconds.<br><br><b>The tail latency problem:</b> In microservices, if a request fans out to 10 services in parallel, the total latency = the slowest service. With 10 services each at p99 = 1%, you'll see p99 behavior on ~10% of requests.<br><br><b>Senior insight:</b> Always define and monitor p95/p99 SLOs. Tail latency disproportionately affects user experience. High p99 is often caused by GC pauses, lock contention, or noisy neighbors.
What causes a Thundering Herd problem?	The Thundering Herd occurs when a large number of processes or requests simultaneously try to access the same resource, overwhelming it.<br><br><b>Common scenarios:</b><br>- A cache key expires → all requests simultaneously miss cache → all hit the DB at once → DB gets crushed.<br>- A server reboots → all clients reconnect simultaneously → server is overwhelmed.<br>- A message queue backs up → when the consumer recovers, it processes thousands of messages at full speed → downstream overwhelmed.<br><br><b>Solutions:</b><br>- <b>Cache lock (mutex/dogpile lock):</b> Only one process fetches from DB; others wait for the result.<br>- <b>Probabilistic early expiration:</b> Refresh cache slightly before it expires, not after.<br>- <b>Staggered reconnect:</b> Add jitter to reconnect intervals so not all clients reconnect simultaneously.<br>- <b>Rate limiting on queue consumers.</b><br><br><b>Senior insight:</b> This is especially dangerous with in-memory caches. Use a distributed lock (Redis SETNX) to prevent simultaneous cache population.
What is Database Connection Pooling?	Connection pooling maintains a pool of pre-opened database connections that are reused across requests, rather than opening/closing a new connection per request.<br><br><b>Why it matters:</b> Opening a DB connection is expensive — TCP handshake, auth, SSL negotiation. At 1000 req/sec this becomes a bottleneck.<br><br><b>How it works:</b><br>- Pool starts with N connections (e.g., 10).<br>- Request borrows a connection from the pool.<br>- After the request, connection is returned to the pool (not closed).<br>- If all connections are in use, new requests wait or fail.<br><br><b>Key settings:</b><br>- <b>Min pool size:</b> Connections kept warm.<br>- <b>Max pool size:</b> Upper limit (don't exceed DB's max_connections).<br>- <b>Connection timeout:</b> How long to wait for a free connection.<br><br><b>Common mistake:</b> Setting max pool size × number of app instances > database max_connections. With 20 app instances × 100 pool size = 2000 connections → most DBs can't handle this.<br><br><b>Senior insight:</b> Use a connection proxy/pooler (PgBouncer for PostgreSQL) to multiplex connections at scale.
What is N+1 Query Problem?	The N+1 query problem occurs when code executes 1 query to fetch a list, then N more queries (one per item) to fetch related data.<br><br><b>Example:</b><br><code>// Fetch 100 orders (1 query)<br>orders = db.query("SELECT * FROM orders")<br>// Then for each order, fetch the user (100 queries!)<br>for order in orders:<br>&nbsp;&nbsp;user = db.query("SELECT * FROM users WHERE id = ?", order.userId)</code><br>Total: 101 queries. With 1000 orders: 1001 queries.<br><br><b>Fix: Use JOINs or eager loading:</b><br><code>SELECT orders.*, users.name FROM orders<br>JOIN users ON orders.user_id = users.id</code><br>Or use ORM eager loading: <code>db.orders.include('user').findAll()</code><br><br><b>Detection:</b> Look for repeated identical queries in DB logs. Tools like Bullet gem (Rails), EF Core query logging, Hibernate SQL logging.<br><br><b>Senior insight:</b> N+1 is the most common performance bug in applications using ORMs. It's invisible in small datasets and catastrophic at scale.
What is Database Indexing and when should you add an index?	An index is a data structure that speeds up data retrieval by creating a lookup structure for specific columns, avoiding a full table scan.<br><br><b>Analogy:</b> A book's index at the back. Instead of reading every page to find "transactions," you look up the index and jump to page 47.<br><br><b>Without index:</b> DB scans every row. O(n) — slow for large tables.<br><b>With index:</b> DB uses B-tree or hash structure. O(log n) — fast.<br><br><b>When to add an index:</b><br>- Columns used in WHERE clauses frequently.<br>- Columns used in JOIN conditions.<br>- Columns used in ORDER BY (avoid full sort).<br><br><b>The trade-off:</b> Indexes slow down writes (INSERT, UPDATE, DELETE must also update index structures). Don't index everything.<br><br><b>Types:</b> B-Tree (general), Hash (equality only), Composite (multiple columns), Partial (subset of rows), Full-text.<br><br><b>Senior insight:</b> Use <code>EXPLAIN ANALYZE</code> in PostgreSQL (or <code>EXPLAIN</code> in MySQL) to see if your query is using indexes. A query with "Seq Scan" on a large table is a red flag.
What is the difference between Optimistic and Pessimistic Locking?	Both prevent concurrent writes from corrupting data, but with different strategies:<br><br><b>Pessimistic Locking:</b> Lock the row when you read it — no one else can read/write until you're done. "Assume conflict will happen."<br><code>SELECT * FROM accounts WHERE id = 1 FOR UPDATE;</code><br>Pros: Prevents conflicts. Cons: Reduces concurrency, risk of deadlocks, holds lock for duration of transaction.<br><br><b>Optimistic Locking:</b> No lock during read. When saving, check if data changed since you read it. If yes, reject the update. "Assume conflict is rare."<br><code>// Read: version = 5<br>// Update: WHERE id = 1 AND version = 5<br>// If 0 rows updated → someone else changed it → retry</code><br><br><b>When to use which:</b><br>- Pessimistic: High contention, short transactions (bank transfers).<br>- Optimistic: Low contention, long transactions, read-heavy workloads.<br><br><b>Senior insight:</b> Most web applications use optimistic locking (version columns). Pessimistic locking can deadlock and tank throughput under load.
What is a Bloom Filter?	A Bloom Filter is a space-efficient probabilistic data structure that tells you: "definitely NOT in the set" or "probably in the set."<br><br><b>Key property:</b> No false negatives (if it says not in set, it's definitely not there). Can have false positives (might say "in set" when it's not).<br><br><b>How it works:</b> Use multiple hash functions to set bits in a bit array. To check membership, verify all bits are set.<br><br><b>Use cases:</b><br>- Check if a URL has been crawled before (web crawlers).<br>- Avoid expensive DB lookups for non-existent keys (cache miss optimization).<br>- Email duplicate detection.<br>- Cassandra uses Bloom Filters to avoid reading SSTables that don't contain the key.<br><br><b>Example flow:</b> Before a cache miss hits the DB, check Bloom Filter. If "definitely not in DB," skip the DB query entirely.<br><br><b>Senior insight:</b> When you see high cache miss rates causing DB load, Bloom Filters can eliminate a large percentage of unnecessary DB lookups.
What is Backpressure in distributed systems?	Backpressure is a mechanism where a downstream service signals to an upstream service that it's overwhelmed, so the upstream slows down — preventing a cascade of failures.<br><br><b>Analogy:</b> A garden hose. If the nozzle is too small (slow consumer), pressure builds up (queue fills). Without backpressure, the hose bursts. With it, the faucet (producer) is throttled.<br><br><b>Without backpressure:</b> Producer floods the consumer → queue grows unboundedly → OOM crash → full system failure.<br><br><b>Backpressure strategies:</b><br>- <b>Drop:</b> Discard excess messages. Simple but lossy.<br>- <b>Buffer:</b> Queue up to a limit, then block the producer.<br>- <b>Throttle:</b> Slow the producer down (rate limiting signals).<br>- <b>Return error:</b> Signal the producer to retry later (HTTP 429, TCP slow window).<br><br><b>In reactive streams:</b> Libraries like RxJava, Project Reactor, and Akka Streams have built-in backpressure protocols.<br><br><b>Senior insight:</b> Backpressure is essential for any data pipeline. Unbounded queues are a time bomb — always set queue size limits and define what happens when the queue is full.
What is the difference between Latency and Jitter?	<b>Latency:</b> The average time for a request to travel from source to destination.<br>Example: Average API response time = 50ms.<br><br><b>Jitter:</b> The variation in latency over time. How inconsistent the latency is.<br>Example: Some requests take 10ms, others take 500ms — high jitter.<br><br><b>Why jitter matters:</b><br>- High jitter = unpredictable user experience.<br>- For real-time systems (video calls, gaming, trading), jitter is often more damaging than high latency. You can buffer for latency; jitter breaks synchronization.<br><br><b>Common causes of jitter:</b><br>- GC pauses (especially Java/Go stop-the-world).<br>- CPU scheduling (noisy neighbors in cloud).<br>- Lock contention.<br>- Variable network conditions.<br>- Connection pool exhaustion causing wait spikes.<br><br><b>Mitigation:</b><br>- Tune GC (use G1GC or ZGC in Java).<br>- Use dedicated resources (avoid burstable instances for latency-sensitive workloads).<br>- Connection pooling and pre-warming.<br><br><b>Senior insight:</b> Tracking p99 and p99.9 latency reveals jitter. If p50 is 20ms and p99 is 2000ms, you have a severe jitter problem.
What is the difference between CPU-bound and I/O-bound workloads?	<b>CPU-bound:</b> The bottleneck is the processor. The task spends most of its time doing computation.<br>Examples: Video encoding, cryptography, machine learning inference, compressing files.<br><br><b>I/O-bound:</b> The bottleneck is waiting for input/output — network, disk, database.<br>Examples: Web APIs (waiting for DB), file processing, external API calls.<br><br><b>Why it matters for scaling:</b><br>- CPU-bound: Add more CPU cores. Async doesn't help much.<br>- I/O-bound: Use async/non-blocking I/O. One thread can handle thousands of concurrent I/O waits.<br><br><b>Language implications:</b><br>- <b>Python:</b> GIL limits CPU parallelism in threads → use multiprocessing for CPU-bound.<br>- <b>Node.js:</b> Single-threaded event loop. Excellent for I/O-bound, terrible for CPU-bound (blocks the loop).<br>- <b>C#:</b> async/await excellent for I/O-bound. Parallel.For/Task for CPU-bound.<br>- <b>Go:</b> Goroutines handle both well. Excellent for concurrent I/O.<br><br><b>Senior insight:</b> Most web services are I/O-bound. Profile before assuming — don't add CPU thinking it's the bottleneck when the real wait is DB latency.
What is Async vs Sync I/O? Why does it matter for scalability?	<b>Synchronous (Blocking) I/O:</b> The thread waits (does nothing) while the I/O operation completes.<br><code>// Thread is BLOCKED for 200ms waiting for DB<br>var result = db.QuerySync("SELECT ...")</code><br>If you have 100 concurrent requests, you need 100 threads — threads are expensive (1MB+ stack each).<br><br><b>Asynchronous (Non-blocking) I/O:</b> The thread initiates the I/O, then goes and does other work. When I/O completes, a callback/continuation runs.<br><code>// Thread is FREE during the await<br>var result = await db.QueryAsync("SELECT ...")</code><br>1 thread can handle thousands of concurrent I/O operations.<br><br><b>Impact on scale:</b><br>- Sync: 10,000 concurrent connections → 10,000 threads → ~10GB memory just for stacks.<br>- Async: 10,000 concurrent connections → ~10-100 threads → vastly lower memory.<br><br><b>Async in languages:</b><br>- C#: <code>async/await</code> + <code>Task</code><br>- JS/Node: <code>async/await</code> + Promises<br>- Python: <code>asyncio</code>, <code>async/await</code><br>- Go: goroutines + channels<br><br><b>Senior insight:</b> Don't mix sync blocking calls in async code paths — it defeats the purpose and can cause thread pool starvation.
What is Connection Draining / Deregistration Delay?	Connection draining (AWS term) or deregistration delay is the time a load balancer waits for in-flight requests to complete before removing a backend instance from the pool.<br><br><b>Without it:</b> LB removes instance → in-progress requests get connection reset errors → users see failures.<br><br><b>With it:</b><br>1. Instance is marked for removal (scale-in, deployment).<br>2. LB stops sending new requests to that instance.<br>3. LB waits up to N seconds (e.g., 30s) for existing requests to complete.<br>4. Instance is terminated.<br><br><b>Pairs with graceful shutdown:</b> The instance stops accepting new connections (readiness probe → unhealthy) while the LB drains existing ones.<br><br><b>Setting:</b> Match deregistration delay to your max expected request duration. If your slowest requests take 25s, set drain timeout to 30s+.<br><br><b>Senior insight:</b> Ignoring connection draining during deployments causes user-visible errors on every deploy. This is one of the most common, easily preventable causes of deployment-time errors.
What is OpenTelemetry (OTEL)?	OpenTelemetry is a vendor-neutral, open-source observability framework that provides standardized APIs, SDKs, and tools to collect logs, metrics, and traces from applications.<br><br><b>Problem it solves:</b> Before OTEL, each observability vendor (Datadog, Jaeger, New Relic) had its own SDK. Switching vendors meant rewriting instrumentation.<br><br><b>OTEL components:</b><br>- <b>API:</b> Interfaces for instrumentation (add to your code).<br>- <b>SDK:</b> Implementation of the API with configuration.<br>- <b>Collector:</b> An agent/proxy that receives, processes, and exports telemetry to any backend.<br>- <b>Exporters:</b> Send data to Jaeger, Prometheus, Datadog, Honeycomb, etc.<br><br><b>Instrumentation:</b><br><code>// C# example<br>using var tracer = TracerProvider.GetTracer("MyService");<br>using var span = tracer.StartActiveSpan("ProcessOrder");<br>span.SetAttribute("orderId", orderId);</code><br><br><b>Senior insight:</b> Adopt OpenTelemetry as your standard from the start. It future-proofs your instrumentation — you can swap backend observability vendors without touching application code.
What is a Span in distributed tracing?	A Span represents a single unit of work in a distributed trace. It records what happened, when it started, when it ended, and metadata about the operation.<br><br><b>A span contains:</b><br>- <b>Trace ID:</b> Links all spans in the same request.<br>- <b>Span ID:</b> Unique ID for this specific span.<br>- <b>Parent Span ID:</b> Which span triggered this one.<br>- <b>Operation name:</b> What this span represents (e.g., "HTTP GET /users", "DB query users table").<br>- <b>Start time and duration.</b><br>- <b>Status:</b> OK or Error.<br>- <b>Attributes/Tags:</b> Key-value pairs (e.g., <code>http.status_code=200</code>, <code>db.statement="SELECT..."</code>).<br>- <b>Events:</b> Timestamped log entries within the span.<br><br><b>Parent-child relationship:</b> Spans form a tree. The root span is the entry point (e.g., incoming HTTP request). Child spans are created for downstream calls.<br><br><b>Senior insight:</b> Always propagate the trace context (W3C Trace Context headers) across HTTP calls and message queues. Without propagation, traces break at service boundaries.
What is Log Aggregation and why do you need a centralized logging system?	Log aggregation collects logs from all services and instances into a single searchable system.<br><br><b>Problem without it:</b> You have 50 microservice instances. A user reports an error. You SSH into each server and grep through log files hoping to find it. Nightmare.<br><br><b>With centralized logging:</b><br>- All logs flow to one place.<br>- Search across all services simultaneously.<br>- Correlate logs by trace ID or user ID.<br>- Set alerts on log patterns.<br>- Retain logs for compliance.<br><br><b>Common stacks:</b><br>- <b>ELK Stack:</b> Elasticsearch (storage/search) + Logstash (ingestion) + Kibana (visualization).<br>- <b>EFK Stack:</b> Elasticsearch + Fluentd + Kibana.<br>- <b>Loki + Grafana:</b> Lightweight, label-based (like Prometheus for logs).<br>- <b>Cloud:</b> AWS CloudWatch Logs, Azure Monitor Logs, Google Cloud Logging.<br><br><b>Senior insight:</b> Index only what you query. Full log indexing in Elasticsearch at scale is expensive. Loki's label-based approach (don't index log content) can be 10x cheaper.
What is Prometheus and how does it collect metrics?	Prometheus is an open-source monitoring and alerting system that collects metrics by <b>scraping</b> HTTP endpoints on a schedule.<br><br><b>How it works:</b><br>1. Your application exposes a <code>/metrics</code> endpoint with metrics in Prometheus text format.<br>2. Prometheus scrapes that endpoint every N seconds (e.g., 15s).<br>3. Metrics are stored in a time-series database (TSDB).<br>4. You query with PromQL and visualize in Grafana.<br>5. AlertManager fires alerts when rules are breached.<br><br><b>Example metric:</b><br><code>http_requests_total{method="GET",status="200"} 14523</code><br><br><b>Key PromQL query:</b><br><code>// Request rate over 5 minutes<br>rate(http_requests_total[5m])<br>// 99th percentile latency<br>histogram_quantile(0.99, http_request_duration_seconds_bucket)</code><br><br><b>Pull vs Push:</b> Prometheus pulls (scrapes). This is different from push-based systems (StatsD, Graphite). Pull has benefits: Prometheus controls the interval, dead services are obvious (scrape fails).<br><br><b>Senior insight:</b> Prometheus + Grafana is the de-facto standard for Kubernetes monitoring. Learn PromQL — it's powerful and ubiquitous.
What is a Correlation ID / Trace ID and how do you propagate it?	A Correlation ID (or Trace ID) is a unique identifier attached to a request that is passed through every service and logged everywhere, allowing you to find all logs and spans for a single user request across all services.<br><br><b>Without it:</b> "User 12345 got an error at 3pm." You search logs across 10 services for anything near 3pm. Impossible to find the root cause.<br><br><b>With it:</b> "Correlation ID: abc-123." Search all logs for <code>correlationId:abc-123</code>. Instantly see every log entry across all services for that request.<br><br><b>How to propagate:</b><br>- Generate UUID at the entry point (API Gateway or first service).<br>- Pass via HTTP header: <code>X-Correlation-ID: abc-123</code> (or use W3C <code>traceparent</code>).<br>- Each service reads the header, logs it, and passes it to downstream calls.<br>- Include in all log entries.<br><br><b>Code pattern (C# middleware):</b><br><code>app.Use(async (ctx, next) => {<br>&nbsp;&nbsp;var id = ctx.Request.Headers["X-Correlation-ID"].FirstOrDefault() ?? Guid.NewGuid().ToString();<br>&nbsp;&nbsp;ctx.Response.Headers["X-Correlation-ID"] = id;<br>&nbsp;&nbsp;using (LogContext.PushProperty("CorrelationId", id)) { await next(); }<br>});</code><br><br><b>Senior insight:</b> Correlation IDs are the minimum viable observability feature. Implement them before anything else.
What is Chaos Engineering?	Chaos Engineering is the practice of intentionally introducing failures into a system in production (or staging) to verify it handles them gracefully — before real failures happen unexpectedly.<br><br><b>Philosophy:</b> "Hope is not a strategy." Don't assume your system is resilient — prove it by breaking it on purpose under controlled conditions.<br><br><b>Netflix's Chaos Monkey:</b> Randomly terminates production instances. Forces teams to build services that survive instance failures.<br><br><b>Chaos experiments examples:</b><br>- Kill a random pod in Kubernetes.<br>- Inject network latency between services.<br>- Simulate a database failure.<br>- Fill up a disk.<br>- Drop network packets between services.<br><br><b>Tools:</b> Chaos Monkey, Gremlin, LitmusChaos (Kubernetes), AWS Fault Injection Simulator.<br><br><b>Process:</b><br>1. Define steady state (what does "healthy" look like?).<br>2. Hypothesize the system survives X failure.<br>3. Inject the failure.<br>4. Observe. Fix weaknesses found.<br><br><b>Senior insight:</b> Start in staging. Graduate to production only when you have good observability and confidence. The goal is learning, not breaking things randomly.
What is the Strangler Fig Pattern for migrating legacy systems?	The Strangler Fig Pattern gradually replaces a legacy system by building new functionality alongside it, routing requests incrementally to the new system until the old one can be decommissioned — without a big-bang rewrite.<br><br><b>Analogy:</b> A strangler fig tree grows around an old tree. Over time the new tree's structure replaces the old tree, which eventually dies and is removed.<br><br><b>Process:</b><br>1. Put a facade (proxy/API gateway) in front of the legacy system.<br>2. Implement one feature/module in the new system.<br>3. Route traffic for that feature to the new system via the facade.<br>4. Repeat for other features over time.<br>5. Once all traffic is routed to the new system, decommission the old one.<br><br><b>Benefits:</b><br>- No big-bang risk. If new code fails, route back to legacy.<br>- Continuous delivery throughout migration.<br>- Team learns incrementally.<br><br><b>Common mistake:</b> Trying to rewrite everything at once. This almost always fails or takes 3x longer than estimated.<br><br><b>Senior insight:</b> The Strangler Fig is the safest, most proven approach to legacy migration. Martin Fowler coined the term. Use it.
What is a Service Mesh?	A service mesh is an infrastructure layer that handles all communication between microservices — providing observability, security (mTLS), traffic management, and resilience — without changing application code.<br><br><b>How it works:</b> A lightweight proxy (sidecar) is injected next to every service container. All traffic goes through the sidecar. The sidecar handles the cross-cutting concerns.<br><br><b>What it provides:</b><br>- <b>Observability:</b> Automatic metrics, traces, and logs for all service-to-service calls.<br>- <b>Security:</b> Mutual TLS (mTLS) between all services automatically.<br>- <b>Traffic control:</b> Canary routing, circuit breaking, retries, timeouts — configured at infrastructure level.<br>- <b>Load balancing:</b> Advanced algorithms at the service level.<br><br><b>Popular options:</b> Istio, Linkerd, Consul Connect, AWS App Mesh.<br><br><b>Trade-off:</b> Service meshes add complexity and latency (extra hop through sidecar). Istio especially has a steep learning curve.<br><br><b>Senior insight:</b> A service mesh is powerful but heavy. Start with simpler solutions (library-level circuit breakers, OTEL instrumentation). Adopt a mesh when you have 20+ services and can't manage cross-cutting concerns at the library level.
What is the difference between Horizontal Pod Autoscaler (HPA) and Vertical Pod Autoscaler (VPA) in Kubernetes?	<b>HPA (Horizontal Pod Autoscaler):</b> Automatically increases or decreases the number of pod replicas based on metrics (CPU, memory, or custom metrics).<br><br>Example: When CPU usage > 70%, HPA scales from 3 pods to 6 pods. When traffic drops, scales back down.<br><code>kubectl autoscale deployment my-app --cpu-percent=70 --min=2 --max=10</code><br><br><b>VPA (Vertical Pod Autoscaler):</b> Automatically adjusts the CPU and memory requests/limits of a pod based on actual usage. Makes individual pods bigger/smaller.<br><br>Example: Pod requests 256MB but consistently uses 1GB → VPA increases the request to 1GB.<br><br><b>When to use which:</b><br>- HPA: Stateless services with variable request load. Most common choice.<br>- VPA: Services where the right resource size is unknown, or for batch jobs. Not recommended for production live traffic — VPA restarts pods to apply new resource values.<br><br><b>KEDA (Kubernetes Event-Driven Autoscaling):</b> Extends HPA to scale on any event source (queue depth, Kafka lag, Prometheus metrics).<br><br><b>Senior insight:</b> Don't use HPA and VPA together on the same deployment — they conflict. Use HPA for horizontal scaling + manually tune resource requests.
What is the difference between an SLA breach and an error budget burn?	<b>SLA breach:</b> You have failed to meet a contractual obligation with a customer. This has legal and financial consequences (service credits, penalties).<br><br><b>Error budget burn:</b> An internal operational concept. You are consuming your allowed failure margin faster than expected — a warning that you're heading toward an SLA breach.<br><br><b>Example:</b><br>- SLO: 99.9% availability per month → 43.8 minutes downtime allowed.<br>- If you're down for 40 minutes in the first week → you've burned 91% of your monthly budget in one week.<br>- Alert: "Error budget burn rate is 10x normal" → stop new deployments, investigate.<br><br><b>Burn rate alerting (Google SRE Book):</b><br>- Fast burn (1-hour window): You'll exhaust the monthly budget in 2 days if this continues.<br>- Slow burn (6-hour window): Gradual degradation that would exhaust budget in 5 days.<br><br><b>Senior insight:</b> Error budget burn alerting is more actionable than raw availability alerts. "You're burning budget 10x faster than normal" is a clearer signal than "availability is 99.85%."
What is Zero-Downtime Deployment and what techniques enable it?	Zero-downtime deployment means releasing new code without any request failures or service interruptions visible to users.<br><br><b>Required conditions:</b><br>- Multiple instances behind a load balancer.<br>- Graceful shutdown on old instances.<br>- Load balancer connection draining.<br>- Backward-compatible DB migrations.<br><br><b>Techniques:</b><br>- <b>Rolling deployment:</b> Replace instances one by one. Some instances run old code, some new during transition. Low risk, gradual.<br>- <b>Blue-Green:</b> Switch all traffic at once. Instant cutover. Requires double infrastructure.<br>- <b>Canary:</b> Route small % of traffic to new version first. Safest for validating changes.<br><br><b>DB migration strategy (expand-contract):</b><br>1. <b>Expand:</b> Add new column, keep old one (both old and new code work).<br>2. <b>Migrate:</b> Deploy new code that uses new column.<br>3. <b>Contract:</b> Remove old column once old code is fully gone.<br><br><b>Common mistake:</b> Deploying a DB migration that removes a column before deploying code that no longer uses it → old running instances break immediately.<br><br><b>Senior insight:</b> Zero-downtime deployment is a system-wide concern, not just a deployment tool feature. Every part of the stack must cooperate.
