What is the Circuit Breaker pattern in cloud architecture?	<b>Circuit Breaker</b> prevents cascading failures by stopping calls to a failing service.<br><br><b>Analogy:</b> Like an electrical circuit breaker — when too many failures happen, it "trips" and stops sending requests, giving the failing service time to recover.<br><br><b>States:</b><br>• <b>Closed</b> — requests flow normally<br>• <b>Open</b> — requests are blocked immediately (fail fast)<br>• <b>Half-Open</b> — a few test requests are allowed through to check recovery<br><br><b>Example (pseudocode):</b><br><code>if (circuitBreaker.isOpen()) {<br>&nbsp;&nbsp;return fallbackResponse();<br>}<br>try {<br>&nbsp;&nbsp;return callService();<br>} catch (Exception e) {<br>&nbsp;&nbsp;circuitBreaker.recordFailure();<br>}</code><br><br><b>Trade-off:</b> You need good fallback logic — returning stale data or a degraded response is better than an error.<br><br><b>Tools:</b> Polly (.NET), Resilience4j (Java), Hystrix (deprecated but common in legacy).

What is the Retry Pattern and when should you NOT use it?	<b>Retry Pattern</b> automatically re-attempts a failed operation, assuming the failure is transient (temporary).<br><br><b>When to use:</b> Network timeouts, rate-limit errors (HTTP 429), temporary unavailability.<br><br><b>When NOT to use:</b><br>• Non-transient errors (HTTP 400 Bad Request — retrying won't help)<br>• If the operation is not idempotent (retrying could duplicate charges, orders, etc.)<br><br><b>Best practice — Exponential Backoff + Jitter:</b><br><code>wait = min(cap, base * 2^attempt) + random_jitter</code><br><br>Without jitter, all clients retry at the same time, creating a <b>thundering herd</b> problem.<br><br><b>Common mistake:</b> Retrying forever without a max attempt limit — this can keep a service overwhelmed.

What is the Bulkhead Pattern?	<b>Bulkhead</b> isolates different parts of a system so one failure doesn't sink the whole ship.<br><br><b>Analogy:</b> Ship compartments (bulkheads) — if one section floods, the others stay dry.<br><br><b>In practice:</b> Separate thread pools, connection pools, or resource limits for each downstream service.<br><br><code>// Example: separate HTTP clients per service<br>var paymentClient = new HttpClient { MaxConnections = 10 };<br>var inventoryClient = new HttpClient { MaxConnections = 10 };</code><br><br>If the payment service is slow and exhausts its pool, the inventory service is unaffected.<br><br><b>Trade-off:</b> Uses more resources upfront, but dramatically improves resilience.<br><br><b>Senior insight:</b> Combine with Circuit Breaker for full defense-in-depth.

What is CQRS (Command Query Responsibility Segregation)?	<b>CQRS</b> separates <b>write operations (commands)</b> from <b>read operations (queries)</b> into different models.<br><br><b>Analogy:</b> A restaurant kitchen (writes food orders) is separate from the menu display (reads what's available).<br><br><b>Why:</b><br>• Reads and writes have very different performance and scaling needs<br>• Read models can be optimized (denormalized, cached, projected)<br>• Write models can enforce business rules cleanly<br><br><b>Simple flow:</b><br><code>Command → CommandHandler → Write DB<br>Query  → QueryHandler  → Read DB (or cache/view)</code><br><br><b>Common mistake:</b> Applying CQRS everywhere — it adds complexity. Use it only when reads and writes have genuinely different scaling or complexity needs.<br><br><b>Often paired with:</b> Event Sourcing, but they are <i>not</i> the same thing.

What is Event Sourcing?	<b>Event Sourcing</b> stores state changes as a sequence of immutable events, not the current state.<br><br><b>Analogy:</b> A bank statement — instead of storing "balance = $500", you store every deposit and withdrawal. The balance is <i>derived</i> from the history.<br><br><b>Benefits:</b><br>• Full audit trail built-in<br>• Can rebuild any past state by replaying events<br>• Enables event-driven integrations naturally<br><br><b>Drawbacks:</b><br>• Querying current state requires replaying events (mitigated with snapshots)<br>• Schema evolution is hard — old events must remain readable<br>• More infrastructure complexity<br><br><b>Common mistake:</b> Confusing Event Sourcing with event-driven architecture — you can have one without the other.<br><br><b>Senior insight:</b> Use snapshots every N events to avoid replaying thousands of events on startup.

What is the Saga Pattern for distributed transactions?	<b>Saga</b> manages long-running transactions across multiple services without a distributed lock.<br><br><b>Problem it solves:</b> In microservices, you can't use a single database transaction across services. If step 3 of 5 fails, you need to undo steps 1 and 2.<br><br><b>Two types:</b><br>• <b>Choreography</b> — each service reacts to events from others (no central coordinator)<br>• <b>Orchestration</b> — a central saga orchestrator tells each service what to do<br><br><b>Compensating transactions:</b> Each step must have an "undo" action.<br><code>CreateOrder → ReserveInventory → ChargePayment<br>          ↓ (if payment fails)<br>ReleaseInventory → CancelOrder</code><br><br><b>Trade-off:</b> No atomicity guarantee — the system is eventually consistent. Design compensations carefully.<br><br><b>Tools:</b> MassTransit Saga (C#), Conductor, Axon (Java).

What is the Sidecar Pattern?	<b>Sidecar</b> deploys a helper container alongside your main application container, sharing its lifecycle.<br><br><b>Analogy:</b> A motorcycle sidecar — the sidecar goes where the motorcycle goes, but handles a different job.<br><br><b>Common uses:</b><br>• Logging agent (Fluentd, Filebeat) — collects and ships logs<br>• Service mesh proxy (Envoy/Istio) — handles TLS, retries, observability<br>• Config sync — keeps configuration up to date<br><br><b>In Kubernetes (pod spec):</b><br><code>containers:<br>&nbsp;&nbsp;- name: app<br>&nbsp;&nbsp;&nbsp;&nbsp;image: myapp:latest<br>&nbsp;&nbsp;- name: log-shipper<br>&nbsp;&nbsp;&nbsp;&nbsp;image: fluentd:latest</code><br><br><b>Trade-off:</b> Doubles container count, adds resource overhead. But it cleanly separates cross-cutting concerns from business logic.

What is the Strangler Fig Pattern?	<b>Strangler Fig</b> is a migration strategy to incrementally replace a legacy monolith with microservices.<br><br><b>Analogy:</b> A strangler fig tree grows around an existing tree, eventually replacing it without cutting it down all at once.<br><br><b>How it works:</b><br>1. Put a façade (API gateway or proxy) in front of the monolith<br>2. Route specific features to new microservices as they're built<br>3. Gradually redirect more traffic until the monolith is gone<br><br><b>Why not rewrite everything at once?</b> "Big Bang" rewrites fail 90% of the time — they take too long, miss edge cases, and risk business continuity.<br><br><b>Senior insight:</b> The façade itself can become a long-term API Gateway, not just a migration tool.

What is the Backends for Frontends (BFF) Pattern?	<b>BFF</b> creates a separate backend API for each type of frontend (web, mobile, third-party).<br><br><b>Problem:</b> A single generic API must satisfy very different clients — mobile needs small payloads, web needs rich data, third-party needs a stable contract.<br><br><b>Solution:</b><br><code>Mobile App → Mobile BFF → Microservices<br>Web App   → Web BFF   → Microservices<br>3rd Party → Public API → Microservices</code><br><br><b>Benefits:</b><br>• Each BFF can optimize for its client's needs<br>• Frontend teams own their BFF<br>• Reduces over-fetching and under-fetching<br><br><b>Trade-off:</b> More services to maintain. Avoid duplication by having BFFs call shared microservices, not reimplement business logic.<br><br><b>Common mistake:</b> Putting business logic in the BFF — it should only aggregate and transform, not own domain rules.

What is the API Gateway Pattern?	<b>API Gateway</b> is a single entry point for all client requests to your backend services.<br><br><b>Responsibilities:</b><br>• Routing requests to the correct microservice<br>• Authentication & authorization<br>• Rate limiting and throttling<br>• SSL termination<br>• Request/response transformation<br>• Logging and observability<br><br><b>Flow:</b><br><code>Client → API Gateway → Service A<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;→ Service B</code><br><br><b>Products:</b> AWS API Gateway, Kong, NGINX, Azure API Management, Traefik.<br><br><b>Trade-off:</b> The gateway can become a bottleneck or single point of failure — deploy it in HA mode and keep it thin (no business logic).<br><br><b>Common mistake:</b> Overloading the gateway with business logic — it should be an infrastructure concern only.

What is a Service Mesh and why do you need one?	A <b>Service Mesh</b> is an infrastructure layer that handles service-to-service communication transparently.<br><br><b>What it provides (without changing app code):</b><br>• Mutual TLS (mTLS) between services<br>• Automatic retries and circuit breaking<br>• Distributed tracing<br>• Traffic shaping (canary, A/B routing)<br>• Observability (metrics per service call)<br><br><b>How it works:</b> A sidecar proxy (e.g., Envoy) is injected next to every pod. All traffic goes through the proxy.<br><br><code>Service A → Envoy Proxy → Envoy Proxy → Service B</code><br><br><b>Products:</b> Istio, Linkerd, Consul Connect.<br><br><b>Trade-off:</b> Significant operational complexity. Only worth it at scale (50+ services). Linkerd is simpler than Istio for getting started.<br><br><b>Senior insight:</b> A service mesh solves the same problems as a library-based approach (Resilience4j, Polly) but at the platform level, removing the burden from developers.

What is the Cache-Aside (Lazy Loading) Pattern?	<b>Cache-Aside</b> loads data into the cache only when it's requested (lazy).<br><br><b>Flow:</b><br>1. App checks cache first<br>2. Cache miss → app fetches from database<br>3. App writes result to cache<br>4. Returns data to caller<br><br><code>var result = cache.Get(key);<br>if (result == null) {<br>&nbsp;&nbsp;result = db.Query(key);<br>&nbsp;&nbsp;cache.Set(key, result, ttl);<br>}<br>return result;</code><br><br><b>Pros:</b> Cache only contains what's actually needed; resilient to cache restarts.<br><b>Cons:</b> First request is always slow (cache cold start); risk of stale data.<br><br><b>Common mistake:</b> Forgetting TTL — stale data lives forever.<br><br><b>Alternative:</b> Write-Through cache (write to cache and DB simultaneously) keeps cache always warm but uses more resources.

What is the Competing Consumers Pattern?	<b>Competing Consumers</b> runs multiple worker instances reading from the same message queue, processing messages in parallel.<br><br><b>Analogy:</b> Multiple cashiers at a supermarket, all drawing from the same customer queue.<br><br><code>Queue → [Worker 1]<br>      → [Worker 2]<br>      → [Worker 3]</code><br><br><b>Benefits:</b><br>• Horizontal scaling — add more workers when load increases<br>• If one worker crashes, others keep processing<br>• Natural load balancing<br><br><b>Important:</b> Messages must be processed idempotently — a message may be delivered more than once (at-least-once delivery).<br><br><b>Tools:</b> AWS SQS + Lambda, Azure Service Bus, RabbitMQ, Kafka consumer groups.<br><br><b>Senior insight:</b> Use visibility timeout (SQS) or lock duration (Service Bus) to prevent two workers from processing the same message simultaneously.

What is the Queue-Based Load Leveling Pattern?	<b>Queue-Based Load Leveling</b> uses a queue as a buffer between producers and consumers to absorb traffic spikes.<br><br><b>Problem:</b> Your API receives 10,000 requests per second during a flash sale, but your backend can only handle 1,000/sec.<br><br><b>Solution:</b> Requests go into a queue. Workers drain the queue at a sustainable rate.<br><br><code>API → Queue → [Workers (1000 req/s)]</code><br><br><b>Benefits:</b><br>• Protects backend from being overwhelmed<br>• Smooths out burst traffic<br>• Callers get instant acknowledgment (async)<br><br><b>Trade-off:</b> Introduces latency — responses are no longer immediate. Only works for workloads that can be async (order processing, email sending, image resizing).<br><br><b>Anti-pattern:</b> Using this for user-facing real-time requests where users expect immediate responses.

What is the Throttling Pattern?	<b>Throttling</b> limits how many requests a client can make in a time window to protect service resources.<br><br><b>Common algorithms:</b><br>• <b>Fixed Window:</b> 100 requests per minute, resets at :00<br>• <b>Sliding Window:</b> 100 requests in any rolling 60-second window<br>• <b>Token Bucket:</b> Tokens added at fixed rate; each request costs a token (allows short bursts)<br>• <b>Leaky Bucket:</b> Requests processed at a fixed rate regardless of burst<br><br><b>HTTP response:</b> Return <code>HTTP 429 Too Many Requests</code> with a <code>Retry-After</code> header.<br><br><b>Where to implement:</b> API Gateway (preferred) or middleware in the service.<br><br><b>Senior insight:</b> Use different rate limits per client tier (free: 100/min, paid: 10,000/min). Store counters in Redis for distributed rate limiting.

What is the Sidecar vs Ambassador vs Adapter pattern distinction?	All three are container patterns for extending functionality without modifying the main container:<br><br><b>Sidecar:</b> Adds functionality to the main container (logging, monitoring, config sync). Tightly coupled to the app lifecycle.<br><br><b>Ambassador:</b> Acts as a proxy to external services. The app talks to localhost, the ambassador handles connection logic (retries, discovery, auth).<br><code>App → Ambassador → External Service</code><br><br><b>Adapter:</b> Translates the interface of the main container to a standard interface — useful when different services expose data differently but need to look the same.<br><br><b>Memory trick:</b><br>• <b>Sidecar</b> = helper<br>• <b>Ambassador</b> = proxy outward<br>• <b>Adapter</b> = translator

What is Cloud-Native architecture?	<b>Cloud-Native</b> is an approach to building apps that fully exploits the advantages of cloud computing.<br><br><b>Core principles (CNCF definition):</b><br>• <b>Microservices</b> — decompose into small, independently deployable services<br>• <b>Containers</b> — package apps for consistency across environments<br>• <b>Dynamic orchestration</b> — Kubernetes manages placement, scaling, healing<br>• <b>Continuous delivery</b> — automated pipelines deploy frequently<br><br><b>Not just "runs in the cloud"</b> — a monolith lifted to EC2 is not cloud-native.<br><br><b>True cloud-native apps are:</b><br>• Horizontally scalable<br>• Fault-tolerant by design<br>• Observable (logs, metrics, traces)<br>• Deployable without downtime<br><br><b>Senior insight:</b> Cloud-native is a culture and architecture shift, not just a technology choice.

What are the 12-Factor App principles?	A methodology for building cloud-native, scalable, maintainable SaaS apps:<br><br>1. <b>Codebase</b> — one codebase, many deploys<br>2. <b>Dependencies</b> — explicitly declare all dependencies<br>3. <b>Config</b> — store config in environment variables, not code<br>4. <b>Backing services</b> — treat databases, queues as attached resources<br>5. <b>Build, Release, Run</b> — strictly separate build and run stages<br>6. <b>Processes</b> — stateless processes; state in backing services<br>7. <b>Port binding</b> — expose services via port binding<br>8. <b>Concurrency</b> — scale out via the process model<br>9. <b>Disposability</b> — fast startup, graceful shutdown<br>10. <b>Dev/prod parity</b> — keep environments as similar as possible<br>11. <b>Logs</b> — treat logs as event streams<br>12. <b>Admin processes</b> — run admin tasks as one-off processes<br><br><b>Most violated factors in practice:</b> Config (hardcoded values) and Dev/prod parity (works on my machine).

What is Immutable Infrastructure?	<b>Immutable Infrastructure</b> means servers/containers are never modified after deployment. To change anything, you deploy a new version.<br><br><b>Analogy:</b> Instead of repainting a car, you replace it with a new one in the color you want.<br><br><b>Traditional (mutable):</b> SSH into a server → change config → restart service. Risk: servers drift, become snowflakes.<br><br><b>Immutable:</b> Build a new Docker image → deploy → old containers die.<br><br><b>Benefits:</b><br>• No configuration drift<br>• Rollback = redeploy previous image<br>• Identical environments (dev = staging = prod)<br>• Security — no running servers accepting SSH<br><br><b>Tools:</b> Docker, Packer, Kubernetes (replace pods, never patch them).<br><br><b>Common mistake:</b> Storing state inside containers — state must live in external services (databases, object storage).

What is GitOps?	<b>GitOps</b> uses Git as the single source of truth for infrastructure and application state. Changes to the system happen only through Git commits.<br><br><b>Flow:</b><br><code>Developer → Git PR → Merge → GitOps agent detects drift → Applies to cluster</code><br><br><b>Key principles:</b><br>• Desired state declared in Git (YAML, Helm, Kustomize)<br>• Automated agent (Flux, ArgoCD) reconciles actual state to desired state<br>• All changes are auditable through Git history<br>• Rollback = revert a commit<br><br><b>Benefits:</b><br>• No manual kubectl apply in production<br>• Full audit trail<br>• Faster recovery (re-sync from Git)<br><br><b>Tools:</b> ArgoCD, Flux CD.<br><br><b>Senior insight:</b> GitOps works best with Kubernetes. For other infrastructure, use Terraform + Atlantis for similar pull-request-driven workflows.

What is Observability in cloud systems? What are the three pillars?	<b>Observability</b> is the ability to understand the internal state of a system from its external outputs — especially when something goes wrong in a way you didn't predict.<br><br><b>The 3 Pillars:</b><br><br>1. <b>Logs</b> — discrete events with context<br><code>{"level":"error","msg":"Payment failed","orderId":"123","userId":"456"}</code><br><br>2. <b>Metrics</b> — numeric measurements over time (latency, error rate, throughput)<br><code>http_request_duration_seconds{route="/checkout"} 0.42</code><br><br>3. <b>Traces</b> — track a single request across multiple services (distributed tracing)<br><code>TraceId: abc123 → Service A (50ms) → Service B (120ms) → DB (30ms)</code><br><br><b>Tools:</b><br>• Logs: ELK Stack, Loki, Datadog<br>• Metrics: Prometheus + Grafana, CloudWatch<br>• Traces: Jaeger, Zipkin, AWS X-Ray, Datadog APM<br><br><b>Senior insight:</b> Correlate all three with a shared <code>traceId</code> / <code>correlationId</code> so you can jump from a metric spike → trace → log entry.

What is a Container and how does it differ from a Virtual Machine?	<b>Container</b> packages an app with its dependencies (code, runtime, libraries) into a single portable unit.<br><br><b>VM vs Container:</b><br><code>VM: Hardware → Hypervisor → Guest OS → App<br>Container: Hardware → Host OS → Container Runtime → App</code><br><br><b>Key differences:</b><br>• Containers share the host OS kernel — much lighter (MBs vs GBs)<br>• VMs are fully isolated with their own OS — stronger security boundary<br>• Containers start in milliseconds; VMs take seconds to minutes<br>• Containers are immutable by design<br><br><b>When to use VMs:</b> Strong isolation requirements (multi-tenant untrusted code), Windows workloads on Linux hosts, legacy applications.<br><br><b>When to use containers:</b> Microservices, consistent deployments, fast scaling, CI/CD pipelines.<br><br><b>Common mistake:</b> Running one container per VM defeats the purpose — containers shine when many run on fewer, larger hosts.

What is Kubernetes and what problems does it solve?	<b>Kubernetes (K8s)</b> is a container orchestration platform that automates deployment, scaling, and management of containerized applications.<br><br><b>Problems it solves:</b><br>• <b>Scheduling:</b> Where should each container run across a cluster of nodes?<br>• <b>Self-healing:</b> Restarts failed containers, replaces dead nodes<br>• <b>Scaling:</b> Horizontal Pod Autoscaler adjusts replicas based on load<br>• <b>Service discovery:</b> Services get stable DNS names inside the cluster<br>• <b>Rolling deployments:</b> Zero-downtime updates<br>• <b>Config management:</b> ConfigMaps and Secrets decouple config from images<br><br><b>Core objects:</b><br>• <b>Pod</b> — smallest unit (one or more containers)<br>• <b>Deployment</b> — manages replica sets and rolling updates<br>• <b>Service</b> — stable network endpoint for pods<br>• <b>Ingress</b> — HTTP routing into the cluster<br><br><b>Senior insight:</b> Kubernetes is powerful but operationally complex. Use managed services (EKS, AKS, GKE) to offload control plane management.

What is Serverless computing and when should you use it?	<b>Serverless</b> means running code without managing servers. You provide a function; the cloud runs it on demand and bills per execution.<br><br><b>Key characteristics:</b><br>• No server provisioning or patching<br>• Automatic scaling from zero to millions of invocations<br>• Pay per invocation (not idle time)<br>• Stateless by default<br><br><b>Best use cases:</b><br>• Event-driven workloads (file uploads, queue processing, webhooks)<br>• Infrequent or unpredictable traffic<br>• Glue code and integrations<br><br><b>Avoid for:</b><br>• Long-running tasks (most platforms cap at 15 minutes)<br>• Low-latency real-time apps (cold starts)<br>• Heavy stateful workloads<br><br><b>Cold start problem:</b> The first invocation spins up a new execution environment — adds 100ms–2s of latency. Mitigation: provisioned concurrency (AWS), keep-alive pings.<br><br><b>Products:</b> AWS Lambda, Azure Functions, Google Cloud Functions, Cloudflare Workers.

What is Service Discovery in microservices?	<b>Service Discovery</b> is the mechanism by which services find each other's network location dynamically, without hardcoding IP addresses.<br><br><b>Problem:</b> In the cloud, service instances start and stop constantly. IPs change. You can't hardcode <code>http://192.168.1.5:8080</code>.<br><br><b>Two types:</b><br><br><b>Client-side discovery:</b> The client queries a registry (Consul, Eureka) and picks an instance to call.<br><code>Client → Registry → gets IP list → calls Service B</code><br><br><b>Server-side discovery:</b> Client calls a load balancer/gateway; it handles discovery.<br><code>Client → Load Balancer → Service B</code><br><br><b>In Kubernetes:</b> Built-in via DNS. Each Service gets a stable DNS name: <code>my-service.my-namespace.svc.cluster.local</code>.<br><br><b>Tools:</b> Consul, AWS Cloud Map, Eureka, Kubernetes DNS.<br><br><b>Senior insight:</b> Kubernetes makes service discovery nearly invisible for in-cluster communication, but you still need a strategy for cross-cluster or hybrid scenarios.

What is Infrastructure as Code (IaC)?	<b>Infrastructure as Code (IaC)</b> manages and provisions infrastructure through machine-readable configuration files instead of manual processes.<br><br><b>Benefits:</b><br>• Reproducible environments (no snowflakes)<br>• Version control for infrastructure<br>• Code review for infrastructure changes<br>• Automated provisioning (CI/CD pipelines)<br><br><b>Types of IaC tools:</b><br>• <b>Declarative</b> (define desired state, tool figures out how): Terraform, Pulumi, CloudFormation, Bicep<br>• <b>Imperative</b> (define exact steps): Ansible, Chef, Puppet<br><br><b>Example (Terraform):</b><br><code>resource "aws_s3_bucket" "my_bucket" {<br>&nbsp;&nbsp;bucket = "my-app-data"<br>&nbsp;&nbsp;tags = { Environment = "prod" }<br>}</code><br><br><b>Common mistake:</b> Storing Terraform state locally — always use remote state (S3 + DynamoDB lock, Terraform Cloud).<br><br><b>Senior insight:</b> Treat IaC like application code: pull requests, linting (tflint), security scanning (Checkov, tfsec), and automated plan review.

What is Blue-Green Deployment?	<b>Blue-Green</b> runs two identical production environments. Only one (Blue) serves traffic. When deploying, you update the other (Green) and switch traffic over.<br><br><code>Users → Load Balancer → [Blue (live)]<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;[Green (new version being deployed)]<br>— switch —<br>Users → Load Balancer → [Green (now live)]<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;[Blue (idle, ready to rollback)]</code><br><br><b>Benefits:</b><br>• Zero-downtime deployments<br>• Instant rollback (flip traffic back to Blue)<br>• Full production testing before switch<br><br><b>Trade-off:</b> Requires double the infrastructure cost during deployment.<br><br><b>Common challenge:</b> Database schema changes must be backward-compatible (both versions must work with the same DB simultaneously).<br><br><b>In Kubernetes:</b> Achievable with two Deployments and a Service selector switch.

What is Canary Deployment?	<b>Canary</b> gradually shifts traffic from the old version to the new version, monitoring for problems before full rollout.<br><br><b>Analogy:</b> Coal miners used canary birds — if the canary died, they knew there was danger before committing everyone.<br><br><code>100% → Old Version<br>  5% → New Version (canary)<br>  ↓ (if metrics look good)<br> 25% → New Version<br> 50% → New Version<br>100% → New Version</code><br><br><b>Benefits:</b><br>• Limits blast radius of bugs to a small % of users<br>• Real production validation before full rollout<br>• Data-driven rollout decisions based on error rate, latency<br><br><b>vs Blue-Green:</b> Blue-Green is all-or-nothing; Canary is gradual.<br><br><b>Tools:</b> Argo Rollouts, Flagger, AWS CodeDeploy, Azure Deployment Slots, Istio traffic splitting.<br><br><b>Senior insight:</b> Define automatic rollback criteria (error rate > 1% → rollback automatically) to make canaries self-managing.

What is Chaos Engineering?	<b>Chaos Engineering</b> intentionally introduces failures into a system to verify it can handle them gracefully in production.<br><br><b>Philosophy:</b> "If it hurts, do it more often in a controlled way" — find weaknesses before customers do.<br><br><b>Common experiments:</b><br>• Kill random pods (Chaos Monkey)<br>• Inject network latency between services<br>• Simulate AZ/region failure<br>• Drain a database connection pool<br>• Corrupt messages in a queue<br><br><b>Process:</b><br>1. Define steady state (normal behavior metrics)<br>2. Hypothesize: "System will remain stable if we kill 1 pod"<br>3. Run experiment in production (or staging first)<br>4. Observe and compare to steady state<br>5. Fix weaknesses found<br><br><b>Tools:</b> Netflix Chaos Monkey, AWS Fault Injection Simulator (FIS), Chaos Mesh, Gremlin.<br><br><b>Senior insight:</b> Start in staging, not production. Have a kill switch. Only run chaos on systems with good observability — otherwise you can't see the results.

What is Feature Flags (Feature Toggles)?	<b>Feature Flags</b> let you enable or disable features at runtime without deploying new code.<br><br><code>if (featureFlags.IsEnabled("new-checkout-flow", userId)) {<br>&nbsp;&nbsp;return NewCheckoutFlow();<br>} else {<br>&nbsp;&nbsp;return OldCheckoutFlow();<br>}</code><br><br><b>Use cases:</b><br>• <b>Trunk-based development:</b> Merge incomplete features safely behind a flag<br>• <b>Canary releases:</b> Enable for 5% of users<br>• <b>A/B testing:</b> Test two variants on different user segments<br>• <b>Kill switch:</b> Instantly disable a broken feature without a hotfix deploy<br><br><b>Types:</b> Release flags, Experiment flags, Ops flags, Permission flags.<br><br><b>Tools:</b> LaunchDarkly, Unleash, AWS AppConfig, Azure App Configuration, Flagsmith.<br><br><b>Common mistake:</b> Never cleaning up old flags — leads to flag debt. Every flag should have an expiry date and owner.<br><br><b>Senior insight:</b> Flags are a short-lived tool; the codebase should eventually have the winning path only.

What is the Shared Responsibility Model in cloud security?	The <b>Shared Responsibility Model</b> defines what the cloud provider secures vs what you (the customer) must secure.<br><br><b>AWS model:</b><br><code>AWS secures: Physical hardware, global network,<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;hypervisor, managed service internals<br><br>You secure: OS patching, app code, IAM config,<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;data encryption, firewall rules, secrets management</code><br><br><b>Varies by service type:</b><br>• <b>IaaS (EC2):</b> You own more — OS, runtime, data<br>• <b>PaaS (RDS, Elastic Beanstalk):</b> AWS owns OS/runtime; you own data and config<br>• <b>SaaS (S3, DynamoDB):</b> AWS owns almost everything; you own access policies and data classification<br><br><b>Common mistake:</b> Assuming "cloud is secure" means your data is automatically protected. S3 buckets are public by default if you don't configure them correctly.<br><br><b>Senior insight:</b> The more managed the service, the less you configure — but also the less control. Understand the trade-off for each service.

What are Cloud Regions and Availability Zones (AZs)?	<b>Region:</b> A geographic location with multiple data centers (e.g., us-east-1, eu-west-1).<br><br><b>Availability Zone (AZ):</b> An isolated subset of data centers within a Region, connected by low-latency links. Each AZ is independent — separate power, cooling, networking.<br><br><code>Region: us-east-1<br>&nbsp;&nbsp;AZ: us-east-1a (data center cluster 1)<br>&nbsp;&nbsp;AZ: us-east-1b (data center cluster 2)<br>&nbsp;&nbsp;AZ: us-east-1c (data center cluster 3)</code><br><br><b>Design rule:</b> Always deploy across at least 2 AZs for high availability. If one AZ fails, traffic routes to the others.<br><br><b>Multi-Region:</b> For disaster recovery or global low-latency. Significantly more complex (data replication, routing, consistency).<br><br><b>RTO/RPO:</b><br>• Multi-AZ: Near-zero downtime, seconds of data loss<br>• Multi-Region: Minutes of downtime, possible data loss depending on replication lag<br><br><b>Cost:</b> Cross-AZ traffic has a small cost. Cross-Region traffic is much more expensive.

What is a Content Delivery Network (CDN)?	A <b>CDN</b> is a globally distributed network of servers that caches content close to end users, reducing latency.<br><br><b>How it works:</b><br>1. User in Tokyo requests your website hosted in us-east-1<br>2. CDN intercepts the request at the nearest edge node (Tokyo PoP)<br>3. If cached: returns content instantly (< 10ms)<br>4. If not cached: fetches from origin, caches, returns<br><br><b>What CDNs cache:</b><br>• Static assets: HTML, JS, CSS, images, fonts<br>• Video streams<br>• API responses (with proper cache headers)<br><br><b>Benefits beyond speed:</b><br>• DDoS protection (absorbs attack traffic at the edge)<br>• Reduced origin load<br>• HTTPS termination at the edge<br><br><b>Products:</b> AWS CloudFront, Cloudflare, Azure CDN, Fastly, Akamai.<br><br><b>Common mistake:</b> Caching API responses without proper Cache-Control headers — users get stale personalized data.<br><br><b>Senior insight:</b> Cloudflare Workers/Lambda@Edge allow running logic at edge nodes — great for auth, A/B testing, and request manipulation.

What is AWS IAM and what are its core concepts?	<b>AWS IAM (Identity and Access Management)</b> controls who can do what in your AWS account.<br><br><b>Core concepts:</b><br>• <b>User:</b> A person or service with long-term credentials<br>• <b>Group:</b> Collection of users sharing the same permissions<br>• <b>Role:</b> Temporary credentials assumed by a service, EC2, Lambda, or external identity<br>• <b>Policy:</b> JSON document defining allowed/denied actions on resources<br><br><b>Example policy (allow S3 read):</b><br><code>{<br>&nbsp;&nbsp;"Effect": "Allow",<br>&nbsp;&nbsp;"Action": ["s3:GetObject"],<br>&nbsp;&nbsp;"Resource": "arn:aws:s3:::my-bucket/*"<br>}</code><br><br><b>Principle of Least Privilege:</b> Grant only the permissions needed — nothing more.<br><br><b>Common mistakes:</b><br>• Using root account for day-to-day tasks<br>• Giving AdministratorAccess to Lambda functions<br>• Storing IAM access keys in code (use roles instead)<br><br><b>Senior insight:</b> Use IAM Roles for EC2/Lambda (they get temporary auto-rotated credentials). Never use IAM Users with static keys for service-to-service auth.

What is AWS S3 and what are its key architectural considerations?	<b>AWS S3 (Simple Storage Service)</b> is object storage — store any file up to 5TB, accessed via HTTP.<br><br><b>Key concepts:</b><br>• <b>Bucket:</b> Container for objects (like a top-level folder)<br>• <b>Object:</b> A file + metadata, identified by a key (path-like string)<br>• <b>Prefix:</b> Simulated folder structure (S3 is flat, not hierarchical)<br><br><b>Storage classes (cost vs access speed):</b><br>• S3 Standard — frequent access<br>• S3 Infrequent Access (IA) — cheaper, higher retrieval cost<br>• S3 Glacier — archival, minutes to hours retrieval<br>• S3 Intelligent-Tiering — auto-moves objects between tiers<br><br><b>Architecture considerations:</b><br>• Use pre-signed URLs for temporary, secure access to private objects<br>• Enable versioning for accidental deletion protection<br>• Use S3 lifecycle policies to auto-move/delete old objects<br>• Bucket policies + Block Public Access — never expose buckets publicly unless intentional<br><br><b>Performance:</b> S3 can handle thousands of requests/second per prefix. If you have millions of objects, randomize key prefixes to avoid hot partitions (though AWS has largely solved this).

What is AWS Lambda and its architectural limits?	<b>AWS Lambda</b> runs your code in response to events without managing servers.<br><br><b>Triggers include:</b> API Gateway, S3 events, SQS, SNS, DynamoDB Streams, EventBridge, CloudWatch Events.<br><br><b>Key limits to architect around:</b><br>• Max execution time: <b>15 minutes</b><br>• Max memory: <b>10 GB</b> (CPU scales proportionally)<br>• Max package size: 50 MB zipped, 250 MB unzipped (use Lambda Layers for shared deps)<br>• Max concurrent executions: 1,000 per region (default, can be increased)<br>• Ephemeral storage: 512 MB–10 GB in /tmp<br><br><b>Cold start:</b> First invocation in a new execution environment has extra latency.<br>• Java/C# worst (~1-2s), Python/Node best (~100ms)<br>• Mitigation: Provisioned Concurrency, SnapStart (Java)<br><br><b>Cost:</b> Billed per 1ms of execution + number of invocations.<br><br><b>Senior insight:</b> Lambda is not always cheapest at high, steady load — compare to always-on containers (Fargate/ECS) for sustained workloads. Lambda shines for spiky, event-driven, or infrequent workloads.

What is AWS RDS vs DynamoDB — when to choose which?	<b>RDS (Relational Database Service):</b> Managed relational databases (PostgreSQL, MySQL, SQL Server, Oracle, Aurora).<br><br><b>DynamoDB:</b> Fully managed NoSQL key-value and document store.<br><br><b>Choose RDS when:</b><br>• Complex queries with JOINs<br>• Strong ACID transactions across tables<br>• Existing relational data model<br>• Reporting and analytics queries<br><br><b>Choose DynamoDB when:</b><br>• Massive scale (millions of requests/sec)<br>• Simple access patterns (get by key, query by partition)<br>• Single-digit millisecond latency at any scale<br>• Serverless / event-driven architecture<br>• No ops overhead (truly serverless)<br><br><b>DynamoDB key concepts:</b><br>• <b>Partition Key:</b> Determines which physical shard stores the item<br>• <b>Sort Key:</b> Enables range queries within a partition<br>• Design access patterns first, then model your tables<br><br><b>Common DynamoDB mistake:</b> Choosing a hot partition key (e.g., a date like "2026-03-27") — all writes go to one shard. Use high-cardinality keys like UUID.<br><br><b>Aurora Serverless v2:</b> Best of both worlds — relational + auto-scaling. Good choice when workloads are variable.

What is AWS VPC and why does it matter for architecture?	<b>AWS VPC (Virtual Private Cloud)</b> is your own isolated network within AWS where you control IP ranges, subnets, routing, and security.<br><br><b>Core components:</b><br>• <b>VPC:</b> Isolated network (e.g., 10.0.0.0/16)<br>• <b>Subnet:</b> Subdivision within a VPC, tied to one AZ<br>• <b>Public subnet:</b> Has a route to the Internet Gateway — for load balancers, NAT gateways<br>• <b>Private subnet:</b> No direct internet access — for databases, app servers<br>• <b>Internet Gateway (IGW):</b> Enables public internet access<br>• <b>NAT Gateway:</b> Allows private subnet resources to initiate outbound internet calls<br>• <b>Security Groups:</b> Stateful firewalls at instance/service level<br>• <b>NACLs:</b> Stateless firewalls at subnet level<br><br><b>Best practice architecture:</b><br><code>Public subnet: Load Balancers, NAT Gateways<br>Private subnet: App servers, containers<br>Isolated subnet: Databases, caches</code><br><br><b>Common mistake:</b> Putting databases in public subnets or allowing 0.0.0.0/0 on Security Groups.<br><br><b>Senior insight:</b> Use VPC Endpoints to access AWS services (S3, DynamoDB) privately without traffic leaving the VPC.

What is AWS ECS vs EKS — when to choose each?	Both run containers on AWS, but with different complexity and control:<br><br><b>ECS (Elastic Container Service):</b><br>• AWS-proprietary orchestrator<br>• Simpler to learn and operate<br>• Deep AWS integration (IAM, CloudWatch, ALB out of the box)<br>• Two launch types: EC2 (manage instances) or Fargate (serverless, no instances)<br><br><b>EKS (Elastic Kubernetes Service):</b><br>• Managed Kubernetes<br>• Industry-standard, portable across clouds<br>• More powerful but more complex<br>• Better for teams with Kubernetes expertise or multi-cloud ambitions<br><br><b>Choose ECS when:</b> Small-to-medium teams, AWS-only, want simplicity, want Fargate (true serverless containers).<br><br><b>Choose EKS when:</b> Large teams, need advanced scheduling, already use Kubernetes, need ecosystem tools (Helm, Argo, Istio), multi-cloud portability.<br><br><b>Fargate:</b> Works with both ECS and EKS — removes EC2 management entirely. Pay per vCPU/memory per second.<br><br><b>Senior insight:</b> Most teams don't need Kubernetes. ECS + Fargate is often the right choice until organizational or technical complexity demands Kubernetes.

What is AWS SQS vs SNS vs EventBridge?	Three different messaging services for different patterns:<br><br><b>SQS (Simple Queue Service):</b><br>• Point-to-point queue<br>• One consumer per message<br>• Messages persist until consumed (or TTL expires)<br>• Use for: work queues, task distribution, load leveling<br><code>Producer → [SQS Queue] → Consumer</code><br><br><b>SNS (Simple Notification Service):</b><br>• Pub/Sub fan-out<br>• One message → multiple subscribers<br>• Push-based (delivers to SQS, Lambda, HTTP, email)<br>• Use for: notifications, fan-out to multiple systems<br><code>Publisher → SNS Topic → SQS, Lambda, Email, SMS</code><br><br><b>EventBridge:</b><br>• Event bus with routing rules<br>• Content-based routing (filter events by their content)<br>• Integrates with 200+ AWS services and SaaS apps<br>• Use for: event-driven architectures, service decoupling<br><code>Event → EventBridge → Rule: if source=orders → Lambda A<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;if detail.status=failed → SQS DLQ</code><br><br><b>Common pattern:</b> SNS + SQS fan-out: SNS topic fans out to multiple SQS queues, each processed independently.

What is Azure's equivalent of key AWS services?	Quick mapping of AWS → Azure equivalents:<br><br><code>AWS → Azure<br>EC2 → Azure VMs<br>ECS/EKS → AKS (Kubernetes), Container Apps<br>Lambda → Azure Functions<br>S3 → Azure Blob Storage<br>RDS → Azure SQL Database, Azure Database for PostgreSQL<br>DynamoDB → Azure Cosmos DB<br>SQS → Azure Service Bus (queues)<br>SNS → Azure Service Bus (topics)<br>EventBridge → Azure Event Grid<br>CloudWatch → Azure Monitor + Log Analytics<br>IAM → Azure Entra ID (formerly Azure AD) + RBAC<br>VPC → Azure Virtual Network (VNet)<br>API Gateway → Azure API Management<br>CloudFront → Azure CDN / Azure Front Door<br>Route 53 → Azure DNS<br>Elastic Beanstalk → Azure App Service<br>CodePipeline → Azure DevOps Pipelines<br>Secrets Manager → Azure Key Vault</code><br><br><b>Key Azure differentiators:</b><br>• Deep Microsoft ecosystem integration (.NET, Active Directory, Office 365)<br>• Azure Hybrid Benefit — discounts for existing Windows/SQL Server licenses<br>• Better enterprise identity via Entra ID<br><br><b>Senior insight:</b> Many enterprises standardize on Azure when they're already Microsoft-heavy (Active Directory, M365, SQL Server).

What is Google Cloud Platform (GCP) and its key differentiators?	<b>GCP (Google Cloud Platform)</b> is Google's public cloud, known for data/analytics strength and Kubernetes leadership (they invented it).<br><br><b>Key services:</b><br><code>Compute: Compute Engine (VMs), GKE (Kubernetes), Cloud Run (serverless containers), Cloud Functions<br>Storage: Cloud Storage (like S3), Cloud SQL, Cloud Spanner, Firestore, BigQuery<br>Messaging: Pub/Sub<br>ML/AI: Vertex AI, BigQuery ML, pre-trained APIs<br>Networking: Cloud Load Balancing, Cloud CDN, VPC</code><br><br><b>GCP differentiators:</b><br>• <b>BigQuery:</b> Serverless data warehouse — analyze petabytes with SQL, billed per query<br>• <b>Cloud Spanner:</b> Globally distributed relational database with ACID at planetary scale<br>• <b>GKE Autopilot:</b> Fully managed Kubernetes (no node management)<br>• <b>Networking:</b> Google's private global fiber network for all GCP traffic<br>• <b>AI/ML:</b> Strongest managed ML platform (Vertex AI, TPUs)<br><br><b>When to choose GCP:</b> Data analytics and ML workloads, companies already using Google Workspace, Kubernetes-first organizations.<br><br><b>Market position:</b> Third largest cloud (behind AWS and Azure) but growing fast in data and AI segments.

What is Multi-Cloud strategy and what are its trade-offs?	<b>Multi-Cloud</b> means using services from two or more cloud providers simultaneously.<br><br><b>Stated reasons companies choose multi-cloud:</b><br>• Avoid vendor lock-in<br>• Best-of-breed services (GCP for ML, AWS for breadth)<br>• Regulatory requirements (data residency in regions only one provider has)<br>• Negotiation leverage<br><br><b>Real trade-offs:</b><br>• Significantly increased operational complexity<br>• Teams must learn multiple toolsets (IAM, networking, monitoring differ per cloud)<br>• Inter-cloud data transfer costs (egress fees)<br>• Latency for cross-cloud service calls<br>• Harder to use cloud-native services (ties you to one provider) vs portable abstractions<br><br><b>Honest senior insight:</b> True multi-cloud is very hard to do well. Most organizations claiming "multi-cloud" actually run different workloads on different clouds (not the same workload on both). This is fine and simpler than genuine active-active multi-cloud.<br><br><b>Mitigation tools:</b> Kubernetes (portable compute), Terraform (portable IaC), Kafka (portable messaging) — but they add their own complexity.

What is Cloud Cost Optimization?	<b>Cloud Cost Optimization</b> is the ongoing practice of reducing cloud spend without sacrificing performance or reliability.<br><br><b>Key strategies:</b><br><br><b>Right-sizing:</b> Match instance size to actual usage. Most teams over-provision by 50%+.<br><code>Tool: AWS Compute Optimizer, Azure Advisor</code><br><br><b>Reserved/Committed use:</b> Pre-commit to 1-3 years of usage for 30-70% discount vs on-demand.<br><code>AWS: Reserved Instances, Savings Plans<br>Azure: Reserved VM Instances<br>GCP: Committed Use Discounts</code><br><br><b>Spot/Preemptible instances:</b> Spare capacity at 60-90% discount — can be terminated with 2-min notice. Use for fault-tolerant batch workloads.<br><br><b>Autoscaling:</b> Scale in during off-hours. A dev environment running 24/7 wastes ~65% of its cost.<br><br><b>Storage lifecycle:</b> Move infrequently accessed data to cheaper tiers automatically.<br><br><b>Tagging:</b> Tag every resource with team/project/env to allocate and track costs accurately.<br><br><b>Common waste sources:</b> Unused load balancers, orphaned EBS volumes, NAT gateway charges, data transfer (egress) costs.<br><br><b>Senior insight:</b> FinOps is a discipline — assign cost ownership to engineering teams, not just central finance.

What is the difference between Horizontal and Vertical Scaling in cloud?	<b>Vertical Scaling (Scale Up):</b> Make the existing server bigger (more CPU, RAM).<br><br><code>t3.medium (2 vCPU, 4GB) → t3.2xlarge (8 vCPU, 32GB)</code><br><br><b>Horizontal Scaling (Scale Out):</b> Add more servers of the same size.<br><br><code>1 × t3.medium → 10 × t3.medium</code><br><br><b>Comparison:</b><br>• Vertical: Simpler but has limits (biggest instance size), requires downtime (usually), single point of failure<br>• Horizontal: Limitless scale, no downtime, fault-tolerant, but requires stateless app design<br><br><b>Cloud preference:</b> Design for horizontal scaling. Use autoscaling groups (AWS ASG, Azure VMSS) to add/remove instances based on load.<br><br><b>What makes horizontal scaling hard:</b> Session state. If a user's session is on Server 1 and their next request hits Server 2 — problem. Solutions: stateless JWT tokens, shared Redis cache, sticky sessions (not recommended).<br><br><b>Senior insight:</b> Stateless application design is a prerequisite for effective horizontal scaling. The 12-Factor App principle #6 enforces this.

What is a Load Balancer and what are the types?	A <b>Load Balancer</b> distributes incoming traffic across multiple backend instances to improve availability and scalability.<br><br><b>Types by OSI layer:</b><br><br><b>Layer 4 (Transport/TCP):</b> Routes based on IP + port. Very fast, no HTTP awareness.<br><code>AWS: NLB (Network Load Balancer)<br>Azure: Azure Load Balancer</code><br><br><b>Layer 7 (Application/HTTP):</b> Routes based on URL path, headers, host. Supports SSL termination, cookie-based routing, content inspection.<br><code>AWS: ALB (Application Load Balancer)<br>Azure: Azure Application Gateway<br>GCP: Cloud Load Balancing</code><br><br><b>Algorithms:</b><br>• Round Robin — each server gets requests in turn<br>• Least Connections — routes to server with fewest active connections<br>• IP Hash — same client always goes to same server (sticky)<br>• Weighted — distribute unequally (canary routing)<br><br><b>Health checks:</b> LBs probe backends; unhealthy instances are removed from rotation automatically.<br><br><b>Common mistake:</b> Using NLB when you need HTTP-level routing, or ALB when you need ultra-low latency TCP throughput.

What is Auto Scaling in cloud and how does it work?	<b>Auto Scaling</b> automatically adjusts the number of running instances based on demand.<br><br><b>Three types of scaling policies:</b><br><br><b>Target Tracking:</b> Keep a metric at a target value.<br><code>"Keep average CPU at 60%"<br>→ Adds instances when CPU > 60%, removes when < 60%</code><br><br><b>Step Scaling:</b> Scale by different amounts based on how far a metric deviates.<br><code>CPU 70-80% → add 1 instance<br>CPU 80-90% → add 2 instances<br>CPU > 90% → add 4 instances</code><br><br><b>Scheduled Scaling:</b> Scale based on time (known traffic patterns).<br><code>"Add 5 instances every weekday at 8am, remove at 7pm"</code><br><br><b>Cooldown period:</b> Prevents scaling from reacting too fast to transient spikes. Default 300 seconds between scaling actions.<br><br><b>Warm-up time:</b> New instances need time to start and serve traffic. Account for this in scaling triggers.<br><br><b>Common mistake:</b> Scaling on CPU alone — scale on application-specific metrics (queue depth, request latency) for better accuracy.<br><br><b>Tools:</b> AWS Auto Scaling Groups, Azure VMSS, Kubernetes HPA (Horizontal Pod Autoscaler), KEDA (event-driven scaling).

What is a Managed Service vs Self-Managed in cloud?	<b>Managed Service:</b> The cloud provider operates the infrastructure for you — patching, HA, backups, scaling.<br><br><b>Self-Managed:</b> You run the software yourself on VMs or containers and handle all operations.<br><br><b>Example — Kafka:</b><br><code>Managed: AWS MSK, Confluent Cloud<br>&nbsp;&nbsp;→ Provider handles broker provisioning, upgrades, HA<br>&nbsp;&nbsp;→ You pay a premium but save ops time<br><br>Self-Managed: Run Kafka on EC2 or Kubernetes<br>&nbsp;&nbsp;→ You handle patching, ZooKeeper/KRaft, monitoring, disk management<br>&nbsp;&nbsp;→ Cheaper at scale but requires Kafka expertise</code><br><br><b>Trade-offs:</b><br>• Managed: Higher cost, less control, less expertise required, faster to set up<br>• Self-managed: Lower cost at scale, full control, requires deep expertise, higher operational burden<br><br><b>General rule:</b> Use managed services by default. Move to self-managed only when costs at your scale are prohibitive AND your team has the expertise.<br><br><b>Senior insight:</b> The hidden cost of self-managed is engineering time — ops, on-call, upgrades. Factor in fully-loaded engineering cost, not just AWS bill.

What is Cloud Security Posture Management (CSPM)?	<b>CSPM</b> tools continuously monitor your cloud environment for security misconfigurations and compliance violations.<br><br><b>Problems it detects:</b><br>• Public S3 buckets<br>• Security groups with 0.0.0.0/0 on port 22 or 3389<br>• Unencrypted storage volumes<br>• Root account without MFA<br>• IAM users with excessive permissions<br>• Logging disabled in regions<br><br><b>Native tools:</b><br><code>AWS: Security Hub, Config, Trusted Advisor<br>Azure: Microsoft Defender for Cloud<br>GCP: Security Command Center</code><br><br><b>Third-party tools:</b> Wiz, Prisma Cloud, Lacework, Orca Security.<br><br><b>How it works:</b> CSPM scans your cloud resource configurations (not traffic) against a rule set (CIS Benchmarks, SOC2, PCI-DSS) and alerts on violations.<br><br><b>Common mistake:</b> Having CSPM installed but ignoring its alerts. CSPM only helps if violations are remediated — ideally automatically via automated remediation workflows.<br><br><b>Senior insight:</b> Shift left — use IaC scanning (Checkov, tfsec) to catch misconfigs before they're deployed, not just after.

What is a Service Level Agreement (SLA), SLO, and SLI in cloud services?	These are related but distinct reliability concepts:<br><br><b>SLI (Service Level Indicator):</b> A metric that measures service behavior.<br><code>Examples: request success rate, p99 latency, availability</code><br><br><b>SLO (Service Level Objective):</b> Your internal target for an SLI.<br><code>Example: "99.9% of requests succeed" or "p99 latency < 200ms"</code><br><br><b>SLA (Service Level Agreement):</b> A formal, often contractual commitment to customers. Violated SLAs trigger financial penalties or credits.<br><code>Example: "We guarantee 99.9% uptime. If we fall below, you get credits."</code><br><br><b>Error Budget:</b> The allowable downtime derived from your SLO.<br><code>99.9% SLO = 0.1% error budget = 8.76 hours/year of downtime allowed<br>99.99% SLO = 0.01% = 52.6 minutes/year</code><br><br><b>How to use error budgets:</b> When error budget is healthy, ship features fast. When it's nearly exhausted, freeze features and focus on reliability.<br><br><b>Senior insight:</b> Set SLOs before incidents, not after. An SLO without an error budget is just a wish. AWS, Azure, and GCP all publish SLAs per service — factor these into your architecture's reliability budget.

What is Zero Trust Security in cloud architecture?	<b>Zero Trust</b> is a security model based on "never trust, always verify" — no user, device, or service is trusted by default, even inside the network perimeter.<br><br><b>Traditional model:</b> Trust everything inside the firewall.<br><code>Inside network → trusted → access granted</code><br><br><b>Zero Trust model:</b><br><code>Every request → verify identity → verify device → check permissions → log everything</code><br><br><b>Core principles:</b><br>• Verify explicitly — always authenticate and authorize (MFA, certificates)<br>• Least privilege access — minimal permissions, time-limited<br>• Assume breach — log all access, detect anomalies, segment blast radius<br><br><b>In cloud practice:</b><br>• mTLS between services (service mesh)<br>• Service accounts with narrow IAM roles<br>• VPC segmentation (not flat networks)<br>• Short-lived credentials (no static keys)<br>• Workload Identity instead of API keys<br><br><b>Tools:</b> AWS IAM + VPC Endpoints, Azure Private Link + Entra ID, Google BeyondCorp, Istio (mTLS), HashiCorp Vault.<br><br><b>Senior insight:</b> Zero Trust is not a product — it's a strategy. Start with identity (strong auth) and network segmentation before buying specialized tools.

What is the difference between Synchronous and Asynchronous communication in cloud services?	<b>Synchronous:</b> Caller waits for a response before continuing.<br><code>Client → API call → waits → gets response → continues</code><br><br><b>Asynchronous:</b> Caller sends a message and continues immediately. Response comes later (or not at all).<br><code>Client → puts message in queue → continues<br>Worker → picks up message → processes → (optionally notifies)</code><br><br><b>Synchronous pros/cons:</b><br>• Simple to reason about<br>• Tight coupling — if the called service is slow/down, caller suffers<br>• Poor for long operations<br><br><b>Asynchronous pros/cons:</b><br>• Loose coupling — services can fail independently<br>• Better scalability (queue absorbs spikes)<br>• More complex error handling and debugging<br>• Not suitable for real-time user interactions<br><br><b>Common hybrid pattern:</b> Sync API accepts the request (returns 202 Accepted), puts work on a queue, worker processes async. Client polls or receives webhook when done.<br><br><b>Senior insight:</b> Over-asyncing everything creates operational complexity. Use sync for user-facing real-time needs; use async for background processing, integration, and resilience.

What is Eventual Consistency vs Strong Consistency in distributed systems?	<b>Strong Consistency:</b> After a write completes, every subsequent read sees that write, regardless of which replica you read from.<br><br><b>Eventual Consistency:</b> After a write, reads may temporarily return old data. Given enough time (no new writes), all replicas converge to the same value.<br><br><b>Why the trade-off exists (CAP Theorem):</b> In a distributed system, you can't have both Consistency and Availability during a network partition — you must choose.<br><br><b>Examples:</b><br><code>Strong: PostgreSQL single instance, DynamoDB (with strong reads — costs 2× read capacity)<br>Eventual: DynamoDB (default), S3 (now strongly consistent for new objects), DNS</code><br><br><b>When eventual consistency is fine:</b> Social media likes, product view counts, search indexes (slight lag is OK).<br><br><b>When you need strong consistency:</b> Bank account balance, inventory "1 item left", seat reservations.<br><br><b>Senior insight:</b> Most cloud databases default to eventual consistency for performance. Always check the consistency model of your database and explicitly choose strong reads when correctness matters. Don't assume consistency — verify it.

What is a Dead Letter Queue (DLQ) and why is it essential?	A <b>Dead Letter Queue (DLQ)</b> is a separate queue where messages are moved after failing to be processed a configured number of times.<br><br><b>Why it's essential:</b> Without a DLQ, a poison message (one that always fails) blocks queue processing forever or gets retried infinitely.<br><br><b>Flow:</b><br><code>SQS Queue → Worker fails to process → retry (3x) → DLQ<br>                ↓<br>Alert triggers → engineer investigates message in DLQ</code><br><br><b>DLQ enables:</b><br>• Inspection of failed messages without losing them<br>• Alerting on DLQ depth (CloudWatch alarm)<br>• Redriving messages back to main queue after fixing the bug<br><br><b>AWS SQS DLQ config:</b><br><code>MaxReceiveCount: 3  // After 3 failures, move to DLQ</code><br><br><b>Common mistake:</b> Not setting up a DLQ at all, or setting it up but never alerting on it.<br><br><b>Senior insight:</b> Every queue in production must have a DLQ with a CloudWatch alarm on the ApproximateNumberOfMessagesVisible metric. Otherwise you'll never know messages are failing.

What is Distributed Tracing and how does it work?	<b>Distributed Tracing</b> tracks a single request as it travels through multiple services, showing where time was spent and where failures occurred.<br><br><b>Problem without tracing:</b> A slow response goes through API Gateway → Service A → Service B → Database. Without tracing, you don't know which hop is slow.<br><br><b>How it works:</b><br>1. First service generates a unique <code>traceId</code> and <code>spanId</code><br>2. Each subsequent service receives the trace context in headers and creates child spans<br>3. All spans are sent to a trace collector<br>4. You can visualize the full request as a timeline/waterfall<br><br><b>Key headers (W3C Trace Context standard):</b><br><code>traceparent: 00-{traceId}-{spanId}-01</code><br><br><b>Trace view example:</b><br><code>Total: 200ms<br>├── API Gateway: 5ms<br>├── Service A: 150ms<br>│   └── DB Query: 140ms ← bottleneck<br>└── Service B: 40ms</code><br><br><b>Tools:</b> AWS X-Ray, Jaeger, Zipkin, Datadog APM, Honeycomb, OpenTelemetry (standard instrumentation).<br><br><b>Senior insight:</b> Use OpenTelemetry for instrumentation — it's vendor-neutral. Then send traces to whichever backend you prefer without changing application code.

What is OpenTelemetry and why does it matter?	<b>OpenTelemetry (OTel)</b> is an open-source, vendor-neutral framework for collecting observability data (traces, metrics, logs) from applications.<br><br><b>Problem it solves:</b> Before OTel, you had to use vendor-specific SDKs (Datadog SDK, Jaeger SDK, X-Ray SDK). Switching vendors meant rewriting instrumentation throughout your codebase.<br><br><b>OTel architecture:</b><br><code>Your App + OTel SDK → OTel Collector → Jaeger / Datadog / Honeycomb / etc.</code><br><br><b>Key components:</b><br>• <b>API:</b> Interfaces for instrumentation (spans, metrics, logs)<br>• <b>SDK:</b> Implementation of the API<br>• <b>Collector:</b> Agent that receives, processes, and exports telemetry<br>• <b>Exporters:</b> Send data to specific backends<br><br><b>Auto-instrumentation:</b> For many languages, OTel instruments HTTP clients, databases, frameworks automatically — no code changes needed.<br><br><code>// .NET — auto-instrumentation<br>services.AddOpenTelemetry()<br>&nbsp;&nbsp;.WithTracing(b => b.AddAspNetCoreInstrumentation().AddHttpClientInstrumentation())<br>&nbsp;&nbsp;.WithMetrics(b => b.AddRuntimeInstrumentation());</code><br><br><b>Senior insight:</b> Adopt OTel as your observability standard now. It's becoming the industry default and gives you full flexibility to change backends without re-instrumenting.

What is a Data Lake vs a Data Warehouse in cloud architecture?	<b>Data Lake:</b> Stores raw, unprocessed data in its native format (files, JSON, Parquet, CSV) at massive scale.<br><br><b>Data Warehouse:</b> Stores structured, processed, query-optimized data for analytics and reporting.<br><br><code>Data Lake: AWS S3, Azure Data Lake Storage, GCP Cloud Storage<br>Data Warehouse: AWS Redshift, Azure Synapse, GCP BigQuery, Snowflake</code><br><br><b>Comparison:</b><br>• Lake: Schema on read (define structure when querying), cheap storage, all data types<br>• Warehouse: Schema on write (structure defined upfront), expensive, fast SQL queries<br><br><b>Modern pattern — Data Lakehouse:</b> Combines both — raw data in the lake, queryable with warehouse-like performance using table formats.<br><code>Tools: Apache Iceberg, Delta Lake, Apache Hudi on S3/ADLS</code><br><br><b>Flow:</b><br><code>Source Systems → [ETL/ELT] → Data Lake (raw) → [Transform] → Data Warehouse (curated) → BI Tools</code><br><br><b>Common mistake:</b> Building a "data swamp" — dumping everything into the lake without governance, cataloging, or schema management.<br><br><b>Senior insight:</b> BigQuery (GCP) blurs the line — it can query data directly in Cloud Storage with near-warehouse performance, making it a natural lakehouse.

What is Disaster Recovery (DR) in cloud and what are the key strategies?	<b>DR</b> is the process of restoring systems after a major failure (region outage, data corruption, ransomware).<br><br><b>Two key metrics:</b><br>• <b>RTO (Recovery Time Objective):</b> How long can you be down? (e.g., 4 hours)<br>• <b>RPO (Recovery Point Objective):</b> How much data can you lose? (e.g., 1 hour of transactions)<br><br><b>Four DR strategies (cheapest → most expensive, worst RTO → best RTO):</b><br><br>1. <b>Backup & Restore:</b> Back up data; restore from scratch. Cheapest. Hours-days RTO.<br>2. <b>Pilot Light:</b> Minimal core infrastructure always running; scale up during disaster. Minutes-hours RTO.<br>3. <b>Warm Standby:</b> Scaled-down version of prod always running. Minutes RTO.<br>4. <b>Multi-Site Active-Active:</b> Full duplicate running in multiple regions. Near-zero RTO and RPO. Most expensive.<br><br><b>Common mistake:</b> Choosing a DR strategy based on cost alone without understanding the business cost of downtime.<br><br><b>AWS tools:</b> AWS Backup, Route 53 failover routing, RDS Multi-AZ vs Cross-Region Read Replicas, S3 Cross-Region Replication.<br><br><b>Senior insight:</b> Test your DR plan regularly. An untested DR plan is not a DR plan — it's a hope.

What is FinOps and how does it relate to cloud architecture decisions?	<b>FinOps</b> (Financial Operations) is the practice of bringing financial accountability to cloud spending through collaboration between engineering, finance, and business teams.<br><br><b>Core principle:</b> Every engineer's technical decision is also a financial decision in the cloud.<br><br><b>FinOps lifecycle:</b><br>• <b>Inform:</b> Make costs visible (tagging, dashboards, cost allocation)<br>• <b>Optimize:</b> Rightsize, use savings plans, eliminate waste<br>• <b>Operate:</b> Continuous improvement, anomaly alerting, cost budgets<br><br><b>Architecture decisions with cost impact:</b><br><code>NAT Gateway: $0.045/GB data processed — avoid for S3 (use VPC Endpoints)<br>Inter-AZ traffic: ~$0.01/GB — minimize cross-AZ calls in hot paths<br>API Gateway: $3.50/million requests — consider ALB ($0.008/LCU) at high volume<br>DynamoDB On-Demand vs Provisioned: 6-7× price difference at steady load</code><br><br><b>Tools:</b> AWS Cost Explorer, AWS Compute Optimizer, Azure Cost Management, Infracost (IaC cost estimation in CI/CD).<br><br><b>Senior insight:</b> Architects who ignore costs create technical debt. Good cloud architecture optimizes for the right trade-off between performance, reliability, and cost — not just one of them.
