#deck:API Design & Communication

What is an API and why does it matter in distributed systems?	An <b>API (Application Programming Interface)</b> is a contract between two pieces of software that defines how they can talk to each other.<br><br>Think of it like a waiter in a restaurant: you (the client) don't go into the kitchen (the server) directly — you give your order to the waiter (the API), and it brings back what you asked for.<br><br><b>Why it matters:</b><br>- Decouples systems so they can evolve independently<br>- Enables reuse across teams, languages, and platforms<br>- Creates stable interfaces over changing implementations<br><br><b>Senior insight:</b> A poorly designed API is expensive to fix once consumers depend on it. Treat API design like a product decision, not just a technical one.

What is REST and what are its core constraints?	<b>REST (Representational State Transfer)</b> is an architectural style for building APIs over HTTP. It has 6 constraints:<br><br>1. <b>Client-Server</b> — UI and data logic are separated<br>2. <b>Stateless</b> — each request contains all info needed; server stores no session<br>3. <b>Cacheable</b> — responses should declare if they can be cached<br>4. <b>Uniform Interface</b> — consistent resource-based URLs and HTTP verbs<br>5. <b>Layered System</b> — client doesn't know if it's talking to a proxy or origin<br>6. <b>Code on Demand (optional)</b> — server can send executable code (e.g., JS)<br><br><b>Common mistake:</b> Many APIs called "REST" are actually just JSON over HTTP (they violate statelessness or uniform interface). True REST is rare — most teams aim for "RESTful enough."

What HTTP verbs are used in REST and what do they mean?	REST maps CRUD operations to HTTP verbs:<br><br><code>GET    /users       → list users (safe, idempotent)</code><br><code>GET    /users/1    → get one user</code><br><code>POST   /users       → create user (not idempotent)</code><br><code>PUT    /users/1    → replace full user</code><br><code>PATCH  /users/1    → partial update</code><br><code>DELETE /users/1    → delete user (idempotent)</code><br><br><b>Safe</b> = no side effects. <b>Idempotent</b> = same result if repeated.<br><br><b>Common mistake:</b> Using POST for everything, or using GET to mutate data (breaks caching and semantics).<br><br><b>Senior insight:</b> PUT vs PATCH matters at scale — PATCH reduces payload size and avoids overwrite conflicts in concurrent systems.

What is GraphQL and how does it differ from REST?	<b>GraphQL</b> is a query language for APIs where the <i>client</i> specifies exactly what data it needs.<br><br><b>REST:</b> Server defines the shape of the response<br><code>GET /users/1 → { id, name, email, address, ... }</code><br><br><b>GraphQL:</b> Client defines the shape<br><code>query { user(id: 1) { name email } }</code><br><br><b>Advantages of GraphQL:</b><br>- No over-fetching (only get what you ask)<br>- No under-fetching (get nested data in one request)<br>- Strongly typed schema acts as documentation<br><br><b>Disadvantages:</b><br>- Complex caching (no simple URL-based caching)<br>- N+1 query problems on the server (solved with DataLoader)<br>- Higher backend complexity<br><br><b>Use GraphQL when:</b> multiple clients (mobile, web) need different data shapes from the same API.

What is gRPC and when should you use it?	<b>gRPC</b> is a high-performance RPC framework by Google that uses <b>Protocol Buffers (protobuf)</b> for serialization and HTTP/2 for transport.<br><br>You define a contract in a <code>.proto</code> file:<br><code>service UserService {<br>  rpc GetUser (UserRequest) returns (UserResponse);<br>}</code><br><br>Then code is auto-generated in any supported language (Go, C#, Python, Java, etc.).<br><br><b>Advantages:</b><br>- Very fast — binary serialization, HTTP/2 multiplexing<br>- Strongly typed, contract-first<br>- Supports streaming (client, server, bidirectional)<br><br><b>Disadvantages:</b><br>- Not human-readable (binary format)<br>- Hard to use from browsers without a proxy (grpc-web)<br>- Steeper learning curve<br><br><b>Use gRPC when:</b> internal microservice communication, high-throughput systems, or polyglot environments needing a shared contract.

What is SOAP and when is it still relevant?	<b>SOAP (Simple Object Access Protocol)</b> is an XML-based messaging protocol with a strict contract defined by a <b>WSDL</b> file.<br><br><code>&lt;soap:Envelope&gt;<br>  &lt;soap:Body&gt;<br>    &lt;GetUser&gt;&lt;UserId&gt;1&lt;/UserId&gt;&lt;/GetUser&gt;<br>  &lt;/soap:Body&gt;<br>&lt;/soap:Envelope&gt;</code><br><br><b>Advantages:</b><br>- Built-in standards for security (WS-Security), transactions, and reliability<br>- Strongly typed and formally verified<br><br><b>Disadvantages:</b><br>- Verbose XML — large payloads<br>- Complex tooling<br>- Overkill for most modern use cases<br><br><b>Still relevant in:</b> Banking, healthcare (HL7), government, and enterprise systems with existing SOAP infrastructure.<br><br><b>Senior insight:</b> If you're integrating with a legacy enterprise system, you'll likely encounter SOAP. Know how to consume it, but don't design new APIs with it.

What are WebSockets and when should you use them?	<b>WebSockets</b> provide a persistent, full-duplex (two-way) connection between client and server over a single TCP connection.<br><br>Unlike HTTP (request → response), WebSockets allow both sides to send messages at any time.<br><br><b>Flow:</b><br>1. Client sends HTTP upgrade request<br>2. Server accepts → connection is now a WebSocket<br>3. Both sides can push messages freely<br><br><b>Use WebSockets when:</b><br>- Real-time features: chat, live dashboards, multiplayer games<br>- Server needs to push updates to the client without polling<br><br><b>Disadvantages:</b><br>- Stateful connections are harder to scale (need sticky sessions or a message broker)<br>- Overhead for maintaining thousands of open connections<br><br><b>Common mistake:</b> Using WebSockets when Server-Sent Events (SSE) would suffice — SSE is simpler for one-way server push.

What are Server-Sent Events (SSE) and how do they differ from WebSockets?	<b>SSE (Server-Sent Events)</b> is a one-way channel where the <b>server pushes</b> updates to the client over a regular HTTP connection.<br><br><b>Client code (JS):</b><br><code>const es = new EventSource('/events');<br>es.onmessage = e => console.log(e.data);</code><br><br><b>SSE vs WebSockets:</b><br><br>| Feature | SSE | WebSocket |<br>|---|---|---|<br>| Direction | Server → Client only | Both ways |<br>| Protocol | HTTP | ws:// |<br>| Reconnect | Auto | Manual |<br>| Complexity | Low | Higher |<br><br><b>Use SSE when:</b> notifications, live feeds, progress updates — client doesn't need to send data back.<br><br><b>Senior insight:</b> SSE works through standard HTTP infrastructure (proxies, load balancers). WebSockets sometimes require extra configuration at the infrastructure level.

What is tRPC and what problem does it solve?	<b>tRPC</b> is a TypeScript-first RPC framework that gives you end-to-end type safety between your frontend and backend — without generating code or schema files.<br><br>If you rename a backend function, TypeScript errors appear immediately in the frontend.<br><br><b>Example:</b><br><code>// Server<br>const appRouter = router({<br>  getUser: procedure.query(() => getUser())<br>});</code><br><br><code>// Client — fully typed<br>const user = await trpc.getUser.query();</code><br><br><b>Best for:</b> Full-stack TypeScript apps (Next.js, Remix) where the team controls both client and server.<br><br><b>Not suitable for:</b> Public APIs consumed by other teams or languages — no interop outside TypeScript.<br><br><b>Senior insight:</b> tRPC eliminates an entire class of runtime errors (type mismatches between client and server), but it's a tight coupling — only viable when client and server are co-deployed.

How do you choose between REST, GraphQL, and gRPC?	Think of it as a decision based on your audience and performance needs:<br><br><b>REST</b> → Public APIs, simple CRUD, broad client support<br><b>GraphQL</b> → Multiple clients with different data needs (mobile vs web)<br><b>gRPC</b> → Internal microservices, high performance, streaming<br><b>WebSockets</b> → Real-time bidirectional (chat, gaming)<br><b>SSE</b> → One-way server push (notifications, feeds)<br><b>SOAP</b> → Legacy enterprise integrations<br><br><b>Decision checklist:</b><br>- Who are the consumers? (external/internal)<br>- Do I need streaming?<br>- How performance-sensitive is this?<br>- Do I control both client and server?<br><br><b>Common mistake:</b> Choosing GraphQL because it's trendy, then struggling with caching, N+1 queries, and operational complexity on a simple CRUD app.

What is the Request/Response messaging pattern?	The simplest messaging pattern: the client sends a <b>request</b> and waits for a <b>response</b>.<br><br>Think of it like a phone call — you ask a question and wait on the line for the answer.<br><br><b>Synchronous example (HTTP REST):</b><br><code>GET /orders/123<br>→ { "id": 123, "status": "shipped" }</code><br><br><b>Characteristics:</b><br>- Simple to understand and debug<br>- Tight coupling — caller blocks until server responds<br>- Failure of one side immediately affects the other<br><br><b>Works well for:</b> User-facing actions that need immediate feedback (login, payment confirmation).<br><br><b>Senior insight:</b> In distributed systems, synchronous request/response chains increase latency and failure surface area. A timeout in one service can cascade. Use async patterns for non-critical paths.

What is the Publish/Subscribe (Pub/Sub) messaging pattern?	In <b>Pub/Sub</b>, a <b>publisher</b> sends messages to a <b>topic</b> without knowing who will receive them. <b>Subscribers</b> listen to topics they care about.<br><br>Think of it like a newspaper: the publisher prints it, multiple subscribers read it — they never interact directly.<br><br><b>Example tools:</b> Kafka, AWS SNS, Google Pub/Sub, Redis Pub/Sub<br><br><b>Flow:</b><br><code>Order Service → publishes "order.created" event<br>Email Service → subscribes to "order.created"<br>Inventory Service → subscribes to "order.created"</code><br><br><b>Advantages:</b><br>- Decoupled — publisher doesn't care about consumers<br>- Easy to add new consumers without changing publisher<br>- Scales consumers independently<br><br><b>Disadvantages:</b><br>- Harder to trace/debug (no direct link between cause and effect)<br>- At-least-once delivery means consumers must be idempotent

What is a Message Queue and how does it differ from Pub/Sub?	A <b>Message Queue</b> stores messages until a consumer processes them. Only <b>one consumer</b> processes each message (competing consumers model).<br><br><b>Pub/Sub:</b> All subscribers get a copy of each message<br><b>Queue:</b> Only one consumer gets each message<br><br>Think of Pub/Sub like a broadcast announcement; a queue is like a help desk ticket that goes to one agent.<br><br><b>Example tools:</b> RabbitMQ, AWS SQS, Azure Service Bus<br><br><b>Use queues when:</b><br>- Work needs to be distributed across workers (load balancing)<br>- Tasks should be processed exactly once<br>- You need durable, reliable delivery with retries<br><br><b>Example:</b> Image resizing jobs. Upload triggers a message; any available worker picks it up.<br><br><b>Senior insight:</b> Dead-Letter Queues (DLQ) are critical — messages that fail repeatedly should be moved there for inspection, not silently dropped.

What is Event Streaming and how does it differ from queues?	<b>Event Streaming</b> (e.g., Apache Kafka) stores events in an ordered, immutable log that multiple consumers can read — each at their own pace.<br><br><b>Queue:</b> Message is deleted after consumption<br><b>Stream:</b> Message is retained — consumers track their own position (offset)<br><br><b>Think of it like:</b> A queue is a to-do list that gets crossed off. A stream is a history log — anyone can re-read from any point.<br><br><b>Advantages of streams:</b><br>- Replay events (great for debugging, new service catch-up)<br>- Multiple independent consumers<br>- Event sourcing and audit trails<br><br><b>Use streaming when:</b><br>- Multiple teams need the same events independently<br>- You need replayability<br>- High-throughput pipelines (analytics, logs)<br><br><b>Common mistake:</b> Using Kafka for simple task queuing — it's operationally heavy. Use SQS/RabbitMQ for simple work queues.

What are Webhooks and how do they work?	A <b>webhook</b> is a reverse API call — instead of your app polling another service for updates, the service calls <i>your</i> endpoint when something happens.<br><br><b>Traditional polling:</b><br><code>Every 5s: GET /payments/status → "pending"... "pending"... "done"</code><br><br><b>Webhook:</b><br><code>Payment provider calls: POST /my-app/webhook → { "status": "done" }</code><br><br><b>Advantages:</b><br>- No wasted polling requests<br>- Near real-time updates<br><br><b>Challenges:</b><br>- Your endpoint must be publicly accessible<br>- Must handle retries (provider will retry on failure)<br>- Must validate the payload (use HMAC signature verification)<br>- Must respond quickly (return 200 immediately, process async)<br><br><b>Security tip:</b> Always verify webhook signatures. Never trust payload content without validation.

What is Long Polling and when is it used?	<b>Long Polling</b> is a technique where the client sends a request and the server holds it open until new data is available (or a timeout occurs).<br><br><b>Flow:</b><br>1. Client: <code>GET /notifications</code><br>2. Server holds request open (up to 30s)<br>3. When event occurs → server responds<br>4. Client immediately sends next request<br><br><b>vs Short Polling:</b><br><code>Short: Client polls every 5s (wasteful)</code><br><code>Long: Client waits; server responds when ready (efficient)</code><br><br><b>Use long polling when:</b><br>- You can't use WebSockets (e.g., corporate firewalls)<br>- Simple real-time needs without full duplex<br><br><b>Disadvantages:</b><br>- Server holds many open connections → resource pressure<br>- Latency spikes at timeout boundaries<br><br><b>Senior insight:</b> Long polling was common before WebSockets were mainstream. Today it's mostly a fallback.

What is the Saga pattern in distributed systems?	The <b>Saga pattern</b> manages distributed transactions by breaking them into a sequence of local transactions, each publishing events or messages to trigger the next step.<br><br>If a step fails, compensating transactions undo previous steps.<br><br><b>Example — Order Saga:</b><br><code>1. Create Order → success<br>2. Reserve Inventory → success<br>3. Charge Payment → FAIL<br>4. Compensate: Release Inventory<br>5. Compensate: Cancel Order</code><br><br><b>Two implementations:</b><br>- <b>Choreography:</b> Each service listens for events and reacts (decentralized)<br>- <b>Orchestration:</b> A central Saga orchestrator tells each service what to do<br><br><b>Trade-off:</b><br>- Choreography = simpler, but hard to trace flow<br>- Orchestration = clearer flow, but central point of complexity<br><br><b>Senior insight:</b> Sagas replace 2-Phase Commit (2PC) in microservices. 2PC doesn't scale across services; Sagas trade consistency for availability.

What is API versioning and what strategies exist?	<b>API versioning</b> lets you evolve an API without breaking existing consumers.<br><br><b>Strategy 1 — URL versioning (most common):</b><br><code>GET /v1/users<br>GET /v2/users</code><br><br><b>Strategy 2 — Header versioning:</b><br><code>Accept: application/vnd.myapi.v2+json</code><br><br><b>Strategy 3 — Query parameter:</b><br><code>GET /users?version=2</code><br><br><b>Strategy 4 — No versioning (Continuous evolution):</b><br>Used by GraphQL — add fields, never remove<br><br><b>Trade-offs:</b><br>- URL versioning: visible, easy to test, but proliferates endpoints<br>- Header versioning: clean URLs, but harder to test in a browser<br>- Query param: flexible, but mixes concerns<br><br><b>Best practice:</b> Use URL versioning for public APIs. Deprecate old versions with a sunset date communicated via <code>Sunset</code> and <code>Deprecation</code> headers.

What is API rate limiting and why is it important?	<b>Rate limiting</b> controls how many requests a client can make in a given time window to protect your API from abuse and ensure fair usage.<br><br><b>Common algorithms:</b><br>- <b>Fixed window:</b> 100 req/min, counter resets every minute<br>- <b>Sliding window:</b> 100 req in any rolling 60s window<br>- <b>Token bucket:</b> Tokens added at a fixed rate; burst allowed until bucket is empty<br>- <b>Leaky bucket:</b> Requests processed at a fixed rate regardless of burst<br><br><b>Response headers to include:</b><br><code>X-RateLimit-Limit: 100<br>X-RateLimit-Remaining: 42<br>X-RateLimit-Reset: 1711500000<br>Retry-After: 30  ← on 429</code><br><br><b>HTTP status:</b> <code>429 Too Many Requests</code><br><br><b>Senior insight:</b> Rate limit per API key/user, not just by IP. IPs can be shared (NAT). Implement rate limiting at the gateway level, not inside each microservice.

What is idempotency in APIs and why does it matter?	An operation is <b>idempotent</b> if performing it multiple times produces the same result as doing it once.<br><br><b>Why it matters:</b> Networks fail. Clients retry. Without idempotency, a retry could create duplicate orders, double charges, etc.<br><br><b>HTTP idempotency by spec:</b><br>- Idempotent: <code>GET, PUT, DELETE, HEAD</code><br>- NOT idempotent: <code>POST</code><br><br><b>Making POST idempotent with Idempotency-Key:</b><br><code>POST /payments<br>Idempotency-Key: a8f3-4e2b-9c01<br><br>{ "amount": 100 }</code><br><br>Server stores the key and returns the same response for duplicate requests.<br><br><b>Implementation:</b> Store <code>(idempotency_key, response)</code> in a cache (Redis) with a TTL (e.g., 24h).<br><br><b>Senior insight:</b> Stripe pioneered this pattern. It's essential for payment APIs and any operation with real-world consequences.

What is pagination and what strategies exist for large datasets?	<b>Pagination</b> breaks large result sets into smaller chunks to avoid overwhelming clients and servers.<br><br><b>Strategy 1 — Offset/Limit (most common):</b><br><code>GET /users?limit=20&offset=40</code><br>- Simple but has "page drift" (new records shift pages)<br>- Slow at large offsets (DB scans all rows to offset N)<br><br><b>Strategy 2 — Cursor-based (recommended for production):</b><br><code>GET /users?limit=20&cursor=eyJpZCI6NDB9</code><br>- Cursor encodes last seen item (usually opaque, base64 encoded)<br>- No page drift; efficient even at large scale<br><br><b>Strategy 3 — Keyset pagination:</b><br><code>GET /users?after_id=40&limit=20</code><br>- Similar to cursor; uses a sortable field (ID, timestamp)<br><br><b>Response envelope:</b><br><code>{ "data": [...], "next_cursor": "abc", "has_more": true }</code><br><br><b>Senior insight:</b> Never use <code>offset</code> on tables with millions of rows. At offset 1M, the DB must scan 1M rows before returning your 20.

What is HATEOAS and should you implement it?	<b>HATEOAS (Hypermedia As The Engine Of Application State)</b> means API responses include links to related actions, so the client can navigate the API without hardcoding URLs.<br><br><b>Example:</b><br><code>{<br>  "id": 1,<br>  "status": "pending",<br>  "_links": {<br>    "self": "/orders/1",<br>    "cancel": "/orders/1/cancel",<br>    "pay": "/orders/1/pay"<br>  }<br>}</code><br><br><b>Benefits:</b><br>- Client doesn't need to know API structure upfront<br>- Server controls available actions per state<br><br><b>Reality check:</b><br>Almost no production APIs implement full HATEOAS. Most teams use consistent URL conventions instead.<br><br><b>Senior insight:</b> HATEOAS is theoretically pure REST, but practically overkill for most internal APIs. Implement it for public platform APIs where discoverability matters (like GitHub's API).

What are the best practices for API error responses?	Good error responses help clients understand what went wrong and what to do next.<br><br><b>Use standard HTTP status codes:</b><br><code>200 OK, 201 Created<br>400 Bad Request (client error)<br>401 Unauthorized (not authenticated)<br>403 Forbidden (authenticated, no permission)<br>404 Not Found<br>409 Conflict (e.g., duplicate)<br>422 Unprocessable Entity (validation fail)<br>429 Too Many Requests<br>500 Internal Server Error</code><br><br><b>Error body format (RFC 7807 Problem Details):</b><br><code>{<br>  "type": "https://api.example.com/errors/validation",<br>  "title": "Validation Failed",<br>  "status": 422,<br>  "detail": "Email is required",<br>  "instance": "/users",<br>  "errors": [{ "field": "email", "message": "required" }]<br>}</code><br><br><b>Common mistakes:</b><br>- Returning 200 with an error in the body<br>- Returning 500 for client mistakes<br>- Exposing stack traces in production responses

What is OpenAPI / Swagger and why should every API have it?	<b>OpenAPI</b> (formerly Swagger) is a standard specification for describing REST APIs in a machine-readable format (YAML/JSON).<br><br><b>What you get:</b><br>- Auto-generated interactive documentation (Swagger UI)<br>- Client SDK generation in any language<br>- Server stub generation<br>- Automated contract testing<br><br><b>Minimal example:</b><br><code>paths:<br>  /users/{id}:<br>    get:<br>      summary: Get user by ID<br>      parameters:<br>        - name: id<br>          in: path<br>          required: true<br>          schema: { type: integer }<br>      responses:<br>        '200':<br>          description: User found</code><br><br><b>Two approaches:</b><br>- <b>Design-first:</b> Write the spec before code (recommended for teams)<br>- <b>Code-first:</b> Generate spec from annotations<br><br><b>Senior insight:</b> Design-first forces you to think about the API contract before implementation, catching design issues early when they're cheap to fix.

What is API authentication and what are the main methods?	<b>Authentication</b> proves who you are. <b>Authorization</b> controls what you can do. Both are needed.<br><br><b>Method 1 — API Key:</b><br><code>Authorization: ApiKey abc123</code><br>Simple, stateless. Good for server-to-server. No user context.<br><br><b>Method 2 — Basic Auth:</b><br><code>Authorization: Basic base64(user:pass)</code><br>Only use over HTTPS. Sends credentials every request.<br><br><b>Method 3 — Bearer Token (JWT):</b><br><code>Authorization: Bearer eyJhbGc...</code><br>Token is self-contained — server validates signature without DB lookup. Stateless.<br><br><b>Method 4 — OAuth 2.0:</b><br>Delegated authorization. User grants your app permission to act on their behalf (e.g., "Sign in with Google").<br><br><b>Method 5 — mTLS (mutual TLS):</b><br>Client and server both present certificates. Used in zero-trust networks.<br><br><b>Senior insight:</b> API keys are simple but hard to rotate safely. JWTs can't be invalidated before expiry (use short TTLs + refresh tokens). OAuth 2.0 is complex but the right choice for user-facing delegated access.

What is JWT (JSON Web Token) and how does it work?	A <b>JWT</b> is a compact, self-contained token with three base64-encoded parts separated by dots:<br><br><code>header.payload.signature</code><br><br><b>Header:</b> algorithm used (e.g., HS256, RS256)<br><b>Payload:</b> claims (user ID, roles, expiry)<br><b>Signature:</b> cryptographic proof of integrity<br><br><b>Example payload:</b><br><code>{<br>  "sub": "user123",<br>  "role": "admin",<br>  "exp": 1711500000<br>}</code><br><br><b>Verification:</b> Server checks signature using secret (symmetric) or public key (asymmetric). No DB lookup needed.<br><br><b>Common mistakes:</b><br>- Storing sensitive data in payload (it's base64, not encrypted — anyone can decode it)<br>- Long expiry times — use short TTL (15 min) + refresh tokens<br>- Using <code>alg: none</code> (always validate the algorithm)<br><br><b>Senior insight:</b> Use RS256 (asymmetric) in microservices — the auth service signs with a private key; other services verify with the public key, never needing access to the secret.

What is OAuth 2.0 and when do you use it?	<b>OAuth 2.0</b> is a framework for delegated authorization — it lets users grant a third-party app access to their data without sharing their password.<br><br><b>Example:</b> "Login with Google" — you authorize an app to read your Gmail, without giving it your Google password.<br><br><b>Key roles:</b><br>- <b>Resource Owner:</b> The user<br>- <b>Client:</b> The app requesting access<br>- <b>Authorization Server:</b> Issues tokens (e.g., Google)<br>- <b>Resource Server:</b> API being accessed<br><br><b>Common flows:</b><br>- <b>Authorization Code + PKCE:</b> For web/mobile apps (most secure)<br>- <b>Client Credentials:</b> For machine-to-machine (no user)<br>- <b>Implicit:</b> Deprecated — never use<br><br><b>Tokens issued:</b><br>- <b>Access Token:</b> Short-lived (15 min), used to call APIs<br>- <b>Refresh Token:</b> Long-lived, used to get new access tokens<br><br><b>Senior insight:</b> Always use PKCE for public clients (mobile/SPA). Never store tokens in localStorage — use httpOnly cookies or secure storage.

What is CORS and how does it work?	<b>CORS (Cross-Origin Resource Sharing)</b> is a browser security mechanism that restricts web pages from making requests to a different domain than the one that served the page.<br><br><b>Example problem:</b><br><code>Frontend: https://myapp.com<br>API: https://api.myapp.com ← different origin → blocked by browser</code><br><br><b>How it works:</b><br>For unsafe requests, the browser sends a <b>preflight OPTIONS request</b> first:<br><code>OPTIONS /api/users<br>Origin: https://myapp.com<br>Access-Control-Request-Method: POST</code><br><br>Server responds with allowed origins/methods:<br><code>Access-Control-Allow-Origin: https://myapp.com<br>Access-Control-Allow-Methods: GET, POST<br>Access-Control-Allow-Headers: Content-Type, Authorization</code><br><br><b>Common mistake:</b> Setting <code>Access-Control-Allow-Origin: *</code> with credentials — browsers reject this. You must specify an exact origin when using cookies/auth.<br><br><b>Senior insight:</b> CORS is a browser enforcement only. Server-to-server calls are never affected. Disable CORS restrictions in server code only — never tell developers to "disable CORS in the browser."

What is content negotiation in REST APIs?	<b>Content negotiation</b> lets clients and servers agree on the format of the data being exchanged using HTTP headers.<br><br><b>Client tells server what it accepts:</b><br><code>Accept: application/json<br>Accept: application/xml<br>Accept: text/csv</code><br><br><b>Client tells server what format it's sending:</b><br><code>Content-Type: application/json</code><br><br><b>Server responds with what it's sending back:</b><br><code>Content-Type: application/json; charset=utf-8</code><br><br><b>If server can't fulfill the Accept header:</b><br>Return <code>406 Not Acceptable</code><br><br><b>Practical use:</b><br>- API returns JSON by default, CSV for exports<br>- Mobile apps might prefer a lighter format<br><br><b>Senior insight:</b> Most APIs only support JSON, but building in content negotiation from the start allows you to support CSV exports, PDF generation, or XML integrations without creating separate endpoints.

What is API caching and what headers control it?	<b>Caching</b> stores responses temporarily to avoid repeated expensive computations or DB queries.<br><br><b>HTTP caching headers:</b><br><code>Cache-Control: max-age=300          ← cache for 5 minutes<br>Cache-Control: no-store               ← never cache (sensitive data)<br>Cache-Control: private                ← only browser, not CDN<br>Cache-Control: public                 ← CDN can cache<br>ETag: "abc123"                         ← version fingerprint<br>Last-Modified: Thu, 01 Jan 2026 ...  ← last change time</code><br><br><b>Conditional requests (avoid sending full body):</b><br><code>GET /users/1<br>If-None-Match: "abc123"<br>→ 304 Not Modified (body empty, reuse cached)</code><br><br><b>Cache layers:</b><br>1. Browser cache<br>2. CDN (CloudFront, Cloudflare)<br>3. Reverse proxy (Nginx, Varnish)<br>4. Application cache (Redis)<br><br><b>Common mistake:</b> Caching user-specific responses at CDN level — always set <code>Cache-Control: private</code> for authenticated endpoints.

What is an API Gateway and what does it do?	An <b>API Gateway</b> is a single entry point in front of your backend services that handles cross-cutting concerns.<br><br>Think of it as the front desk of a building — visitors don't walk directly to departments; they check in first.<br><br><b>What API Gateways handle:</b><br>- <b>Routing:</b> Forward requests to the right microservice<br>- <b>Authentication:</b> Validate tokens before reaching services<br>- <b>Rate limiting:</b> Protect services from overload<br>- <b>Load balancing:</b> Distribute traffic<br>- <b>SSL termination:</b> Handle HTTPS once<br>- <b>Request/response transformation</b><br>- <b>Logging and monitoring</b><br><br><b>Examples:</b> AWS API Gateway, Kong, NGINX, Traefik, Apigee<br><br><b>Senior insight:</b> Don't put business logic in the API Gateway — it becomes a bottleneck and a deployment risk. Keep it as infrastructure. Business logic belongs in the services.

What is the Backend for Frontend (BFF) pattern?	<b>BFF (Backend for Frontend)</b> is a pattern where you create a dedicated backend for each frontend type (mobile, web, TV) instead of one generic API.<br><br><b>Problem it solves:</b><br>A generic API returns the same data to everyone. Mobile needs less data (small screens, slow networks). Web might need more. Without BFF, you either over-fetch or under-fetch.<br><br><b>Structure:</b><br><code>Mobile App → Mobile BFF → Microservices<br>Web App   → Web BFF   → Microservices<br>Partner   → Public API → Microservices</code><br><br><b>Each BFF:</b><br>- Aggregates calls from multiple services<br>- Shapes the response for its specific client<br>- Handles auth and session for that client type<br><br><b>Advantages:</b><br>- Optimized payloads per client<br>- Client teams can own their BFF<br>- Change mobile behavior without affecting web<br><br><b>Disadvantages:</b><br>- Code duplication if BFFs share too much logic<br>- More services to maintain

What is the strangler fig pattern for API migration?	The <b>Strangler Fig</b> pattern gradually replaces a legacy system by routing traffic through a facade that incrementally redirects requests to the new system.<br><br>Named after a fig tree that grows around an existing tree and eventually replaces it.<br><br><b>How it works:</b><br><code>All traffic → Facade (proxy)<br>  ↓ (route by path/feature)<br>Legacy System ← old features<br>New System    ← migrated features</code><br><br>As more features migrate, more traffic shifts. When done, the legacy system is gone.<br><br><b>API migration use case:</b><br>- Running v1 (legacy) and v2 (new) behind the same URL<br>- Migrating from monolith to microservices feature-by-feature<br><br><b>Advantages:</b><br>- Zero big-bang rewrites (which almost always fail)<br>- Rollback is easy — just re-route to legacy<br>- Incremental, testable steps<br><br><b>Senior insight:</b> This is the safest way to modernize. Never attempt a full rewrite of a production system in one shot.

What is contract testing and why is it better than end-to-end tests for APIs?	<b>Contract testing</b> verifies that a consumer and provider agree on an API contract — independently of each other.<br><br><b>The problem with E2E tests:</b><br>- Slow, brittle, require all services to be running<br>- Don't pinpoint which service broke the contract<br><br><b>How contract testing works (Pact framework):</b><br>1. <b>Consumer</b> writes a test that defines what it expects<br>2. Pact generates a <b>contract file (pact)</b><br>3. <b>Provider</b> runs its tests against that contract<br>4. If provider changes break the contract → test fails<br><br><b>Example consumer expectation:</b><br><code>GET /users/1 → { id: 1, name: "Alice" }</code><br><br>If the provider renames <code>name</code> to <code>fullName</code>, the contract test catches it before production.<br><br><b>Senior insight:</b> Contract tests shift the feedback left — you find integration bugs locally, not during deployment. Essential for teams with many independent services.

What are the key HTTP status code categories every developer must know?	HTTP status codes are grouped into 5 categories:<br><br><b>1xx — Informational</b><br><code>100 Continue</code> — keep sending the request<br><br><b>2xx — Success</b><br><code>200 OK, 201 Created, 204 No Content</code><br><br><b>3xx — Redirection</b><br><code>301 Moved Permanently, 302 Found, 304 Not Modified</code><br><br><b>4xx — Client Error (you did something wrong)</b><br><code>400 Bad Request — malformed syntax<br>401 Unauthorized — not authenticated<br>403 Forbidden — authenticated but no permission<br>404 Not Found<br>409 Conflict — state conflict (e.g. duplicate)<br>422 Unprocessable Entity — validation failed<br>429 Too Many Requests — rate limited</code><br><br><b>5xx — Server Error (we did something wrong)</b><br><code>500 Internal Server Error<br>502 Bad Gateway — upstream service failed<br>503 Service Unavailable — overloaded/down<br>504 Gateway Timeout — upstream timed out</code><br><br><b>Common mistake:</b> Returning 200 with <code>{ "error": "not found" }</code> in the body. Use the correct status code.

What is exponential backoff with jitter and when should you use it?	When an API call fails, don't retry immediately — that creates thundering herd problems (all clients retry at the same moment, amplifying the outage).<br><br><b>Exponential backoff:</b> Double the wait time on each retry<br><code>Retry 1: wait 1s<br>Retry 2: wait 2s<br>Retry 3: wait 4s<br>Retry 4: wait 8s</code><br><br><b>Add jitter (random offset):</b><br><code>wait = min(cap, base * 2^attempt) + random(0, 1000ms)</code><br><br>Jitter spreads retries across time, reducing server spike.<br><br><b>When to retry:</b><br>- <code>429 Too Many Requests</code> (use <code>Retry-After</code> header)<br>- <code>503 Service Unavailable</code><br>- Network timeouts<br><br><b>When NOT to retry:</b><br>- <code>400, 401, 403, 404, 422</code> — these won't succeed on retry<br><br><b>Senior insight:</b> Only retry idempotent operations. Retrying a POST that creates a resource could cause duplicates. Use idempotency keys.

What is the circuit breaker pattern in API communication?	A <b>circuit breaker</b> monitors calls to a remote service and "opens" (stops calls) when the failure rate exceeds a threshold — protecting both caller and callee.<br><br><b>States:</b><br>- <b>Closed:</b> Normal operation, requests pass through<br>- <b>Open:</b> Requests fail immediately (no network call), return fallback<br>- <b>Half-Open:</b> Let a test request through; if it succeeds, close the circuit<br><br><b>Analogy:</b> Like an electrical circuit breaker — instead of letting a fault burn down the whole system, it trips and isolates the problem.<br><br><b>Benefits:</b><br>- Prevents cascading failures (one slow service taking down everything)<br>- Fails fast instead of waiting for timeout<br>- Allows downstream service time to recover<br><br><b>Libraries:</b> Polly (C#), Resilience4j (Java), Hystrix (deprecated), opossum (Node.js)<br><br><b>Senior insight:</b> Always combine circuit breakers with timeouts and retries. A circuit breaker without a timeout can still block threads waiting for a response that never comes.

What are the best practices for API endpoint naming?	Good API naming makes your API intuitive without needing documentation.<br><br><b>Use nouns, not verbs (for REST):</b><br><code>✅ GET /users<br>❌ GET /getUsers<br>✅ POST /orders<br>❌ POST /createOrder</code><br><br><b>Use plural nouns for collections:</b><br><code>✅ /users, /products, /orders<br>❌ /user, /product</code><br><br><b>Nest related resources (but max 2 levels):</b><br><code>✅ /users/1/orders<br>❌ /users/1/orders/5/items/3/details</code><br><br><b>Use kebab-case for multi-word paths:</b><br><code>✅ /order-items<br>❌ /orderItems, /order_items</code><br><br><b>Query params for filtering, sorting, pagination:</b><br><code>GET /orders?status=shipped&sort=created_at&limit=20</code><br><br><b>Actions that don't fit CRUD — use a verb sub-resource:</b><br><code>POST /orders/1/cancel<br>POST /users/1/resend-verification</code>

What is API backward compatibility and how do you maintain it?	<b>Backward compatibility</b> means old clients keep working after you update the API.<br><br><b>Safe changes (non-breaking):</b><br>- Adding new optional fields to response<br>- Adding new optional request parameters<br>- Adding new endpoints<br>- Adding new enum values (but clients must handle unknowns!)<br><br><b>Breaking changes:</b><br>- Removing or renaming fields<br>- Changing field types (string → int)<br>- Changing HTTP method or status codes<br>- Making optional fields required<br><br><b>Strategies to maintain compatibility:</b><br>- Use API versioning for breaking changes<br>- Deprecate fields with warnings before removal<br>- Use the <b>Expand/Contract</b> pattern: add new field, migrate consumers, remove old field<br>- Use <code>Sunset</code> header to announce deprecation date<br><br><b>Senior insight:</b> Consumer-driven contract tests automatically catch breaking changes before deployment. Without them, you'll break clients silently.

What is the difference between synchronous and asynchronous APIs?	<b>Synchronous API:</b><br>Client sends request and waits until the server finishes and responds. The client is blocked.<br><br><code>POST /reports → waits 30s → 200 OK { data: ... }</code><br><br><b>Asynchronous API:</b><br>Client sends request, server acknowledges immediately, processes in the background, and notifies when done.<br><br><code>POST /reports → 202 Accepted { jobId: "abc" }<br>GET /reports/abc → { status: "processing" }<br>GET /reports/abc → { status: "done", url: "..." }</code><br><br><b>When to use async:</b><br>- Long-running operations (report generation, video encoding)<br>- Operations that depend on third parties<br>- Batch processing<br><br><b>202 Accepted</b> is the correct status for async tasks — it means "I got your request and I'm working on it."<br><br><b>Senior insight:</b> Don't hold HTTP connections open for more than ~30s. For long tasks, always go async with a polling or webhook callback.

What is GraphQL N+1 problem and how do you solve it?	The <b>N+1 problem</b> occurs when a GraphQL query fetches a list of N items, then makes N additional queries to fetch related data for each item.<br><br><b>Example:</b><br><code>query {<br>  users {          ← 1 query for 100 users<br>    orders {       ← 100 queries, one per user<br>      total<br>    }<br>  }<br>}</code><br><br>Result: 1 + 100 = 101 DB queries for one request.<br><br><b>Solution: DataLoader (batch + cache)</b><br>DataLoader collects all requested IDs, then fetches them in one batch query:<br><code>// Instead of 100 individual queries:<br>SELECT * FROM orders WHERE user_id IN (1,2,...,100)</code><br><br><b>How it works:</b><br>1. Collects IDs during the tick<br>2. Batches them into one query<br>3. Caches results within the request<br><br><b>Available in:</b> JavaScript (DataLoader library), Python (Strawberry/graphene-django), Java (java-dataloader)<br><br><b>Senior insight:</b> N+1 is one of the most common GraphQL performance issues. Always monitor query counts when introducing GraphQL.

What is Protocol Buffers (protobuf) and why is it faster than JSON?	<b>Protocol Buffers (protobuf)</b> is Google's binary serialization format used by gRPC. Messages are defined in <code>.proto</code> files and compiled to code.<br><br><b>Proto definition:</b><br><code>message User {<br>  int32 id = 1;<br>  string name = 2;<br>  string email = 3;<br>}</code><br><br><b>Why faster than JSON:</b><br>- <b>Binary format</b> (not text) — 3-10x smaller payloads<br>- No field name repetition — uses field numbers instead<br>- Faster parse/serialize (no string parsing)<br><br><b>JSON equivalent payload:</b><br><code>{"id":1,"name":"Alice","email":"a@b.com"} ← 40 bytes</code><br><br><b>Protobuf equivalent:</b><br><code>[binary: ~10 bytes]</code><br><br><b>Disadvantages:</b><br>- Not human-readable (can't curl and inspect easily)<br>- Requires schema to decode<br>- Breaking schema changes cause data corruption without care<br><br><b>Senior insight:</b> For internal high-throughput APIs (millions of req/s), the bandwidth and latency savings of protobuf are significant. For external/public APIs, JSON's debuggability wins.

What are WebHook security best practices?	Webhooks receive POST requests from external services — if not secured, attackers can send fake events.<br><br><b>1. Validate HMAC signatures:</b><br><code># Provider sends:<br>X-Signature: sha256=abc123<br><br># You verify:<br>expected = HMAC-SHA256(secret, body)<br>if expected != received → reject</code><br><br><b>2. Return 200 immediately, process async:</b><br><code># Wrong: process synchronously (provider may timeout and retry)<br># Right:<br>return 200 OK<br>queue.push(event)  # process in background</code><br><br><b>3. Handle retries idempotently:</b><br>Providers retry on failure. Use the event ID to detect and skip duplicates.<br><br><b>4. Validate event type before processing:</b><br>Don't assume the event type — check it explicitly.<br><br><b>5. Use HTTPS only:</b><br>Never expose a webhook endpoint over plain HTTP.<br><br><b>6. Whitelist provider IPs if possible</b><br><br><b>Senior insight:</b> Most webhook security issues stem from not validating signatures or not being idempotent. Both are easy to implement and critical.

What is GraphQL schema design best practices?	<b>1. Design for the client, not the database:</b><br>GraphQL types should reflect domain concepts, not DB tables.<br><br><b>2. Use meaningful, domain-specific type names:</b><br><code>type Order { ... }  ✅<br>type OrderRecord { ... }  ❌  (leaks DB concept)</code><br><br><b>3. Make fields non-nullable only if truly guaranteed:</b><br><code>name: String!   ← always present<br>bio: String      ← can be null</code><br><br><b>4. Prefer specific over generic fields:</b><br><code>createdAt: DateTime  ✅<br>metadata: JSON       ❌ (loses type safety)</code><br><br><b>5. Use connections (Relay spec) for paginated lists:</b><br><code>orders(first: 10, after: "cursor") {<br>  edges { node { id total } cursor }<br>  pageInfo { hasNextPage }<br>}</code><br><br><b>6. Avoid breaking changes — add, deprecate, remove over time:</b><br><code>oldField: String @deprecated(reason: "Use newField")</code><br><br><b>Senior insight:</b> Schema design is public contract. Once clients depend on a type, removing or renaming it is a breaking change. Design carefully upfront.

What is API throttling vs rate limiting?	These terms are often confused — they're related but different mechanisms:<br><br><b>Rate Limiting:</b><br>Enforces a hard cap — once exceeded, requests are rejected with <code>429 Too Many Requests</code>.<br><code>You get 100 requests/minute. Request 101 → rejected.</code><br><br><b>Throttling:</b><br>Slows down requests instead of rejecting them — adds delay to keep throughput manageable.<br><code>You're going too fast → we'll respond, but slowly.</code><br><br>In practice, many use the terms interchangeably. What matters is the implementation choice:<br><br><b>Hard limit (rate limiting):</b> Better for protecting infrastructure, fair usage enforcement<br><b>Soft limit (throttling):</b> Better for graceful degradation, less disruptive to clients<br><br><b>Token bucket (common algorithm):</b><br>- Tokens refill at a fixed rate<br>- Each request costs one token<br>- Burst allowed until bucket empties<br>- Empty bucket → reject or throttle<br><br><b>Senior insight:</b> Differentiate limits by tier (free vs paid) and endpoint type (read vs write). A search endpoint should have lower limits than a simple GET.

What is API observability and what should you measure?	<b>Observability</b> means you can understand what your API is doing from the outside by looking at its outputs (logs, metrics, traces).<br><br><b>The three pillars:</b><br><br><b>1. Metrics (what's happening):</b><br>- Request rate (req/s)<br>- Error rate (% of 5xx)<br>- Latency (p50, p90, p99)<br>- Active connections<br><br><b>2. Logs (what happened):</b><br>- Structured logs (JSON): <code>{ "method": "GET", "path": "/users", "status": 200, "duration_ms": 42 }</code><br>- Include correlation/trace IDs<br><br><b>3. Distributed Traces (how it happened):</b><br>- Follow a request across services<br>- Identify which service caused latency<br>- Tools: Jaeger, Zipkin, AWS X-Ray, Datadog<br><br><b>Key header for tracing:</b><br><code>X-Request-Id: uuid  ← pass through every service</code><br><br><b>Senior insight:</b> Always track p99 latency — averages hide the pain of slow users. A 200ms average with a 10s p99 means 1% of users are suffering.

What is gRPC streaming and what are the four types?	<b>gRPC supports four communication patterns:</b><br><br><b>1. Unary RPC (standard request/response):</b><br><code>rpc GetUser (UserRequest) returns (UserResponse);</code><br><br><b>2. Server-side streaming:</b><br>Client sends one request, server streams multiple responses.<br><code>rpc ListLogs (LogRequest) returns (stream LogEntry);</code><br>Use case: tail logs, live feed<br><br><b>3. Client-side streaming:</b><br>Client streams multiple messages, server replies once.<br><code>rpc UploadChunks (stream Chunk) returns (UploadResult);</code><br>Use case: file upload, telemetry batch<br><br><b>4. Bidirectional streaming:</b><br>Both sides stream independently.<br><code>rpc Chat (stream Message) returns (stream Message);</code><br>Use case: real-time chat, game server, collaborative editing<br><br><b>Senior insight:</b> gRPC streaming uses HTTP/2 multiplexing — all streams share one TCP connection, which is much more efficient than multiple connections. This is a major advantage over REST for high-frequency communication.

What is the difference between REST and RPC-style APIs?	<b>REST (Resource-Oriented):</b><br>Organizes the API around <i>resources</i> (nouns) and uses HTTP verbs to express operations.<br><code>GET  /users/1<br>DELETE /orders/5</code><br><br><b>RPC (Remote Procedure Call):</b><br>Organizes the API around <i>actions</i> (verbs). You call functions on the server.<br><code>POST /getUserById     { id: 1 }<br>POST /cancelOrder     { orderId: 5 }</code><br><br><b>Neither is universally better:</b><br>- REST is great for CRUD-heavy domains with clear resources<br>- RPC is natural for command-heavy domains (actions, workflows)<br><br><b>JSON-RPC is a lightweight RPC standard:</b><br><code>{<br>  "jsonrpc": "2.0",<br>  "method": "getUserById",<br>  "params": { "id": 1 },<br>  "id": 1<br>}</code><br><br><b>gRPC is RPC over HTTP/2 with protobuf</b><br><br><b>Senior insight:</b> Many internal APIs that claim to be REST are actually RPC-style. That's fine — just be deliberate about it. The worst APIs are REST-shaped RPC that's inconsistently applied.

What are the security best practices for REST APIs?	<b>Authentication & Authorization:</b><br>- Use HTTPS everywhere — never HTTP in production<br>- Validate JWT signatures; never trust payload without verification<br>- Enforce authorization at the resource level, not just the endpoint<br><br><b>Input Validation:</b><br>- Validate all input (type, format, length) — never trust client data<br>- Prevent SQL injection — use parameterized queries<br>- Prevent NoSQL injection — sanitize filter objects<br><br><b>Output Safety:</b><br>- Never return more data than needed (avoid "data dumps")<br>- Don't expose internal errors or stack traces in production<br>- Use allowlists for returned fields, not blocklists<br><br><b>Transport Security:</b><br>- Set <code>Strict-Transport-Security</code> header (HSTS)<br>- Set proper <code>CORS</code> policies — don't use <code>*</code> for authenticated APIs<br><br><b>Rate Limiting & Monitoring:</b><br>- Rate limit all endpoints, especially auth endpoints<br>- Log and alert on authentication failures<br><br><b>Common vulnerabilities (OWASP API Top 10):</b><br>Broken object-level auth, broken function-level auth, excessive data exposure, mass assignment, injection.

What is mass assignment vulnerability in APIs?	<b>Mass assignment</b> occurs when an API blindly binds all properties from a client request to a data model, including fields the client shouldn't be able to set.<br><br><b>Vulnerable example (C# / .NET):</b><br><code>// Client sends: { "name": "Alice", "role": "admin" }<br>var user = JsonSerializer.Deserialize&lt;User&gt;(body);<br>db.Update(user);  // ← role just became "admin"!</code><br><br><b>Fix — use DTOs (Data Transfer Objects):</b><br><code>// Only accept what's allowed<br>var dto = JsonSerializer.Deserialize&lt;UpdateUserDto&gt;(body);<br>// UpdateUserDto only has: name, email<br>// role field doesn't exist on the DTO</code><br><br><b>Fix in Python (Pydantic):</b><br><code>class UpdateUserDto(BaseModel):<br>    name: str<br>    email: str<br>    # role is NOT here — can't be set</code><br><br><b>Senior insight:</b> Never use your database entity as your API request model. Always create explicit request DTOs that only include fields the client is allowed to set. This is OWASP API Security #3.

What is the API design principle of "fail fast"?	<b>Fail fast</b> means detecting and reporting errors as early as possible — at the input boundary, before doing expensive work.<br><br><b>Example — bad (fails late):</b><br><code>POST /orders { "quantity": -5 }<br>→ API starts processing<br>→ calls inventory service<br>→ calls payment service<br>→ fails with cryptic DB error</code><br><br><b>Example — good (fails fast):</b><br><code>POST /orders { "quantity": -5 }<br>→ input validation fails immediately<br>→ 422 Unprocessable Entity<br>→ { "errors": [{ "field": "quantity", "message": "must be > 0" }] }</code><br><br><b>Benefits:</b><br>- Saves server resources (no wasted downstream calls)<br>- Clear, actionable error messages for clients<br>- Easier to debug<br><br><b>Validate at the boundary:</b><br>- Required fields present<br>- Types correct<br>- Values in allowed ranges<br>- Business rules (e.g., can't order more than stock)<br><br><b>Senior insight:</b> Validate early, validate completely. Return all validation errors at once (not just the first one) — clients shouldn't have to fix one error to discover the next.

What is the command-query separation (CQS) principle in API design?	<b>CQS (Command-Query Separation)</b> states that a method (or API endpoint) should either:<br>- <b>Return data (Query)</b> — read-only, no side effects<br>- <b>Change state (Command)</b> — side effects, no return data<br><br>Never both at the same time.<br><br><b>Applied to REST:</b><br><code>GET /users/1          ← Query (safe, idempotent)<br>POST /users           ← Command (creates user, returns 201 + Location)<br>POST /orders/1/cancel ← Command (side effects; return 204 or result)</code><br><br><b>Why it matters:</b><br>- GET requests can be safely cached and retried<br>- Mixing reads and writes makes caching impossible<br>- Cleaner mental model for API consumers<br><br><b>Common violation:</b><br><code>GET /users/1/activate  ← GET that mutates — terrible idea<br>                         Browsers prefetch, crawlers hit it, caches replay it</code><br><br><b>Senior insight:</b> CQRS (Command Query Responsibility Segregation) is the architectural extension of CQS — separate read models from write models at the service level for high-scale systems.

What is API documentation and what makes it good?	Good documentation makes your API usable without reading source code.<br><br><b>Must-haves:</b><br>- <b>Authentication:</b> How to get credentials and use them<br>- <b>Endpoint reference:</b> All endpoints, params, request/response bodies<br>- <b>Error reference:</b> All error codes and what they mean<br>- <b>Rate limits:</b> Limits and how to handle 429<br>- <b>Changelog:</b> What changed and when<br><br><b>Nice-to-haves:</b><br>- Getting started tutorial (working example in 5 minutes)<br>- Code samples in multiple languages (curl, Python, JS, C#)<br>- Interactive API explorer (Swagger UI, Postman collection)<br>- Versioning and deprecation notices<br><br><b>Tools:</b><br>- <b>OpenAPI + Swagger UI / Redoc</b> — most common<br>- <b>Postman / Bruno</b> — shareable collections<br>- <b>Mintlify, Readme.io</b> — documentation platforms<br><br><b>Senior insight:</b> The best documentation has a working "Hello World" example that developers can run in under 5 minutes. Time-to-first-successful-call is the most important metric for developer experience.

What is the difference between API design first vs code first?	<b>Design First (API-first):</b><br>Write the OpenAPI spec before writing any code. The spec is the source of truth.<br><br><b>Workflow:</b><br><code>1. Write openapi.yaml<br>2. Mock server from spec (Prism, Stoplight)<br>3. Frontend integrates against mock<br>4. Backend implements spec<br>5. Contract tests verify implementation matches spec</code><br><br><b>Benefits:</b><br>- Teams can work in parallel (frontend + backend)<br>- Forces design thinking before implementation<br>- Catches design mistakes when they're cheap<br><br><b>Code First:</b><br>Write code with annotations; generate the spec from code.<br><br><code>[HttpGet("/users/{id}")]<br>[ProducesResponseType&lt;User&gt;(200)]<br>public User GetUser(int id) { ... }</code><br><br><b>Benefits:</b><br>- Faster for small teams or prototypes<br>- Spec always in sync with code<br><br><b>Senior insight:</b> For APIs consumed by external teams or public users, design-first is almost always better. For internal APIs where you control all consumers, code-first is pragmatic.

What is eventual consistency and how does it affect API design?	<b>Eventual consistency</b> means that after a write, different services or DB replicas may temporarily return different values — but they will converge to the same state <i>eventually</i>.<br><br><b>Example:</b><br><code>POST /orders → 201 Created (written to primary DB)<br>GET /orders  → returns empty list (replica hasn't synced yet)</code><br><br>This is disorienting for clients expecting immediate consistency.<br><br><b>API design strategies for eventual consistency:</b><br><br><b>1. Return the created resource in the POST response</b> — don't make the client re-fetch<br><code>POST /orders → 201 { "id": 5, "status": "pending", ... }</code><br><br><b>2. Use Location header to point to the new resource</b><br><code>Location: /orders/5</code><br><br><b>3. Document the consistency model</b> — tell clients to expect a brief delay<br><br><b>4. Use read-your-own-writes consistency</b> — route the creating user's reads to the primary for a brief window<br><br><b>Senior insight:</b> Many API bugs that look like race conditions are actually eventual consistency issues at the infrastructure level. Understanding your consistency model is essential for debugging distributed systems.

What is service mesh and how does it relate to API communication?	A <b>service mesh</b> is an infrastructure layer that handles service-to-service communication in microservices, offloading concerns from application code.<br><br>Think of it as a network of smart "sidecars" alongside each service that intercept all traffic.<br><br><b>What a service mesh handles:</b><br>- <b>mTLS:</b> Automatic encryption between services<br>- <b>Load balancing:</b> Smart routing with circuit breaking<br>- <b>Retries and timeouts:</b> Configured centrally<br>- <b>Observability:</b> Metrics, traces, logs without code changes<br>- <b>Traffic splitting:</b> Canary deployments (send 5% to v2)<br><br><b>Popular service meshes:</b><br>- <b>Istio</b> — feature-rich, complex<br>- <b>Linkerd</b> — lightweight, simpler<br>- <b>AWS App Mesh</b> — AWS-native<br><br><b>Sidecar pattern:</b><br>A proxy (Envoy) runs alongside each service pod, intercepting all traffic. App code doesn't change.<br><br><b>Senior insight:</b> Service meshes add significant operational complexity. Only introduce one when you have enough microservices (10+) and operational maturity. For smaller systems, handle retries and mTLS in code.

What is API design for idempotency in different languages?	Implementing idempotency requires storing the key and response. Here's how across languages:<br><br><b>C# / .NET:</b><br><code>// Middleware checks idempotency key<br>string key = request.Headers["Idempotency-Key"];<br>if (cache.TryGet(key, out var cached))<br>    return cached;  // return stored response<br>var result = await ProcessRequest();<br>cache.Set(key, result, TimeSpan.FromHours(24));<br>return result;</code><br><br><b>Python / FastAPI:</b><br><code>@app.post("/payments")<br>async def create_payment(idempotency_key: str = Header(...)):<br>    cached = await redis.get(idempotency_key)<br>    if cached:<br>        return json.loads(cached)<br>    result = await process_payment()<br>    await redis.setex(idempotency_key, 86400, result.json())<br>    return result</code><br><br><b>Node.js / Express:</b><br><code>const key = req.headers['idempotency-key'];<br>const cached = await redis.get(key);<br>if (cached) return res.json(JSON.parse(cached));</code><br><br><b>Key: Use Redis with TTL</b> — in-memory caches won't survive restarts and don't work across multiple instances.

What is the difference between polling, long polling, SSE, and WebSockets for real-time APIs?	Four patterns for getting real-time updates from a server, in order of complexity:<br><br><b>1. Short Polling (simplest, worst):</b><br><code>setInterval(() => fetch('/status'), 5000);</code><br>- Constant requests, mostly wasted<br>- High server load, stale data between polls<br><br><b>2. Long Polling (better):</b><br>- Server holds request until data is ready<br>- Less wasteful, but complex connection management<br><br><b>3. SSE — Server-Sent Events (good for one-way):</b><br><code>const es = new EventSource('/stream');<br>es.onmessage = e => update(e.data);</code><br>- Auto-reconnect, works over HTTP, simple<br>- Client → server not supported<br><br><b>4. WebSockets (full duplex):</b><br><code>const ws = new WebSocket('wss://api.example.com');<br>ws.onmessage = e => update(e.data);<br>ws.send(JSON.stringify(msg));</code><br>- Both directions, real-time, stateful<br>- Harder to scale, no auto-reconnect<br><br><b>Decision guide:</b><br>- Read-only updates → SSE<br>- Bidirectional → WebSocket<br>- Can't use either → Long Polling<br>- Prototype → Short Polling

What are common API anti-patterns to avoid?	<b>1. Chatty APIs:</b><br>Requiring 10 requests to load a single page. Solution: composite endpoints or GraphQL.<br><br><b>2. God endpoints:</b><br><code>POST /doEverything</code><br>Endpoints that do too much — hard to version, cache, or secure.<br><br><b>3. Leaky abstractions:</b><br>Exposing DB column names, table structures, or internal IDs in the API. Implementation details become public contract.<br><br><b>4. Boolean parameters that control behavior:</b><br><code>GET /users?includeDeleted=true&includeAdmins=true&verbose=true</code><br>Creates hidden complexity. Use separate endpoints or query params with clear semantics.<br><br><b>5. Ignoring HTTP semantics:</b><br>- Using POST for reads (breaks caching)<br>- Returning 200 for errors<br>- Using GET with a body<br><br><b>6. No pagination:</b><br><code>GET /users → returns 500,000 records</code><br><br><b>7. No versioning strategy:</b><br>Breaking changes with no warning to consumers.<br><br><b>8. Anemic error responses:</b><br><code>{ "error": "bad request" }</code> — tells the client nothing actionable.<br><br><b>Senior insight:</b> Most anti-patterns come from treating the API as an afterthought. Treat it as a product with real consumers who can't easily update.

What is GraphQL federation and why does it matter for microservices?	<b>GraphQL Federation</b> (Apollo Federation) allows multiple GraphQL services to each own part of the schema, combined into a single unified graph by a gateway.<br><br><b>Without federation:</b><br>One giant GraphQL schema → one service owns it all → doesn't scale across teams<br><br><b>With federation:</b><br><code>User Service schema:<br>  type User @key(fields: "id") { id name email }<br><br>Order Service schema:<br>  extend type User @key(fields: "id") {<br>    id: ID! @external<br>    orders: [Order]   ← User gets orders from this service<br>  }</code><br><br>Gateway merges all subgraphs transparently. Client queries the unified schema.<br><br><b>Benefits:</b><br>- Teams own their own subgraph<br>- Schema evolves per team<br>- Single graph for clients<br><br><b>Challenges:</b><br>- Complex gateway setup<br>- Schema composition can conflict<br>- Distributed tracing harder<br><br><b>Senior insight:</b> Federation is the right architecture when multiple teams own different parts of the data model. Without it, either one team becomes a bottleneck, or schema quality degrades as everyone touches the same file.
