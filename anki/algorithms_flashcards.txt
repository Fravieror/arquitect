1→What is Floyd's Cycle Detection Algorithm?	Floyd's algorithm (also called the "tortoise and hare") detects cycles in a linked list using two pointers that move at different speeds. The slow pointer moves 1 step at a time; the fast pointer moves 2 steps. If there's a cycle, they WILL eventually meet inside it. If there's no cycle, the fast pointer reaches null. Analogy: two runners on a circular track — the faster one laps the slower one eventually.
2→Why does Floyd's algorithm work? What's the mathematical guarantee?	If a cycle exists, the fast pointer gains 1 step on the slow pointer each iteration. Eventually the gap between them becomes a multiple of the cycle length, so they meet. The distance to the meeting point has a direct mathematical relationship to where the cycle starts. Senior insight: the algorithm is deterministic — it always detects a cycle in O(n) time if one exists.
3→What are the two phases of Floyd's algorithm?	Phase 1 — Detect: move slow 1 step, fast 2 steps until they meet (proves a cycle exists). Phase 2 — Find entry point: reset one pointer to the head, keep the other at the meeting point, then move BOTH one step at a time. Where they meet again is the START of the cycle. This is the part most developers forget.
4→What is the time and space complexity of Floyd's Cycle Detection?	Time: O(n) — both pointers traverse the list at most twice. Space: O(1) — only two pointer variables, no extra data structures. Compare to the naive approach: using a HashSet to track visited nodes is O(n) time but also O(n) space. Floyd's is strictly better on memory.
5→What is a common mistake when implementing Floyd's algorithm?	Not checking for null before advancing the fast pointer. Always verify fast != null AND fast.next != null before moving fast two steps — otherwise you'll get a NullPointerException/NullReferenceException. Also: forgetting Phase 2 and returning only "cycle exists" instead of the actual cycle start node.
6→Where is Floyd's algorithm used in real-world systems?	1) Detecting infinite loops in process scheduling. 2) Detecting circular references in garbage collectors (conceptually). 3) Interview staple for linked list problems. 4) Database query planners can use similar ideas to detect cyclic dependencies in join graphs. Senior insight: the two-pointer technique generalizes far beyond cycles — it appears in finding the middle of a list, palindrome checks, and more.
7→How does Floyd's algorithm differ from using a visited set?	HashSet approach: simple, intuitive, O(n) time AND O(n) space. Floyd's: more complex to understand but O(1) space. Trade-off: if memory is tight (e.g., embedded systems, huge linked structures), Floyd's wins. In most application code, the HashSet version is more readable and acceptable. Prefer clarity unless space is a real constraint.
8→Implement Floyd's cycle detection in pseudocode (detection phase only).	slow = head; fast = head; while fast != null AND fast.next != null: slow = slow.next; fast = fast.next.next; if slow == fast: return "cycle detected"; return "no cycle". Key: pointer equality (same node reference), NOT value equality. In Java/C# this is reference comparison (==). In Python, use 'is' not '=='.
9→What is the "Missing Number — XOR Trick"?	Given an array of n-1 distinct numbers from 1 to n, find the missing one using XOR. XOR every number from 1 to n with every element in the array. Because x XOR x = 0 (anything XOR itself cancels), all present numbers cancel out, leaving only the missing number. Analogy: imagine pairing up socks — unpaired socks are "missing partners," and XOR is how you find the loner.
10→Why does XOR work for finding a missing number? Explain the math.	XOR properties: x XOR x = 0, x XOR 0 = x, XOR is commutative and associative. So: XOR(1..n) XOR XOR(array) = XOR of all numbers that appear an odd number of times. Every number from 1 to n appears exactly once in the full range and once in the array — except the missing one, which appears only once total. Result: the missing number.
11→What is the time and space complexity of the XOR missing number trick?	Time: O(n) — one pass to XOR the full range, one pass through the array. Space: O(1) — just one integer variable accumulates the XOR result. Compare: sorting is O(n log n). Sum formula (n*(n+1)/2 - array sum) is also O(n) time, O(1) space and arguably simpler. XOR shines when overflow is a concern.
12→When should you prefer the XOR trick over the sum formula for missing number?	Sum formula: simple, readable, but can overflow with very large n (sum of 1..n exceeds int32 range at ~65k). XOR trick: never overflows — it only flips bits, never accumulates large values. In languages with fixed-width integers (C, C++, Java, C#) and large n, XOR is safer. In Python (arbitrary precision ints), sum formula overflow is not an issue.
13→What are common XOR trick misconceptions?	1) "It only works for 1 to n" — false, you can adapt it for any known set. 2) "Order matters" — false, XOR is commutative. 3) Confusing bitwise XOR (^) with logical XOR. In C#/Java/JS: use ^. In Python: also ^. Common bug: using bitwise OR (|) instead of XOR (^). Another bug: initializing result to 1 instead of 0 — always start with 0.
14→How does the XOR trick extend to finding two missing numbers?	XOR of all missing = m1 XOR m2. Find the rightmost set bit (it differs between m1 and m2). Split all numbers into two groups based on that bit and XOR each group separately — this isolates m1 and m2. Senior insight: this extension is an interview signal — it shows you understand WHY XOR works, not just that it works.
15→Give a real-world use case for XOR beyond missing numbers.	1) Swap two integers without a temp variable: a ^= b; b ^= a; a ^= b. 2) Parity checks in network protocols and RAID storage. 3) Cryptographic one-time pads (XOR with a random key). 4) Finding the single non-duplicate in an array where all others appear twice (LeetCode 136). Senior insight: XOR is a fundamental building block in error detection/correction algorithms (Hamming codes, CRC).
16→What is Kadane's Algorithm?	Kadane's finds the maximum sum contiguous subarray in O(n) time. It tracks two values: current_max (the best subarray ending HERE) and global_max (the best seen so far). At each element: current_max = max(element, current_max + element). This asks: "is it better to start fresh here, or extend the previous subarray?" Then: global_max = max(global_max, current_max).
17→What is the intuition behind Kadane's Algorithm?	At each position, decide: "Does adding the previous subarray help me, or hurt me?" If current_max + element < element, the previous subarray is dragging you down — abandon it and start fresh. Analogy: if a business partnership is losing you money, you're better off going solo. Kadane's greedily makes this decision at every step.
18→What is the time and space complexity of Kadane's Algorithm?	Time: O(n) — single pass through the array. Space: O(1) — only two variables (current_max and global_max). This beats the brute-force O(n²) approach (check all subarrays) and the divide-and-conquer O(n log n) approach. For this specific problem, Kadane's is optimal — you cannot do better than O(n).
19→What is a critical edge case in Kadane's Algorithm?	All-negative arrays. If you initialize global_max = 0, you return 0 (empty subarray), which may be wrong if the problem requires a non-empty subarray. Fix: initialize global_max = array[0] and start loop from index 1. Also handle empty arrays (length 0) as a precondition check. This is the #1 mistake in interviews.
20→How do you modify Kadane's to return the actual subarray, not just the sum?	Track start, end, and temp_start indices. When current_max resets to element (fresh start), set temp_start = i. When global_max updates, set start = temp_start and end = i. The subarray is array[start..end]. This adds O(1) space overhead and is critical in production systems where you need to report WHICH records caused an anomaly.
21→What are real-world applications of Kadane's Algorithm?	1) Stock market: maximum profit window over a price-change array. 2) Signal processing: finding the strongest signal segment. 3) Image processing: maximum sum rectangle (2D extension). 4) Genomics: finding regions of highest gene expression. Senior insight: the 2D extension (maximum sum submatrix) uses Kadane's as a subroutine and runs in O(n³) — a common senior-level follow-up question.
22→How does Kadane's Algorithm behave differently in various languages?	Python: clean with built-in max(), no overflow issues. JavaScript: watch for -Infinity as initial value for all-negative case (Number.NEGATIVE_INFINITY). C#/Java: use int.MinValue / Integer.MIN_VALUE as initial global_max to handle all-negatives correctly — avoid initializing to 0. Common mistake in JS: comparing with === instead of Math.max().
23→What is the 2D extension of Kadane's Algorithm?	Fix left and right column boundaries (O(n²) pairs). For each pair, compress the 2D submatrix into a 1D array by summing each row between those columns. Run Kadane's on that 1D array. Track the maximum across all column pairs. Total complexity: O(n³). This is how you find the maximum sum rectangle in a 2D matrix.
24→What does it mean for a binary tree to be symmetric (mirrored)?	A tree is symmetric if its left subtree is a mirror image of its right subtree. Specifically: the left child of the left subtree should equal the right child of the right subtree at every level, and vice versa. Analogy: hold the tree up to a mirror — it should look identical. Single-node trees and null trees are always symmetric.
25→How do you check if a binary tree is symmetric recursively?	Define a helper isMirror(left, right): if both null → true. If one null and one not → false. If values differ → false. Return isMirror(left.left, right.right) AND isMirror(left.right, right.left). Call isMirror(root.left, root.right). This recursively checks the "cross" pattern — outer pair and inner pair at each level.
26→How do you check tree symmetry iteratively (without recursion)?	Use a queue. Enqueue root.left and root.right together as pairs. Each iteration: dequeue a pair. If both null → continue. If one null or values differ → return false. Enqueue: (left.left, right.right) and (left.right, right.left). Return true if queue empties without failure. Senior insight: iterative is safer for very deep trees — avoids stack overflow from deep recursion.
27→What is the time and space complexity of the Tree Symmetry check?	Time: O(n) — every node is visited exactly once. Space: O(n) worst case — the recursion stack (or queue) holds up to n/2 nodes at the widest level of the tree. For a balanced tree, space is O(log n). For a completely unbalanced tree (list-like), recursion depth is O(n) — risk of stack overflow in Python (default limit ~1000) or deep C# stacks.
28→What are common mistakes when implementing the tree mirror check?	1) Comparing node references instead of values (== vs .val ==). 2) Forgetting the base cases for null nodes — causes NullPointerException. 3) Checking left.left vs right.left (same-side) instead of left.left vs right.right (cross-side). 4) Assuming a "balanced" tree is always symmetric — wrong. Balance and symmetry are independent properties.
29→How does tree symmetry relate to tree serialization?	A serialized in-order traversal of a symmetric tree is a palindrome. This is sometimes used as an alternative check: serialize in-order (including nulls), then check if the result is a palindrome. Senior insight: this approach is less efficient (O(n) space for the string) but reveals the deep connection between tree structure and string patterns — useful in compression and pattern matching.
30→What is BFS (Breadth-First Search) and how does it find shortest paths?	BFS explores nodes layer by layer — all nodes at distance 1, then distance 2, then distance 3, etc. In an unweighted graph, this guarantees the FIRST time you reach a node, you've found the shortest path to it. Analogy: ripples in a pond — the wave front expands outward, hitting the nearest shores first.
31→Why does BFS guarantee the shortest path in unweighted graphs?	BFS processes nodes in order of increasing distance from the source. The first time a node is dequeued, it's via the shortest route — any later path would be at least as long (since BFS processes shorter paths first). This guarantee ONLY holds for unweighted graphs. For weighted graphs, use Dijkstra's algorithm instead.
32→Walk through the BFS shortest path algorithm step by step.	1) Create a queue and enqueue the start node. 2) Create a visited set and mark start as visited. 3) While queue is not empty: dequeue a node. If it's the target, return distance. Enqueue all unvisited neighbors, mark them visited, record distance as current + 1. 4) If queue empties without finding target, return -1 (unreachable). Distance tracking: use a separate dict or store (node, distance) pairs in the queue.
33→What is the time and space complexity of BFS?	Time: O(V + E) — V vertices and E edges, each visited once. Space: O(V) — the queue and visited set can hold all vertices in the worst case (e.g., a star graph where every node connects to center). Senior insight: for dense graphs (E ≈ V²), the edge traversal dominates; for sparse graphs (E ≈ V), vertex traversal dominates.
34→What is the difference between BFS and DFS for shortest paths?	BFS = shortest path (unweighted). DFS = NOT guaranteed shortest path — DFS can find A path but it may be longer. DFS is better for: detecting cycles, topological sort, finding connected components, maze generation. BFS is better for: shortest path, level-order traversal, finding nodes within k hops. Senior insight: in interviews, defaulting to DFS when BFS is needed is a common mistake.
35→How do you reconstruct the actual shortest path (not just the length) using BFS?	Maintain a parent/predecessor dictionary: when you first visit a node, record which node you came from. After reaching the target, backtrack through the parent map from target to source, then reverse the path. This adds O(V) space but is essential when you need to report the route, not just the distance.
36→What are common BFS pitfalls and bugs?	1) Not marking nodes as visited WHEN ENQUEUED (mark when dequeuing is too late — causes re-processing and infinite loops). 2) Using a list as a queue (Python list.pop(0) is O(n) — use collections.deque). 3) Not handling disconnected graphs — BFS from one component won't reach others. 4) Applying BFS to weighted graphs expecting shortest path — wrong, use Dijkstra's.
37→How does BFS behave across languages — Python, JavaScript, C#?	Python: use collections.deque for O(1) enqueue/dequeue. JS: no built-in queue — arrays with shift() are O(n); use a custom queue or index pointer trick. C#: use Queue<T> from System.Collections.Generic (O(1) operations). Java: use LinkedList or ArrayDeque as Queue. Senior insight: using the wrong data structure (array as queue) causes BFS to silently become O(n²).
38→What is multi-source BFS and when do you use it?	Instead of starting from one node, enqueue ALL source nodes at distance 0 simultaneously. BFS then finds the shortest distance from ANY source to each node. Use case: "find the nearest exit in a maze with multiple exits," "find the nearest 1 in a binary matrix," "walls closing in from all sides at once." This is a powerful technique that many developers don't know exists.
39→How does BFS apply to real-world system design?	1) Social networks: "degrees of separation" (LinkedIn's 2nd/3rd connections). 2) Routing protocols: OSPF uses BFS-like flooding to build network topology. 3) Web crawlers: BFS ensures pages are crawled by proximity to the seed URL. 4) GPS navigation: BFS on road graphs (then optimized to Dijkstra's for weighted roads). 5) Dependency resolution: package managers find minimal install sets.
40→What is the relationship between BFS and Dijkstra's algorithm?	BFS is Dijkstra's algorithm where all edge weights = 1. Dijkstra's generalizes BFS to weighted graphs by replacing the simple queue with a min-heap (priority queue). If you ever catch yourself doing BFS on a weighted graph and wondering why results are wrong — switch to Dijkstra's. Senior insight: 0-1 BFS (weights are only 0 or 1) uses a deque and sits between plain BFS and Dijkstra's in complexity.
41→How do Floyd's, XOR, Kadane's, Mirror Check, and BFS relate to each other conceptually?	All five are examples of elegant linear-time algorithms that replace brute force with a clever invariant. Floyd's: two-speed pointer invariant. XOR: self-cancellation invariant. Kadane's: local vs global optimum invariant. Mirror: recursive symmetry invariant. BFS: distance-ordering invariant. Senior insight: mastering these patterns trains your mind to look for invariants — a skill that separates senior engineers from juniors.
42→What common interview mistakes span all five of these algorithms?	1) Ignoring edge cases: empty input, all-same values, single-element arrays, null trees. 2) Off-by-one errors in loop boundaries. 3) Wrong data structure choice (list vs deque for BFS, wrong initialization for Kadane's). 4) Confusing reference equality with value equality (tree node comparison). 5) Forgetting that Floyd's Phase 2 exists. 6) Applying unweighted-graph algorithms (BFS) to weighted problems.
43→How do you recognize which algorithm to apply to a new problem?	Linked list + cycle → Floyd's. Array with known range, find missing/duplicate → XOR or sum trick. Contiguous subarray maximum → Kadane's. Binary tree structural comparison → recursive mirror/symmetry. Shortest path in unweighted graph → BFS. Senior insight: pattern recognition comes from deliberate practice — after solving 3-4 variations of each, you start seeing the signature instantly.
