What is Floyd's Cycle Detection Algorithm?	<div style="text-align:left">Floyd's algorithm (also called the "tortoise and hare") detects cycles in a linked list using two pointers that move at different speeds. The <code>slow</code> pointer moves 1 step at a time; the <code>fast</code> pointer moves 2 steps. If there's a cycle, they WILL eventually meet inside it. If there's no cycle, the <code>fast</code> pointer reaches <code>null</code>.<br><br>Analogy: two runners on a circular track — the faster one laps the slower one eventually.</div>
Why does Floyd's algorithm work? What's the mathematical guarantee?	<div style="text-align:left">If a cycle exists, the <code>fast</code> pointer gains 1 step on the <code>slow</code> pointer each iteration. Eventually the gap between them becomes a multiple of the cycle length, so they meet. The distance to the meeting point has a direct mathematical relationship to where the cycle starts.<br><br>Senior insight: the algorithm is deterministic — it always detects a cycle in O(n) time if one exists.</div>
What are the two phases of Floyd's algorithm?	<div style="text-align:left">Phase 1 — Detect: move <code>slow</code> 1 step, <code>fast</code> 2 steps until they meet (proves a cycle exists).<br><br>Phase 2 — Find entry point: reset one pointer to the head, keep the other at the meeting point, then move BOTH one step at a time. Where they meet again is the START of the cycle.<br><br>This is the part most developers forget.</div>
What is the time and space complexity of Floyd's Cycle Detection?	<div style="text-align:left">Time: O(n) — both pointers traverse the list at most twice.<br>Space: O(1) — only two pointer variables, no extra data structures.<br><br>Compare to the naive approach: using a <code>HashSet</code> to track visited nodes is O(n) time but also O(n) space. Floyd's is strictly better on memory.</div>
What is a common mistake when implementing Floyd's algorithm?	<div style="text-align:left">Not checking for <code>null</code> before advancing the fast pointer. Always verify <code>fast != null</code> AND <code>fast.next != null</code> before moving fast two steps — otherwise you'll get a <code>NullPointerException</code>/<code>NullReferenceException</code>.<br><br>Also: forgetting Phase 2 and returning only "cycle exists" instead of the actual cycle start node.</div>
Where is Floyd's algorithm used in real-world systems?	<div style="text-align:left">1) Detecting infinite loops in process scheduling.<br>2) Detecting circular references in garbage collectors (conceptually).<br>3) Interview staple for linked list problems.<br>4) Database query planners can use similar ideas to detect cyclic dependencies in join graphs.<br><br>Senior insight: the two-pointer technique generalizes far beyond cycles — it appears in finding the middle of a list, palindrome checks, and more.</div>
How does Floyd's algorithm differ from using a visited set?	<div style="text-align:left"><code>HashSet</code> approach: simple, intuitive, O(n) time AND O(n) space.<br>Floyd's: more complex to understand but O(1) space.<br><br>Trade-off: if memory is tight (e.g., embedded systems, huge linked structures), Floyd's wins. In most application code, the HashSet version is more readable and acceptable. Prefer clarity unless space is a real constraint.</div>
Implement Floyd's cycle detection in pseudocode (detection phase only).	<div style="text-align:left"><code>slow = head<br>fast = head<br>while fast != null AND fast.next != null:<br>&nbsp;&nbsp;&nbsp;&nbsp;slow = slow.next<br>&nbsp;&nbsp;&nbsp;&nbsp;fast = fast.next.next<br>&nbsp;&nbsp;&nbsp;&nbsp;if slow == fast: return "cycle detected"<br>return "no cycle"</code><br><br>Key: pointer equality (same node reference), NOT value equality. In Java/C# this is reference comparison (<code>==</code>). In Python, use <code>is</code> not <code>==</code>.</div>
What is the "Missing Number — XOR Trick"?	<div style="text-align:left">Given an array of n-1 distinct numbers from 1 to n, find the missing one using XOR. XOR every number from 1 to n with every element in the array. Because <code>x XOR x = 0</code> (anything XOR itself cancels), all present numbers cancel out, leaving only the missing number.<br><br>Analogy: imagine pairing up socks — unpaired socks are "missing partners," and XOR is how you find the loner.</div>
Why does XOR work for finding a missing number? Explain the math.	<div style="text-align:left">XOR properties: <code>x XOR x = 0</code>, <code>x XOR 0 = x</code>, XOR is commutative and associative.<br><br>So: <code>XOR(1..n) XOR XOR(array)</code> = XOR of all numbers that appear an odd number of times. Every number from 1 to n appears exactly once in the full range and once in the array — except the missing one, which appears only once total.<br><br>Result: the missing number.</div>
What is the time and space complexity of the XOR missing number trick?	<div style="text-align:left">Time: O(n) — one pass to XOR the full range, one pass through the array.<br>Space: O(1) — just one integer variable accumulates the XOR result.<br><br>Compare: sorting is O(n log n). Sum formula (<code>n*(n+1)/2 - array sum</code>) is also O(n) time, O(1) space and arguably simpler. XOR shines when overflow is a concern.</div>
When should you prefer the XOR trick over the sum formula for missing number?	<div style="text-align:left">Sum formula: simple, readable, but can overflow with very large n (sum of 1..n exceeds int32 range at ~65k).<br>XOR trick: never overflows — it only flips bits, never accumulates large values.<br><br>In languages with fixed-width integers (C, C++, Java, C#) and large n, XOR is safer. In Python (arbitrary precision ints), sum formula overflow is not an issue.</div>
What are common XOR trick misconceptions?	<div style="text-align:left">1) "It only works for 1 to n" — false, you can adapt it for any known set.<br>2) "Order matters" — false, XOR is commutative.<br>3) Confusing bitwise XOR (<code>^</code>) with logical XOR. In C#/Java/JS: use <code>^</code>. In Python: also <code>^</code>.<br><br>Common bug: using bitwise OR (<code>|</code>) instead of XOR (<code>^</code>).<br>Another bug: initializing result to <code>1</code> instead of <code>0</code> — always start with <code>0</code>.</div>
How does the XOR trick extend to finding two missing numbers?	<div style="text-align:left">XOR of all missing = <code>m1 XOR m2</code>. Find the rightmost set bit (it differs between <code>m1</code> and <code>m2</code>). Split all numbers into two groups based on that bit and XOR each group separately — this isolates <code>m1</code> and <code>m2</code>.<br><br>Senior insight: this extension is an interview signal — it shows you understand WHY XOR works, not just that it works.</div>
Give a real-world use case for XOR beyond missing numbers.	<div style="text-align:left">1) Swap two integers without a temp variable: <code>a ^= b; b ^= a; a ^= b</code>.<br>2) Parity checks in network protocols and RAID storage.<br>3) Cryptographic one-time pads (XOR with a random key).<br>4) Finding the single non-duplicate in an array where all others appear twice (LeetCode 136).<br><br>Senior insight: XOR is a fundamental building block in error detection/correction algorithms (Hamming codes, CRC).</div>
What is Kadane's Algorithm?	<div style="text-align:left">Kadane's finds the maximum sum contiguous subarray in O(n) time. It tracks two values: <code>current_max</code> (the best subarray ending HERE) and <code>global_max</code> (the best seen so far).<br><br>At each element: <code>current_max = max(element, current_max + element)</code><br>This asks: "is it better to start fresh here, or extend the previous subarray?"<br><br>Then: <code>global_max = max(global_max, current_max)</code></div>
What is the intuition behind Kadane's Algorithm?	<div style="text-align:left">At each position, decide: "Does adding the previous subarray help me, or hurt me?" If <code>current_max + element &lt; element</code>, the previous subarray is dragging you down — abandon it and start fresh.<br><br>Analogy: if a business partnership is losing you money, you're better off going solo. Kadane's greedily makes this decision at every step.</div>
What is the time and space complexity of Kadane's Algorithm?	<div style="text-align:left">Time: O(n) — single pass through the array.<br>Space: O(1) — only two variables (<code>current_max</code> and <code>global_max</code>).<br><br>This beats the brute-force O(n²) approach (check all subarrays) and the divide-and-conquer O(n log n) approach. For this specific problem, Kadane's is optimal — you cannot do better than O(n).</div>
What is a critical edge case in Kadane's Algorithm?	<div style="text-align:left">All-negative arrays. If you initialize <code>global_max = 0</code>, you return 0 (empty subarray), which may be wrong if the problem requires a non-empty subarray.<br><br>Fix: initialize <code>global_max = array[0]</code> and start loop from index 1. Also handle empty arrays (length 0) as a precondition check.<br><br>This is the #1 mistake in interviews.</div>
How do you modify Kadane's to return the actual subarray, not just the sum?	<div style="text-align:left">Track <code>start</code>, <code>end</code>, and <code>temp_start</code> indices.<br>When <code>current_max</code> resets to element (fresh start), set <code>temp_start = i</code>.<br>When <code>global_max</code> updates, set <code>start = temp_start</code> and <code>end = i</code>.<br><br>The subarray is <code>array[start..end]</code>. This adds O(1) space overhead and is critical in production systems where you need to report WHICH records caused an anomaly.</div>
What are real-world applications of Kadane's Algorithm?	<div style="text-align:left">1) Stock market: maximum profit window over a price-change array.<br>2) Signal processing: finding the strongest signal segment.<br>3) Image processing: maximum sum rectangle (2D extension).<br>4) Genomics: finding regions of highest gene expression.<br><br>Senior insight: the 2D extension (maximum sum submatrix) uses Kadane's as a subroutine and runs in O(n³) — a common senior-level follow-up question.</div>
How does Kadane's Algorithm behave differently in various languages?	<div style="text-align:left">Python: clean with built-in <code>max()</code>, no overflow issues.<br>JavaScript: watch for <code>-Infinity</code> as initial value for all-negative case (<code>Number.NEGATIVE_INFINITY</code>).<br>C#/Java: use <code>int.MinValue</code> / <code>Integer.MIN_VALUE</code> as initial <code>global_max</code> to handle all-negatives correctly — avoid initializing to <code>0</code>.<br><br>Common mistake in JS: comparing with <code>===</code> instead of <code>Math.max()</code>.</div>
What is the 2D extension of Kadane's Algorithm?	<div style="text-align:left">Fix left and right column boundaries (O(n²) pairs).<br>For each pair, compress the 2D submatrix into a 1D array by summing each row between those columns.<br>Run Kadane's on that 1D array.<br>Track the maximum across all column pairs.<br><br>Total complexity: O(n³). This is how you find the maximum sum rectangle in a 2D matrix.</div>
What does it mean for a binary tree to be symmetric (mirrored)?	<div style="text-align:left">A tree is symmetric if its left subtree is a mirror image of its right subtree. Specifically: the left child of the left subtree should equal the right child of the right subtree at every level, and vice versa.<br><br>Analogy: hold the tree up to a mirror — it should look identical. Single-node trees and <code>null</code> trees are always symmetric.</div>
How do you check if a binary tree is symmetric recursively?	<div style="text-align:left">Define a helper <code>isMirror(left, right)</code>:<br>— if both <code>null</code> → <code>true</code><br>— if one <code>null</code> and one not → <code>false</code><br>— if values differ → <code>false</code><br>— return <code>isMirror(left.left, right.right)</code> AND <code>isMirror(left.right, right.left)</code><br><br>Call <code>isMirror(root.left, root.right)</code>. This recursively checks the "cross" pattern — outer pair and inner pair at each level.</div>
How do you check tree symmetry iteratively (without recursion)?	<div style="text-align:left">Use a queue. Enqueue <code>root.left</code> and <code>root.right</code> together as pairs.<br><br>Each iteration: dequeue a pair.<br>— If both <code>null</code> → continue<br>— If one <code>null</code> or values differ → return <code>false</code><br>— Enqueue: <code>(left.left, right.right)</code> and <code>(left.right, right.left)</code><br><br>Return <code>true</code> if queue empties without failure.<br><br>Senior insight: iterative is safer for very deep trees — avoids stack overflow from deep recursion.</div>
What is the time and space complexity of the Tree Symmetry check?	<div style="text-align:left">Time: O(n) — every node is visited exactly once.<br>Space: O(n) worst case — the recursion stack (or queue) holds up to n/2 nodes at the widest level of the tree.<br><br>For a balanced tree, space is O(log n). For a completely unbalanced tree (list-like), recursion depth is O(n) — risk of stack overflow in Python (default limit ~1000) or deep C# stacks.</div>
What are common mistakes when implementing the tree mirror check?	<div style="text-align:left">1) Comparing node references instead of values (<code>==</code> vs <code>.val ==</code>).<br>2) Forgetting the base cases for <code>null</code> nodes — causes <code>NullPointerException</code>.<br>3) Checking <code>left.left</code> vs <code>right.left</code> (same-side) instead of <code>left.left</code> vs <code>right.right</code> (cross-side).<br>4) Assuming a "balanced" tree is always symmetric — wrong. Balance and symmetry are independent properties.</div>
How does tree symmetry relate to tree serialization?	<div style="text-align:left">A serialized in-order traversal of a symmetric tree is a palindrome. This is sometimes used as an alternative check: serialize in-order (including <code>null</code>s), then check if the result is a palindrome.<br><br>Senior insight: this approach is less efficient (O(n) space for the string) but reveals the deep connection between tree structure and string patterns — useful in compression and pattern matching.</div>
What is BFS (Breadth-First Search) and how does it find shortest paths?	<div style="text-align:left">BFS explores nodes layer by layer — all nodes at distance 1, then distance 2, then distance 3, etc. In an unweighted graph, this guarantees the FIRST time you reach a node, you've found the shortest path to it.<br><br>Analogy: ripples in a pond — the wave front expands outward, hitting the nearest shores first.</div>
Why does BFS guarantee the shortest path in unweighted graphs?	<div style="text-align:left">BFS processes nodes in order of increasing distance from the source. The first time a node is dequeued, it's via the shortest route — any later path would be at least as long (since BFS processes shorter paths first).<br><br>This guarantee ONLY holds for unweighted graphs. For weighted graphs, use Dijkstra's algorithm instead.</div>
Walk through the BFS shortest path algorithm step by step.	<div style="text-align:left">1) Create a queue and enqueue the start node.<br>2) Create a visited set and mark start as visited.<br>3) While queue is not empty: dequeue a node. If it's the target, return distance. Enqueue all unvisited neighbors, mark them visited, record distance as current + 1.<br>4) If queue empties without finding target, return -1 (unreachable).<br><br>Distance tracking: use a separate dict or store <code>(node, distance)</code> pairs in the queue.</div>
What is the time and space complexity of BFS?	<div style="text-align:left">Time: O(V + E) — V vertices and E edges, each visited once.<br>Space: O(V) — the queue and visited set can hold all vertices in the worst case (e.g., a star graph where every node connects to center).<br><br>Senior insight: for dense graphs (E ≈ V²), the edge traversal dominates; for sparse graphs (E ≈ V), vertex traversal dominates.</div>
What is the difference between BFS and DFS for shortest paths?	<div style="text-align:left">BFS = shortest path (unweighted).<br>DFS = NOT guaranteed shortest path — DFS can find A path but it may be longer.<br><br>DFS is better for: detecting cycles, topological sort, finding connected components, maze generation.<br>BFS is better for: shortest path, level-order traversal, finding nodes within k hops.<br><br>Senior insight: in interviews, defaulting to DFS when BFS is needed is a common mistake.</div>
How do you reconstruct the actual shortest path (not just the length) using BFS?	<div style="text-align:left">Maintain a parent/predecessor dictionary: when you first visit a node, record which node you came from. After reaching the target, backtrack through the parent map from target to source, then reverse the path.<br><br>This adds O(V) space but is essential when you need to report the route, not just the distance.</div>
What are common BFS pitfalls and bugs?	<div style="text-align:left">1) Not marking nodes as visited WHEN ENQUEUED (mark when dequeuing is too late — causes re-processing and infinite loops).<br>2) Using a list as a queue (Python <code>list.pop(0)</code> is O(n) — use <code>collections.deque</code>).<br>3) Not handling disconnected graphs — BFS from one component won't reach others.<br>4) Applying BFS to weighted graphs expecting shortest path — wrong, use Dijkstra's.</div>
How does BFS behave across languages — Python, JavaScript, C#?	<div style="text-align:left">Python: use <code>collections.deque</code> for O(1) enqueue/dequeue.<br>JS: no built-in queue — arrays with <code>shift()</code> are O(n); use a custom queue or index pointer trick.<br>C#: use <code>Queue&lt;T&gt;</code> from <code>System.Collections.Generic</code> (O(1) operations).<br>Java: use <code>LinkedList</code> or <code>ArrayDeque</code> as <code>Queue</code>.<br><br>Senior insight: using the wrong data structure (array as queue) causes BFS to silently become O(n²).</div>
What is multi-source BFS and when do you use it?	<div style="text-align:left">Instead of starting from one node, enqueue ALL source nodes at distance 0 simultaneously. BFS then finds the shortest distance from ANY source to each node.<br><br>Use case: "find the nearest exit in a maze with multiple exits," "find the nearest 1 in a binary matrix," "walls closing in from all sides at once."<br><br>This is a powerful technique that many developers don't know exists.</div>
How does BFS apply to real-world system design?	<div style="text-align:left">1) Social networks: "degrees of separation" (LinkedIn's 2nd/3rd connections).<br>2) Routing protocols: OSPF uses BFS-like flooding to build network topology.<br>3) Web crawlers: BFS ensures pages are crawled by proximity to the seed URL.<br>4) GPS navigation: BFS on road graphs (then optimized to Dijkstra's for weighted roads).<br>5) Dependency resolution: package managers find minimal install sets.</div>
What is the relationship between BFS and Dijkstra's algorithm?	<div style="text-align:left">BFS is Dijkstra's algorithm where all edge weights = 1. Dijkstra's generalizes BFS to weighted graphs by replacing the simple queue with a min-heap (priority queue).<br><br>If you ever catch yourself doing BFS on a weighted graph and wondering why results are wrong — switch to Dijkstra's.<br><br>Senior insight: 0-1 BFS (weights are only 0 or 1) uses a deque and sits between plain BFS and Dijkstra's in complexity.</div>
How do Floyd's, XOR, Kadane's, Mirror Check, and BFS relate to each other conceptually?	<div style="text-align:left">All five are examples of elegant linear-time algorithms that replace brute force with a clever invariant.<br><br>Floyd's: two-speed pointer invariant.<br>XOR: self-cancellation invariant.<br>Kadane's: local vs global optimum invariant.<br>Mirror: recursive symmetry invariant.<br>BFS: distance-ordering invariant.<br><br>Senior insight: mastering these patterns trains your mind to look for invariants — a skill that separates senior engineers from juniors.</div>
What common interview mistakes span all five of these algorithms?	<div style="text-align:left">1) Ignoring edge cases: empty input, all-same values, single-element arrays, <code>null</code> trees.<br>2) Off-by-one errors in loop boundaries.<br>3) Wrong data structure choice (list vs deque for BFS, wrong initialization for Kadane's).<br>4) Confusing reference equality with value equality (tree node comparison).<br>5) Forgetting that Floyd's Phase 2 exists.<br>6) Applying unweighted-graph algorithms (BFS) to weighted problems.</div>
How do you recognize which algorithm to apply to a new problem?	<div style="text-align:left">Linked list + cycle → Floyd's.<br>Array with known range, find missing/duplicate → XOR or sum trick.<br>Contiguous subarray maximum → Kadane's.<br>Binary tree structural comparison → recursive mirror/symmetry.<br>Shortest path in unweighted graph → BFS.<br><br>Senior insight: pattern recognition comes from deliberate practice — after solving 3-4 variations of each, you start seeing the signature instantly.</div>
