#deck:Kubernetes

What is Kubernetes and why does it exist?	<div style="text-align:left">Kubernetes (K8s) is an open-source <b>container orchestration platform</b> — it manages the deployment, scaling, and operation of containerized applications across a cluster of machines.<br><br>Think of it as the <b>OS for your data center</b>: just like an OS manages processes on a single machine, Kubernetes manages containers across many machines.<br><br><b>Why it exists:</b> Running Docker on one machine is easy. Running 100 containers across 50 servers, auto-healing when one crashes, scaling under load, and rolling out updates without downtime — that's hard. Kubernetes automates all of that.<br><br><b>Senior insight:</b> K8s doesn't just schedule containers — it continuously reconciles <i>desired state</i> vs <i>actual state</i>. If something drifts, it self-corrects.</div>

What is the difference between a Container, a Pod, and a Node in Kubernetes?	<div style="text-align:left"><b>Container:</b> A single runnable unit (Docker image). Think of it as one process in a box.<br><br><b>Pod:</b> The smallest deployable unit in K8s. A Pod wraps one or more tightly coupled containers that share the same network namespace and storage. Think of a Pod as an apartment — the containers inside share the same address (IP) and hallways (volumes).<br><br><b>Node:</b> A physical or virtual machine that runs Pods. Think of a Node as a building — it houses many apartments (Pods).<br><br><b>Key rule:</b> You never deploy containers directly in K8s — you deploy Pods. Containers inside the same Pod communicate via <code>localhost</code>.<br><br><b>Common mistake:</b> Putting unrelated services in the same Pod just because they're related — they'll scale together and can't be placed independently.</div>

What is a Kubernetes Cluster?	<div style="text-align:left">A Kubernetes <b>cluster</b> is the full system: a set of Nodes managed together by Kubernetes.<br><br>It has two types of nodes:<br>- <b>Control Plane (Master):</b> The brain — runs the API server, scheduler, and controller manager<br>- <b>Worker Nodes:</b> The muscle — run your actual application Pods<br><br>Analogy: Think of a cluster as a company. The Control Plane is the management team making decisions; Worker Nodes are the employees doing the actual work.<br><br><b>Senior insight:</b> In production, the Control Plane should be highly available (3+ replicas). Cloud providers (EKS, GKE, AKS) manage the Control Plane for you — you only manage Worker Nodes.</div>

What components make up the Kubernetes Control Plane?	<div style="text-align:left">The Control Plane is the brain of the cluster. Key components:<br><br><b>kube-apiserver:</b> The front door — all communication goes through it (kubectl, controllers, nodes). It's the only component that talks to etcd.<br><br><b>etcd:</b> A distributed key-value store — the cluster's source of truth. All cluster state lives here. Treat it like a sacred database: back it up!<br><br><b>kube-scheduler:</b> Decides <i>which Node</i> a new Pod should run on, based on resource requests, node affinity, taints, etc.<br><br><b>kube-controller-manager:</b> Runs controllers that watch state and reconcile it (ReplicaSet controller, Node controller, Job controller, etc.).<br><br><b>cloud-controller-manager:</b> Interfaces with cloud provider APIs (AWS, GCP, Azure) to manage load balancers, volumes, and node lifecycle.<br><br><b>Senior insight:</b> All components are stateless except etcd. Losing etcd means losing the cluster state. Always run etcd with 3 or 5 members for quorum.</div>

What runs on each Kubernetes Worker Node?	<div style="text-align:left">Every Worker Node runs these core components:<br><br><b>kubelet:</b> The node agent — talks to the API server, ensures containers described in PodSpecs are running and healthy. Think of it as the node's foreman.<br><br><b>kube-proxy:</b> Manages network rules on the node — routes traffic to the right Pod. Implements the Service abstraction at the network level.<br><br><b>Container Runtime:</b> The software that actually runs containers — typically <b>containerd</b> (Docker was deprecated in K8s 1.24+). Others: CRI-O, Docker (via shim).<br><br><b>Senior insight:</b> kubelet uses the Container Runtime Interface (CRI) to communicate with the runtime — this is why Docker was swappable. Always check what runtime your managed K8s service uses. EKS and GKE both use containerd by default now.</div>

What is a Deployment in Kubernetes?	<div style="text-align:left">A <b>Deployment</b> is the standard way to run stateless applications in Kubernetes. It manages a ReplicaSet, which manages Pods.<br><br>You declare: "I want 3 replicas of this container image running at all times." The Deployment controller ensures that's always true — if one Pod dies, it creates a new one.<br><br>Example:<br><code>apiVersion: apps/v1<br>kind: Deployment<br>metadata:<br>  name: my-app<br>spec:<br>  replicas: 3<br>  selector:<br>    matchLabels:<br>      app: my-app<br>  template:<br>    metadata:<br>      labels:<br>        app: my-app<br>    spec:<br>      containers:<br>      - name: my-app<br>        image: my-app:1.0</code><br><br><b>Key features:</b> Rolling updates, rollback, scaling<br><b>Common mistake:</b> Using a Deployment for stateful apps (databases) — use StatefulSet instead.</div>

What is the difference between a ReplicaSet and a Deployment?	<div style="text-align:left">A <b>ReplicaSet</b> ensures a specified number of identical Pod replicas are running at all times — it's the "keep N copies alive" mechanism.<br><br>A <b>Deployment</b> wraps a ReplicaSet and adds:<br>- <b>Rolling updates</b> (create new ReplicaSet, scale it up, scale old one down)<br>- <b>Rollback</b> capability (kubectl rollout undo)<br>- <b>Update history</b><br><br>Analogy: ReplicaSet is a factory line that replaces broken parts. Deployment is the factory manager who can upgrade the factory line without stopping production.<br><br><b>Rule:</b> Never create ReplicaSets directly — always use Deployments. The only time you'll touch a ReplicaSet directly is when debugging.<br><br><b>Senior insight:</b> <code>kubectl rollout history deployment/my-app</code> shows past revisions. You can rollback to any of them.</div>

What is a StatefulSet and when should you use it?	<div style="text-align:left">A <b>StatefulSet</b> manages Pods that need stable, persistent identity — used for stateful applications like databases, message queues, and caches.<br><br>Unlike Deployments, StatefulSets give each Pod:<br>- A <b>stable, predictable hostname</b>: <code>pod-0</code>, <code>pod-1</code>, <code>pod-2</code><br>- <b>Persistent storage</b> that survives Pod restarts (via VolumeClaimTemplates)<br>- <b>Ordered, sequential startup and shutdown</b><br><br>Use cases: PostgreSQL, MySQL, Kafka, Zookeeper, Redis Cluster, Elasticsearch<br><br><b>Key difference:</b><br>Deployment Pod: <code>my-app-7d4f8b-xkz9p</code> (random ID, ephemeral)<br>StatefulSet Pod: <code>my-db-0</code>, <code>my-db-1</code> (stable, predictable)<br><br><b>Senior insight:</b> StatefulSets don't solve all statefulness problems — you still need to handle leader election, replication, and backup yourself (or use an Operator).</div>

What is a DaemonSet in Kubernetes?	<div style="text-align:left">A <b>DaemonSet</b> ensures that one instance of a Pod runs on <b>every Node</b> in the cluster (or a subset via node selectors). When a new Node joins, a Pod is automatically scheduled on it. When a Node is removed, its Pod is garbage collected.<br><br><b>Classic use cases:</b><br>- Log collectors (Fluentd, Filebeat)<br>- Monitoring agents (Datadog agent, Prometheus node-exporter)<br>- Network plugins (CNI plugins like Calico, Weave)<br>- Storage drivers<br><br>Analogy: Like a janitor in every floor of a building — one per floor, always present, handles floor-level duties.<br><br><b>Senior insight:</b> DaemonSets bypass the normal scheduler's bin-packing logic — they guarantee one-per-node regardless of resources. Be careful with resource requests; a misbehaving DaemonSet can starve all Pods on a node.</div>

What is a Job and a CronJob in Kubernetes?	<div style="text-align:left"><b>Job:</b> Runs one or more Pods to completion — used for batch tasks, data migrations, one-off scripts. Unlike a Deployment, a Job finishes and stops.<br><br>Example: Process 10,000 records in parallel using 5 worker Pods.<br><br><b>CronJob:</b> Creates Jobs on a schedule (like Linux cron). Uses standard cron syntax.<br><br><code>spec:<br>  schedule: "0 3 * * *"  # daily at 3am<br>  jobTemplate:<br>    spec:<br>      template:<br>        spec:<br>          containers:<br>          - name: backup<br>            image: backup-tool<br>          restartPolicy: OnFailure</code><br><br><b>Common mistakes:</b><br>- Not setting <code>activeDeadlineSeconds</code> — a stuck Job runs forever<br>- Not setting <code>concurrencyPolicy: Forbid</code> — overlapping CronJob runs pile up<br>- Not setting <code>successfulJobsHistoryLimit</code> — old completed Jobs accumulate</div>

What is a Kubernetes Service and why is it needed?	<div style="text-align:left">A <b>Service</b> provides a stable network endpoint to access a set of Pods. Pods are ephemeral — they die and get new IPs. A Service gives you a fixed IP and DNS name that always routes to healthy Pods.<br><br>Think of it like a load balancer's DNS name — the backend servers (Pods) change, but you always call the same address.<br><br><b>How it works:</b> Services use <b>label selectors</b> to find matching Pods and route traffic to them.<br><br><code>spec:<br>  selector:<br>    app: my-app   # routes to all Pods with this label<br>  ports:<br>  - port: 80<br>    targetPort: 8080</code><br><br><b>Senior insight:</b> Services create a stable DNS entry: <code>my-service.my-namespace.svc.cluster.local</code>. This is how microservices discover each other in K8s — use the Service name, never hardcode Pod IPs.</div>

What are the different types of Kubernetes Services?	<div style="text-align:left"><b>ClusterIP</b> (default): Internal only. Exposes the service on a cluster-internal IP — only reachable within the cluster. Use for microservice-to-microservice communication.<br><br><b>NodePort:</b> Exposes the service on a static port on every Node's IP (range 30000–32767). Reachable from outside the cluster via <code>NodeIP:NodePort</code>. Rarely used in production — use for dev/testing.<br><br><b>LoadBalancer:</b> Creates an external load balancer (cloud provider's LB: AWS ELB, GCP LB). The standard way to expose services to the internet in cloud environments.<br><br><b>ExternalName:</b> Maps a Service to an external DNS name — used to reference external services (like a managed DB) by a K8s-style name inside the cluster.<br><br><b>Senior insight:</b> In production, use Ingress + ClusterIP instead of many LoadBalancer services — one LB for multiple services is cheaper and more manageable. Each LoadBalancer Service provisions a cloud LB, which costs money.</div>

What is a Kubernetes Ingress?	<div style="text-align:left">An <b>Ingress</b> is an API object that manages <b>HTTP/HTTPS routing</b> from outside the cluster to Services inside the cluster. It provides host/path-based routing, TLS termination, and more — all through one entry point.<br><br>Without Ingress: you need one LoadBalancer per service (expensive, unmanageable).<br>With Ingress: one LoadBalancer, route by domain/path.<br><br>Example:<br><code>rules:<br>- host: api.example.com<br>  http:<br>    paths:<br>    - path: /users<br>      backend:<br>        service: users-svc<br>    - path: /orders<br>      backend:<br>        service: orders-svc</code><br><br><b>Ingress Controller:</b> Ingress is just a config — you need a controller to implement it (nginx-ingress, Traefik, AWS ALB Ingress, Kong). Install one per cluster.<br><br><b>Senior insight:</b> Ingress doesn't support non-HTTP protocols (gRPC needs special config). For raw TCP/UDP routing use a LoadBalancer Service or a Gateway API.</div>

What is a ConfigMap in Kubernetes?	<div style="text-align:left">A <b>ConfigMap</b> stores non-sensitive configuration data as key-value pairs — decoupling config from container images.<br><br>Instead of baking config into your image, you inject it at runtime:<br><code>apiVersion: v1<br>kind: ConfigMap<br>metadata:<br>  name: app-config<br>data:<br>  DATABASE_URL: "postgres://db:5432/mydb"<br>  LOG_LEVEL: "info"</code><br><br><b>How to use it:</b><br>1. As environment variables: <code>envFrom: - configMapRef: name: app-config</code><br>2. As mounted files: mount as a volume — great for config files (nginx.conf, app.yaml)<br><br><b>Common mistake:</b> Storing secrets in ConfigMaps — they're not encrypted and visible to anyone with namespace access. Use Secrets instead.<br><br><b>Senior insight:</b> ConfigMap changes don't automatically restart Pods. You need to either restart manually, use a config reloader sidecar, or use rolling updates.</div>

What is a Kubernetes Secret and how does it differ from a ConfigMap?	<div style="text-align:left">A <b>Secret</b> stores sensitive data (passwords, API keys, TLS certs) separately from your container images.<br><br>Differences from ConfigMap:<br>- Secret values are <b>base64-encoded</b> (not encrypted by default — just encoded!)<br>- Access can be restricted via RBAC<br>- Can be encrypted at rest if etcd encryption is enabled<br>- Marked as sensitive in audit logs<br><br><code>apiVersion: v1<br>kind: Secret<br>metadata:<br>  name: db-secret<br>type: Opaque<br>data:<br>  password: cGFzc3dvcmQxMjM=  # base64 of "password123"</code><br><br><b>Common mistake:</b> Thinking base64 = encryption. It's not — anyone with kubectl access can decode it. For real security, use:<br>- etcd encryption at rest<br>- External secrets managers: HashiCorp Vault, AWS Secrets Manager, Sealed Secrets<br><br><b>Senior insight:</b> Never commit Secret YAML to git. Use tools like Sealed Secrets or External Secrets Operator to sync secrets from a vault into K8s Secrets.</div>

What are Kubernetes Namespaces?	<div style="text-align:left"><b>Namespaces</b> provide a mechanism for isolating groups of resources within a single cluster — like virtual sub-clusters.<br><br>Think of namespaces as folders: you can have the same filename in different folders. Similarly, you can have a Service named <code>api</code> in both <code>development</code> and <code>production</code> namespaces.<br><br><b>Default namespaces:</b><br>- <code>default</code>: Where resources go if no namespace is specified<br>- <code>kube-system</code>: K8s internal components (DNS, kube-proxy)<br>- <code>kube-public</code>: Publicly readable (rarely used)<br>- <code>kube-node-lease</code>: Node heartbeat leases<br><br><b>What namespaces scope:</b> Pods, Services, Deployments, ConfigMaps, Secrets (most resources)<br><b>What they don't scope:</b> Nodes, PersistentVolumes, StorageClasses, ClusterRoles (cluster-scoped resources)<br><br><b>Senior insight:</b> Namespaces are not a security boundary — they're an organization tool. A compromised Pod can still reach other namespaces unless NetworkPolicies are applied.</div>

What are Labels and Selectors in Kubernetes?	<div style="text-align:left"><b>Labels</b> are key-value pairs attached to Kubernetes objects (Pods, Services, Nodes, etc.). They're used to organize and select objects.<br><br><code>metadata:<br>  labels:<br>    app: my-app<br>    env: production<br>    version: "1.2"</code><br><br><b>Selectors</b> are queries that find objects matching specific labels. This is how Services find their Pods, and how Deployments manage their ReplicaSets.<br><br>Two types:<br>- <b>Equality-based:</b> <code>app=my-app, env!=staging</code><br>- <b>Set-based:</b> <code>env in (production, staging)</code><br><br><b>Annotations</b> are similar but for non-identifying metadata (build info, tool config, URLs) — not used for selection.<br><br><b>Senior insight:</b> Labels are the glue of Kubernetes. A broken label selector is one of the most common reasons a Service has no Endpoints and traffic doesn't flow. Always verify with <code>kubectl get endpoints service-name</code>.</div>

What is a Kubernetes Namespace ResourceQuota and LimitRange?	<div style="text-align:left"><b>ResourceQuota:</b> Sets aggregate limits on resource consumption per namespace — prevents one team from consuming all cluster resources.<br><br><code>spec:<br>  hard:<br>    pods: "20"<br>    requests.cpu: "10"<br>    requests.memory: 20Gi<br>    limits.cpu: "20"<br>    limits.memory: 40Gi</code><br><br><b>LimitRange:</b> Sets default and max/min resource requests/limits per individual Pod or container — prevents Pods from running without resource declarations.<br><br><code>spec:<br>  limits:<br>  - type: Container<br>    default:<br>      cpu: 200m<br>      memory: 256Mi<br>    defaultRequest:<br>      cpu: 100m<br>      memory: 128Mi</code><br><br><b>Senior insight:</b> Without ResourceQuotas, a misconfigured Deployment requesting 100 replicas can starve other teams. In multi-tenant clusters, always enforce quotas per namespace. LimitRanges protect against OOM kills from unbounded containers.</div>

What are resource requests and limits in Kubernetes?	<div style="text-align:left"><b>Requests:</b> The minimum resources a container needs. The scheduler uses this to decide which Node to place the Pod on.<br><br><b>Limits:</b> The maximum resources a container can use. If exceeded, CPU is throttled; memory causes OOM kill.<br><br><code>resources:<br>  requests:<br>    memory: "128Mi"<br>    cpu: "250m"   # 0.25 CPU cores<br>  limits:<br>    memory: "256Mi"<br>    cpu: "500m"</code><br><br><b>CPU:</b> 1000m = 1 core. CPU is <i>compressible</i> — it throttles, doesn't kill.<br><b>Memory:</b> Memory is <i>incompressible</i> — exceeding limit = OOM kill (container restart).<br><br><b>QoS classes based on settings:</b><br>- <b>Guaranteed:</b> requests == limits (highest priority, never evicted first)<br>- <b>Burstable:</b> requests < limits<br>- <b>BestEffort:</b> no requests/limits set (evicted first under pressure)<br><br><b>Senior insight:</b> Always set requests. Without them, the scheduler places Pods blindly and nodes get overcommitted. Limits without requests = bad (limits default to requests if only limits are set, which wastes capacity).</div>

What is Horizontal Pod Autoscaler (HPA)?	<div style="text-align:left"><b>HPA</b> automatically scales the number of Pod replicas in a Deployment (or StatefulSet) based on observed metrics.<br><br>Default metric: CPU utilization. You can also scale on memory, custom metrics (Prometheus), or external metrics (queue depth, request rate).<br><br><code>apiVersion: autoscaling/v2<br>kind: HorizontalPodAutoscaler<br>spec:<br>  scaleTargetRef:<br>    kind: Deployment<br>    name: my-app<br>  minReplicas: 2<br>  maxReplicas: 20<br>  metrics:<br>  - type: Resource<br>    resource:<br>      name: cpu<br>      target:<br>        type: Utilization<br>        averageUtilization: 70</code><br><br><b>How it works:</b> Every 15s (default), it checks metrics and adjusts replicas to meet the target utilization.<br><br><b>Senior insight:</b> HPA requires metrics-server to be installed. Don't use HPA and manual replicas together — HPA will fight your manual changes. Always set meaningful <code>requests</code> — HPA calculates utilization as actual/requested.</div>

What is Vertical Pod Autoscaler (VPA)?	<div style="text-align:left"><b>VPA</b> automatically adjusts the CPU and memory <b>requests</b> (not replicas) of containers based on actual usage over time.<br><br>Think of it as the complement to HPA:<br>- <b>HPA:</b> scales <i>out</i> (more replicas)<br>- <b>VPA:</b> scales <i>up</i> (bigger containers)<br><br><b>VPA modes:</b><br>- <code>Off</code>: Just recommends, no changes applied (great for right-sizing analysis)<br>- <code>Initial</code>: Only applies on Pod creation<br>- <code>Auto</code>: Evicts and restarts Pods to apply new recommendations (disruptive!)<br><br><b>Senior insight:</b> <b>Do NOT use VPA and HPA together on the same metric</b> (both on CPU = conflicts). Safe combo: VPA in Off/Initial mode + HPA on custom metrics, or VPA on memory + HPA on CPU. VPA in Auto mode causes Pod restarts — not safe for stateless apps in production without testing.</div>

What is Cluster Autoscaler (CA)?	<div style="text-align:left"><b>Cluster Autoscaler</b> automatically adjusts the <b>number of Nodes</b> in a cluster when:<br>- Pods fail to schedule due to insufficient resources → <b>scale up</b> (add nodes)<br>- Nodes are underutilized for a sustained period → <b>scale down</b> (remove nodes)<br><br>It works with cloud provider node groups (AWS ASG, GCP MIG, Azure VMSS).<br><br><b>Scale-up trigger:</b> A Pod is in <code>Pending</code> state because no Node has enough capacity.<br><b>Scale-down trigger:</b> A Node's utilization is below 50% for 10+ minutes and its Pods can be rescheduled elsewhere.<br><br><b>Senior insight:</b><br>- CA won't remove a Node if a Pod has <code>PodDisruptionBudget</code> blocking eviction<br>- CA won't remove a Node with local storage Pods<br>- Set <code>cluster-autoscaler.kubernetes.io/safe-to-evict: "false"</code> annotation on Pods that must not be evicted (critical, long-running batch jobs)<br>- CA + HPA is the standard production scaling combo: HPA scales replicas, CA scales nodes</div>

What is a PersistentVolume (PV) and PersistentVolumeClaim (PVC)?	<div style="text-align:left">In Kubernetes, storage is abstracted into two objects:<br><br><b>PersistentVolume (PV):</b> A piece of storage provisioned in the cluster (NFS, AWS EBS, GCP PD, etc.) — like a physical hard drive in the data center. Created by admins or dynamically.<br><br><b>PersistentVolumeClaim (PVC):</b> A request for storage by a user/Pod — like a ticket saying "I need 10Gi of SSD storage." K8s finds a matching PV and binds them.<br><br><code>kind: PersistentVolumeClaim<br>spec:<br>  accessModes: [ReadWriteOnce]<br>  storageClassName: fast-ssd<br>  resources:<br>    requests:<br>      storage: 10Gi</code><br><br><b>Access modes:</b><br>- <code>ReadWriteOnce (RWO)</code>: One node at a time (typical block storage)<br>- <code>ReadOnlyMany (ROX)</code>: Many nodes, read-only<br>- <code>ReadWriteMany (RWX)</code>: Many nodes, read-write (NFS, EFS)<br><br><b>Senior insight:</b> Dynamic provisioning via StorageClass is the standard — you define a PVC and K8s automatically creates the underlying disk. No manual PV creation needed.</div>

What is a StorageClass in Kubernetes?	<div style="text-align:left">A <b>StorageClass</b> defines how storage is dynamically provisioned — it's a template that tells Kubernetes which provisioner to use and with what parameters.<br><br>Think of it like storage tiers: you define a "fast-ssd" class and a "cheap-hdd" class. Developers just pick the class they need in their PVC — no knowledge of the underlying infrastructure required.<br><br><code>apiVersion: storage.k8s.io/v1<br>kind: StorageClass<br>metadata:<br>  name: fast-ssd<br>provisioner: ebs.csi.aws.com<br>parameters:<br>  type: gp3<br>  iops: "3000"<br>reclaimPolicy: Delete<br>allowVolumeExpansion: true</code><br><br><b>Reclaim policies:</b><br>- <code>Delete</code>: When PVC is deleted, the underlying volume is deleted (default for cloud)<br>- <code>Retain</code>: Volume is kept — manual cleanup needed (safer for production data)<br><br><b>Senior insight:</b> Always set <code>Retain</code> for databases in production — losing data because a PVC was accidentally deleted is catastrophic. Use <code>Delete</code> only for ephemeral/test workloads.</div>

What is a Kubernetes Liveness Probe and Readiness Probe?	<div style="text-align:left"><b>Liveness Probe:</b> Checks if a container is <b>alive</b>. If it fails, Kubernetes <b>restarts</b> the container. Detects deadlocks or hung processes.<br><br><b>Readiness Probe:</b> Checks if a container is <b>ready to receive traffic</b>. If it fails, the Pod is removed from the Service's load balancing pool. No restart — just no traffic sent to it.<br><br><b>Startup Probe:</b> Used for slow-starting containers (legacy apps). Delays liveness checks until the app has started. Prevents premature restarts.<br><br><code>livenessProbe:<br>  httpGet:<br>    path: /healthz<br>    port: 8080<br>  initialDelaySeconds: 10<br>  periodSeconds: 10<br>  failureThreshold: 3<br><br>readinessProbe:<br>  httpGet:<br>    path: /ready<br>    port: 8080<br>  initialDelaySeconds: 5<br>  periodSeconds: 5</code><br><br><b>Senior insight:</b> A bad liveness probe that hits a slow endpoint can cause infinite restart loops under load. Use a lightweight <code>/healthz</code> endpoint that only checks if the process is alive — not DB connections. Readiness should check dependencies (DB reachable, cache warm).</div>

What is the Kubernetes Scheduler and how does it assign Pods to Nodes?	<div style="text-align:left">The <b>kube-scheduler</b> is responsible for assigning newly created Pods (those without a Node) to an appropriate Node.<br><br><b>Scheduling process (two phases):</b><br><br>1. <b>Filtering:</b> Eliminates Nodes that can't run the Pod (insufficient CPU/memory, taints don't match, node selectors don't match, etc.)<br><br>2. <b>Scoring:</b> Ranks remaining Nodes by multiple criteria (resource balance, affinity preferences, spread constraints) and picks the best one<br><br><b>Key scheduling controls:</b><br>- <b>nodeSelector:</b> Basic label-based node targeting<br>- <b>Node Affinity:</b> Rich rules for node targeting (required vs preferred)<br>- <b>Pod Affinity/Anti-Affinity:</b> Place Pods near or away from other Pods<br>- <b>Taints & Tolerations:</b> Mark nodes as special; Pods must tolerate to be scheduled there<br>- <b>Topology Spread Constraints:</b> Spread Pods evenly across zones/nodes<br><br><b>Senior insight:</b> When a Pod is stuck in <code>Pending</code>, run <code>kubectl describe pod &lt;name&gt;</code> and look at the Events section — the scheduler tells you exactly why it can't find a node.</div>

What are Taints and Tolerations in Kubernetes?	<div style="text-align:left"><b>Taints</b> are placed on Nodes to <b>repel</b> Pods — they mark a Node as special and prevent regular Pods from being scheduled there.<br><br><b>Tolerations</b> are placed on Pods to <b>allow</b> them to be scheduled on tainted Nodes despite the taint.<br><br>Analogy: A taint is a "No Entry" sign on a node. A toleration is the keycard that lets your Pod through anyway.<br><br><b>Taint effects:</b><br>- <code>NoSchedule</code>: Don't schedule Pods without matching toleration<br>- <code>PreferNoSchedule</code>: Try to avoid, but not hard<br>- <code>NoExecute</code>: Evict existing Pods that don't tolerate this<br><br><code># Taint a node for GPU workloads only<br>kubectl taint nodes gpu-node1 gpu=true:NoSchedule<br><br># Pod toleration<br>tolerations:<br>- key: "gpu"<br>  operator: "Equal"<br>  value: "true"<br>  effect: "NoSchedule"</code><br><br><b>Senior insight:</b> Control Plane nodes are tainted <code>node-role.kubernetes.io/control-plane:NoSchedule</code> to prevent workloads from running on them. Use taints to dedicate nodes to specific workloads (GPU, spot instances, compliance zones).</div>

What is Node Affinity and Pod Affinity in Kubernetes?	<div style="text-align:left"><b>Node Affinity:</b> Constrain which Nodes a Pod can be scheduled on based on Node labels. More expressive than nodeSelector.<br><br>Two modes:<br>- <code>requiredDuringSchedulingIgnoredDuringExecution</code>: Hard requirement — Pod won't schedule if no node matches<br>- <code>preferredDuringSchedulingIgnoredDuringExecution</code>: Soft preference — K8s tries, but will ignore if needed<br><br><b>Pod Affinity/Anti-Affinity:</b> Schedule Pods relative to <i>other Pods</i>:<br>- <b>Affinity:</b> "Run this Pod near Pods with label X" — e.g., co-locate app and cache<br>- <b>Anti-Affinity:</b> "Spread replicas across different Nodes/Zones" — e.g., don't put all replicas on one node<br><br><code>affinity:<br>  podAntiAffinity:<br>    requiredDuringSchedulingIgnoredDuringExecution:<br>    - labelSelector:<br>        matchLabels:<br>          app: my-app<br>      topologyKey: kubernetes.io/hostname</code><br><br><b>Senior insight:</b> Pod Anti-Affinity with <code>topologyKey: topology.kubernetes.io/zone</code> is the standard way to spread replicas across availability zones for high availability.</div>

What is a Kubernetes NetworkPolicy?	<div style="text-align:left">A <b>NetworkPolicy</b> controls how Pods communicate with each other and with the outside world — essentially a firewall for Pod-to-Pod traffic.<br><br>By default, all Pods in a cluster can talk to all other Pods. NetworkPolicies let you restrict this.<br><br><code>apiVersion: networking.k8s.io/v1<br>kind: NetworkPolicy<br>spec:<br>  podSelector:<br>    matchLabels:<br>      app: database<br>  policyTypes: [Ingress]<br>  ingress:<br>  - from:<br>    - podSelector:<br>        matchLabels:<br>          app: backend  # only backend can talk to database<br>    ports:<br>    - port: 5432</code><br><br><b>Default deny pattern:</b> Apply an empty-selector NetworkPolicy to a namespace to deny all traffic, then explicitly allow what's needed.<br><br><b>Important:</b> NetworkPolicies require a CNI plugin that supports them (Calico, Cilium, Weave). The default kubenet/flannel does NOT enforce NetworkPolicies!<br><br><b>Senior insight:</b> NetworkPolicies are namespace-scoped. In high-security environments, combine with a service mesh (Istio/Linkerd) for mTLS between services.</div>

What is RBAC in Kubernetes?	<div style="text-align:left"><b>RBAC (Role-Based Access Control)</b> controls who can do what in a Kubernetes cluster. It answers: "Can this user/service account perform this action on this resource?"<br><br><b>Key objects:</b><br>- <b>Role:</b> Defines permissions within a namespace (e.g., "can read Pods in default namespace")<br>- <b>ClusterRole:</b> Like Role but cluster-wide (across all namespaces or cluster-scoped resources)<br>- <b>RoleBinding:</b> Grants a Role to a user/group/ServiceAccount within a namespace<br>- <b>ClusterRoleBinding:</b> Grants a ClusterRole cluster-wide<br><br><code>kind: Role<br>rules:<br>- apiGroups: ["apps"]<br>  resources: ["deployments"]<br>  verbs: ["get", "list", "watch"]</code><br><br><b>Principle of least privilege:</b> Give only the permissions needed — nothing more.<br><br><b>Senior insight:</b> Always audit ServiceAccount permissions. A compromised Pod with a high-privileged ServiceAccount can take over the entire cluster. Use <code>kubectl auth can-i --as system:serviceaccount:namespace:sa-name get secrets</code> to check what a SA can do.</div>

What is a ServiceAccount in Kubernetes?	<div style="text-align:left">A <b>ServiceAccount</b> provides an identity to a Pod for interacting with the Kubernetes API. Every Pod runs with a ServiceAccount — the default is <code>default</code> in its namespace.<br><br>When a Pod has a ServiceAccount, a token is automatically mounted at:<br><code>/var/run/secrets/kubernetes.io/serviceaccount/token</code><br><br>This token can be used to call the Kubernetes API — useful for operators, controllers, and CI/CD tools running inside the cluster.<br><br><b>Usage:</b><br><code>spec:<br>  serviceAccountName: my-app-sa</code><br><br><b>Common mistake:</b> Leaving Pods with the <code>default</code> ServiceAccount and giving that account broad permissions. One compromise = broad API access.<br><br><b>Senior insight:</b> Set <code>automountServiceAccountToken: false</code> on ServiceAccounts (or Pods) that don't need API access — reduces attack surface. With K8s 1.22+, tokens are time-limited and audience-bound by default, replacing the old static long-lived tokens.</div>

What is a Kubernetes Operator?	<div style="text-align:left">A <b>Kubernetes Operator</b> is a custom controller that extends Kubernetes to manage complex, stateful applications using Kubernetes-native patterns.<br><br>Operators encode operational knowledge: "How do I deploy, scale, upgrade, backup, and recover this specific application?" — expressed as code that runs inside the cluster.<br><br>Built using <b>Custom Resources (CRDs)</b> and a controller that watches them:<br>- You create a <code>PostgreSQLCluster</code> custom resource<br>- The Postgres Operator watches it and creates the actual Pods, Services, PVCs, manages failover, handles backups, etc.<br><br><b>Popular Operators:</b><br>- Prometheus Operator (monitoring)<br>- cert-manager (TLS certificate lifecycle)<br>- Strimzi (Kafka on K8s)<br>- CloudNativePG (PostgreSQL)<br>- Vault Operator (secrets management)<br><br><b>Senior insight:</b> An Operator is just a Pod running a controller loop (List/Watch → Reconcile). The reconcile loop asks: "Does the actual state match the desired state? If not, fix it." This is the same pattern K8s uses internally — Operators extend that pattern to your apps.</div>

What is a Custom Resource Definition (CRD)?	<div style="text-align:left">A <b>CRD (Custom Resource Definition)</b> extends the Kubernetes API to add your own resource types. Once you create a CRD, you can use <code>kubectl</code> to create, read, update, and delete instances of your new resource just like built-in resources (Pods, Deployments, etc.).<br><br>Example: cert-manager adds a <code>Certificate</code> resource type:<br><code>apiVersion: cert-manager.io/v1<br>kind: Certificate<br>metadata:<br>  name: my-tls<br>spec:<br>  secretName: my-tls-secret<br>  dnsNames: ["example.com"]<br>  issuerRef:<br>    name: letsencrypt</code><br><br>kubectl now knows about <code>Certificate</code> objects. A controller watches them and provisions real TLS certs.<br><br><b>CRDs are the foundation of:</b> Operators, service meshes, GitOps tools (ArgoCD, Flux), and all major K8s ecosystem projects.<br><br><b>Senior insight:</b> CRDs have a validation schema (OpenAPI v3) — always define it to prevent invalid resources from being created. Upgrading a CRD schema is tricky: removing fields is a breaking change and requires careful migration.</div>

What is Helm and why is it used?	<div style="text-align:left"><b>Helm</b> is the package manager for Kubernetes — like apt/npm/pip but for K8s. It allows you to define, install, and upgrade complex K8s applications as versioned packages called <b>Charts</b>.<br><br><b>A Helm Chart is:</b> A collection of YAML templates + a <code>values.yaml</code> file for configuration.<br><br><code># Install a chart from a repo<br>helm repo add bitnami https://charts.bitnami.com/bitnami<br>helm install my-postgres bitnami/postgresql \<br>  --set auth.postgresPassword=secret \<br>  --namespace db<br><br># Upgrade<br>helm upgrade my-postgres bitnami/postgresql --set image.tag=15.3<br><br># Rollback<br>helm rollback my-postgres 1</code><br><br><b>Why it matters:</b><br>- Packages 20+ K8s objects into one deployable unit<br>- Parameterizes config (dev vs prod values files)<br>- Tracks release history<br><br><b>Senior insight:</b> Don't blindly install community charts in production — audit them. Use <code>helm template</code> to inspect generated YAML before applying. Store your Helm values files in git, not secrets in CI/CD variables.</div>

What is Kustomize and how does it differ from Helm?	<div style="text-align:left"><b>Kustomize</b> is a Kubernetes-native configuration management tool — it patches and overlays plain YAML files without templates. It's built into kubectl (<code>kubectl apply -k</code>).<br><br><b>Key concepts:</b><br>- <b>Base:</b> Common/shared YAML resources<br>- <b>Overlay:</b> Environment-specific patches applied on top of the base<br><br>Structure:<br><code>base/<br>  deployment.yaml<br>  service.yaml<br>  kustomization.yaml<br>overlays/<br>  production/<br>    kustomization.yaml  # patches replicas, image tag, resources<br>  staging/<br>    kustomization.yaml  # different values</code><br><br><b>Helm vs Kustomize:</b><br>- Helm: Templating engine, good for distributable packages (community charts), has release management<br>- Kustomize: Patching/overlaying, good for your own apps, no templating syntax to learn, simpler<br><br><b>Senior insight:</b> Many teams use <b>both</b>: Helm to install third-party tools (Prometheus, nginx-ingress), Kustomize to manage their own application config. This is the standard GitOps pattern with ArgoCD and Flux.</div>

What is GitOps and how is it implemented with Kubernetes?	<div style="text-align:left"><b>GitOps</b> is a practice where Git is the single source of truth for the desired state of your infrastructure and applications. All changes go through Git (PRs, reviews, commits) — a controller in the cluster automatically syncs the live state to match.<br><br><b>Core principles:</b><br>1. Declarative: All config stored as YAML in Git<br>2. Versioned: Git history = audit trail + rollback<br>3. Automated: Controller pulls and applies changes<br>4. Reconciled: Controller continuously corrects drift<br><br><b>Tools:</b><br>- <b>ArgoCD:</b> UI-rich, app-centric GitOps. Each app is an ArgoCD Application pointing at a Git repo/path<br>- <b>Flux:</b> Lightweight, GitOps-native, better Helm integration<br><br><b>Flow:</b> Developer pushes image tag change to Git → ArgoCD detects change → applies new Deployment to cluster → cluster reconciles<br><br><b>Senior insight:</b> GitOps eliminates "configuration drift" — the cluster can't be in a state not reflected in Git (if reconciliation is enforced). Combine with branch-based environment promotion (dev → staging → prod) and PR approvals for a mature release process.</div>

What is a Kubernetes rolling update strategy?	<div style="text-align:left">A <b>rolling update</b> gradually replaces old Pods with new ones, ensuring zero downtime. It's the default update strategy for Deployments.<br><br><code>spec:<br>  strategy:<br>    type: RollingUpdate<br>    rollingUpdate:<br>      maxSurge: 1        # max extra Pods above desired count<br>      maxUnavailable: 0  # min Pods that must stay available</code><br><br><b>With maxSurge: 1, maxUnavailable: 0:</b> Creates 1 new Pod, waits for it to be ready, then terminates 1 old Pod. Repeats until all are updated. No downtime, but briefly uses extra capacity.<br><br><b>Other strategies:</b><br>- <b>Recreate:</b> Kill all old Pods, then create new ones. Has downtime — use only for stateful apps where you can't run two versions concurrently<br><br><b>Senior insight:</b> Rolling updates can cause issues if your new and old versions are incompatible (e.g., DB schema changes). Address this with backwards-compatible migrations (expand-contract pattern). Also set <code>minReadySeconds</code> to give new Pods time to stabilize before the next Pod is replaced.</div>

What is a Blue/Green Deployment in Kubernetes?	<div style="text-align:left">A <b>Blue/Green deployment</b> runs two identical environments (Blue = current, Green = new version). Traffic instantly switches from Blue to Green when ready — enabling instant rollback if issues occur.<br><br><b>In Kubernetes:</b><br>1. Deploy the new version as a separate Deployment with label <code>version: green</code><br>2. Verify the Green environment is healthy<br>3. Update the Service selector to point to Green<br>4. Keep Blue running for fast rollback<br>5. Delete Blue when confident<br><br><code># Switch service to green<br>kubectl patch service my-app \<br>  -p '{"spec":{"selector":{"version":"green"}}}'</code><br><br><b>Pros:</b> Instant traffic switch, instant rollback, zero downtime<br><b>Cons:</b> Requires 2x capacity during deployment, need to manage DB compatibility<br><br><b>Senior insight:</b> Blue/Green is ideal for major version upgrades or when you need to test at full prod load before committing. Use tools like Argo Rollouts for automated Blue/Green and Canary with traffic analysis.</div>

What is a Canary Deployment in Kubernetes?	<div style="text-align:left">A <b>Canary deployment</b> gradually shifts traffic to a new version — starting with a small percentage and increasing as confidence grows. Named after "canary in a coal mine" — the small percentage of users are the early warning system.<br><br><b>Simple implementation with two Deployments:</b><br>- Stable: 9 replicas (90% traffic)<br>- Canary: 1 replica (10% traffic)<br>- Same Service selector hits both (traffic split by replica ratio)<br><br><b>Advanced with Argo Rollouts or Istio:</b><br>- Precise traffic splitting (5% → 20% → 50% → 100%)<br>- Automated analysis (error rate, latency)<br>- Auto-rollback if metrics degrade<br><br><code>analysis:<br>  templates:<br>  - templateName: success-rate<br>  args:<br>  - name: threshold<br>    value: "99"</code><br><br><b>Senior insight:</b> Canary is the gold standard for production deployments at scale. The replica-count approach is crude — use Istio or Argo Rollouts for percentage-based traffic splitting independent of replica count. Always define rollback conditions based on real metrics (error rate, p99 latency), not just manual observation.</div>

What is a PodDisruptionBudget (PDB) in Kubernetes?	<div style="text-align:left">A <b>PodDisruptionBudget</b> limits the number of Pods of a replicated application that can be simultaneously unavailable during <b>voluntary disruptions</b> (node drains, cluster upgrades, scaling down).<br><br>It protects against: kubectl drain, cluster upgrades, node group rolling updates, and Cluster Autoscaler scale-down.<br><br><code>apiVersion: policy/v1<br>kind: PodDisruptionBudget<br>spec:<br>  selector:<br>    matchLabels:<br>      app: my-app<br>  minAvailable: 2    # at least 2 Pods must always be running<br>  # OR<br>  maxUnavailable: 1  # at most 1 Pod can be down at a time</code><br><br><b>Real-world scenario:</b> Without a PDB, draining 3 nodes simultaneously can take down all replicas at once. With <code>minAvailable: 2</code>, kubectl drain waits until a new Pod is running before draining the next node.<br><br><b>Senior insight:</b> PDBs only work for voluntary disruptions — involuntary (node failure) is not covered. Always create PDBs for production services. A PDB that's too strict can block cluster upgrades indefinitely if Pods are unhealthy.</div>

What is etcd and why is it critical in Kubernetes?	<div style="text-align:left"><b>etcd</b> is a distributed, consistent key-value store that serves as Kubernetes' database — it stores all cluster state: Pods, Services, ConfigMaps, Secrets, Deployments, RBAC, etc.<br><br>Think of etcd as the cluster's memory. If etcd is lost and has no backup, the cluster loses all its state — though running workloads continue until they restart.<br><br><b>Key properties:</b><br>- Uses the <b>Raft consensus algorithm</b> for distributed consistency<br>- Requires quorum (majority) for writes: 3-node cluster needs 2 alive, 5-node needs 3<br>- Strong consistency: every read returns the latest data<br><br><b>etcd sizing rule:</b> Always run odd numbers (3 or 5). Never 2 or 4 — you lose quorum if one fails with 2.<br><br><b>Performance:</b> etcd is sensitive to disk I/O latency. Use dedicated SSDs (nvme). Never run etcd on spinning disks in production. Use local SSDs, not network-attached storage.<br><br><b>Senior insight:</b> Back up etcd regularly (at least daily). Use <code>etcdctl snapshot save</code>. In cloud K8s (EKS/GKE/AKS), the cloud provider manages and backs up etcd. In self-managed clusters, etcd backup is the most important operational task.</div>

What is the Kubernetes API server and how does a kubectl command work?	<div style="text-align:left">The <b>kube-apiserver</b> is the central management endpoint of a Kubernetes cluster. All interactions (kubectl, controllers, kubelet, scheduler) go through it as REST API calls.<br><br><b>What happens when you run <code>kubectl apply -f deployment.yaml</code>:</b><br><br>1. kubectl reads your kubeconfig for server address and credentials<br>2. kubectl sends an HTTP request to the API server<br>3. API server <b>authenticates</b>: Who are you? (cert, token, OIDC)<br>4. API server <b>authorizes</b>: Are you allowed to do this? (RBAC check)<br>5. API server runs <b>Admission Controllers</b>: Validate and mutate the request (webhooks, resource quotas)<br>6. Object is validated against its schema<br>7. Object is written to etcd<br>8. A controller/scheduler watches the API and acts on the new object<br><br><b>Admission Controllers are powerful:</b><br>- Enforce policies (no latest image tags, resource limits required)<br>- Inject sidecars (service mesh)<br>- Validate security constraints (Pod Security Admission)<br><br><b>Senior insight:</b> The API server is stateless (state is in etcd) — you can run multiple replicas behind a load balancer for HA. All audit logging happens here.</div>

What are Kubernetes Admission Controllers?	<div style="text-align:left"><b>Admission Controllers</b> are plugins that intercept API requests to the Kubernetes API server <b>after authentication and authorization but before persistence</b> to etcd. They can validate or mutate (modify) requests.<br><br><b>Two types:</b><br>- <b>Validating:</b> Accept or reject the request (e.g., reject Pods without resource limits)<br>- <b>Mutating:</b> Modify the request before it's stored (e.g., inject a sidecar container, add default labels)<br><br><b>Built-in controllers (examples):</b><br>- <code>LimitRanger</code>: Enforces LimitRange policies<br>- <code>ResourceQuota</code>: Enforces namespace quotas<br>- <code>PodSecurity</code>: Enforces Pod Security Standards<br>- <code>DefaultStorageClass</code>: Sets default storage class on PVCs<br><br><b>Webhook Admission Controllers:</b> Dynamic, extensible — you write a webhook server and register it. Used by OPA/Gatekeeper, Kyverno, service meshes (Istio injects sidecars this way).<br><br><b>Senior insight:</b> Mutating webhooks run before Validating webhooks. A flaky webhook can block all cluster operations — always set appropriate timeouts and <code>failurePolicy: Ignore</code> for non-critical webhooks. Test webhooks in dry-run mode first.</div>

What is Pod Security in Kubernetes (PSA/PSP)?	<div style="text-align:left">Kubernetes provides mechanisms to enforce security constraints on Pods at the admission level.<br><br><b>History:</b><br>- <b>PodSecurityPolicy (PSP):</b> Deprecated in K8s 1.21, removed in 1.25 — complex, had usability issues<br>- <b>Pod Security Admission (PSA):</b> Built-in replacement from K8s 1.25+<br><br><b>Pod Security Standards (three levels):</b><br>- <code>privileged</code>: No restrictions — for trusted system workloads<br>- <code>baseline</code>: Prevents known privilege escalations — reasonable default<br>- <code>restricted</code>: Heavily restricted — follows pod hardening best practices<br><br><b>Applied per namespace via label:</b><br><code>kubectl label namespace my-ns \<br>  pod-security.kubernetes.io/enforce=restricted</code><br><br><b>Modes:</b> <code>enforce</code> (reject), <code>audit</code> (log), <code>warn</code> (warn user)<br><br><b>Senior insight:</b> For more granular policy control, use <b>Kyverno</b> or <b>OPA Gatekeeper</b> — they go beyond PSA (custom policies, image signing verification, label enforcement, etc.). Start with PSA for baseline hardening, add OPA/Kyverno as policies grow.</div>

What is a Service Mesh and when would you use one with Kubernetes?	<div style="text-align:left">A <b>Service Mesh</b> adds a layer of infrastructure for handling service-to-service communication — providing mTLS, observability, traffic management, and retries without changing application code.<br><br>It works by injecting a <b>sidecar proxy</b> (Envoy) into every Pod. All traffic goes through the proxy, which enforces policies and emits telemetry.<br><br><b>What a service mesh gives you:</b><br>- <b>mTLS everywhere:</b> Automatic mutual TLS between all services — zero-trust networking<br>- <b>Observability:</b> Automatic distributed tracing, metrics, and topology maps<br>- <b>Traffic control:</b> Canary releases, circuit breaking, retries, timeouts, fault injection<br>- <b>Access policy:</b> Fine-grained authorization (Service A can't call Service B)<br><br><b>Popular options:</b> Istio (most feature-rich, complex), Linkerd (simpler, lighter), Cilium (eBPF-based, no sidecars)<br><br><b>When to use it:</b> 10+ microservices, compliance requiring encryption in transit, need for advanced traffic control<br><br><b>Senior insight:</b> Service meshes add significant operational complexity and resource overhead (each sidecar consumes ~50-100MB). Don't add a mesh "just in case" — earn the complexity with a real requirement.</div>

What is Kubernetes DNS and how does service discovery work?	<div style="text-align:left">Kubernetes includes a built-in DNS server (usually <b>CoreDNS</b>) that automatically creates DNS records for Services and Pods.<br><br><b>Service DNS format:</b><br><code>service-name.namespace.svc.cluster.local</code><br><br><b>Examples:</b><br>- Same namespace: <code>my-service</code> (short name works)<br>- Different namespace: <code>my-service.other-namespace</code><br>- Fully qualified: <code>my-service.other-namespace.svc.cluster.local</code><br><br><b>How it works:</b><br>1. You create a Service<br>2. K8s creates a DNS record in CoreDNS<br>3. All Pods get <code>/etc/resolv.conf</code> pointing to CoreDNS<br>4. Pods resolve service names to ClusterIP<br>5. kube-proxy routes the ClusterIP to a healthy Pod<br><br><b>Pod DNS:</b> Pods also get DNS: <code>pod-ip.namespace.pod.cluster.local</code> (dots in IP replaced with dashes)<br><br><b>Senior insight:</b> CoreDNS is a critical cluster component — tune its cache TTL and resources for high-traffic clusters. DNS lookup failures are a common cause of cascading failures. Monitor CoreDNS error rates. Use <code>ndots:5</code> awareness — K8s adds search domains, causing 5 DNS lookups per external hostname.</div>

What is the difference between an InitContainer and a regular container?	<div style="text-align:left">An <b>InitContainer</b> is a specialized container that runs and completes <b>before</b> any regular (app) containers start in a Pod. They run sequentially, and all must succeed for the Pod to start.<br><br><b>Use cases:</b><br>- Wait for a database to be ready before starting the app<br>- Run database migrations before the app server starts<br>- Download config files or secrets before startup<br>- Register the service in a service registry<br><br><code>spec:<br>  initContainers:<br>  - name: wait-for-db<br>    image: busybox<br>    command: ['sh', '-c', 'until nc -z db-service 5432; do sleep 2; done']<br>  containers:<br>  - name: my-app<br>    image: my-app:1.0</code><br><br><b>Key differences:</b><br>- InitContainers always run to completion (exit 0) before app containers start<br>- They run sequentially (not in parallel)<br>- They can have different images and security settings than app containers<br>- They don't support lifecycle hooks or readiness probes<br><br><b>Senior insight:</b> Use InitContainers instead of sleep hacks in your main container. They're cleaner, reusable, and kubectl shows their status separately during debugging.</div>

What is a sidecar container pattern in Kubernetes?	<div style="text-align:left">The <b>sidecar pattern</b> runs a helper container alongside the main app container in the same Pod — extending or enhancing the main container's functionality without modifying it.<br><br>Both containers share the same network and can share volumes.<br><br><b>Common sidecars:</b><br>- <b>Log shipper:</b> Reads app logs from a shared volume, forwards to Elasticsearch/Loki (Filebeat, Fluentd)<br>- <b>Proxy:</b> Envoy proxy in service meshes (Istio injects this automatically)<br>- <b>Secret syncer:</b> Refreshes certs or tokens without restarting the app<br>- <b>Config reloader:</b> Watches ConfigMap changes and triggers a reload<br>- <b>Metrics exporter:</b> Exposes app metrics in Prometheus format<br><br><code>spec:<br>  containers:<br>  - name: app<br>    image: my-app<br>    volumeMounts:<br>    - name: logs<br>      mountPath: /var/log<br>  - name: log-shipper<br>    image: filebeat<br>    volumeMounts:<br>    - name: logs<br>      mountPath: /var/log</code><br><br><b>Senior insight:</b> K8s 1.28+ introduced native Sidecar containers (via <code>initContainers</code> with <code>restartPolicy: Always</code>) — they start before app containers and don't block startup, solving ordering issues.</div>

How does Kubernetes handle container image pulls and what is imagePullPolicy?	<div style="text-align:left"><b>imagePullPolicy</b> controls when Kubernetes pulls the container image for a Pod:<br><br>- <code>Always</code>: Pull the image every time a Pod starts — slower but ensures the latest image is used. Useful in development.<br>- <code>IfNotPresent</code>: Pull only if not cached on the node. Default when image tag is not <code>latest</code>.<br>- <code>Never</code>: Never pull — image must be pre-loaded on the node. Rare use case.<br><br><b>Default behavior based on tag:</b><br>- Image with <code>:latest</code> tag → defaults to <code>Always</code><br>- Image with a specific tag or digest → defaults to <code>IfNotPresent</code><br><br><b>Common mistake:</b> Using <code>:latest</code> tag in production — it makes deployments non-deterministic. Always use an immutable tag (git SHA, semantic version).<br><br><b>Image Pull Secrets:</b> For private registries, create a secret and reference it:<br><code>spec:<br>  imagePullSecrets:<br>  - name: my-registry-secret</code><br><br><b>Senior insight:</b> Use image digests (<code>myimage@sha256:abc123</code>) instead of tags for 100% reproducible deployments. Tags can be overwritten; digests are immutable. This is critical for security and auditability.</div>

What is Kubernetes pod lifecycle and what are the possible Pod phases?	<div style="text-align:left">Every Kubernetes Pod goes through distinct phases during its lifetime:<br><br><b>Pod Phases:</b><br>- <code>Pending</code>: Pod accepted by K8s but containers not yet running (waiting for scheduling, image pull)<br>- <code>Running</code>: Pod is bound to a node, at least one container is running<br>- <code>Succeeded</code>: All containers exited with 0 (success) — Jobs use this<br>- <code>Failed</code>: All containers have terminated, at least one failed<br>- <code>Unknown</code>: Node communication lost (node down)<br><br><b>Container states within a running Pod:</b><br>- <code>Waiting</code>: Not yet running (pulling image, init containers running)<br>- <code>Running</code>: Executing normally<br>- <code>Terminated</code>: Finished execution<br><br><b>Container restart policies:</b><br>- <code>Always</code>: Always restart (default for Deployments)<br>- <code>OnFailure</code>: Restart only if exit code is non-zero (Jobs)<br>- <code>Never</code>: Never restart<br><br><b>Senior insight:</b> When a container crashes, K8s restarts it with exponential backoff (10s, 20s, 40s... up to 5min). This is <code>CrashLoopBackOff</code> — one of the most common issues you'll debug. Check <code>kubectl logs --previous</code> to see the crashed container's last logs.</div>

What is CrashLoopBackOff and how do you debug it?	<div style="text-align:left"><b>CrashLoopBackOff</b> means a container is repeatedly crashing and Kubernetes keeps restarting it with increasing delays (backoff). It's not a cause — it's a symptom.<br><br><b>Common root causes:</b><br>1. Application bug/exception on startup<br>2. Wrong configuration (bad env vars, wrong DB URL)<br>3. Missing dependency (secret not found, ConfigMap missing)<br>4. Port conflict<br>5. Insufficient resources (OOM killed)<br>6. Bad liveness probe (configured too aggressively)<br><br><b>Debugging steps:</b><br><code># Check current logs<br>kubectl logs pod/my-pod<br><br># Check PREVIOUS container's logs (before crash)<br>kubectl logs pod/my-pod --previous<br><br># Describe pod for events and exit codes<br>kubectl describe pod my-pod<br><br># Check exit code in describe output<br># Exit code 1: app error | 137: OOM kill | 126: permission denied<br># 132: SIGILL | 139: segfault</code><br><br><b>Senior insight:</b> Exit code 137 = OOM kill → increase memory limit. Exit code 1 → app crashed, check logs. If logs show nothing, the container might be crashing before it can log — use <code>kubectl describe</code> Events to see the error.</div>

What is the Kubernetes Endpoint and EndpointSlice?	<div style="text-align:left">When you create a Service with a selector, Kubernetes automatically creates an <b>Endpoints</b> object that lists the IP addresses and ports of all matching Pods.<br><br>Think of Endpoints as the real-time routing table behind a Service. kube-proxy reads Endpoints to configure traffic routing.<br><br><code># Check if a service has endpoints (are Pods matching the selector?)<br>kubectl get endpoints my-service<br># NAME         ENDPOINTS           AGE<br># my-service   10.0.1.5:8080,...   5m</code><br><br>If Endpoints is empty → no Pods match the Service selector → your label selectors don't match.<br><br><b>EndpointSlices</b> (K8s 1.17+): Replace Endpoints for large-scale clusters. A single Endpoints object for a 1000-Pod service caused massive etcd/API server load on every Pod change. EndpointSlices split them into chunks of 100 — scalable for large deployments.<br><br><b>Senior insight:</b> Empty Endpoints is one of the first things to check when a Service isn't routing traffic. Always run <code>kubectl get endpoints service-name</code> when debugging connectivity — it tells you immediately if the selector is wrong.</div>

What are Kubernetes Init Containers used for in real production scenarios?	<div style="text-align:left">InitContainers are used for pre-startup tasks that must complete before the app starts. Real production scenarios:<br><br><b>1. Database migration gating:</b><br>Run <code>alembic upgrade head</code> or <code>rails db:migrate</code> before the app server starts — ensures schema is ready.<br><br><b>2. Dependency readiness check:</b><br><code>command: ['sh', '-c', 'until pg_isready -h postgres-svc; do sleep 2; done']</code><br>Waits for Postgres to be accepting connections before starting the API.<br><br><b>3. File seeding:</b><br>Download config files from S3 or a secret store and place them in a shared volume before the app reads them.<br><br><b>4. Permission fixing:</b><br>Some images run as non-root but mounted volumes default to root ownership — init container runs <code>chown</code> to fix permissions.<br><br><b>5. Certificate provisioning:</b><br>Generate or fetch a TLS cert into a shared volume before the main process starts its HTTPS listener.<br><br><b>Senior insight:</b> InitContainers are retried independently of the main container. If an InitContainer fails (non-zero exit), the Pod stays in <code>Init:CrashLoopBackOff</code>. Check <code>kubectl logs pod/my-pod -c init-container-name</code> to debug them.</div>

What is Kubernetes resource management and QoS classes?	<div style="text-align:left">Kubernetes assigns a <b>QoS (Quality of Service) class</b> to every Pod based on its resource requests and limits. This class determines eviction priority when a node is under memory pressure.<br><br><b>QoS Classes:</b><br><br><b>Guaranteed (highest priority, never evicted first):</b><br>- Every container has both requests and limits set<br>- requests == limits for both CPU and memory<br><br><b>Burstable (middle priority):</b><br>- At least one container has requests or limits<br>- requests ≠ limits<br><br><b>BestEffort (lowest priority, evicted first):</b><br>- No resource requests or limits set on any container<br><br><b>Eviction order under memory pressure:</b><br>BestEffort → Burstable (over limit) → Guaranteed<br><br><b>Senior insight:</b> For production critical services, use <b>Guaranteed</b> QoS — set requests == limits. This prevents eviction and also reduces CPU scheduling jitter (no throttling). For batch/dev workloads, BestEffort or Burstable is fine. Never run databases or payment services as BestEffort.</div>

What is the Kubernetes control loop / reconciliation loop?	<div style="text-align:left">The <b>reconciliation loop</b> (or control loop) is the fundamental pattern that makes Kubernetes self-healing. Every controller in K8s works this way:<br><br>1. <b>Observe:</b> Watch the current state of the cluster<br>2. <b>Compare:</b> Desired state (what you declared) vs Actual state (what exists)<br>3. <b>Act:</b> Take action to make actual match desired<br><br>Example — Deployment controller:<br>- Desired: 3 replicas<br>- Actual: 2 Pods running (one crashed)<br>- Action: Create 1 new Pod<br><br>This loop runs continuously and handles:<br>- Node failures (reschedule Pods)<br>- Pod crashes (restart/replace)<br>- Manual deletions (recreate to match desired count)<br>- Config changes (rolling update)<br><br><b>Key insight:</b> Kubernetes is always trying to reach desired state — you declare what you want, K8s figures out how to get there and stay there. This is <b>declarative</b> (what) vs imperative (how) infrastructure.<br><br><b>Senior insight:</b> Understanding this loop helps debug: if K8s isn't taking action you expect, something in the loop is blocked — an admission webhook, a stuck finalizer, a controller not running. Check controller logs.</div>

What are Kubernetes Finalizers and when are they used?	<div style="text-align:left"><b>Finalizers</b> are string values in an object's metadata that prevent deletion until specific cleanup actions are completed. When you delete an object with a finalizer, Kubernetes sets a <code>deletionTimestamp</code> but doesn't delete the object — it waits for the finalizer to be removed.<br><br><code>metadata:<br>  finalizers:<br>  - kubernetes.io/pvc-protection<br>  - my-operator.example.com/cleanup</code><br><br><b>How it works:</b><br>1. You delete a PVC<br>2. K8s sets <code>deletionTimestamp</code> and waits<br>3. A controller sees the deletion request, performs cleanup (e.g., detaches cloud disk)<br>4. Controller removes the finalizer from the object<br>5. Object is now fully deleted<br><br><b>Common examples:</b><br>- <code>kubernetes.io/pvc-protection</code>: Prevents PVC deletion while a Pod is using it<br>- Operators use finalizers to clean up external resources (cloud LBs, DNS records, DB users) before the K8s object is removed<br><br><b>Senior insight:</b> Objects stuck in <code>Terminating</code> state forever = a finalizer that's never being removed (controller is dead or buggy). Force-remove a stuck finalizer with: <code>kubectl patch resource/name -p '{"metadata":{"finalizers":[]}}' --type=merge</code> — but understand why it's stuck first!</div>

What are Kubernetes Events and how do you use them for debugging?	<div style="text-align:left">Kubernetes <b>Events</b> are informational records of what's happening in the cluster — created by components (scheduler, kubelet, controllers) to describe state changes and errors.<br><br><b>Key commands:</b><br><code># Get events for a specific pod<br>kubectl describe pod my-pod<br># Events section at the bottom is most useful<br><br># Get all events in a namespace, sorted by time<br>kubectl get events --sort-by='.metadata.creationTimestamp'<br><br># Watch events in real time<br>kubectl get events -w<br><br># Get events for a specific object<br>kubectl get events --field-selector involvedObject.name=my-pod</code><br><br><b>Common events to know:</b><br>- <code>Scheduled</code>: Pod assigned to a node<br>- <code>Pulling/Pulled</code>: Image pull status<br>- <code>Started</code>: Container started<br>- <code>BackOff</code>: Restarting failed container<br>- <code>FailedScheduling</code>: Scheduler couldn't find a node (with reason!)<br>- <code>OOMKilled</code>: Memory limit exceeded<br><br><b>Senior insight:</b> Events are ephemeral — they're garbage collected after 1 hour (default) or 1000 events. For long-term audit and debugging, ship events to a logging system (Loki, Elasticsearch). Tools like <code>event-exporter</code> or <code>kube-state-metrics</code> expose them as metrics.</div>

What is Prometheus and how does monitoring work in Kubernetes?	<div style="text-align:left"><b>Prometheus</b> is the standard monitoring system for Kubernetes. It <b>scrapes</b> (pulls) metrics from HTTP endpoints at regular intervals and stores them as time-series data.<br><br><b>How K8s monitoring works:</b><br>- Components expose metrics at <code>/metrics</code> in Prometheus format<br>- Prometheus discovers targets via Service Discovery (reads K8s API to find Pods/Services with specific annotations)<br>- Grafana visualizes the metrics<br><br><b>Key components of the kube-prometheus-stack:</b><br>- <b>Prometheus:</b> Metrics collection and alerting<br>- <b>Alertmanager:</b> Routes alerts (email, Slack, PagerDuty)<br>- <b>Grafana:</b> Dashboards and visualization<br>- <b>kube-state-metrics:</b> Exposes K8s object state as metrics (replicas, pod phases, resource limits)<br>- <b>node-exporter:</b> Exposes host-level metrics (CPU, memory, disk) as a DaemonSet<br><br><b>Auto-discovery annotation:</b><br><code>annotations:<br>  prometheus.io/scrape: "true"<br>  prometheus.io/port: "9090"</code><br><br><b>Senior insight:</b> Instrument your apps with client libraries (prom-client, prometheus-client) to expose business metrics. Don't rely only on infrastructure metrics — track request rate, error rate, and latency (the RED method: Rate, Errors, Duration).</div>

What is the difference between kubectl apply and kubectl create?	<div style="text-align:left"><b>kubectl create:</b> Creates a resource for the first time. Fails if the resource already exists.<br><br><b>kubectl apply:</b> Creates a resource if it doesn't exist, or <b>updates</b> it if it does — using a 3-way merge between the last applied config, the current live state, and the new config. This is the <b>declarative</b> approach.<br><br><code># Declarative (use this in production/CI-CD)<br>kubectl apply -f deployment.yaml<br><br># Imperative (use for quick one-offs or exploration)<br>kubectl create deployment my-app --image=my-app:1.0</code><br><br><b>kubectl apply stores the last applied config</b> as an annotation on the resource (<code>kubectl.kubernetes.io/last-applied-configuration</code>) — this is used for the 3-way merge.<br><br><b>kubectl replace:</b> Completely replaces the resource — equivalent to delete + create. Fails if the resource doesn't exist.<br><br><b>Senior insight:</b> Always use <code>apply</code> in GitOps and CI/CD pipelines — it's idempotent and safe to run multiple times. Use <code>kubectl diff -f file.yaml</code> to preview changes before applying. Server-side apply (<code>kubectl apply --server-side</code>) is the modern approach and handles conflicts better in multi-team environments.</div>

What is kubectl and what are the most important commands for a DevOps engineer?	<div style="text-align:left"><b>kubectl</b> is the command-line tool for interacting with Kubernetes clusters.<br><br><b>Essential commands:</b><br><code># Inspect resources<br>kubectl get pods -n namespace -o wide<br>kubectl describe pod pod-name<br>kubectl logs pod-name -c container-name --previous<br>kubectl exec -it pod-name -- /bin/sh<br><br># Apply/manage<br>kubectl apply -f file.yaml<br>kubectl delete -f file.yaml<br>kubectl scale deployment my-app --replicas=5<br><br># Rollouts<br>kubectl rollout status deployment/my-app<br>kubectl rollout history deployment/my-app<br>kubectl rollout undo deployment/my-app<br><br># Debugging<br>kubectl get events --sort-by=.metadata.creationTimestamp<br>kubectl top pods<br>kubectl top nodes<br>kubectl port-forward pod/my-pod 8080:8080<br>kubectl cp pod-name:/path/to/file ./local-file<br><br># Context switching<br>kubectl config get-contexts<br>kubectl config use-context prod-cluster</code><br><br><b>Senior insight:</b> Master <code>kubectl get -o jsonpath</code> and <code>-o custom-columns</code> for scripting. Use <code>kubectx</code> and <code>kubens</code> tools for faster context/namespace switching. Never run <code>kubectl delete pod --all</code> in production without being absolutely sure of the namespace.</div>

What is a Kubernetes Context and how do you manage multiple clusters?	<div style="text-align:left">A <b>kubeconfig context</b> combines a cluster, a user, and a namespace into a named configuration — letting you switch between different clusters with a single command.<br><br>The kubeconfig file (<code>~/.kube/config</code>) stores:<br>- <b>clusters:</b> API server addresses and CA certs<br>- <b>users:</b> Credentials (certs, tokens, OIDC)<br>- <b>contexts:</b> Named combinations of cluster + user + namespace<br><br><code># View all contexts<br>kubectl config get-contexts<br><br># Switch context<br>kubectl config use-context production<br><br># Set default namespace for a context<br>kubectl config set-context --current --namespace=my-team<br><br># Merge multiple kubeconfig files<br>export KUBECONFIG=~/.kube/config:~/.kube/prod-config<br>kubectl config view --merge --flatten > ~/.kube/merged-config</code><br><br><b>Tools for multi-cluster management:</b><br>- <code>kubectx</code>: Fast context switching (<code>kubectx prod</code>)<br>- <code>kubens</code>: Fast namespace switching (<code>kubens my-app</code>)<br>- <code>k9s</code>: Terminal UI for K8s<br><br><b>Senior insight:</b> Store per-cluster kubeconfig files separately and merge them. Never store credentials in plain text in shared systems. Use OIDC/SSO-based authentication for team access.</div>

What is Kubernetes Federation and multi-cluster management?	<div style="text-align:left">As organizations grow, they run multiple Kubernetes clusters across regions, clouds, or environments. Managing these is called <b>multi-cluster management</b>.<br><br><b>Why multiple clusters?</b><br>- Regulatory isolation (EU data must stay in EU)<br>- Blast radius reduction (one cluster's failure doesn't affect others)<br>- Different environments (dev/staging/prod)<br>- Multi-cloud strategy<br>- Geographic distribution / latency<br><br><b>Challenges:</b><br>- Consistent policy enforcement across clusters<br>- Workload placement and failover<br>- Unified observability<br>- Secret/config synchronization<br><br><b>Tools:</b><br>- <b>Cluster API:</b> Declarative cluster lifecycle management (provision clusters with K8s manifests)<br>- <b>ArgoCD/Flux:</b> Multi-cluster GitOps deployment<br>- <b>Rancher/Anthos/Azure Arc:</b> Commercial multi-cluster management platforms<br>- <b>Submariner:</b> Cross-cluster network connectivity<br>- <b>Karmada:</b> Multi-cluster workload distribution<br><br><b>Senior insight:</b> Kubernetes Federation v2 was abandoned. The community moved toward GitOps-based multi-cluster management. ArgoCD ApplicationSets is the modern standard for deploying the same app to multiple clusters simultaneously.</div>

What is the difference between a headless service and a regular ClusterIP service?	<div style="text-align:left">A <b>regular ClusterIP Service</b> has a single stable virtual IP. DNS resolves to this single IP, and kube-proxy load-balances traffic to individual Pods.<br><br>A <b>Headless Service</b> has <code>clusterIP: None</code> — no virtual IP is allocated. Instead, DNS resolves <b>directly to the individual Pod IPs</b>.<br><br><code>spec:<br>  clusterIP: None   # headless<br>  selector:<br>    app: my-db</code><br><br><b>DNS behavior:</b><br>- Regular: <code>my-service</code> → <code>10.96.0.1</code> (single VIP)<br>- Headless: <code>my-service</code> → <code>10.0.1.5, 10.0.1.6, 10.0.1.7</code> (all Pod IPs)<br><br>StatefulSets also get individual Pod DNS records:<br><code>my-db-0.my-db-svc.namespace.svc.cluster.local</code><br><br><b>Use cases for headless services:</b><br>- StatefulSets (each Pod has its own identity — Postgres primary vs replica)<br>- Client-side load balancing (gRPC, Cassandra drivers)<br>- Service discovery where the client wants to know all endpoints<br><br><b>Senior insight:</b> gRPC doesn't work well with ClusterIP services because HTTP/2 multiplexes connections — all requests go to one Pod. Use headless service + client-side load balancing or a service mesh for gRPC.</div>

What is Kubernetes node drain and cordon?	<div style="text-align:left"><b>kubectl cordon node-name:</b> Marks a node as <b>unschedulable</b> — no new Pods will be placed on it. Existing Pods continue running.<br><br>Use when: You want to prevent new scheduling without disrupting existing workloads (e.g., debugging a node issue).<br><br><b>kubectl drain node-name:</b> Cordons the node AND evicts all running Pods to other nodes — empties the node for maintenance (OS upgrade, hardware replacement).<br><br><code>kubectl cordon node-1<br>kubectl drain node-1 \<br>  --ignore-daemonsets \<br>  --delete-emptydir-data \<br>  --force</code><br><br><b>What drain does:</b><br>1. Cordons the node (no new pods)<br>2. Evicts Pods one by one, respecting PodDisruptionBudgets<br>3. Waits for Pods to be rescheduled elsewhere<br><br><b>Flags needed for drain:</b><br>- <code>--ignore-daemonsets</code>: DaemonSet Pods can't be rescheduled, so ignore them<br>- <code>--delete-emptydir-data</code>: Allow eviction of Pods using emptyDir volumes (data will be lost)<br><br><b>Senior insight:</b> Always drain nodes before maintenance — never just reboot. A PDB with <code>minAvailable: 1</code> will cause drain to block indefinitely if only 1 replica is running and it can't be evicted. Scale up first, then drain.</div>

What is the Kubernetes Gateway API?	<div style="text-align:left">The <b>Gateway API</b> is the next-generation successor to Ingress, designed to solve its limitations and provide richer traffic management capabilities for Kubernetes.<br><br><b>Problems with Ingress:</b><br>- Limited to HTTP/HTTPS<br>- Non-standard annotations for advanced features (timeouts, retries vary by controller)<br>- Single resource type (no separation of concerns between infra and app teams)<br><br><b>Gateway API introduces role-oriented resources:</b><br>- <b>GatewayClass:</b> Defines the type of gateway (infrastructure provider configures this)<br>- <b>Gateway:</b> Configures a listener (port, protocol, TLS) — cluster ops team manages this<br>- <b>HTTPRoute:</b> Configures routing rules — application teams manage this<br>- <b>TCPRoute, TLSRoute, GRPCRoute:</b> Support non-HTTP protocols<br><br><code>kind: HTTPRoute<br>spec:<br>  parentRefs:<br>  - name: my-gateway<br>  rules:<br>  - matches:<br>    - path: {type: PathPrefix, value: /api}<br>    backendRefs:<br>    - name: api-service<br>      port: 80</code><br><br><b>Senior insight:</b> Gateway API is now GA (stable) in K8s. Major controllers (nginx, Traefik, Istio, AWS ALB) support it. For new clusters, prefer Gateway API over Ingress — it's more expressive, portable, and officially supported going forward.</div>

What is eBPF and how is it changing Kubernetes networking?	<div style="text-align:left"><b>eBPF (extended Berkeley Packet Filter)</b> is a Linux kernel technology that allows running sandboxed programs inside the kernel — without changing kernel source or loading kernel modules.<br><br>In Kubernetes, eBPF is revolutionizing networking and observability:<br><br><b>Traditional K8s networking:</b> kube-proxy uses iptables rules — at 10,000+ services, iptables becomes a bottleneck (O(n) rule matching, hard to debug)<br><br><b>eBPF-based networking (Cilium):</b><br>- Replaces kube-proxy entirely with eBPF programs that run in the kernel<br>- O(1) service lookup regardless of service count<br>- Native pod-to-pod encryption without a sidecar proxy<br>- Kernel-level observability (no sidecar injection needed)<br>- Better performance: 2-3x lower latency at high load<br><br><b>eBPF tools in K8s ecosystem:</b><br>- <b>Cilium:</b> CNI + service mesh replacement (Hubble for observability)<br>- <b>Falco:</b> Runtime security using eBPF (syscall monitoring)<br>- <b>Pixie:</b> Zero-instrumentation observability<br><br><b>Senior insight:</b> Cilium with eBPF is the leading choice for high-performance K8s clusters (10,000+ nodes, Google and Meta use it). It's now the default CNI for Amazon EKS and Google GKE Autopilot. It replaces kube-proxy AND a service mesh's data plane — a significant consolidation.</div>

What is the Container Network Interface (CNI)?	<div style="text-align:left">The <b>CNI (Container Network Interface)</b> is a specification and set of libraries for configuring network interfaces in Linux containers. It defines how the container runtime sets up networking when a container starts.<br><br>In Kubernetes, a CNI plugin is responsible for:<br>1. Assigning IP addresses to Pods<br>2. Configuring routing so Pods can communicate across nodes<br>3. Enforcing NetworkPolicies<br><br><b>K8s networking model requirements:</b><br>- Every Pod gets a unique IP<br>- Pods can communicate with any other Pod without NAT<br>- Nodes can communicate with any Pod without NAT<br><br><b>Popular CNI plugins:</b><br>- <b>Flannel:</b> Simple overlay network, doesn't support NetworkPolicies<br>- <b>Calico:</b> BGP-based, supports NetworkPolicies, widely used<br>- <b>Cilium:</b> eBPF-based, best performance, full NetworkPolicy + service mesh<br>- <b>Weave:</b> Simple, supports NetworkPolicies<br>- <b>AWS VPC CNI:</b> Uses actual VPC IPs for Pods (native AWS routing)<br><br><b>Senior insight:</b> CNI choice affects performance, scalability, and feature availability significantly. AWS VPC CNI gives native routing (no overlay) but has IP address limitations per node. Calico or Cilium are better for on-prem or multi-cloud. You can't easily change CNI after cluster creation without rebuilding.</div>

What is Kubernetes cert-manager and how does TLS management work?	<div style="text-align:left"><b>cert-manager</b> is a Kubernetes Operator that automates the issuance and renewal of TLS certificates. It integrates with Let's Encrypt, HashiCorp Vault, Venafi, and self-signed CAs.<br><br><b>Key CRDs it adds:</b><br>- <b>Issuer/ClusterIssuer:</b> Defines how to obtain certs (ACME/Let's Encrypt, self-signed, Vault)<br>- <b>Certificate:</b> Declares the desired certificate<br>- <b>CertificateRequest:</b> Internal — cert-manager creates these automatically<br><br><code>apiVersion: cert-manager.io/v1<br>kind: Certificate<br>spec:<br>  secretName: my-app-tls<br>  dnsNames: ["app.example.com"]<br>  issuerRef:<br>    name: letsencrypt-prod<br>    kind: ClusterIssuer</code><br><br>cert-manager creates a Kubernetes Secret containing the cert and key, then automatically renews it before expiry.<br><br><b>ACME challenge types:</b><br>- <b>HTTP-01:</b> Serves a file over HTTP — requires public internet access<br>- <b>DNS-01:</b> Creates a DNS TXT record — works for internal/wildcard certs<br><br><b>Senior insight:</b> Annotate your Ingress to have cert-manager automatically issue certs:<br><code>cert-manager.io/cluster-issuer: letsencrypt-prod</code><br>cert-manager handles the entire certificate lifecycle. Never manage TLS certs manually in K8s — it's error-prone and cert-manager handles renewal 30 days before expiry automatically.</div>

What is Kubernetes resource garbage collection?	<div style="text-align:left">Kubernetes automatically cleans up objects whose owner has been deleted — this is called <b>garbage collection</b>. It uses <b>owner references</b> to track parent-child relationships.<br><br>Every Kubernetes object can have <code>ownerReferences</code> pointing to its owner:<br>- A Pod has its ReplicaSet as owner<br>- A ReplicaSet has its Deployment as owner<br>- When you delete a Deployment → it cascades: Deployment → ReplicaSet → Pods<br><br><b>Deletion propagation policies:</b><br>- <b>Foreground:</b> Owner waits until all dependents are deleted (safe, slower)<br>- <b>Background:</b> Owner deleted immediately; garbage collector cleans dependents async (default)<br>- <b>Orphan:</b> Delete owner but keep dependents (set their ownerReferences to nil)<br><br><code># Delete deployment but keep pods running (orphan)<br>kubectl delete deployment my-app \<br>  --cascade=orphan</code><br><br><b>Other GC responsibilities:</b><br>- Cleaning up completed Jobs and their Pods (<code>ttlSecondsAfterFinished</code>)<br>- Removing unused container images from nodes (kubelet's image GC)<br>- Removing terminated Pods that exceed <code>terminated-pod-gc-threshold</code><br><br><b>Senior insight:</b> Orphaning is useful when migrating resources between owners or renaming Deployments without restarting Pods. Understanding cascade deletion prevents accidental data loss when deleting parent objects.</div>

What is the Kubernetes Vertical Scaling challenge and how does it relate to Pod restarts?	<div style="text-align:left">Unlike horizontal scaling (adding replicas), <b>vertical scaling</b> in Kubernetes (changing CPU/memory limits) traditionally requires restarting the Pod — because container resources are set at container creation time by the Linux kernel's cgroups.<br><br><b>Traditional flow with VPA Auto mode:</b><br>1. VPA recommends higher memory limit<br>2. VPA evicts the Pod<br>3. New Pod starts with updated resource limits<br>4. This causes a brief disruption<br><br><b>In-Place Pod Resource Resize (K8s 1.27+ beta):</b><br>- Allows changing CPU and memory limits <b>without restarting the container</b><br>- Supported for CPU (always resizable) and memory (resizable if new limit > old limit)<br>- Memory downsizing still requires restart<br><br><code>spec:<br>  containers:<br>  - name: app<br>    resources:<br>      requests:<br>        memory: "512Mi"  # can be changed in-place<br>      limits:<br>        memory: "512Mi"</code><br><br><b>Senior insight:</b> Until in-place resize is stable, the practical approach for production is: combine HPA (handles traffic spikes) + VPA in <code>Off</code> mode (gives recommendations) + periodic right-sizing during maintenance windows. Avoid VPA Auto on latency-sensitive services.</div>

What is the difference between Kubernetes Ingress and a Load Balancer Service?	<div style="text-align:left">Both expose services to external traffic, but they work at different levels:<br><br><b>LoadBalancer Service:</b><br>- Works at <b>Layer 4</b> (TCP/UDP) — no HTTP awareness<br>- Creates one external cloud load balancer per Service<br>- Each gets its own public IP<br>- Cost: 1 LB = ~$15-25/month on AWS/GCP<br>- Good for: non-HTTP services, raw TCP, databases (with allowance)<br><br><b>Ingress:</b><br>- Works at <b>Layer 7</b> (HTTP/HTTPS) — aware of hostnames and URL paths<br>- One Ingress Controller (one LB) routes to many Services<br>- Cost-efficient: many apps share one LB<br>- Features: SSL termination, host/path routing, redirects, rewrites<br><br><b>Economics at scale:</b><br>Without Ingress: 50 HTTP microservices = 50 cloud LBs = ~$750-1250/month<br>With Ingress: 50 HTTP microservices = 1 cloud LB + Ingress Controller = ~$15/month<br><br><b>Senior insight:</b> Always use Ingress for HTTP workloads. Use LoadBalancer Services only for non-HTTP services (TCP databases, game servers, MQTT brokers) or when Layer 4 termination is required. Consider Gateway API as the evolution beyond Ingress for complex routing requirements.</div>

What is the Kubernetes Topology Spread Constraint?	<div style="text-align:left"><b>Topology Spread Constraints</b> tell the scheduler how to distribute Pods across topology domains (nodes, zones, regions) — ensuring your app is spread for high availability and performance.<br><br><code>spec:<br>  topologySpreadConstraints:<br>  - maxSkew: 1<br>    topologyKey: topology.kubernetes.io/zone<br>    whenUnsatisfiable: DoNotSchedule<br>    labelSelector:<br>      matchLabels:<br>        app: my-app</code><br><br><b>Fields:</b><br>- <code>topologyKey</code>: The node label to group by (<code>hostname</code>, <code>zone</code>, <code>region</code>)<br>- <code>maxSkew</code>: Max allowed difference in Pod count between any two topology groups<br>- <code>whenUnsatisfiable</code>: What to do if constraint can't be met: <code>DoNotSchedule</code> (hard) or <code>ScheduleAnyway</code> (soft)<br><br><b>Example:</b> 6 Pods across 3 zones with maxSkew: 1 → at most 2 Pods per zone, max 1 difference between zones.<br><br><b>Senior insight:</b> Topology Spread Constraints replaced the complex Pod Anti-Affinity approach for zone spreading. Use them instead of Pod Anti-Affinity with topologyKey for zone distribution — they're more predictable and work better with Cluster Autoscaler. Always spread across zones for HA — a zone failure should take down at most 1/3 of your capacity.</div>

What is Kubernetes Pod Priority and Preemption?	<div style="text-align:left"><b>Pod Priority</b> assigns numerical priority to Pods — higher priority Pods are scheduled before lower priority ones, and can <b>preempt</b> (evict) lower priority Pods to free up resources.<br><br><code>apiVersion: scheduling.k8s.io/v1<br>kind: PriorityClass<br>metadata:<br>  name: high-priority<br>value: 1000000<br>globalDefault: false<br>preemptionPolicy: PreemptLowerPriority<br>---<br>spec:<br>  priorityClassName: high-priority</code><br><br><b>How preemption works:</b><br>1. High-priority Pod is Pending (no room)<br>2. Scheduler finds a node where evicting low-priority Pods would make room<br>3. Low-priority Pods are gracefully terminated<br>4. High-priority Pod is scheduled<br><br><b>Built-in priority classes:</b><br>- <code>system-cluster-critical</code>: DNS, metrics-server (value: 2000000000)<br>- <code>system-node-critical</code>: kubelet, etcd (value: 2000001000)<br><br><b>Senior insight:</b> Use priority classes to protect critical services. Without them, a deployment storm of low-priority batch Pods can evict your production API Pods. Common pattern: <code>critical</code> (production API) > <code>default</code> (regular apps) > <code>batch</code> (background jobs). PDB + Priority together form the complete resource protection strategy.</div>

What are common Kubernetes security best practices for production?	<div style="text-align:left">A checklist of critical K8s security practices for production:<br><br><b>Image security:</b><br>- Use minimal base images (distroless, alpine)<br>- Scan images for vulnerabilities (Trivy, Snyk, Grype)<br>- Use image digests, not mutable tags<br>- Sign images (Sigstore/cosign) and verify in admission webhooks<br><br><b>Runtime security:</b><br>- Run containers as non-root: <code>runAsNonRoot: true, runAsUser: 1000</code><br>- Read-only root filesystem: <code>readOnlyRootFilesystem: true</code><br>- Drop all Linux capabilities: <code>capabilities: {drop: ["ALL"]}</code><br>- Disable privilege escalation: <code>allowPrivilegeEscalation: false</code><br>- Use Pod Security Admission (restricted profile)<br><br><b>Network security:</b><br>- Apply default-deny NetworkPolicies<br>- Restrict egress as well as ingress<br>- Use mTLS (service mesh) for service-to-service<br><br><b>Access control:</b><br>- Principle of least privilege for all ServiceAccounts<br>- Disable automount of service account tokens<br>- Use OIDC for user authentication, not static kubeconfig<br>- Audit RBAC regularly<br><br><b>Supply chain:</b><br>- Keep K8s version up to date (within 2 minor versions of latest)<br>- Encrypt etcd at rest<br>- Enable audit logging<br><br><b>Senior insight:</b> Start with Pod Security Admission enforced at <code>baseline</code> cluster-wide, then move specific namespaces to <code>restricted</code>. Add OPA/Gatekeeper or Kyverno for policies that PSA can't enforce (image registry allowlisting, label requirements, etc.).</div>

What is Kubernetes audit logging?	<div style="text-align:left"><b>Audit logging</b> records all requests to the Kubernetes API server — who did what, when, and what the response was. It's essential for security, compliance, and forensic investigation.<br><br><b>Audit policy levels:</b><br>- <code>None</code>: Don't log<br>- <code>Metadata</code>: Log request metadata (user, verb, resource) but not body<br>- <code>Request</code>: Log metadata + request body<br>- <code>RequestResponse</code>: Log metadata + request + response body (most verbose)<br><br><b>Example audit policy (partial):</b><br><code>rules:<br>- level: None<br>  users: ["system:kube-proxy"]<br>  verbs: ["watch"]<br>- level: Metadata<br>  resources:<br>  - group: ""<br>    resources: ["secrets"]<br>- level: RequestResponse<br>  verbs: ["create", "delete"]<br>  resources:<br>  - group: ""<br>    resources: ["pods"]</code><br><br><b>Senior insight:</b> Always audit Secret access (even Metadata level) — this helps detect credential theft. Ship audit logs to a SIEM (Splunk, ELK, Datadog) immediately — API server audit logs rotate and aren't persisted long. Critical events to alert on: secret reads by unexpected service accounts, role escalation, exec into containers, node changes.</div>

What is Kubernetes Operator Framework and how do you build an Operator?	<div style="text-align:left">The <b>Operator pattern</b> automates the management of complex, stateful applications by extending Kubernetes with application-specific knowledge.<br><br><b>Building blocks:</b><br>1. <b>CRD:</b> Define your custom resource (e.g., <code>MySQLCluster</code>)<br>2. <b>Controller:</b> A reconciliation loop that watches your CRD and acts<br>3. <b>RBAC:</b> Permissions for the controller to manage K8s resources<br><br><b>Popular frameworks:</b><br>- <b>Operator SDK</b> (Go, Ansible, Helm): From the Red Hat/K8s community<br>- <b>Kubebuilder:</b> Google-backed, pure Go, generates boilerplate<br>- <b>Kopf</b> (Python): Simple Python Operator framework<br><br><b>Reconcile loop pattern (Go):</b><br><code>func (r *MySQLClusterReconciler) Reconcile(ctx context.Context,<br>    req ctrl.Request) (ctrl.Result, error) {<br>  // 1. Fetch the MySQLCluster resource<br>  // 2. Check if Deployment exists, create if not<br>  // 3. Check if config matches, update if not<br>  // 4. Update status subresource<br>  return ctrl.Result{}, nil<br>}</code><br><br><b>Operator Maturity Model levels:</b><br>Level 1: Basic install → Level 2: Upgrades → Level 3: Backup/recovery → Level 4: Deep insights → Level 5: Auto-pilot<br><br><b>Senior insight:</b> Write idempotent reconcilers — they run repeatedly. Never assume one-shot execution. Use <code>ctrl.Result{RequeueAfter: time.Minute}</code> to re-check state periodically.</div>

What is Kubernetes Pod Affinity vs Topology Spread Constraints — when to use each?	<div style="text-align:left">Both control Pod placement relative to other Pods, but with different goals and trade-offs:<br><br><b>Pod Anti-Affinity:</b><br>- Hard/soft rule: "Don't place this Pod on the same node/zone as another Pod with label X"<br>- Binary: either satisfied or not<br>- Useful for: ensuring no two replicas share a single node (simple HA requirement)<br>- Limitation: with many replicas, becomes hard to satisfy; doesn't balance evenly<br><br><code>podAntiAffinity:<br>  requiredDuringSchedulingIgnoredDuringExecution:<br>  - labelSelector:<br>      matchLabels: {app: my-app}<br>    topologyKey: kubernetes.io/hostname</code><br><br><b>Topology Spread Constraints:</b><br>- Distribution rule: "Spread Pods evenly across zones, allow max 1 skew"<br>- Gradient: optimizes for balance, not binary<br>- Useful for: multi-zone HA with many replicas, ensuring even load distribution<br>- Better for: large deployments where affinity rules become too restrictive<br><br><b>When to use which:</b><br>- 2-5 replicas, simple HA: Pod Anti-Affinity per hostname<br>- 6+ replicas, zone-based HA: Topology Spread Constraints<br>- Co-location (keep near another service): Pod Affinity<br><br><b>Senior insight:</b> You can combine them. Common pattern: Anti-Affinity to prevent same-node placement + Topology Spread Constraint for zone balancing. Topology Spread Constraints are the future — they're more flexible and composable.</div>

What are common Kubernetes performance bottlenecks and how do you troubleshoot them?	<div style="text-align:left">Key K8s performance issues and how to diagnose them:<br><br><b>1. API Server overload:</b><br>- Symptom: kubectl slow, controllers lagging, events delayed<br>- Cause: Too many controllers doing frequent List/Watch, large objects, too many CRDs<br>- Fix: Use informers with caches, reduce watch frequency, tune API priority and fairness<br><br><b>2. etcd latency:</b><br>- Symptom: API server slow, timeouts in controllers<br>- Cause: Disk I/O contention, etcd compaction lag, fragmented DB<br>- Fix: Dedicated NVMe disks, tune compaction, defrag etcd periodically<br><br><b>3. DNS resolution bottleneck:</b><br>- Symptom: Intermittent connection failures, high latency on first request<br>- Cause: CoreDNS overloaded, ndots:5 causing 5 DNS lookups per request<br>- Fix: Scale CoreDNS, enable NodeLocal DNSCache, use FQDN in service URLs<br><br><b>4. kube-proxy iptables lag:</b><br>- Symptom: New Service endpoints not routing correctly at scale<br>- Cause: iptables with 10,000+ rules is slow to update<br>- Fix: Switch to IPVS mode or use Cilium (eBPF)<br><br><b>5. Image pull latency:</b><br>- Symptom: Pods slow to start, especially on new nodes<br>- Fix: Use image caching (Karpenter pre-fetching), reduce image size, use pull-through cache (Harbor, ECR mirror)<br><br><b>Senior insight:</b> Always use <code>kubectl top</code> and node-exporter metrics first. The four golden signals (latency, traffic, errors, saturation) applied to cluster components give you the full picture.</div>
