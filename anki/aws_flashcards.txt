What is an AWS Region?	A <b>Region</b> is a physical geographic area (e.g., <code>us-east-1</code>, <code>eu-west-1</code>) that contains multiple isolated data centers called Availability Zones.<br><br>Think of it like a city: the city is the Region, and the neighborhoods are the AZs.<br><br><b>Key points:</b><br>- Each Region is completely independent<br>- Data does NOT automatically replicate across Regions<br>- You choose a Region based on latency, compliance, and cost<br>- Not all services are available in all Regions<br><br><b>Senior insight:</b> Always design for multi-Region when RPO/RTO requirements are strict. Regional failures are rare but happen.
What is an AWS Availability Zone (AZ)?	An <b>Availability Zone</b> is one or more discrete data centers within a Region, each with redundant power, networking, and connectivity.<br><br>Think of AZs like separate buildings in the same city — if one burns down, the others still work.<br><br><b>Key points:</b><br>- A Region has 2–6 AZs (usually 3)<br>- AZs are connected with low-latency fiber links<br>- Deploying across multiple AZs = high availability<br>- AZ names like <code>us-east-1a</code> are account-specific (your 1a may be a different physical AZ than mine)<br><br><b>Common mistake:</b> Assuming single-AZ deployment is "good enough" — always spread across at least 2 AZs.
What is an AWS Edge Location?	<b>Edge Locations</b> are mini data centers distributed worldwide used by services like <b>CloudFront</b> (CDN) and <b>Route 53</b> to serve content closer to end users.<br><br>Think of them as local delivery hubs — instead of shipping from a warehouse (Region), the package is already at a nearby depot (Edge Location).<br><br><b>Key points:</b><br>- There are 400+ Edge Locations globally<br>- Used for caching static assets, DNS resolution, DDoS protection (Shield)<br>- NOT the same as Regions or AZs — you can't run EC2 there<br><br><b>Senior insight:</b> CloudFront + S3 is one of the cheapest and most scalable ways to serve static websites globally.
What is IAM in AWS?	<b>IAM (Identity and Access Management)</b> controls who can do what in your AWS account.<br><br>Think of it as the security guard + key card system for your entire AWS building.<br><br><b>Core concepts:</b><br>- <b>Users</b>: individual people or applications<br>- <b>Groups</b>: collections of users sharing the same permissions<br>- <b>Roles</b>: temporary permissions assumed by services or users<br>- <b>Policies</b>: JSON documents that define allowed/denied actions<br><br><b>Example policy (allow S3 read):</b><br><code>{"Effect":"Allow","Action":["s3:GetObject"],"Resource":"arn:aws:s3:::my-bucket/*"}</code><br><br><b>Senior insight:</b> Never use root account for day-to-day work. Apply least-privilege principle. Prefer roles over long-term access keys.
What is the difference between IAM Users, Groups, and Roles?	<b>User:</b> A permanent identity for a person or application. Has long-term credentials (password + access keys).<br><br><b>Group:</b> A collection of users. Assign policies to the group, not individual users. Easier to manage at scale.<br><br><b>Role:</b> Temporary credentials. Used by AWS services (e.g., EC2 calling S3), cross-account access, or federated users.<br><br><b>Analogy:</b><br>- User = employee badge<br>- Group = department (all devs get the same access)<br>- Role = visitor pass (temporary, for a specific purpose)<br><br><b>Common mistake:</b> Hardcoding access keys in code instead of using IAM Roles. Keys can leak; roles rotate automatically.
What is an IAM Policy? What types exist?	An <b>IAM Policy</b> is a JSON document that defines permissions — what actions are allowed or denied on which resources.<br><br><b>Types:</b><br>- <b>Identity-based policies</b>: attached to users, groups, or roles<br>- <b>Resource-based policies</b>: attached directly to a resource (e.g., S3 bucket policy)<br>- <b>Permission boundaries</b>: set the maximum permissions an identity can have<br>- <b>Service Control Policies (SCPs)</b>: org-wide guardrails in AWS Organizations<br>- <b>Session policies</b>: temporary, passed when assuming a role<br><br><b>Evaluation logic:</b> Default DENY → explicit ALLOW → explicit DENY wins always<br><br><b>Senior insight:</b> An explicit <code>Deny</code> always overrides an <code>Allow</code>. Use SCPs to prevent even admins from doing dangerous things like disabling CloudTrail.
What is the AWS Shared Responsibility Model?	AWS and you share security responsibilities — the split depends on the service type.<br><br><b>AWS is responsible for:</b> Security <i>of</i> the cloud — physical hardware, data centers, network infrastructure, hypervisor.<br><br><b>You are responsible for:</b> Security <i>in</i> the cloud — OS patching, firewall rules, IAM policies, data encryption, application security.<br><br><b>Example:</b><br>- EC2: YOU patch the OS<br>- RDS: AWS patches the DB engine; YOU manage access and encryption<br>- Lambda: AWS manages everything below your code; YOU secure your code<br><br><b>Senior insight:</b> The more managed the service (Lambda > RDS > EC2), the more AWS takes on — but you never fully outsource security.
What is a VPC in AWS?	A <b>VPC (Virtual Private Cloud)</b> is your own isolated, private network within AWS — like building your own virtual office network in the cloud.<br><br><b>Key components:</b><br>- <b>Subnets</b>: segments of the VPC IP range, tied to an AZ<br>- <b>Route Tables</b>: control where traffic goes<br>- <b>Internet Gateway (IGW)</b>: connects the VPC to the internet<br>- <b>NAT Gateway</b>: lets private subnets reach the internet without being reachable from it<br>- <b>Security Groups</b>: stateful firewalls per resource<br>- <b>NACLs</b>: stateless firewalls per subnet<br><br><b>Default VPC:</b> AWS creates one per Region for you. Fine for testing, but use a custom VPC for production.<br><br><b>Senior insight:</b> Always use dedicated VPCs per environment (dev/staging/prod). Use CIDR planning to avoid overlap when using VPC Peering.
What is the difference between a Public and Private Subnet?	<b>Public Subnet:</b> Has a route to the Internet Gateway. Resources here can be reached from the internet (if they have a public IP).<br><br><b>Private Subnet:</b> No direct route to the internet. Resources here are NOT directly accessible from outside. They access the internet through a NAT Gateway.<br><br><b>Typical 3-tier architecture:</b><br>- Public subnet: Load Balancers<br>- Private subnet: App servers (EC2, ECS)<br>- Private subnet: Databases (RDS)<br><br><b>Common mistake:</b> Putting databases or app servers in public subnets "for convenience." This is a major security risk.<br><br><b>Senior insight:</b> Even if something is in a private subnet, you still need Security Groups to control inter-service communication.
What is the difference between Security Groups and NACLs?	Both control traffic in a VPC, but they work differently.<br><br><b>Security Groups (SGs):</b><br>- Applied to individual resources (EC2, RDS, etc.)<br>- <b>Stateful</b>: if you allow inbound, the response is automatically allowed outbound<br>- Only ALLOW rules (no explicit deny)<br>- Evaluated together — all rules apply<br><br><b>NACLs (Network ACLs):</b><br>- Applied to subnets<br>- <b>Stateless</b>: you must allow both inbound AND outbound explicitly<br>- Support ALLOW and DENY rules<br>- Rules evaluated in order by rule number — first match wins<br><br><b>Analogy:</b><br>- SG = doorman at an apartment (knows who lives there)<br>- NACL = security checkpoint at the building entrance (checks everyone, forgets them instantly)<br><br><b>Senior insight:</b> Use SGs for most traffic control. Use NACLs only for broad subnet-level blocks (e.g., blocking a malicious IP range).
What is a NAT Gateway?	A <b>NAT Gateway</b> allows resources in a <b>private subnet</b> to initiate outbound connections to the internet, while preventing the internet from initiating connections back.<br><br><b>Use case:</b> Your app server needs to download a package or call an external API, but you don't want it directly exposed to the internet.<br><br><b>Flow:</b> Private EC2 → NAT Gateway (in public subnet) → Internet Gateway → Internet<br><br><b>Key points:</b><br>- NAT Gateway is managed by AWS (no patching needed)<br>- Charged per hour + per GB of data processed<br>- Must be in a public subnet<br>- Not free-tier eligible<br><br><b>Common mistake:</b> Creating one NAT Gateway and routing all AZs through it — this creates a single point of failure. Use one NAT Gateway per AZ for HA.<br><br><b>Senior insight:</b> NAT Gateway costs add up quickly at high data volumes. Consider VPC Endpoints for S3/DynamoDB to avoid NAT costs entirely.
What is a VPC Endpoint?	A <b>VPC Endpoint</b> lets your VPC resources privately connect to AWS services without going through the public internet, NAT Gateway, or IGW.<br><br><b>Types:</b><br>- <b>Gateway Endpoint</b>: for S3 and DynamoDB — free, added to route tables<br>- <b>Interface Endpoint (PrivateLink)</b>: for most other AWS services — creates an ENI in your subnet, costs money<br><br><b>Analogy:</b> Instead of leaving the building to pick up your mail, there's an internal mailroom — faster, more private, no external traffic.<br><br><b>Benefits:</b><br>- Improved security (traffic never leaves AWS network)<br>- Reduced NAT Gateway costs<br>- Lower latency<br><br><b>Senior insight:</b> Always use Gateway Endpoints for S3 and DynamoDB in production — they're free and eliminate unnecessary NAT traffic.
What is Amazon EC2?	<b>EC2 (Elastic Compute Cloud)</b> provides resizable virtual machines in the cloud — you pick the OS, CPU, RAM, and storage.<br><br><b>Key concepts:</b><br>- <b>Instance types</b>: define CPU/memory ratio (t3, m5, c5, r5, etc.)<br>- <b>AMI (Amazon Machine Image)</b>: the OS + pre-installed software template<br>- <b>Key pairs</b>: SSH access to Linux instances<br>- <b>User Data</b>: bootstrap script that runs on first launch<br>- <b>Instance metadata</b>: accessible at <code>169.254.169.254</code> — contains IAM credentials, instance ID, etc.<br><br><b>Common mistake:</b> Treating EC2 as a pet (manually configured, hard to replace). Treat them as cattle — disposable, auto-scaled, launched from AMIs.<br><br><b>Senior insight:</b> Use IMDSv2 (token-based) instead of IMDSv1 to prevent SSRF attacks that steal instance metadata.
What are EC2 instance families and when do you use each?	AWS groups EC2 instances by workload type:<br><br><b>General Purpose (t3, m6i):</b> Balanced CPU/memory. Good for web apps, dev environments. T-series are "burstable" — cheap but throttle under sustained load.<br><br><b>Compute Optimized (c6i, c7g):</b> High CPU. Good for batch processing, game servers, scientific modeling.<br><br><b>Memory Optimized (r6i, x2idn):</b> High RAM. Good for in-memory databases, real-time analytics, large caches.<br><br><b>Storage Optimized (i3, d3):</b> High I/O. Good for NoSQL databases, data warehousing, log processing.<br><br><b>Accelerated Computing (p4, inf2, g5):</b> GPUs/FPGAs. Good for ML training, video rendering, HPC.<br><br><b>Common mistake:</b> Using T-series instances for production workloads that need sustained CPU — they run out of CPU credits and throttle hard.<br><br><b>Senior insight:</b> Graviton (arm-based) instances (c7g, m7g) offer ~20% better price-performance for most workloads.
What are EC2 pricing models?	<b>On-Demand:</b> Pay per second/hour, no commitment. Most expensive per unit. Best for unpredictable workloads or testing.<br><br><b>Reserved Instances (RI):</b> 1 or 3-year commitment. Up to 72% discount. Best for steady-state workloads. Comes in Standard (locked), Convertible (flexible), and Scheduled.<br><br><b>Savings Plans:</b> Flexible commitment ($/hour spend). Covers EC2, Lambda, Fargate. Simpler than RIs.<br><br><b>Spot Instances:</b> Bid on unused capacity. Up to 90% discount. AWS can reclaim with 2-minute notice. Best for fault-tolerant, stateless, batch jobs.<br><br><b>Dedicated Hosts:</b> Physical server exclusively for you. Used for licensing compliance (Oracle, Windows Server).<br><br><b>Common mistake:</b> Running everything On-Demand — even committing 1 year on baseline capacity saves significantly.<br><br><b>Senior insight:</b> Use a mix: RIs/Savings Plans for baseline + Spot for burst + On-Demand for overflow. This is the "3-layer strategy."
What is an Auto Scaling Group (ASG)?	An <b>Auto Scaling Group (ASG)</b> automatically adds or removes EC2 instances based on demand — keeping your app available and your costs optimized.<br><br><b>Key components:</b><br>- <b>Launch Template</b>: defines the instance config (AMI, type, SG, user data)<br>- <b>Min/Max/Desired capacity</b>: hard limits and target count<br>- <b>Scaling policies</b>: when to scale in/out<br><br><b>Scaling types:</b><br>- <b>Target Tracking</b>: maintain a metric (e.g., CPU at 60%) — simplest<br>- <b>Step Scaling</b>: scale by fixed amounts at thresholds<br>- <b>Scheduled Scaling</b>: scale at specific times (e.g., 9 AM Mon-Fri)<br>- <b>Predictive Scaling</b>: ML-based forecast<br><br><b>Common mistake:</b> Setting min=0. If all instances terminate, the app goes down and the ASG won't self-heal fast enough.<br><br><b>Senior insight:</b> Always use a minimum of 2 instances across 2 AZs. Pair ASGs with ALBs and use health checks to auto-replace unhealthy instances.
What is Elastic Load Balancing (ELB)? What are the types?	<b>ELB</b> distributes incoming traffic across multiple targets (EC2, containers, IPs) to improve availability and fault tolerance.<br><br><b>Types:</b><br><b>ALB (Application Load Balancer):</b><br>- Layer 7 (HTTP/HTTPS)<br>- Routes by path (<code>/api/*</code>), hostname, headers, query strings<br>- Best for microservices and containerized apps<br>- Supports WebSocket and HTTP/2<br><br><b>NLB (Network Load Balancer):</b><br>- Layer 4 (TCP/UDP/TLS)<br>- Ultra-low latency, handles millions of requests/sec<br>- Preserves source IP<br>- Best for gaming, IoT, financial apps<br><br><b>GWLB (Gateway Load Balancer):</b><br>- Layer 3 — for routing traffic through third-party virtual appliances (firewalls, IDS)<br><br><b>Classic LB (deprecated):</b> Avoid for new deployments.<br><br><b>Senior insight:</b> ALB is the default choice for web apps. NLB when you need TLS passthrough or static IPs. ALB + WAF = standard security combo.
What is Amazon S3?	<b>S3 (Simple Storage Service)</b> is object storage — store any file (objects) in containers called <b>buckets</b>. Infinitely scalable, 11 9's of durability (99.999999999%).<br><br><b>Key concepts:</b><br>- <b>Bucket</b>: globally unique namespace container<br>- <b>Object</b>: a file + metadata, up to 5TB<br>- <b>Key</b>: the object's full path/name in the bucket<br>- <b>Prefix</b>: fake "folder" structure (there are no real folders)<br><br><b>Common operations:</b><br><code>aws s3 cp file.txt s3://my-bucket/folder/file.txt</code><br><code>aws s3 sync ./local-dir s3://my-bucket/</code><br><br><b>Storage classes:</b> Standard, Standard-IA, One Zone-IA, Glacier, Glacier Deep Archive, Intelligent-Tiering<br><br><b>Common mistake:</b> Making buckets public by accident. Since 2023, AWS blocks all public access by default — keep it that way unless it's an intentional static site.<br><br><b>Senior insight:</b> S3 is not a file system. It's a key-value store. "Folders" are just key prefixes. Don't treat it like a directory tree.
What are S3 Storage Classes and when do you use them?	<b>Standard:</b> Frequent access. High availability. Most expensive. Default for active data.<br><br><b>Standard-IA (Infrequent Access):</b> Lower cost, but retrieval fee. Good for backups accessed monthly.<br><br><b>One Zone-IA:</b> Like Standard-IA but stored in 1 AZ only. Cheaper, but data lost if AZ fails. Good for re-creatable data.<br><br><b>Intelligent-Tiering:</b> AWS automatically moves data between tiers based on usage. Good when access patterns are unknown.<br><br><b>Glacier Instant Retrieval:</b> Archive data, millisecond retrieval. Good for data accessed ~quarterly.<br><br><b>Glacier Flexible Retrieval:</b> Minutes to hours retrieval. Good for backups.<br><br><b>Glacier Deep Archive:</b> Cheapest. 12-hour retrieval. Good for 7+ year compliance archives.<br><br><b>Senior insight:</b> Use S3 Lifecycle Policies to automatically transition objects between classes. E.g., move logs to Glacier after 30 days, delete after 1 year.
What is S3 Versioning and why does it matter?	<b>S3 Versioning</b> keeps multiple versions of an object in a bucket. When enabled, every upload creates a new version instead of overwriting.<br><br><b>Benefits:</b><br>- Recover from accidental deletes (delete creates a "delete marker," not a real deletion)<br>- Recover from bad overwrites<br>- Audit history of changes<br><br><b>Key behavior:</b><br>- Once enabled, versioning can be suspended but NOT disabled<br>- Storing multiple versions increases storage costs<br>- Use Lifecycle policies to expire old versions<br><br><b>Common mistake:</b> Enabling versioning without lifecycle policies — storage costs balloon as every overwrite keeps all history.<br><br><b>Senior insight:</b> Always enable versioning on production buckets, especially for Terraform state files (S3 backend). Combine with MFA Delete for critical data to prevent accidental purges.
What is S3 Object Lock and MFA Delete?	<b>S3 Object Lock:</b> Prevents objects from being deleted or overwritten for a fixed retention period or indefinitely. Uses WORM (Write Once, Read Many) model.<br><br><b>Modes:</b><br>- <b>Governance Mode</b>: users with special permissions can override<br>- <b>Compliance Mode</b>: NO ONE can delete, not even root<br><br><b>MFA Delete:</b> Requires MFA authentication to permanently delete versioned objects or change versioning state.<br><br><b>Use cases:</b><br>- Regulatory compliance (SEC, FINRA, HIPAA)<br>- Ransomware protection<br>- Audit log immutability<br><br><b>Senior insight:</b> For compliance workloads, use Compliance Mode + Object Lock. These are the tools that make auditors happy. Glacier Vault Lock is the equivalent for Glacier.
What is S3 Cross-Region Replication (CRR)?	<b>CRR</b> automatically replicates objects from a source bucket to a destination bucket in a different AWS Region.<br><br><b>Requirements:</b><br>- Versioning must be enabled on both buckets<br>- An IAM role with permissions to read source and write destination<br>- Works for new objects only (existing objects need a one-time S3 Batch Operations job)<br><br><b>Use cases:</b><br>- Disaster recovery (secondary Region)<br>- Latency reduction (serve objects from closer Region)<br>- Data sovereignty compliance<br><br><b>Also available: SRR (Same-Region Replication)</b> for log aggregation or cross-account backup within same Region.<br><br><b>Common mistake:</b> Thinking CRR is synchronous. It's asynchronous — there may be replication lag.<br><br><b>Senior insight:</b> CRR does NOT replicate delete markers by default (configurable). Also doesn't replicate objects encrypted with customer-managed keys unless you set up cross-account KMS access.
What is AWS Lambda?	<b>Lambda</b> is serverless compute — you upload code and AWS runs it in response to events, without you managing servers.<br><br><b>How it works:</b> Event → Lambda triggers → your function runs → response returned. AWS handles scaling, patching, infrastructure.<br><br><b>Supported runtimes:</b> Python, Node.js, Java, C#/.NET, Go, Ruby, and custom runtimes.<br><br><b>Key limits:</b><br>- Max execution time: 15 minutes<br>- Memory: 128MB – 10GB (CPU scales with memory)<br>- Deployment package: 50MB zipped, 250MB unzipped (or use Lambda Layers)<br>- Concurrent executions: 1,000 per Region (soft limit, can be raised)<br><br><b>Common mistake:</b> Putting long-running tasks in Lambda. For anything over 15 min, use ECS Fargate, Step Functions, or AWS Batch.<br><br><b>Senior insight:</b> Lambda pricing = requests + duration (GB-seconds). Very cheap at low volume, but can be more expensive than Fargate at very high, sustained throughput.
What is a Lambda Cold Start?	A <b>cold start</b> happens when Lambda has to initialize a new execution environment (container) for your function because there's no warm instance available.<br><br><b>Timeline:</b> Download code → Start container → Initialize runtime → Run your handler<br><br><b>Cold start adds latency</b> — typically 100ms–10s depending on runtime and package size.<br><br><b>Warmest runtimes:</b> Python, Node.js (fast) | Coldest: Java, .NET (slow JVM startup)<br><br><b>Mitigation strategies:</b><br>- <b>Provisioned Concurrency</b>: pre-warm N instances (costs money even when idle)<br>- Use smaller deployment packages<br>- Avoid heavy initialization outside the handler<br>- Use Snap Start for Java (restores from snapshot)<br><br><b>Common mistake:</b> Blaming Lambda for slow p99 latency without checking if it's cold starts. Use X-Ray to identify initialization time.<br><br><b>Senior insight:</b> Cold starts only matter for latency-sensitive APIs. For async/batch workloads, they're irrelevant.
What is Amazon ECS?	<b>ECS (Elastic Container Service)</b> is AWS's managed container orchestration service — it runs Docker containers at scale without you managing Kubernetes.<br><br><b>Key concepts:</b><br>- <b>Task Definition</b>: blueprint (what image, CPU, memory, env vars, ports)<br>- <b>Task</b>: a running instance of a task definition<br>- <b>Service</b>: ensures N tasks are always running, integrates with ALB<br>- <b>Cluster</b>: logical grouping of tasks/services<br><br><b>Launch types:</b><br>- <b>Fargate</b>: serverless — AWS manages the underlying EC2. You pay per task CPU/memory.<br>- <b>EC2</b>: you manage the EC2 instances. More control, better for GPU/custom OS needs.<br><br><b>Common mistake:</b> Choosing EC2 launch type without a good reason. Fargate eliminates node patching and bin-packing complexity.<br><br><b>Senior insight:</b> ECS + Fargate is the sweet spot for most containerized workloads. Only use EKS if you need Kubernetes-specific features or ecosystem (Helm, operators).
What is Amazon EKS?	<b>EKS (Elastic Kubernetes Service)</b> is a managed Kubernetes control plane on AWS — AWS runs the masters, you manage the worker nodes (or use Fargate for those too).<br><br><b>When to use EKS over ECS:</b><br>- Need Kubernetes-native tools (Helm, Kustomize, ArgoCD, Istio)<br>- Already have K8s expertise on the team<br>- Need advanced scheduling (taints, tolerations, affinity)<br>- Multi-cloud or hybrid deployments<br><br><b>Key AWS integrations:</b><br>- <b>IRSA (IAM Roles for Service Accounts)</b>: pods get fine-grained IAM permissions<br>- <b>ALB Ingress Controller</b>: ALB provisioned per Ingress resource<br>- <b>EBS/EFS CSI drivers</b>: persistent volumes<br>- <b>Karpenter</b>: faster, smarter node autoscaling than Cluster Autoscaler<br><br><b>Common mistake:</b> Using EKS just because Kubernetes is familiar — ECS is simpler, cheaper, and sufficient for most use cases.<br><br><b>Senior insight:</b> EKS control plane costs $0.10/hour (~$72/month) per cluster. Don't run multiple clusters just for environment separation — use namespaces + RBAC.
What is Amazon ECR?	<b>ECR (Elastic Container Registry)</b> is AWS's fully managed Docker container registry — like Docker Hub, but private, secure, and integrated with IAM.<br><br><b>Key features:</b><br>- Private and public repositories<br>- Image scanning for vulnerabilities (Enhanced scanning via Inspector)<br>- Lifecycle policies to auto-delete old images<br>- Cross-region and cross-account replication<br>- Integrated with ECS, EKS, Lambda (container images)<br><br><b>Example push:</b><br><code>aws ecr get-login-password | docker login --username AWS --password-stdin 123.dkr.ecr.us-east-1.amazonaws.com</code><br><code>docker push 123.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0</code><br><br><b>Common mistake:</b> Not setting lifecycle policies — old image tags accumulate and you pay for storage you don't need.<br><br><b>Senior insight:</b> Enable ECR image scanning on push. Add it to your CI pipeline gate: fail builds if HIGH/CRITICAL CVEs are found.
What is Amazon RDS?	<b>RDS (Relational Database Service)</b> is a managed relational database — AWS handles provisioning, patching, backups, and failover.<br><br><b>Supported engines:</b> MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, and Aurora.<br><br><b>Key features:</b><br>- Automated backups (up to 35 days retention)<br>- Manual snapshots (kept until you delete them)<br>- Multi-AZ deployments for high availability<br>- Read Replicas for read scaling<br>- Encryption at rest (KMS) and in transit (SSL/TLS)<br><br><b>Common mistake:</b> Using Single-AZ in production. A hardware failure means downtime. Multi-AZ adds a standby that auto-promotes on failure.<br><br><b>Senior insight:</b> Read Replicas ≠ HA. They don't auto-failover. Use Multi-AZ for HA and Read Replicas for scaling reads. Aurora has both built-in and is usually the better choice.
What is Amazon Aurora?	<b>Aurora</b> is AWS's cloud-native relational database, compatible with MySQL and PostgreSQL, but engineered for performance and scale.<br><br><b>Why Aurora is different:</b><br>- Storage auto-scales up to 128TB, in 10GB increments<br>- Storage is replicated 6 ways across 3 AZs (very durable)<br>- Up to 15 Aurora Read Replicas with <10ms replication lag<br>- Failover in ~30 seconds (faster than standard RDS Multi-AZ)<br>- Up to 5x faster than MySQL, 3x faster than PostgreSQL<br><br><b>Aurora Serverless v2:</b> Auto-scales capacity in fine-grained increments. Great for variable or unpredictable workloads.<br><br><b>Global Database:</b> Low-latency reads and DR across Regions with <1 second replication lag.<br><br><b>Common mistake:</b> Choosing standard RDS over Aurora for new projects. Aurora is usually better and competitively priced for most production workloads.<br><br><b>Senior insight:</b> Use Aurora Serverless v2 for dev/staging environments — scales to zero when idle, huge cost savings.
What is Amazon DynamoDB?	<b>DynamoDB</b> is a fully managed, serverless NoSQL database — key-value and document store with single-digit millisecond performance at any scale.<br><br><b>Core concepts:</b><br>- <b>Table</b>: like a table in SQL, but schema-less (except the key)<br>- <b>Primary Key</b>: Partition Key (required) + Sort Key (optional)<br>- <b>Item</b>: like a row — a collection of attributes<br>- <b>Attribute</b>: like a column — can be different per item<br><br><b>Capacity modes:</b><br>- <b>On-Demand</b>: pay per request, auto-scales. Best for unpredictable traffic.<br>- <b>Provisioned</b>: set read/write capacity units (RCU/WCU). Use Auto Scaling.<br><br><b>Common mistake:</b> Designing DynamoDB like a relational DB with lots of tables and joins. Design for access patterns first — denormalization is expected.<br><br><b>Senior insight:</b> DynamoDB is incredibly powerful when designed correctly, but painful when you need to query in ways you didn't plan for. Know your access patterns before you start.
What are DynamoDB GSIs and LSIs?	<b>GSI (Global Secondary Index):</b><br>- A completely different partition key and optional sort key<br>- Allows querying by non-primary-key attributes<br>- Created at any time<br>- Stored separately — has its own capacity and costs<br>- Eventually consistent only<br><br><b>LSI (Local Secondary Index):</b><br>- Same partition key as the table, different sort key<br>- Must be created at table creation time (cannot add later)<br>- Shares capacity with the base table<br>- Can be strongly consistent<br>- Max 5 per table<br><br><b>Analogy:</b><br>The base table is a book index by title. A GSI is like adding an index by author. An LSI is like adding an index by page count within the same title section.<br><br><b>Common mistake:</b> Not planning GSIs before launch — you can't change the schema easily after data is in DynamoDB.<br><br><b>Senior insight:</b> Avoid hot partitions by ensuring GSI partition keys have high cardinality. A GSI with few distinct values will bottleneck.
What is Amazon ElastiCache?	<b>ElastiCache</b> is a managed in-memory caching service — dramatically speeds up applications by storing frequently accessed data in memory instead of hitting a database.<br><br><b>Two engines:</b><br>- <b>Redis</b>: more feature-rich. Supports persistence, pub/sub, sorted sets, TTL, replication, clustering. Best for sessions, leaderboards, queues.<br>- <b>Memcached</b>: simple, fast key-value cache. Multi-threaded. No persistence. Best for simple caching of DB results.<br><br><b>Common use cases:</b><br>- Database query result caching<br>- Session storage<br>- Rate limiting counters<br>- Real-time leaderboards<br><br><b>Common mistake:</b> Caching mutable data without proper invalidation — serving stale data is worse than slow data in some cases.<br><br><b>Senior insight:</b> For new projects, almost always choose Redis over Memcached. The extra features (replication, persistence, data structures) are nearly always useful. ElastiCache Serverless for Redis is now available for truly auto-scaling caches.
What is Amazon SQS?	<b>SQS (Simple Queue Service)</b> is a fully managed message queue — decouples components so they can communicate asynchronously without depending on each other being available simultaneously.<br><br><b>Key concepts:</b><br>- <b>Producer</b>: sends message to queue<br>- <b>Consumer</b>: polls queue, processes message, deletes it<br>- <b>Visibility Timeout</b>: time a message is hidden after being received (prevents double-processing)<br>- <b>DLQ (Dead Letter Queue)</b>: messages that fail repeatedly are moved here<br><br><b>Queue types:</b><br>- <b>Standard</b>: best-effort ordering, at-least-once delivery, very high throughput<br>- <b>FIFO</b>: exactly-once processing, strict ordering, 300 TPS (3,000 with batching)<br><br><b>Common mistake:</b> Not setting up a DLQ. Without it, poison messages loop forever, consuming resources and blocking the queue.<br><br><b>Senior insight:</b> SQS is the backbone of event-driven architecture. Combine SQS + Lambda for auto-scaling consumers. Use Long Polling (20 sec) to reduce costs and empty receive calls.
What is Amazon SNS?	<b>SNS (Simple Notification Service)</b> is a fully managed pub/sub messaging service — one message can fan out to multiple subscribers simultaneously.<br><br><b>How it works:</b><br>Publisher → SNS Topic → [SQS Queue, Lambda, Email, HTTP endpoint, SMS, Mobile Push]<br><br><b>Key concepts:</b><br>- <b>Topic</b>: the communication channel<br>- <b>Subscription</b>: an endpoint that receives messages from a topic<br>- <b>Fan-out pattern</b>: SNS → multiple SQS queues for parallel processing<br><br><b>SNS vs SQS:</b><br>- SQS: one-to-one, consumer pulls, persists messages<br>- SNS: one-to-many, pushes to subscribers, no persistence<br><br><b>Common mistake:</b> Sending important notifications via SNS email subscriptions with no retry. If the email server is down, the message is lost. Use SNS → SQS for durability.<br><br><b>Senior insight:</b> The SNS → SQS fan-out pattern is one of the most powerful AWS patterns. S3 Events → SNS → multiple SQS queues is a classic decoupled architecture.
What is Amazon EventBridge?	<b>EventBridge</b> is a serverless event bus — routes events from AWS services, your apps, or SaaS partners to targets like Lambda, SQS, Step Functions, etc.<br><br><b>Key concepts:</b><br>- <b>Event Bus</b>: the pipeline events flow through (default, custom, or partner)<br>- <b>Rule</b>: pattern matching logic (route events matching condition X to target Y)<br>- <b>Event Pattern</b>: JSON filter that matches against event structure<br>- <b>Schema Registry</b>: auto-discovers and catalogs event schemas<br><br><b>EventBridge vs SNS:</b><br>- EventBridge: content-based routing, schema registry, 100+ SaaS integrations, more complex rules<br>- SNS: simpler, higher throughput, better for internal pub/sub<br><br><b>EventBridge Scheduler:</b> Create one-time or recurring scheduled events (replaces CloudWatch Events for scheduling).<br><br><b>Senior insight:</b> Use EventBridge for event-driven architectures that react to AWS resource changes (e.g., EC2 state change → Lambda). It's more flexible than SNS for routing logic.
What is Amazon Kinesis?	<b>Kinesis</b> is a platform for real-time streaming data at massive scale — think of it as a highway for continuous data flows.<br><br><b>Services:</b><br><b>Kinesis Data Streams:</b><br>- Low-latency real-time streaming<br>- Data retained 1–365 days<br>- Multiple consumers can read the same stream<br>- You manage shards (throughput units)<br><br><b>Kinesis Data Firehose:</b><br>- Fully managed delivery of streams to S3, Redshift, OpenSearch, Splunk<br>- Near real-time (60 sec buffer)<br>- No shard management needed<br><br><b>Kinesis Data Analytics:</b> SQL or Apache Flink on streaming data in real-time.<br><br><b>Kinesis vs SQS:</b><br>- Kinesis: multiple consumers, time-ordered, replay data, real-time analytics<br>- SQS: single consumer, message deleted after processing, simpler<br><br><b>Senior insight:</b> Use Kinesis for analytics pipelines, log streaming, and IoT telemetry. Use SQS for job queues and task distribution. They solve different problems.
What is AWS CloudFormation?	<b>CloudFormation</b> is AWS's native Infrastructure as Code (IaC) service — you define your infrastructure in YAML/JSON templates and CloudFormation provisions and manages it.<br><br><b>Key concepts:</b><br>- <b>Template</b>: YAML/JSON file describing resources<br>- <b>Stack</b>: a deployed instance of a template<br>- <b>Change Set</b>: preview what will change before applying<br>- <b>Drift Detection</b>: find resources that were manually changed<br>- <b>StackSets</b>: deploy stacks across multiple accounts/regions<br><br><b>Example resource:</b><br><code>MyBucket:<br>  Type: AWS::S3::Bucket<br>  Properties:<br>    BucketName: my-unique-bucket</code><br><br><b>Common mistake:</b> Manually modifying resources managed by CloudFormation — this causes drift and can break future stack updates.<br><br><b>Senior insight:</b> Use Change Sets before every production stack update. A change set shows you what will be replaced vs. updated in-place — surprises here can cause downtime.
What is AWS CDK?	<b>CDK (Cloud Development Kit)</b> lets you define AWS infrastructure using real programming languages (TypeScript, Python, Java, C#, Go) instead of YAML/JSON.<br><br><b>How it works:</b> CDK code → synthesizes to CloudFormation templates → deployed via CloudFormation<br><br><b>Benefits over raw CloudFormation:</b><br>- Reusable abstractions (Constructs)<br>- Loops, conditions, functions — real code<br>- Type safety and IDE autocomplete<br>- Higher-level patterns (e.g., <code>aws-ecs-patterns.ApplicationLoadBalancedFargateService</code>)<br><br><b>Example (TypeScript):</b><br><code>new s3.Bucket(this, 'MyBucket', {<br>  versioned: true,<br>  encryption: s3.BucketEncryption.S3_MANAGED<br>});</code><br><br><b>Common mistake:</b> Writing CDK code that's too imperative — CDK is declarative at heart. Avoid complex runtime logic in synth.<br><br><b>Senior insight:</b> CDK is the recommended approach for greenfield projects. Use CDK Constructs to share patterns across teams. Combine with CDK Pipelines for self-mutating CI/CD.
What is AWS CloudWatch?	<b>CloudWatch</b> is AWS's observability service — collect, monitor, and alert on metrics, logs, and traces from AWS resources and applications.<br><br><b>Key components:</b><br>- <b>Metrics</b>: numeric time-series data (CPU, request count, error rate)<br>- <b>Logs</b>: text log streams from EC2, Lambda, ECS, etc.<br>- <b>Alarms</b>: trigger notifications or actions when metrics breach thresholds<br>- <b>Dashboards</b>: visualize metrics in real-time<br>- <b>CloudWatch Agent</b>: collect custom metrics + logs from EC2<br>- <b>Container Insights</b>: ECS/EKS metrics (CPU/memory per task/pod)<br><br><b>Log Insights:</b> Query logs with SQL-like syntax:<br><code>fields @timestamp, @message | filter @message like /ERROR/ | sort @timestamp desc | limit 20</code><br><br><b>Common mistake:</b> Relying only on default metrics. Most useful metrics (memory, disk, app errors) require the CloudWatch Agent or custom metrics.<br><br><b>Senior insight:</b> Set alarms on SLI/SLO metrics, not just CPU. Alert on error rates, latency p99, and queue depth for a true picture of system health.
What is AWS CloudTrail?	<b>CloudTrail</b> records every API call made in your AWS account — who did what, from where, and when. It's your audit log for the entire account.<br><br><b>What it captures:</b><br>- API calls via Console, CLI, SDK, or other AWS services<br>- The identity (user/role/service) that made the call<br>- Source IP address<br>- Request parameters and response<br><br><b>Event types:</b><br>- <b>Management Events</b>: control plane operations (create/delete/modify resources) — on by default<br>- <b>Data Events</b>: S3 object operations, Lambda invocations — must enable explicitly (expensive)<br>- <b>Insights Events</b>: detects unusual API activity<br><br><b>Common mistake:</b> Not enabling CloudTrail. By default it's on, but logs only 90 days in Event History. You need a Trail to store logs to S3 long-term.<br><br><b>Senior insight:</b> Send CloudTrail logs to a separate, locked-down S3 bucket in a separate account. This prevents attackers who compromise your main account from covering their tracks.
What is AWS X-Ray?	<b>X-Ray</b> is AWS's distributed tracing service — it maps how requests flow through your application and identifies performance bottlenecks and errors.<br><br><b>Key concepts:</b><br>- <b>Trace</b>: the full journey of a single request through your system<br>- <b>Segment</b>: work done by a single service within a trace<br>- <b>Subsegment</b>: granular detail within a segment (DB call, HTTP call)<br>- <b>Service Map</b>: visual graph of all connected services<br><br><b>Works with:</b> Lambda, EC2, ECS, API Gateway, SQS, SNS, and more<br><br><b>Integration example (Python):</b><br><code>from aws_xray_sdk.core import xray_recorder<br>@xray_recorder.capture('my-function')<br>def my_function():</code><br><br><b>Common mistake:</b> Enabling X-Ray on every Lambda without sampling rules — at high throughput, tracing every request becomes expensive. Use sampling rules to trace 5% or 1 req/sec.<br><br><b>Senior insight:</b> X-Ray Service Map is invaluable during incidents to quickly identify which downstream service is causing latency. Combine with CloudWatch Synthetics for proactive monitoring.
What is AWS CodePipeline?	<b>CodePipeline</b> is AWS's fully managed CI/CD orchestration service — defines the stages a code change goes through from source to production.<br><br><b>Pipeline stages typically include:</b><br>1. <b>Source</b>: CodeCommit, GitHub, S3, ECR<br>2. <b>Build</b>: CodeBuild (compile, test, package)<br>3. <b>Test</b>: integration tests, approvals<br>4. <b>Deploy</b>: CodeDeploy, CloudFormation, ECS, Lambda<br><br><b>Key features:</b><br>- Manual approval gates between stages<br>- Parallel execution stages<br>- Integration with 3rd party tools (Jenkins, GitHub Actions)<br>- EventBridge triggers on pipeline events<br><br><b>Common mistake:</b> Having no manual approval gate before production. Even in fully automated pipelines, a human gate before prod is a good safety net.<br><br><b>Senior insight:</b> CodePipeline is simpler than Jenkins but less flexible. For complex pipelines, GitHub Actions or GitLab CI are often better. Use CodePipeline when you want deep AWS-native integrations with minimal setup.
What is AWS CodeBuild?	<b>CodeBuild</b> is a fully managed build service — it compiles your code, runs tests, and produces deployment artifacts without you managing build servers.<br><br><b>Key concepts:</b><br>- <b>buildspec.yml</b>: defines build commands and phases<br>- <b>Build environment</b>: a Docker container with your chosen runtime<br>- <b>Artifacts</b>: output files (zip, Docker image, JAR) stored in S3 or ECR<br>- Scales automatically — no idle build servers<br><br><b>Example buildspec.yml:</b><br><code>phases:<br>  install:<br>    commands: npm install<br>  build:<br>    commands: npm test && npm run build<br>artifacts:<br>  files: dist/**</code><br><br><b>Common mistake:</b> Not caching dependencies — every build downloading all npm/pip/maven packages is slow and costly. Use CodeBuild cache to S3.<br><br><b>Senior insight:</b> CodeBuild charges per build minute. For high-frequency, fast builds, it's very cost-effective. For long builds (20+ min) at high volume, consider self-hosted runners.
What is AWS CodeDeploy?	<b>CodeDeploy</b> automates deployment of application code to EC2, Lambda, or ECS — handling rolling updates, blue/green deployments, and rollback.<br><br><b>Deployment types:</b><br>- <b>In-place (EC2/on-prem)</b>: stops old version, installs new version on same instances. Causes downtime unless combined with load balancer.<br>- <b>Blue/Green (EC2)</b>: provision new instances, deploy new version, shift traffic, terminate old.<br>- <b>Blue/Green (Lambda)</b>: shift % of traffic between Lambda versions (canary, linear, all-at-once).<br>- <b>Blue/Green (ECS)</b>: deploy new task set, shift ALB traffic, drain old tasks.<br><br><b>Lambda traffic shifting example:</b><br>- Canary: 10% → 100% after X minutes<br>- Linear: shift 10% every minute<br><br><b>Common mistake:</b> Not defining rollback triggers — CodeDeploy can auto-rollback if CloudWatch alarms fire during deployment.<br><br><b>Senior insight:</b> Blue/Green with automatic rollback is the gold standard for zero-downtime deployments. Always define alarm-based rollback conditions.
What is AWS API Gateway?	<b>API Gateway</b> is a fully managed service to create, publish, secure, and scale APIs — typically used as the front door for Lambda functions or backend services.<br><br><b>Types:</b><br>- <b>REST API</b>: full-featured, more control, request/response transformation<br>- <b>HTTP API</b>: simpler, cheaper, lower latency. Best for Lambda and HTTP backends.<br>- <b>WebSocket API</b>: persistent connections for real-time apps (chat, live updates)<br><br><b>Key features:</b><br>- Authentication (IAM, Cognito, Lambda Authorizers, API Keys)<br>- Throttling and quotas per API/stage<br>- Request/response transformation<br>- Caching (REST API)<br>- Custom domain names<br><br><b>Common mistake:</b> Using REST API when HTTP API is sufficient — HTTP APIs are 70% cheaper and have lower latency. REST API is only needed for advanced features like request validation, transformation, or WAF integration.<br><br><b>Senior insight:</b> Lambda + API Gateway is the canonical serverless API pattern. Use a Lambda Proxy integration to keep the code simple and let your app handle routing.
What is AWS Step Functions?	<b>Step Functions</b> is a serverless workflow orchestration service — coordinates multiple AWS services (Lambda, ECS, SQS, etc.) into visual state machines.<br><br><b>When to use it:</b><br>- Workflows with complex branching logic<br>- Long-running processes (days/months)<br>- Retry and error handling across multiple services<br>- Human approval workflows<br>- Replace complex Lambda chains<br><br><b>State types:</b> Task, Choice, Wait, Parallel, Map, Pass, Succeed, Fail<br><br><b>Workflow types:</b><br>- <b>Standard</b>: exactly-once, up to 1 year, auditable (all state transitions logged). For long workflows.<br>- <b>Express</b>: at-least-once, up to 5 min, higher throughput. For high-volume event processing.<br><br><b>Common mistake:</b> Using Step Functions for simple two-step Lambda chains — overkill. Use it when you have 3+ steps with complex flow control.<br><br><b>Senior insight:</b> Step Functions with Activity Tasks enables human-in-the-loop workflows — e.g., send email → wait for approval → continue.
What is AWS KMS?	<b>KMS (Key Management Service)</b> is a managed service for creating and controlling cryptographic keys used to encrypt your data across AWS services.<br><br><b>Key types:</b><br>- <b>AWS Managed Keys</b>: automatically created per service (e.g., <code>aws/s3</code>). Free, no control.<br>- <b>Customer Managed Keys (CMKs)</b>: you create and control them. Can set rotation, access policies, audit via CloudTrail. $1/month each.<br>- <b>Custom Key Store (CloudHSM)</b>: keys backed by dedicated hardware. For compliance requirements.<br><br><b>Key usage:</b><br>- Encrypt S3, EBS, RDS, SSM, Secrets Manager, DynamoDB at rest<br>- Sign/verify data<br>- Generate data encryption keys (envelope encryption)<br><br><b>Common mistake:</b> Thinking encryption is free — KMS charges per API call ($0.03 per 10,000). High-throughput services making frequent <code>Decrypt</code> calls can surprise you with costs.<br><br><b>Senior insight:</b> Always use CMKs for sensitive data. They allow cross-account access, key policy auditing, and can be disabled/deleted if compromised.
What is AWS Secrets Manager vs Systems Manager Parameter Store?	Both store configuration and secrets, but are designed for different use cases.<br><br><b>Secrets Manager:</b><br>- Designed for secrets (passwords, API keys, DB credentials)<br>- Automatic secret rotation (built-in for RDS, custom Lambda for others)<br>- Cross-account access<br>- Costs ~$0.40/secret/month<br><br><b>Parameter Store:</b><br>- General-purpose config + secrets<br>- <b>Standard tier</b>: free, 4KB, no rotation<br>- <b>Advanced tier</b>: costs money, 8KB, supports parameter policies<br>- Hierarchical naming: <code>/myapp/prod/db-password</code><br><br><b>When to use which:</b><br>- Use Secrets Manager for database passwords and credentials that need auto-rotation<br>- Use Parameter Store for non-sensitive config (feature flags, environment URLs) and simple secrets<br><br><b>Common mistake:</b> Hardcoding secrets in environment variables or code. Always use Secrets Manager or Parameter Store, even for dev.<br><br><b>Senior insight:</b> Use IAM + resource policies to control which services can access which secrets. Integrate Secrets Manager with RDS for zero-downtime credential rotation.
What is AWS WAF?	<b>WAF (Web Application Firewall)</b> protects web applications from common exploits — SQL injection, XSS, bot attacks, and OWASP Top 10 vulnerabilities.<br><br><b>How it works:</b> Inspect HTTP(S) requests before they reach your ALB, API Gateway, CloudFront, or AppSync.<br><br><b>Key components:</b><br>- <b>Web ACL</b>: the ruleset<br>- <b>Rules</b>: conditions to match (IP, geo, header, body, rate)<br>- <b>Managed Rule Groups</b>: pre-built rules from AWS or Marketplace<br>- <b>Rate-based rules</b>: block IPs exceeding X requests/5 min<br><br><b>AWS Managed Rules include:</b> Core Rule Set (OWASP), SQL injection, Linux/Windows OS rules, Known bad inputs<br><br><b>Common mistake:</b> Running WAF in COUNT mode and never switching to BLOCK — always test in count mode first, but set a date to switch to block.<br><br><b>Senior insight:</b> WAF + Shield Standard is the minimum security baseline for any public-facing application. Add Bot Control Managed Rule Group to deal with scrapers and credential stuffing.
What is AWS Shield?	<b>Shield</b> provides DDoS (Distributed Denial of Service) protection for AWS resources.<br><br><b>Two tiers:</b><br><b>Shield Standard:</b><br>- Free for all AWS customers automatically<br>- Protects against common Layer 3/4 DDoS attacks (SYN floods, UDP reflection)<br>- Always on, no configuration needed<br><br><b>Shield Advanced:</b><br>- $3,000/month per organization<br>- Protects against sophisticated Layer 7 attacks<br>- 24/7 access to AWS DRT (DDoS Response Team)<br>- Cost protection: AWS credits you for resource scale-out during attacks<br>- Enhanced visibility and attack diagnostics<br>- Applies to EC2, ALB, CloudFront, Route 53, Global Accelerator<br><br><b>Common mistake:</b> Assuming Shield Standard covers everything. It only covers volumetric attacks. For application-layer DDoS, you need Shield Advanced + WAF.<br><br><b>Senior insight:</b> Shield Advanced cost protection alone can justify the price for high-traffic applications — during a DDoS attack, your bill won't explode from auto-scaling defenses.
What is Amazon Route 53?	<b>Route 53</b> is AWS's scalable, highly available DNS service — translates domain names to IP addresses, and routes users to the right resources.<br><br><b>Key features:</b><br>- Domain registration<br>- DNS hosting<br>- Health checks<br>- Traffic routing policies<br><br><b>Routing policies:</b><br>- <b>Simple</b>: single resource<br>- <b>Weighted</b>: split traffic (e.g., 90/10 between versions — canary)<br>- <b>Latency-based</b>: route to lowest latency Region<br>- <b>Failover</b>: primary + standby, uses health checks<br>- <b>Geolocation</b>: route by user's country/continent<br>- <b>Geoproximity</b>: route by geographic distance with bias<br>- <b>Multi-value</b>: return multiple IPs (basic load balancing with health checks)<br><br><b>Common mistake:</b> Using simple routing for multi-region without health checks — if your primary goes down, Route 53 keeps sending traffic there.<br><br><b>Senior insight:</b> Route 53 health checks + Failover routing is the simplest DNS-based disaster recovery pattern. Combine with CloudFront for a fast, resilient global setup.
What is Amazon CloudFront?	<b>CloudFront</b> is AWS's Content Delivery Network (CDN) — distributes your content globally by caching it at 400+ Edge Locations closer to your users.<br><br><b>Key concepts:</b><br>- <b>Distribution</b>: the CloudFront config<br>- <b>Origin</b>: where content comes from (S3, ALB, EC2, API Gateway, custom HTTP)<br>- <b>Cache Behavior</b>: rules for what/how to cache by path pattern<br>- <b>TTL</b>: how long objects are cached at the edge<br>- <b>Invalidation</b>: force-expire cached objects (first 1,000/month free)<br><br><b>Benefits:</b><br>- Reduced latency for users worldwide<br>- Reduces load on origin<br>- HTTPS via ACM certificates (free!)<br>- WAF integration at the edge<br>- Signed URLs and Cookies for private content<br><br><b>Common mistake:</b> Short TTLs to avoid stale content, but negating all caching benefits. Use long TTLs + cache-busting (versioned filenames) instead.<br><br><b>Senior insight:</b> S3 + CloudFront + Route 53 + ACM = fully serverless, globally distributed, HTTPS static site for near-zero cost.
What is AWS Direct Connect?	<b>Direct Connect</b> is a dedicated, private network connection from your on-premises data center to AWS — bypassing the public internet entirely.<br><br><b>Why use it:</b><br>- Consistent, predictable bandwidth (not subject to internet congestion)<br>- Lower latency than VPN<br>- Reduced data transfer costs for large volumes<br>- Required for compliance in some industries<br><br><b>Connection speeds:</b> 50Mbps to 100Gbps<br><br><b>Direct Connect vs VPN:</b><br>- VPN: encrypted, over public internet, setup in minutes, cheaper, variable latency<br>- Direct Connect: private fiber, faster, consistent, expensive to set up (physical cabling)<br><br><b>HA design:</b> Single Direct Connect = single point of failure. Always pair with a VPN backup or use two Direct Connect connections from different providers.<br><br><b>Common mistake:</b> Assuming Direct Connect is encrypted by default — it's private but NOT encrypted. Encrypt using MACsec or a VPN over the Direct Connect connection.<br><br><b>Senior insight:</b> Direct Connect Gateway lets you connect your on-prem to multiple VPCs across multiple Regions with a single Direct Connect connection.
What is AWS VPN?	<b>AWS VPN</b> creates an encrypted tunnel between your on-premises network and your AWS VPC over the public internet.<br><br><b>Two types:</b><br><b>Site-to-Site VPN:</b><br>- Connects on-prem network to VPC<br>- Uses IPSec tunnels<br>- Two tunnels per connection (redundancy)<br>- ~1.25 Gbps max bandwidth<br>- $0.05/hour + data transfer costs<br><br><b>Client VPN:</b><br>- Lets individual users connect securely to AWS from their laptops<br>- OpenVPN-based<br>- Supports MFA (Active Directory, SAML, Certificate-based)<br><br><b>Common mistake:</b> Relying on a single VPN tunnel — always use both tunnels that Site-to-Site VPN provides for redundancy.<br><br><b>Senior insight:</b> VPN is the right choice for getting started quickly and for lower-bandwidth needs. For consistent, high-bandwidth hybrid connectivity, graduate to Direct Connect.
What is VPC Peering?	<b>VPC Peering</b> creates a direct network connection between two VPCs — traffic routes through the AWS backbone, not the internet.<br><br><b>Key properties:</b><br>- Works across accounts and Regions<br>- Traffic stays on AWS network (low latency, no extra cost for same-Region)<br>- NOT transitive: if A peers with B and B peers with C, A cannot reach C through B<br><br><b>Required steps:</b> Initiate peering request → Accept → Update route tables in BOTH VPCs → Update Security Groups<br><br><b>VPC Peering vs Transit Gateway:</b><br>- Peering: one-to-one, no transitive routing, simpler, no cost (same Region)<br>- Transit Gateway: hub-and-spoke, transitive, scales to thousands of VPCs, costs per attachment and data<br><br><b>Common mistake:</b> Overlapping CIDR ranges — you cannot peer VPCs with the same IP range. Plan your CIDR blocks across all VPCs before you start.<br><br><b>Senior insight:</b> For 2-3 VPCs, peering is fine. For 5+, use Transit Gateway. Non-transitivity makes peering messy at scale (N*(N-1)/2 peering connections needed for full mesh).
What is AWS Transit Gateway?	<b>Transit Gateway (TGW)</b> is a network hub that connects multiple VPCs, on-premises networks, and VPNs through a single gateway — like a cloud router.<br><br><b>Key benefits:</b><br>- Hub-and-spoke model: connect thousands of VPCs without mesh peering<br>- Transitive routing: VPC A can reach VPC C through TGW<br>- Route table isolation: control which VPCs can talk to which<br>- Works across accounts and Regions<br>- Supports VPN and Direct Connect attachments<br><br><b>Architecture example:</b><br>Dev VPC → TGW → Prod VPC (blocked via route tables)<br>All VPCs → TGW → Shared Services VPC (DNS, monitoring)<br><br><b>Common mistake:</b> Allowing full mesh routing in TGW by default. Use separate TGW route tables per environment to enforce isolation between dev/staging/prod.<br><br><b>Senior insight:</b> TGW replaces VPC Peering at scale. Use TGW + Network Firewall for centralized egress inspection — route all outbound traffic through a security VPC before it exits to the internet.
What is AWS IAM Roles for Service Accounts (IRSA) in EKS?	<b>IRSA</b> lets Kubernetes pods assume AWS IAM Roles and get temporary credentials — so each pod gets only the permissions it needs, with no shared credentials.<br><br><b>How it works:</b><br>1. EKS cluster has an OIDC provider<br>2. IAM Role trusts that OIDC provider and a specific K8s service account<br>3. Pod annotated with the service account gets temporary AWS credentials injected automatically<br><br><b>Why it matters:</b><br>Without IRSA: all pods on a node share the node's IAM role → overprivileged<br>With IRSA: each pod has its own minimal IAM role → least privilege<br><br><b>Example annotation:</b><br><code>eks.amazonaws.com/role-arn: arn:aws:iam::123:role/my-pod-role</code><br><br><b>Common mistake:</b> Giving EC2 nodes broad IAM permissions "for convenience" instead of using IRSA per workload.<br><br><b>Senior insight:</b> IRSA is the foundation of security for EKS workloads. Always pair it with OPA/Kyverno policies to enforce that pods declare their service accounts and don't run as root.
What is AWS Organizations and Service Control Policies (SCPs)?	<b>AWS Organizations</b> manages multiple AWS accounts centrally — grouping them in Organizational Units (OUs) with shared policies and consolidated billing.<br><br><b>Key features:</b><br>- Consolidated billing: one invoice, volume discounts<br>- Organizational Units (OUs): group accounts (e.g., Prod OU, Dev OU)<br>- Service Control Policies (SCPs): guardrails on what accounts CAN do<br>- AWS SSO / IAM Identity Center: centralized login across accounts<br><br><b>SCPs:</b><br>- Define the MAXIMUM permissions for all IAM entities in an account<br>- Even the account root user is constrained by SCPs<br>- Use to prevent: disabling CloudTrail, creating root access keys, using non-approved regions<br><br><b>Example SCP (deny all non-approved regions):</b><br><code>{"Effect":"Deny","Action":"*","Resource":"*","Condition":{"StringNotEquals":{"aws:RequestedRegion":"us-east-1"}}}</code><br><br><b>Senior insight:</b> A good landing zone (multi-account strategy) uses SCPs extensively. They're your safety net — even a compromised admin account can't do things SCPs forbid.
What is AWS Config?	<b>AWS Config</b> continuously monitors and records your AWS resource configurations — and evaluates them against desired rules for compliance.<br><br><b>What it does:</b><br>- Maintains a history of resource configurations<br>- Shows what a resource looked like at any point in time<br>- Evaluates resources against managed or custom rules<br>- Sends notifications when resources become non-compliant<br><br><b>Example rules:</b><br>- <code>s3-bucket-public-read-prohibited</code>: fails if any S3 bucket allows public read<br>- <code>encrypted-volumes</code>: fails if EBS volumes are unencrypted<br>- <code>restricted-ssh</code>: fails if security group allows 0.0.0.0/0 on port 22<br><br><b>Remediation:</b> Trigger SSM Automation documents to auto-fix non-compliant resources<br><br><b>Common mistake:</b> Enabling Config without acting on findings — Config is only valuable if someone reviews and remediates non-compliance.<br><br><b>Senior insight:</b> AWS Config is your compliance enforcement plane. Combine with Security Hub for a unified view across all accounts. Use Conformance Packs to apply multi-rule compliance frameworks (CIS, PCI-DSS) at once.
What is Amazon GuardDuty?	<b>GuardDuty</b> is a managed threat detection service — it continuously analyzes AWS account activity and network traffic to identify malicious behavior and unauthorized access.<br><br><b>Data sources analyzed:</b><br>- CloudTrail logs (API calls)<br>- VPC Flow Logs (network traffic)<br>- DNS logs<br>- S3 data events (optional)<br>- EKS audit logs (optional)<br>- Lambda network activity (optional)<br><br><b>Example findings:</b><br>- Instance communicating with a known crypto-mining C2 server<br>- Root credentials used from an unusual location<br>- S3 bucket exfiltration behavior<br>- Port scanning from an EC2 instance<br><br><b>Common mistake:</b> Not enabling GuardDuty because "we have nothing valuable." Every compromised AWS account is used for crypto-mining at minimum — and you pay the EC2 bill.<br><br><b>Senior insight:</b> GuardDuty is cheap and entirely passive (no agents to install). Enable it everywhere. Route findings to Security Hub → EventBridge → Lambda for automated response (e.g., isolate compromised instance).
What is Amazon Inspector?	<b>Inspector</b> is an automated vulnerability management service — continuously scans EC2 instances, Lambda functions, and container images for software vulnerabilities and unintended network exposure.<br><br><b>What it scans:</b><br>- <b>EC2</b>: OS and application CVEs, network reachability<br>- <b>Lambda</b>: function code and layer CVEs<br>- <b>ECR container images</b>: OS and package CVEs (on push and continuously)<br><br><b>Key features:</b><br>- Continuously re-evaluates as new CVEs are published (not just on scan schedule)<br>- Risk scoring based on exploitability + reachability<br>- Integrates with Security Hub and ECR<br><br><b>Common mistake:</b> Confusing Inspector with GuardDuty. Inspector = vulnerability scanning (what's vulnerable). GuardDuty = threat detection (active attacks).<br><br><b>Senior insight:</b> Enable Inspector for ECR to get CVE reports before images are deployed. Fail CI/CD pipelines on CRITICAL CVEs using Inspector findings via EventBridge.
What is AWS Security Hub?	<b>Security Hub</b> is a centralized security dashboard — aggregates findings from GuardDuty, Inspector, Config, Macie, and third-party tools into a single pane of glass.<br><br><b>Key features:</b><br>- Automated security checks against standards (CIS AWS Foundations, PCI-DSS, AWS Foundational Security Best Practices)<br>- Normalized finding format (ASFF — Amazon Security Finding Format)<br>- Cross-account aggregation via Organizations<br>- Integration with EventBridge for automated remediation<br><br><b>Security score:</b> Security Hub gives you an overall percentage score per standard. Track this as a KPI for your security posture.<br><br><b>Common mistake:</b> Treating Security Hub as a fire-and-forget dashboard. It needs a process: regular review, ticketing for findings, remediation SLAs.<br><br><b>Senior insight:</b> Security Hub + AWS Organizations = one place to see the security posture of all accounts. Enable "Auto-enable new accounts" so new accounts are automatically onboarded into your security monitoring.
What is Amazon Macie?	<b>Macie</b> is a managed data security service that uses ML to automatically discover, classify, and protect sensitive data in S3 — things like PII, financial data, credentials.<br><br><b>What it detects:</b><br>- Personally Identifiable Information (PII): names, SSNs, passports, credit cards<br>- AWS credentials and private keys accidentally uploaded<br>- Health information (PHI/HIPAA data)<br>- Custom sensitive data patterns you define<br><br><b>How it works:</b><br>- Scans S3 objects on schedule or one-time<br>- Generates findings in Security Hub<br>- Also alerts on bucket-level security issues (public access, unencrypted buckets)<br><br><b>Common mistake:</b> Thinking "we don't store PII." Every application collects some PII — Macie finds it even when you didn't know it was there.<br><br><b>Senior insight:</b> Run a Macie discovery job on all S3 buckets when taking over a legacy account. You'll often find S3 buckets with sensitive data that nobody knew about.
What is AWS Trusted Advisor?	<b>Trusted Advisor</b> is an automated advisor that analyzes your AWS account and gives recommendations across 5 categories:<br><br>1. <b>Cost Optimization</b>: idle resources, underutilized instances, reserved instance recommendations<br>2. <b>Performance</b>: high-utilization instances, CloudFront optimization<br>3. <b>Security</b>: open security groups, MFA on root, IAM keys not rotated<br>4. <b>Fault Tolerance</b>: no Multi-AZ, backups not enabled, unhealthy health checks<br>5. <b>Service Limits</b>: approaching service quotas<br><br><b>Tiers:</b><br>- Basic (free): 6 core security checks + service limits<br>- Business/Enterprise Support: all checks + API access<br><br><b>Common mistake:</b> Ignoring Trusted Advisor findings. The cost and security recommendations are almost always actionable and save real money/risk.<br><br><b>Senior insight:</b> Pull Trusted Advisor findings into a weekly review. The "Cost Optimization" category alone typically identifies thousands of dollars in waste in most accounts.
What is the AWS Well-Architected Framework?	The <b>Well-Architected Framework</b> is AWS's set of best practices for designing and operating reliable, secure, efficient, and cost-effective systems in the cloud.<br><br><b>6 Pillars:</b><br>1. <b>Operational Excellence</b>: run and monitor, improve processes (IaC, CI/CD, runbooks)<br>2. <b>Security</b>: protect data and systems (least privilege, encryption, detection)<br>3. <b>Reliability</b>: recover from failures (multi-AZ, auto-scaling, backups, chaos engineering)<br>4. <b>Performance Efficiency</b>: use resources efficiently (right-sizing, serverless, caching)<br>5. <b>Cost Optimization</b>: avoid unnecessary costs (right-pricing, lifecycle policies, waste elimination)<br>6. <b>Sustainability</b>: minimize environmental impact (efficient architectures, managed services)<br><br><b>Well-Architected Tool:</b> Free AWS tool to review workloads against these pillars interactively.<br><br><b>Senior insight:</b> Do a Well-Architected Review before major launches and annually for critical workloads. It surfaces architectural risks before they become incidents.
What is the difference between RTO and RPO in AWS DR planning?	<b>RPO (Recovery Point Objective):</b> How much data can you afford to lose? Measured in time since the last backup/recovery point.<br>Example: RPO = 1 hour means you can tolerate losing up to 1 hour of data.<br><br><b>RTO (Recovery Time Objective):</b> How long can the system be down? Time from failure to recovery.<br>Example: RTO = 30 minutes means you must be back online within 30 minutes.<br><br><b>DR Strategies (slowest/cheapest → fastest/most expensive):</b><br>1. <b>Backup & Restore</b>: RTO hours, RPO hours. Just backups in S3/Glacier.<br>2. <b>Pilot Light</b>: RTO hours, RPO minutes. Minimal infrastructure always running (just the DB).<br>3. <b>Warm Standby</b>: RTO minutes, RPO seconds. Scaled-down duplicate running.<br>4. <b>Multi-Site Active/Active</b>: RTO near-zero, RPO near-zero. Full duplicate in another Region.<br><br><b>Senior insight:</b> Most organizations think they need Active/Active but actually need Warm Standby. Match the strategy to the business impact — not to what feels "safest."
What is AWS Backup?	<b>AWS Backup</b> is a centralized managed service for creating, managing, and restoring backups across AWS services from one place.<br><br><b>Supported services:</b> EC2, EBS, RDS, Aurora, DynamoDB, EFS, FSx, S3, DocumentDB, Neptune, VMware workloads<br><br><b>Key features:</b><br>- <b>Backup Plans</b>: schedule, retention, and lifecycle rules<br>- <b>Backup Vault</b>: isolated storage for backups, encrypted with KMS<br>- <b>Vault Lock</b>: WORM protection for backups (prevents deletion — compliance)<br>- Cross-region and cross-account backup copy<br>- AWS Organizations integration: enforce backup policies across all accounts<br><br><b>Common mistake:</b> Backing up everything to the same account. A ransomware attack or compromised account that deletes prod resources will also delete those backups. Use cross-account backup copies.<br><br><b>Senior insight:</b> Apply Vault Lock to your backup vault. This protects backups even from root users in the account — your last line of defense against ransomware.
What is Amazon EFS vs EBS vs S3?	Three different AWS storage types for different use cases:<br><br><b>EBS (Elastic Block Store):</b><br>- Block storage attached to a single EC2 instance (usually)<br>- Like a hard drive for your server<br>- Low latency, high performance<br>- Use for OS volumes, databases, filesystems<br>- Multi-Attach available (io1/io2 only, multiple instances same AZ)<br><br><b>EFS (Elastic File System):</b><br>- Managed NFS (Network File System)<br>- Can be mounted by thousands of EC2/Lambda instances simultaneously<br>- Scales automatically<br>- Use for shared content, CMS assets, developer environments<br>- More expensive than EBS (~3x)<br><br><b>S3 (Simple Storage Service):</b><br>- Object storage via API (not mountable as filesystem, without tools)<br>- Unlimited scale, cheapest per GB<br>- Use for data lakes, backups, static assets, logs<br><br><b>Senior insight:</b> Default to S3. Use EBS when you need a filesystem on a single server. Use EFS when multiple servers need to share a filesystem. Avoid EFS if S3 + a tool (Mountpoint) will do.
What is Amazon Redshift?	<b>Redshift</b> is AWS's fully managed cloud data warehouse — designed for analytical queries (OLAP) over petabytes of data, not transactional workloads (OLTP).<br><br><b>Key concepts:</b><br>- Columnar storage (fast for aggregations and scans)<br>- Massively Parallel Processing (MPP) across nodes<br>- PostgreSQL-compatible SQL syntax<br>- Leader node + compute nodes architecture<br><br><b>Key features:</b><br>- <b>Redshift Spectrum</b>: query S3 data directly without loading into Redshift<br>- <b>Redshift Serverless</b>: auto-scales, no cluster management<br>- <b>Data sharing</b>: share live data across Redshift clusters<br>- <b>RA3 nodes</b>: compute and storage scale independently<br><br><b>Common mistake:</b> Using Redshift for OLTP workloads. It's optimized for large sequential scans, not point queries or frequent row updates — use RDS for transactional data.<br><br><b>Senior insight:</b> Use Redshift Spectrum to query your S3 data lake without copying data into Redshift. This is the foundation of the modern "lakehouse" architecture.
What is AWS Glue?	<b>AWS Glue</b> is a fully managed ETL (Extract, Transform, Load) service — discovers, catalogs, and transforms data from various sources for analytics.<br><br><b>Key components:</b><br>- <b>Glue Data Catalog</b>: central metadata repository — databases, tables, schemas. Works with Athena, Redshift Spectrum, and EMR.<br>- <b>Glue Crawlers</b>: scan data sources and auto-create/update schema in the Catalog<br>- <b>Glue Jobs</b>: Spark or Python ETL scripts that transform and move data<br>- <b>Glue Studio</b>: visual ETL designer<br>- <b>Glue DataBrew</b>: no-code data preparation<br><br><b>Common mistake:</b> Running Glue Crawlers too frequently — each run has costs and startup time. Schedule crawlers based on data update frequency.<br><br><b>Senior insight:</b> The Glue Data Catalog is the "glue" (pun intended) of your data lake. Even if you don't use Glue for ETL, use the Catalog to give Athena, Redshift Spectrum, and EMR a shared schema layer.
What is Amazon Athena?	<b>Athena</b> is an interactive, serverless query service — run SQL queries directly against data in S3 without loading it into a database.<br><br><b>Key features:</b><br>- No infrastructure to manage<br>- Pay per query (per TB of data scanned, ~$5/TB)<br>- Supports CSV, JSON, Parquet, ORC, Avro formats<br>- Uses Presto/Trino under the hood<br>- Works with Glue Data Catalog for schemas<br><br><b>Performance & cost tips:</b><br>- Convert data to <b>columnar format (Parquet/ORC)</b> — query only needed columns<br>- <b>Partition</b> data by date/region — scan only relevant partitions<br>- Compress data (Snappy, GZIP) — less to scan<br>- These can reduce costs by 90%+<br><br><b>Example query:</b><br><code>SELECT status, COUNT(*) FROM access_logs WHERE year='2024' AND month='01' GROUP BY status</code><br><br><b>Common mistake:</b> Querying large unpartitioned CSV files — you'll scan terabytes and pay dollars for queries that should cost cents.<br><br><b>Senior insight:</b> CloudTrail + Athena = ad-hoc security forensics. Store CloudTrail logs in S3 and query them with Athena during incidents.
What is Amazon SNS FIFO vs Standard?	<b>Standard SNS Topic:</b><br>- Best-effort ordering<br>- At-least-once delivery (duplicates possible)<br>- Very high throughput (millions of messages/sec)<br>- Subscribers: SQS, Lambda, HTTP, Email, SMS, Mobile Push<br><br><b>FIFO SNS Topic:</b><br>- Strict message ordering<br>- Exactly-once message delivery<br>- Throughput: 300 messages/sec (3,000 with batching)<br>- Can ONLY deliver to SQS FIFO queues<br>- Message Group ID used to maintain order per group<br><br><b>When to use FIFO:</b><br>- Financial transactions where order matters<br>- Inventory updates that must be processed in sequence<br>- Any event where out-of-order processing causes incorrect state<br><br><b>Common mistake:</b> Using Standard SNS for workflows that require strict ordering — occasional duplicates and out-of-order delivery WILL cause bugs in order-sensitive logic.<br><br><b>Senior insight:</b> FIFO + Exactly-Once comes with the cost of lower throughput. For most microservice event buses, Standard SNS is fine if your consumers are idempotent.
What is the concept of idempotency in AWS messaging?	<b>Idempotency</b> means processing the same message multiple times has the same result as processing it once — the operation is safe to repeat.<br><br><b>Why it matters:</b><br>SQS Standard delivers messages <b>at least once</b> — duplicates can occur. If your consumer is not idempotent, you'll process orders twice, charge customers twice, etc.<br><br><b>How to achieve idempotency:</b><br>- Include a unique <code>MessageDeduplicationId</code> with each message<br>- On consumer side, track processed message IDs in DynamoDB<br>- Use conditional writes (DynamoDB <code>ConditionExpression</code>) to detect duplicates<br>- Design operations to be naturally idempotent (e.g., SET vs INCREMENT)<br><br><b>Example:</b><br><code>-- NOT idempotent: UPDATE orders SET quantity = quantity + 1<br>-- Idempotent: UPDATE orders SET quantity = 5 WHERE id = 123</code><br><br><b>Senior insight:</b> Design all Lambda + SQS consumers to be idempotent. It makes your system resilient to retries, network issues, and infrastructure restarts without data corruption.
What is AWS Fargate?	<b>Fargate</b> is a serverless compute engine for containers — run ECS or EKS workloads without managing the underlying EC2 instances.<br><br><b>How it differs from EC2 launch type:</b><br>- No nodes to provision, patch, or scale<br>- Pay per task (CPU + memory) per second<br>- AWS handles OS patching and kernel updates<br>- Isolated compute per task (no shared kernel like Docker on EC2)<br><br><b>When Fargate is NOT ideal:</b><br>- GPU workloads (not supported)<br>- Windows containers (limited support)<br>- Very high density workloads (EC2 + Graviton can be cheaper at scale)<br>- Custom networking modes (limited vs EC2)<br><br><b>Common mistake:</b> Overprovisioning CPU/memory in task definitions — Fargate charges exactly what you configure, even if the app uses 10% of it. Right-size with Container Insights metrics.<br><br><b>Senior insight:</b> Fargate Spot (like EC2 Spot) can cut costs 70% for fault-tolerant tasks. Use in your ECS Service capacity provider strategy for batch or background workers.
What is AWS Cost Explorer and how do you optimize costs?	<b>Cost Explorer</b> is AWS's built-in cost analysis tool — visualize, understand, and manage your AWS spending over time.<br><br><b>Key features:</b><br>- Break down costs by service, account, region, tag, or resource<br>- Forecast future costs<br>- Reserved Instance and Savings Plan recommendations<br>- Rightsizing recommendations (underutilized EC2)<br><br><b>Cost optimization strategies:</b><br>1. <b>Rightsizing</b>: match instance size to actual usage<br>2. <b>Reserved Instances/Savings Plans</b>: commit for discounts on predictable workloads<br>3. <b>Spot Instances</b>: for fault-tolerant batch workloads<br>4. <b>Lifecycle policies</b>: auto-delete/move unused S3, EBS snapshots<br>5. <b>Use managed services</b>: Lambda/Fargate over EC2 for variable workloads<br>6. <b>Delete idle resources</b>: ELBs, NAT Gateways, unused EIPs, unattached EBS volumes<br><br><b>Common mistake:</b> No tagging strategy — without tags, you can't attribute costs to teams, projects, or environments.<br><br><b>Senior insight:</b> Activate AWS Cost Anomaly Detection. It uses ML to alert you when spending deviates unexpectedly — catches runaway costs before they compound.
What are AWS Savings Plans?	<b>Savings Plans</b> offer discounts (up to 72%) in exchange for a commitment to a consistent amount of usage ($/hour) for 1 or 3 years.<br><br><b>Types:</b><br><b>Compute Savings Plans:</b><br>- Most flexible: applies to EC2 (any instance family, size, region, OS), Fargate, Lambda<br>- Up to 66% discount<br><br><b>EC2 Instance Savings Plans:</b><br>- Applies to a specific instance family in a specific Region (e.g., M5 in us-east-1)<br>- Up to 72% discount<br>- Less flexible but deeper discount<br><br><b>SageMaker Savings Plans:</b><br>- For SageMaker workloads specifically<br><br><b>Savings Plans vs Reserved Instances:</b><br>- Savings Plans: more flexible, covers Fargate/Lambda<br>- RIs: deeper discounts for EC2 (Convertible RIs offer flexibility with slightly less discount)<br><br><b>Common mistake:</b> Committing to a high $/hour amount based on peak usage. Commit to your baseline — use On-Demand for peaks.<br><br><b>Senior insight:</b> AWS Cost Explorer has a built-in Savings Plans recommendation engine. Run it quarterly and commit to what it recommends at the level you're comfortable with.
What is Amazon Cognito?	<b>Cognito</b> is AWS's fully managed identity service — handles user sign-up, sign-in, and access control for web and mobile applications.<br><br><b>Two main components:</b><br><b>User Pools:</b><br>- User directory (username/password, social login, SAML/OIDC federation)<br>- Issues JWT tokens (ID, Access, Refresh) after authentication<br>- MFA, email/phone verification, custom Lambda triggers<br><br><b>Identity Pools (Federated Identities):</b><br>- Exchange tokens (from User Pools, Google, Facebook, SAML) for temporary AWS credentials (STS)<br>- Lets users directly access AWS services (S3, DynamoDB) with their own IAM permissions<br><br><b>Common flow:</b><br>App → Cognito User Pool (authenticate) → JWT → API Gateway/ALB (authorize) → Backend<br><br><b>Common mistake:</b> Using Cognito Identity Pools to give users direct DynamoDB access without proper IAM conditions — users can access other users' data if conditions aren't set.<br><br><b>Senior insight:</b> Cognito is good for apps that need auth without building it yourself. For enterprise SSO with SAML/OIDC, use IAM Identity Center instead.
What is AWS IAM Identity Center (SSO)?	<b>IAM Identity Center</b> (formerly AWS SSO) is a centralized single sign-on service — one login to access multiple AWS accounts, business applications, and custom apps.<br><br><b>Key features:</b><br>- Integrates with external identity providers (Okta, Azure AD, Google Workspace) via SAML 2.0 / OIDC<br>- Permission Sets: IAM policy collections assigned to users/groups per account<br>- Manage access to dozens of AWS accounts from one place<br>- Provides temporary credentials via AWS CLI (<code>aws sso login</code>)<br><br><b>How it differs from IAM Users:</b><br>- IAM Users: permanent, per-account, need separate credentials per account<br>- IAM Identity Center: federated, centralized, single login for all accounts, credentials auto-rotate<br><br><b>Common mistake:</b> Creating IAM Users for human access in a multi-account setup. This is unmaintainable — use IAM Identity Center with your corporate IdP.<br><br><b>Senior insight:</b> IAM Identity Center + AWS Organizations is the foundation of a proper landing zone. No long-lived human IAM users, no shared credentials, full audit trail of who accessed what.
What is AWS STS (Security Token Service)?	<b>STS</b> issues temporary, limited-privilege security credentials for AWS access — instead of long-term access keys.<br><br><b>Key operations:</b><br>- <code>AssumeRole</code>: assume an IAM role and get temporary credentials (key, secret, session token) — expires in 15min–12h<br>- <code>AssumeRoleWithWebIdentity</code>: used by Cognito, OIDC (IRSA for EKS)<br>- <code>AssumeRoleWithSAML</code>: SAML federation (corporate IdP)<br>- <code>GetSessionToken</code>: get temporary credentials for MFA<br>- <code>GetFederationToken</code>: for custom federation brokers<br><br><b>Example (CLI):</b><br><code>aws sts assume-role --role-arn arn:aws:iam::123:role/CrossAccountRole --role-session-name mysession</code><br><br><b>Common mistake:</b> Not setting a meaningful <code>role-session-name</code> — in CloudTrail, all assume-role calls look the same. Use a meaningful name (username + date) for audit trails.<br><br><b>Senior insight:</b> All temporary credentials have an expiry. Your code must handle credential refresh. Use the AWS SDK — it automatically refreshes credentials before they expire.
What is AWS Service Quotas?	<b>Service Quotas</b> are the limits AWS places on your resource usage per account/region — e.g., max 5 VPCs per region, 1,000 Lambda concurrent executions, 200 EC2 instances.<br><br><b>Why quotas exist:</b><br>- Protect AWS infrastructure from runaway resource consumption<br>- Protect customers from accidental over-provisioning costs<br><br><b>Two types:</b><br>- <b>Default quotas</b>: apply to new accounts (conservative)<br>- <b>Adjustable quotas</b>: can be increased via support ticket or Service Quotas console<br><br><b>Key quotas to know for DevOps:</b><br>- Lambda: 1,000 concurrent executions (soft)<br>- EC2: varies by instance type<br>- ELB: 20 load balancers per region<br>- VPCs: 5 per region<br><br><b>Common mistake:</b> Discovering quota limits during a high-traffic event or launch. Check and raise quotas proactively, weeks before planned launches.<br><br><b>Senior insight:</b> Use AWS Service Quotas + CloudWatch alarms on quota usage. Set alarms at 80% of quota so you get notified before hitting limits.
What is Amazon SES (Simple Email Service)?	<b>SES</b> is a scalable, cost-effective cloud email service for sending transactional and marketing emails from your applications.<br><br><b>Use cases:</b><br>- Transactional emails (order confirmations, password resets)<br>- Marketing emails (newsletters, promotions)<br>- Automated notifications from applications<br><br><b>Key concepts:</b><br>- <b>Sending limits</b>: sandbox by default (only to verified emails). Request production access for unlimited sending.<br>- <b>Sending methods</b>: SMTP interface or SES API<br>- <b>Dedicated IPs</b>: for high-volume senders needing warm IP reputation<br>- <b>Bounce/Complaint handling</b>: critical for deliverability — suppress bad addresses<br>- <b>Configuration Sets</b>: track sends, opens, clicks, bounces via Kinesis/SNS<br><br><b>Common mistake:</b> Ignoring bounce/complaint rates. Too many bounces (>5%) or complaints (>0.1%) will result in your account being suspended by AWS.<br><br><b>Senior insight:</b> Implement a suppression list for hard bounces. Use SES event publishing to track all email events and build dashboards on deliverability health.
What is AWS App Mesh?	<b>App Mesh</b> is AWS's managed service mesh — provides consistent networking controls, observability, and traffic management for microservices, without changing application code.<br><br><b>Key capabilities:</b><br>- Traffic shaping (canary deployments, A/B testing, weighted routing)<br>- mTLS between services (mutual authentication)<br>- Circuit breaking<br>- Retries and timeouts<br>- End-to-end observability (CloudWatch, X-Ray)<br><br><b>How it works:</b><br>Envoy proxy sidecar is injected next to each service container. App Mesh controls sidecar behavior via APIs.<br><br><b>Works with:</b> ECS, EKS, EC2, and even on-premises<br><br><b>Common mistake:</b> Implementing retries in both the application code AND the service mesh — double retries amplify load during failures. Choose one layer to own retries.<br><br><b>Senior insight:</b> For EKS teams already familiar with Istio, App Mesh adds less value. App Mesh shines in mixed ECS/EKS/EC2 environments where you need one consistent mesh across all compute types.
What is AWS Network Firewall?	<b>AWS Network Firewall</b> is a managed, stateful network firewall deployed within your VPC — provides deep packet inspection, intrusion prevention (IPS), and domain filtering.<br><br><b>Key capabilities:</b><br>- Stateful traffic inspection (tracks connection state)<br>- Intrusion prevention rules (Suricata-compatible rule format)<br>- Domain-based filtering (block <code>*.malware.com</code>)<br>- TLS inspection (decrypt, inspect, re-encrypt)<br>- Managed threat intelligence rules<br><br><b>Deployment pattern:</b><br>Deploy in a dedicated "inspection VPC." Route all inter-VPC and egress traffic through Transit Gateway → Inspection VPC → Network Firewall → Internet.<br><br><b>Common mistake:</b> Placing Network Firewall in each application VPC instead of centralizing it — this gets expensive and complex fast. Use a centralized architecture.<br><br><b>Senior insight:</b> Network Firewall is the missing piece between Security Groups/NACLs (which work at port/IP level) and WAF (which works at HTTP level). Use it for east-west traffic inspection and DNS-based threat blocking.
What is the difference between horizontal and vertical scaling in AWS?	<b>Vertical Scaling (Scale Up):</b><br>Make the instance bigger — more CPU, more RAM (e.g., upgrade from m5.large to m5.4xlarge).<br>- Has upper limits (biggest instance type)<br>- Usually requires downtime for EC2<br>- Simple to implement<br>- Good for databases (RDS scale storage, Aurora Serverless v2 scales vertically in real-time)<br><br><b>Horizontal Scaling (Scale Out):</b><br>Add more instances — run 10 smaller instances instead of 1 big one.<br>- Theoretically unlimited scale<br>- No single point of failure<br>- Requires stateless application design (session stored externally, e.g., ElastiCache)<br>- Works with Auto Scaling Groups + Load Balancers<br><br><b>AWS preference:</b><br>AWS services are designed for horizontal scaling. Prefer stateless services that scale out over stateful ones that scale up.<br><br><b>Senior insight:</b> Stateful applications (user sessions in memory) can't horizontally scale without moving state out. Use ElastiCache for sessions or JWT tokens to make apps stateless before scaling out.
What is Amazon EBS and what types exist?	<b>EBS (Elastic Block Store)</b> provides persistent block storage volumes for EC2 instances — like a hard drive that persists even after the instance is stopped.<br><br><b>Volume types:</b><br><b>SSD-based:</b><br>- <b>gp3</b>: General Purpose SSD. Up to 16,000 IOPS, independently configurable throughput. Default choice.<br>- <b>gp2</b>: Older general purpose. IOPS tied to size. Prefer gp3 for new volumes.<br>- <b>io2 Block Express</b>: High performance. Up to 256,000 IOPS. For critical databases (Oracle, SQL Server).<br><br><b>HDD-based (not for boot):</b><br>- <b>st1</b>: Throughput Optimized HDD. Sequential workloads (log processing, MapReduce).<br>- <b>sc1</b>: Cold HDD. Cheapest. For infrequently accessed data.<br><br><b>Key facts:</b><br>- EBS volumes are locked to one AZ<br>- Snapshots (stored in S3) are AZ-independent<br>- gp3 is always cheaper AND better than gp2<br><br><b>Common mistake:</b> Still using gp2 on existing instances. Migrate to gp3 — it's cheaper and has higher baseline performance.<br><br><b>Senior insight:</b> EBS Snapshots are incremental and go to S3. Automate snapshots with Data Lifecycle Manager (DLM) or AWS Backup.
What is Amazon CloudWatch Logs Insights?	<b>CloudWatch Logs Insights</b> is an interactive query tool for CloudWatch Logs — analyze your log data with a purpose-built query language.<br><br><b>Key query capabilities:</b><br>- Filter, aggregate, and visualize log data<br>- Join multiple log groups<br>- Detect patterns and anomalies<br><br><b>Example queries:</b><br><code># Top 20 URLs with highest error rate<br>filter status >= 400<br>| stats count(*) as errors by requestUri<br>| sort errors desc | limit 20</code><br><br><code># Lambda duration P99<br>filter @type = "REPORT"<br>| stats pct(@duration, 99) as p99 by bin(5m)</code><br><br><b>Common mistake:</b> Using CloudWatch Logs Insights for compliance log archival — it's a query tool, not a storage strategy. Store long-term logs in S3 and query with Athena.<br><br><b>Senior insight:</b> Save frequently used queries and add them to CloudWatch Dashboards for operational visibility. Combine with CloudWatch Contributor Insights to automatically identify top-N contributors to spikes (e.g., top IP addresses generating errors).
What is Elastic Beanstalk?	<b>Elastic Beanstalk</b> is a Platform as a Service (PaaS) that automatically handles deployment, scaling, load balancing, and monitoring — you just upload your application code.<br><br><b>Supported platforms:</b> Node.js, Python, Java, PHP, Ruby, Go, .NET, Docker<br><br><b>What Beanstalk manages for you:</b><br>- EC2 instances<br>- Auto Scaling Groups<br>- Load Balancers<br>- RDS (optional, not recommended to attach)<br>- CloudWatch monitoring<br><br><b>Deployment policies:</b> All at once, Rolling, Rolling with additional batch, Immutable, Blue/Green<br><br><b>Common mistake:</b> Storing the database inside Beanstalk's lifecycle — if you delete the environment, the DB gets deleted. Always create RDS outside Beanstalk.<br><br><b>Senior insight:</b> Beanstalk is great for rapid prototyping and small teams. For anything serious, you'll quickly want more control than it offers. Beanstalk creates underlying CloudFormation stacks — but they're "owned" by Beanstalk, making direct modification risky.
What is Amazon Lightsail vs EC2?	<b>Lightsail</b> is a simplified VPS (Virtual Private Server) service designed for developers who want a straightforward cloud experience without AWS complexity.<br><br><b>Lightsail includes (bundled):</b> Virtual machine + SSD storage + data transfer + DNS + static IP — for a fixed monthly price (starting ~$3.50/month).<br><br><b>When to use Lightsail:</b><br>- Simple websites, blogs (WordPress)<br>- Small applications with predictable resources<br>- Developers new to cloud<br>- Dev/test environments with fixed budgets<br><br><b>When to use EC2:</b><br>- Production workloads requiring fine-grained control<br>- Auto Scaling, custom networking (VPC), complex IAM<br>- Instance types beyond what Lightsail offers<br>- Integration with other AWS services<br><br><b>Common mistake:</b> Using Lightsail for production at scale — it has simplified networking that doesn't integrate with VPC, making it hard to connect to other AWS services securely.<br><br><b>Senior insight:</b> Lightsail is excellent for agency developers managing many small client sites. For DevOps engineers building scalable systems, EC2/ECS/EKS is almost always the right answer.
What is Amazon EMR?	<b>EMR (Elastic MapReduce)</b> is AWS's managed big data platform — run Apache Hadoop, Spark, Hive, HBase, Flink, and other frameworks to process massive datasets.<br><br><b>Key concepts:</b><br>- <b>Cluster</b>: a collection of EC2 instances (master + core + task nodes)<br>- <b>Master node</b>: coordinates the cluster (YARN, HDFS NameNode)<br>- <b>Core nodes</b>: run tasks and store data<br>- <b>Task nodes</b>: run tasks only (no data storage) — ideal for Spot Instances<br><br><b>EMR on EKS:</b> Run Spark jobs on your EKS cluster — no separate EMR cluster to manage.<br><br><b>EMR Serverless:</b> Submit Spark/Hive jobs without managing clusters — auto-scales, pay per use.<br><br><b>Common mistake:</b> Running persistent clusters for batch workloads — spin up clusters when needed and terminate after the job completes (transient clusters).<br><br><b>Senior insight:</b> Use Spot Instances for task nodes (fault-tolerant, stateless). For cost-optimized Spark, use Graviton + Spot + EMR Serverless. Store data in S3 (not HDFS) to decouple storage from compute.
What is AWS Batch?	<b>AWS Batch</b> is a fully managed service for running batch computing jobs at scale — handles job queuing, scheduling, and dynamic provisioning of compute resources.<br><br><b>Key concepts:</b><br>- <b>Job Definition</b>: template defining the Docker container, CPU, memory, IAM role<br>- <b>Job Queue</b>: where submitted jobs wait<br>- <b>Compute Environment</b>: EC2 or Fargate resources that run jobs<br><br><b>Why use Batch over Lambda:</b><br>- No 15-minute execution limit<br>- Can use GPUs<br>- Handles job dependencies (Job A must complete before Job B)<br>- Can use Spot Instances automatically<br>- For CPU/memory-intensive workloads<br><br><b>AWS Batch vs Step Functions:</b><br>- Batch: resource provisioning and large-scale job scheduling<br>- Step Functions: workflow orchestration with branching logic<br>- Often combined: Step Functions orchestrates Batch jobs<br><br><b>Common mistake:</b> Using EC2 compute environment when Fargate works — Fargate is simpler and removes node management overhead for most batch workloads.<br><br><b>Senior insight:</b> Use Spot compute environments for cost savings (up to 90%). Batch automatically retries Spot interruptions. Set <code>attempts: 3</code> in job definitions.
What is Amazon MSK (Managed Streaming for Kafka)?	<b>MSK</b> is a fully managed Apache Kafka service — run Kafka clusters without managing ZooKeeper (or KRaft), brokers, OS patching, or storage scaling.<br><br><b>Key features:</b><br>- Fully Kafka API-compatible (use any Kafka client library)<br>- Multi-AZ deployment for HA<br>- Automatic broker failure detection and replacement<br>- TLS encryption in transit, KMS encryption at rest<br>- MSK Connect: run Kafka connectors (S3 sink, DynamoDB, etc.) as managed service<br>- MSK Serverless: no capacity planning, auto-scales<br><br><b>MSK vs Kinesis Data Streams:</b><br>- MSK: Kafka ecosystem, open standards, portable<br>- Kinesis: AWS-proprietary, simpler, tighter AWS integration, Firehose delivery<br>- MSK: for teams already using Kafka or needing Kafka ecosystem tools (Kafka Streams, ksqlDB, Schema Registry)<br><br><b>Common mistake:</b> Running Kafka on EC2 manually when MSK is available — managing ZooKeeper quorums and broker replacement is painful operational overhead.<br><br><b>Senior insight:</b> MSK Serverless removes the biggest pain point of Kafka: capacity planning. Use MSK Serverless for dev/staging and MSK provisioned for production with predictable load.
What is AWS AppSync?	<b>AppSync</b> is a fully managed GraphQL API service — connect your frontend to multiple backends (DynamoDB, Lambda, RDS, HTTP endpoints) through a single GraphQL schema.<br><br><b>Key features:</b><br>- GraphQL subscriptions for real-time updates (via WebSockets)<br>- Built-in offline support for mobile apps<br>- Data sources: DynamoDB, Lambda, RDS (Aurora Serverless), OpenSearch, HTTP APIs<br>- Caching at the resolver level<br>- Authentication: Cognito, IAM, API key, Lambda authorizer, OIDC<br><br><b>When to use AppSync:</b><br>- Mobile apps needing offline + real-time sync<br>- Frontend teams wanting to query multiple backends in one call<br>- Rapid prototyping with DynamoDB<br><br><b>Common mistake:</b> Using AppSync for simple CRUD REST APIs — the GraphQL overhead isn't worth it. AppSync excels when you need subscriptions or federation across multiple data sources.<br><br><b>Senior insight:</b> AppSync + Amplify is a rapid development stack for full-stack serverless apps. For production GraphQL at scale, evaluate Apollo Server on Lambda or ECS as an alternative with more customization.
What is AWS Global Accelerator?	<b>Global Accelerator</b> improves availability and performance of global applications by routing traffic through AWS's private network instead of the public internet.<br><br><b>How it works:</b><br>User → nearest AWS Edge Location → AWS private backbone → your ALB/NLB/EC2 in target Region<br><br><b>Key features:</b><br>- Two static anycast IP addresses (don't change even as backends change)<br>- Automatic failover across Regions in <30 seconds<br>- Traffic dials to shift % of traffic to endpoints (blue/green)<br>- Works with ALB, NLB, EC2, Elastic IPs<br><br><b>Global Accelerator vs CloudFront:</b><br>- CloudFront: caches content at edge (CDN). For static assets and cacheable HTTP.<br>- Global Accelerator: NO caching. Optimizes the network path for dynamic requests (gaming, APIs, VoIP). Reduces latency by ~60% for non-cacheable traffic.<br><br><b>Common mistake:</b> Using CloudFront for non-cacheable API traffic and wondering why it doesn't help much. Use Global Accelerator for APIs and dynamic content.<br><br><b>Senior insight:</b> Global Accelerator's static IPs are valuable for whitelisting at enterprise firewalls. Give your enterprise customers two IPs that never change, regardless of how your backend evolves.
What is AWS Outposts?	<b>AWS Outposts</b> brings AWS infrastructure, services, APIs, and tools to on-premises locations — run AWS services in your own data center.<br><br><b>Why use Outposts:</b><br>- Ultra-low latency requirements (factory automation, healthcare devices)<br>- Data residency — data cannot leave specific physical location<br>- Modernize on-prem workloads using AWS APIs while staying local<br><br><b>What runs on Outposts:</b><br>EC2, ECS, EKS, RDS, S3 (Outposts S3), ElastiCache, ALB — same APIs as cloud<br><br><b>Management:</b><br>- AWS manages the hardware (owns, deploys, patches the physical servers)<br>- You manage your workloads on the Outpost<br>- Control plane is in the parent AWS Region (requires connectivity)<br><br><b>Common mistake:</b> Treating Outposts as an alternative to the cloud — it's an extension of the cloud to your location, not a replacement. If the parent Region loses connectivity, some management capabilities are affected.<br><br><b>Senior insight:</b> Outposts is expensive and operationally complex. Exhaust alternatives (Local Zones, Wavelength Zones, VPN/Direct Connect latency optimization) before committing.
What is AWS Local Zones?	<b>Local Zones</b> are AWS infrastructure deployments closer to large population centers — extending Regions to metropolitan areas for single-digit millisecond latency.<br><br><b>How they work:</b><br>- A Local Zone is an extension of a parent Region<br>- Available in cities like Los Angeles, Dallas, Boston, and internationally<br>- Run EC2, ECS, RDS, ElastiCache in the Local Zone<br>- Connected to the parent Region via AWS private network<br><br><b>Use cases:</b><br>- Media & entertainment (video rendering, live production)<br>- Real-time gaming<br>- Machine learning inference at the edge<br>- Applications where <10ms to users in specific cities is required<br><br><b>Local Zones vs Outposts vs Wavelength:**<br>- Local Zones: AWS-managed, in large cities, best for latency to urban users<br>- Outposts: in YOUR data center, for private/compliance needs<br>- Wavelength: embedded in 5G carrier networks, for ultra-low latency mobile apps<br><br><b>Senior insight:</b> Local Zones are a middle ground between a full Region and Outposts. Good for media companies and game studios that need sub-10ms to their content creation teams without the complexity of Outposts.
What is Amazon SageMaker in a DevOps context?	<b>SageMaker</b> is AWS's fully managed ML platform — build, train, and deploy ML models. From a DevOps perspective, key concerns are MLOps, pipeline automation, and infrastructure.<br><br><b>DevOps-relevant components:</b><br>- <b>SageMaker Pipelines</b>: CI/CD for ML — define training/evaluation/deployment workflows as code<br>- <b>SageMaker Model Registry</b>: version and approve models before production<br>- <b>SageMaker Endpoints</b>: deploy models as managed REST APIs (auto-scaling)<br>- <b>SageMaker Feature Store</b>: centralized feature repository for training and serving<br>- <b>SageMaker Ground Truth</b>: managed data labeling<br><br><b>MLOps pattern:</b><br>Code commit → SageMaker Pipeline → train → evaluate → Model Registry (approval gate) → deploy endpoint → monitor with Model Monitor<br><br><b>Common mistake:</b> Treating ML deployments like regular code deployments. Models need data drift detection, retraining triggers, and canary deployments — SageMaker Model Monitor handles drift detection.<br><br><b>Senior insight:</b> Infrastructure for SageMaker training jobs is ephemeral — the cluster spins up, trains, terminates. Use Spot training instances (up to 90% savings) with checkpointing to handle interruptions.
What is AWS CloudFormation StackSets?	<b>StackSets</b> extend CloudFormation to deploy stacks across multiple AWS accounts and regions in a single operation — perfect for multi-account governance at scale.<br><br><b>Key concepts:</b><br>- <b>Stack Set</b>: defines the template and deployment targets<br>- <b>Stack Instance</b>: a stack in a specific account/region combination<br>- <b>Service-managed StackSets</b>: integrate with AWS Organizations — automatically deploy to new accounts<br><br><b>Use cases:</b><br>- Deploy baseline security controls (CloudTrail, Config, GuardDuty) to all accounts<br>- Enforce logging buckets, IAM roles, or security baselines<br>- Enable Security Hub across all Regions<br><br><b>Deployment modes:</b><br>- <b>Sequential</b>: deploy to one account/region at a time (safer)<br>- <b>Parallel</b>: deploy concurrently (faster)<br><br><b>Common mistake:</b> Not setting Failure Tolerance and Max Concurrent settings — by default, one failure stops all deployments. Set <code>--failure-tolerance-count 1</code> for rolling deployments.<br><br><b>Senior insight:</b> Use Service-managed StackSets with "Auto deployment" enabled. When new accounts join the org, baseline security stacks are automatically deployed — zero-touch compliance onboarding.
What is Amazon ECS Service Discovery?	<b>ECS Service Discovery</b> allows ECS services to automatically register themselves in DNS so other services can find them by name — no hardcoded IPs or manual configuration.<br><br><b>How it works:</b><br>- Uses AWS Cloud Map (managed service discovery)<br>- Each ECS task registers with a namespace (e.g., <code>myapp.local</code>)<br>- Other services resolve: <code>auth.myapp.local</code> → IP of healthy auth service task<br>- Health checks automatically deregister unhealthy tasks<br><br><b>Two types of discovery:</b><br>- <b>DNS-based</b>: Route 53 A/SRV records<br>- <b>API-based</b>: Cloud Map API for non-DNS scenarios<br><br><b>ECS Service Discovery vs ALB:</b><br>- ALB: for external or service-to-service HTTP traffic needing advanced routing<br>- Service Discovery: for internal service-to-service communication (lower cost, lower latency)<br><br><b>Common mistake:</b> Using ALBs for all inter-service communication inside ECS — every ALB costs ~$20/month. Use Service Discovery for internal traffic and ALBs only for external-facing services.<br><br><b>Senior insight:</b> In EKS, this is handled natively by kube-dns + CoreDNS. In ECS, Service Discovery is an explicit setup step — don't skip it for microservice architectures.
What is the difference between AWS Managed Policies, Inline Policies, and Customer Managed Policies?	<b>AWS Managed Policies:</b><br>- Created and maintained by AWS<br>- Pre-built for common use cases (e.g., <code>AmazonS3ReadOnlyAccess</code>, <code>AdministratorAccess</code>)<br>- Updated by AWS when new services/features are added<br>- Can be attached to multiple identities<br>- Cannot be modified<br><br><b>Customer Managed Policies:</b><br>- Created and managed by you<br>- Standalone, reusable — attach to multiple users/groups/roles<br>- Version-controlled (up to 5 versions)<br>- Best for custom permission sets used across your org<br><br><b>Inline Policies:</b><br>- Embedded directly into a single user, group, or role<br>- Deleted when the identity is deleted<br>- Not reusable<br>- Use for one-off permissions you never want applied elsewhere<br><br><b>Common mistake:</b> Using AWS Managed Policies like <code>AdministratorAccess</code> for production service roles — these are far too broad. Create least-privilege Customer Managed Policies.<br><br><b>Senior insight:</b> Prefer Customer Managed Policies over Inline. They're reusable, versionable, and easier to audit. Use AWS Managed Policies only as a starting point, then scope them down.
What is AWS Resource Access Manager (RAM)?	<b>AWS RAM</b> allows you to share AWS resources with other AWS accounts or organizational units — without duplicating resources or using complex cross-account IAM setups.<br><br><b>Shareable resources (examples):</b><br>- VPC Subnets (share subnets from a central account to workload accounts)<br>- Transit Gateway<br>- Route 53 Resolver rules<br>- License Manager configurations<br>- Aurora DB clusters (read-only)<br><br><b>Most common use case — Shared VPC:</b><br>Central Networking account owns the VPC. Other accounts deploy EC2/ECS/RDS into the shared subnets. Resources are isolated per account but share the network infrastructure.<br><br><b>Benefits:</b><br>- One VPC to manage instead of peering dozens<br>- Centralized networking team controls routing, NACLs<br>- Application teams control their own EC2/SGs within shared subnets<br><br><b>Common mistake:</b> Giving all accounts their own VPCs and using VPC Peering at scale — this explodes into an unmanageable mesh. RAM + Shared VPC or Transit Gateway is the scalable answer.<br><br><b>Senior insight:</b> Shared VPC via RAM is a key pattern in AWS Landing Zone / Control Tower architectures. Central networking account + RAM = clean separation of concerns.
What is AWS Control Tower?	<b>Control Tower</b> is AWS's opinionated, automated landing zone service — sets up and governs a multi-account AWS environment following AWS best practices.<br><br><b>What it sets up automatically:</b><br>- Management, Log Archive, and Audit accounts<br>- AWS Organizations structure with OUs<br>- IAM Identity Center (SSO)<br>- CloudTrail and Config across all accounts<br>- Guardrails (SCPs and Config rules) for compliance<br><br><b>Key concepts:</b><br>- <b>Landing Zone</b>: the baseline multi-account environment<br>- <b>Account Factory</b>: self-service provisioning of new accounts via Service Catalog<br>- <b>Guardrails</b>: preventive (SCPs) or detective (Config rules) controls<br>- <b>Drift Detection</b>: alerts if manual changes break the governance setup<br><br><b>Common mistake:</b> Building a custom landing zone manually before learning what Control Tower offers. Control Tower often covers 80% of what teams build by hand, faster.<br><br><b>Senior insight:</b> Account Factory for Terraform (AFT) extends Control Tower with GitOps-style account provisioning. Commit a Terraform file → new AWS account with all guardrails, baseline config, and SSO access automatically applied.
What is AWS Artifact?	<b>AWS Artifact</b> is a self-service portal for accessing AWS compliance reports and security documentation.<br><br><b>What you can find:</b><br>- AWS SOC reports (SOC 1, SOC 2, SOC 3)<br>- ISO certifications (ISO 27001, 27017, 27018)<br>- PCI DSS compliance documents<br>- HIPAA BAA (Business Associate Agreement)<br>- FedRAMP documentation<br>- GDPR resources<br><br><b>Why it matters for DevOps:</b><br>When enterprise customers or auditors ask "Is AWS compliant with XYZ?", Artifact is where you get the evidence. Download AWS's compliance reports and include them in your own audit packages.<br><br><b>Artifact Agreements:</b><br>- Accept legal agreements (BAA, NDA) in-console without back-and-forth with AWS<br>- HIPAA BAA is a prerequisite for storing Protected Health Information (PHI) on AWS<br><br><b>Common mistake:</b> Thinking your workload is automatically compliant because AWS has certifications. AWS's compliance covers THEIR infrastructure — YOUR application still needs to meet requirements on top.<br><br><b>Senior insight:</b> AWS Shared Responsibility + Artifact = your compliance strategy. Use Artifact to document what AWS covers, then layer your own controls on top.
What is Amazon Timestream?	<b>Timestream</b> is a fully managed, serverless time-series database optimized for IoT and operational telemetry — storing and querying time-stamped data efficiently.<br><br><b>Key features:</b><br>- Automatically moves recent data to memory, older data to magnetic store (tiered)<br>- Scales up to trillions of events/day<br>- Built-in time-series analytics functions<br>- Native integration with Grafana<br>- Query with SQL-like syntax plus time-series functions<br><br><b>Use cases:</b><br>- IoT sensor data (temperature, humidity, GPS)<br>- DevOps metrics (CPU, latency, request rates)<br>- Application performance monitoring<br>- Industrial equipment telemetry<br><br><b>Timestream vs DynamoDB vs RDS for time-series:</b><br>- Timestream: purpose-built, automatic tiering, time-aware queries, best for pure time-series<br>- DynamoDB: general NoSQL, flexible but no built-in time-series analytics<br>- RDS: general relational, slow for high-cardinality time-series queries<br><br><b>Senior insight:</b> For DevOps metric storage at scale with Grafana visualization, Timestream + Grafana is a strong alternative to a self-managed Prometheus + Thanos stack.
What is Amazon OpenSearch Service?	<b>OpenSearch Service</b> (formerly Elasticsearch) is a managed search and analytics engine — search, analyze, and visualize large volumes of data in near real-time.<br><br><b>Key use cases:</b><br>- Full-text search for applications<br>- Log analytics (alternative to ELK stack)<br>- Metrics and APM data<br>- Security analytics (SIEM-lite)<br><br><b>Common AWS architecture — log pipeline:</b><br>CloudWatch Logs → Subscription Filter → Kinesis Firehose → OpenSearch → Kibana/OpenSearch Dashboards<br><br><b>Key features:</b><br>- Auto-scaling with UltraWarm (S3-backed warm storage)<br>- Cold storage tier for rarely accessed indexes<br>- OpenSearch Security: fine-grained access control, SAML, IAM auth<br>- Anomaly detection built-in<br><br><b>Common mistake:</b> Indexing everything with default settings — high cardinality fields (UUIDs, IPs) as keyword fields balloon index size. Design mappings carefully.<br><br><b>Senior insight:</b> OpenSearch Serverless (2023) removes cluster management entirely. Pay per OCU (OpenSearch Compute Unit). Best for variable search workloads where managing cluster sizing is painful.
What is AWS DataSync?	<b>DataSync</b> is an online data transfer service that moves large amounts of data to and from AWS — accelerating migrations, data pipeline feeds, and archival.<br><br><b>Key capabilities:</b><br>- Transfer between on-premises NFS/SMB/HDFS and S3, EFS, FSx<br>- Transfer between AWS storage services (S3 ↔ EFS ↔ FSx)<br>- Preserves metadata (timestamps, permissions, ACLs)<br>- Handles network optimization, encryption, integrity verification<br>- Schedules for ongoing sync jobs<br><br><b>DataSync vs Storage Gateway vs Transfer Family:</b><br>- DataSync: bulk, scheduled, or one-time data movement. Migration focus.<br>- Storage Gateway: hybrid cloud, ongoing access to S3/EBS from on-prem as NFS/SMB/iSCSI<br>- Transfer Family: SFTP/FTP/FTPS server frontend for S3/EFS (for B2B file exchange partners)<br><br><b>Speed:</b> DataSync can use 10Gbps links and achieves up to 20x faster transfer than manual tools like rsync<br><br><b>Common mistake:</b> Using S3 CLI for large-scale migrations — DataSync handles retries, bandwidth throttling, and verification automatically.<br><br><b>Senior insight:</b> DataSync is often overlooked in migration projects. It significantly reduces migration risk by handling data integrity checks, change tracking, and incremental syncs before cutover.
