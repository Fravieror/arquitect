1→1→What is the difference between NUnit, xUnit, and MSTest in C#?	All three are test frameworks for C#/.NET, but with key differences. NUnit: oldest, feature-rich, uses [TestFixture] and [Test] attributes. xUnit: modern, created by NUnit's original author, uses [Fact] and [Theory], no shared state between tests by default (each test gets a fresh class instance). MSTest: Microsoft's built-in framework, tightly integrated with Visual Studio, uses [TestClass] and [TestMethod]. Senior insight: xUnit's design forces better test isolation. MSTest wins on tooling integration in enterprise shops. NUnit shines for parameterized test flexibility.
2→2→When would you choose xUnit over NUnit or MSTest for a new C# project?	Choose xUnit when you want opinionated test isolation (new instance per test prevents shared state bugs), modern C# idioms (IDisposable for cleanup instead of [TearDown]), and strong OSS ecosystem. Choose MSTest for heavy Azure DevOps integration or team familiarity. Choose NUnit when you need rich parameterized tests with [TestCaseSource] or migrating a legacy NUnit project. Mistake to avoid: treating them as interchangeable — xUnit's constructor/dispose lifecycle vs NUnit's [SetUp]/[TearDown] behaves very differently.
3→3→How does test setup and teardown differ between NUnit, xUnit, and MSTest?	NUnit: uses [SetUp] / [TearDown] methods and [OneTimeSetUp] / [OneTimeTearDown] for class-level. MSTest: uses [TestInitialize] / [TestCleanup] and [ClassInitialize] / [ClassCleanup]. xUnit: no special attributes — use the class constructor for setup and implement IDisposable for teardown. For class-level shared state use IClassFixture<T>. Senior insight: xUnit's approach makes the lifecycle explicit in the type system, reducing invisible side effects across tests.
4→4→What is a [Theory] in xUnit and how does it differ from a [Fact]?	[Fact] is a single test with no parameters — it either passes or fails. [Theory] is a parameterized test that runs multiple times with different input data, driven by [InlineData], [MemberData], or [ClassData] attributes. Example: [Theory] [InlineData(2, 3, 5)] [InlineData(0, 0, 0)] public void Add_Returns_Sum(int a, int b, int expected). Equivalent in NUnit is [TestCase]. Senior insight: theories are essential for boundary value testing without writing duplicate test methods.
5→5→What is the difference between unit, integration, functional, and smoke tests?	Unit: tests a single class/method in complete isolation using mocks — fast, no I/O. Integration: tests real interactions between components (e.g., service + database) — slower, needs infrastructure. Functional (end-to-end): tests full user flows through the system — slowest, most realistic. Smoke: a minimal subset of tests run after deployment to confirm the system is alive (login works, home page loads). Analogy: unit = testing a car engine in a lab; integration = engine + transmission on a test bench; functional = full test drive; smoke = just turn the key and check it starts.
6→6→What is the Testing Pyramid and why does it matter for architecture?	The Testing Pyramid (by Mike Cohn) says: write MANY unit tests (fast, cheap), SOME integration tests, FEW end-to-end tests. The pyramid shape reflects cost and speed. Inverting it (many E2E, few unit) creates a "testing ice cream cone" — slow CI, flaky tests, hard to debug failures. Senior insight: the pyramid is a guideline, not a law. A CRUD API might legitimately have more integration tests than unit tests if the domain logic is thin. Match test distribution to where your risk actually lives.
7→7→What is defect prevention vs defect detection, and why does it matter at the architect level?	Defect detection: finding bugs after the code is written (testing, QA). Defect prevention: making bugs impossible or unlikely before code is written (code reviews, pair programming, static analysis, type systems, design-by-contract). Prevention is always cheaper — a bug caught at design costs 1x; at development 10x; at QA 100x; in production 1000x. Senior insight: invest in type safety, linting, contract testing, and design reviews. Testing is important but it is your last safety net, not your first.
8→8→What is a regression suite and why is it critical for large systems?	A regression suite is a collection of tests (usually automated) that verify previously working functionality still works after a change. It is the automated memory of the system. Without it, every release risks silently breaking existing features. Senior insight: the suite must be fast enough to run on every PR (or at minimum every merge to main). Slow regression suites get skipped, defeating their purpose. Prioritize: unit tests in the suite first, targeted integration tests second, full E2E regression third (often nightly).
9→9→What is frame stability in a test automation framework?	Frame stability means the test framework itself does not introduce flakiness or inconsistency — tests fail because of real defects, not because of timing issues, driver bugs, or test infra problems. A stable framework has: reliable waits (no hard sleeps), isolated state, deterministic setup/teardown, and pinned dependency versions. Analogy: you want your weighing scale to be accurate — if the scale wobbles, you can't trust the reading. Senior insight: framework instability is the #1 killer of test suite trust. Teams stop caring about failures if the framework cries wolf.
10→10→What is platform parity in test automation?	Platform parity means the automated tests run correctly across all target environments — OS versions, browser versions, .NET runtimes, container images. A test that passes on Windows but fails on Linux due to path separators or case sensitivity is a platform parity problem. Senior insight: solve this with containerized test execution (Docker), matrix builds in CI (GitHub Actions matrix strategy), and environment-agnostic test data (avoid absolute paths, locale-sensitive date formats, etc.).
11→11→What is telemetry-driven testing?	Telemetry-driven testing uses production observability data (logs, traces, metrics) to guide what to test, detect failures, and measure test coverage of real user flows. Instead of only writing tests based on requirements, you analyze which code paths users actually hit in production and ensure those are covered. Senior insight: tools like Application Insights, OpenTelemetry, and distributed tracing let you map real traffic to code coverage. This catches the "we tested the happy path but users do X" problem.
12→12→How can telemetry inform test case generation?	Analyze production traces to find: 1) high-frequency paths (must be covered by fast unit/integration tests), 2) error-prone paths (add targeted regression tests), 3) unused code paths (deprioritize testing, consider deleting), 4) latency outliers (add performance assertions). Example: if telemetry shows 30% of users hit a specific filter combination, that combination needs an explicit test case. Senior insight: telemetry-driven testing closes the loop between QA and production reality — tests reflect what users actually do.
13→13→What is a flaky test and why is it dangerous?	A flaky test is one that produces inconsistent results (passes sometimes, fails sometimes) without any code change. Causes: timing issues, shared state, network dependency, random data, test order dependence. Danger: teams start ignoring red builds ("it's probably just flaky"), which masks real failures. Senior insight: flaky tests erode trust in the entire test suite. Track flakiness rates per test. Quarantine flaky tests immediately (move to a separate suite), fix root cause, then re-promote.
14→14→How do you implement a retry wrapper for flaky test steps using async/await in C#?	Use a helper that retries the action up to N times with a delay: public static async Task RetryAsync(Func<Task> action, int maxAttempts = 3, int delayMs = 500) { for (int i = 0; i < maxAttempts; i++) { try { await action(); return; } catch when (i < maxAttempts - 1) { await Task.Delay(delayMs); } } await action(); } Usage: await RetryAsync(() => clickButton()); Senior insight: only retry at the step level for known flaky infrastructure interactions (UI rendering, network jitter). Never retry assertions — that hides real bugs.
15→15→What is the risk of using retry wrappers in test automation?	Retry wrappers mask root causes. A test that passes on the 3rd attempt is still a flaky test — it just doesn't show as red. Risks: 1) real failures are hidden behind retries, 2) test suite runtime increases, 3) flakiness compounds (if each of 100 tests retries 3x in the worst case, CI time triples). Mitigation: log every retry with reason, track retry frequency in telemetry, treat high-retry tests as candidates for quarantine and fix. Retry wrappers are a band-aid, not a cure.
16→16→How does async/await in C# affect test design?	async/await allows non-blocking I/O in tests, which is essential for testing HTTP clients, gRPC calls, database operations, and message queues. In xUnit, test methods can return Task directly: public async Task GetUser_Returns200(). NUnit and MSTest also support async test methods. Common mistake: using .Result or .Wait() in async tests — this can cause deadlocks in certain synchronization contexts (especially in ASP.NET). Always await all async operations in tests.
17→17→How do you use HttpClient in C# tests to send and verify HTTP requests?	Use HttpClient to call real or test-server endpoints. For integration tests with ASP.NET Core: use WebApplicationFactory<Program> to spin up an in-memory test server and get a pre-configured HttpClient. Example: var client = factory.CreateClient(); var response = await client.GetAsync("/api/users/1"); response.EnsureSuccessStatusCode(); var body = await response.Content.ReadFromJsonAsync<UserDto>(); Senior insight: avoid new HttpClient() in test loops — it exhausts sockets. WebApplicationFactory handles lifetime correctly.
18→18→What is WebApplicationFactory and why is it the preferred way to integration-test ASP.NET Core APIs?	WebApplicationFactory<TEntryPoint> spins up your real ASP.NET Core application in-memory without starting a real HTTP server. Tests run against the actual middleware pipeline, routing, and DI container — but fast, without network overhead. You can override services (swap real DB for in-memory, swap external APIs with mocks) via WithWebHostBuilder. Senior insight: this gives you high confidence (real app stack) with unit-test speed (no network). It is the gold standard for ASP.NET Core API integration testing.
19→19→How do you test gRPC services in C#?	Option 1 — Unit test: create the service class directly, pass mock dependencies, call methods like regular C# methods. Option 2 — Integration test: use WebApplicationFactory with GrpcChannel.ForAddress pointed at the in-process server. Example: var channel = GrpcChannel.ForAddress("http://localhost", new GrpcChannelOptions { HttpClient = factory.CreateClient() }); var client = new Greeter.GreeterClient(channel); var reply = await client.SayHelloAsync(new HelloRequest { Name = "World" }); Senior insight: always test the proto-generated contract, not just internal logic — breaking the proto breaks all consumers.
20→20→What is the difference between mocking and stubbing in unit tests?	A stub returns a pre-programmed response ("when called with X, return Y") — used to control inputs to the system under test. A mock also verifies behavior ("was this method called? how many times? with what args?") — used to assert interactions. In C# with Moq: stub = mock.Setup(x => x.GetUser(1)).Returns(user); verify = mock.Verify(x => x.GetUser(1), Times.Once). Mistake: over-mocking (mocking value objects, private methods, concrete classes) makes tests brittle and tied to implementation details rather than behavior.
21→21→What is the difference between a mock and a fake?	Mock: a test double configured to verify specific interactions, generated by a library (Moq, NSubstitute). Fake: a simplified but working implementation of a dependency — e.g., an in-memory repository that stores data in a Dictionary instead of a real database. Fakes are more realistic and less brittle than mocks. Senior insight: prefer fakes for infrastructure (repositories, caches) and mocks for verifying side-effect interactions (email sent, event published). Overusing mocks couples tests to implementation structure.
22→22→What is contract testing and how does it differ from integration testing?	Contract testing verifies that two services agree on their API contract (request/response shape, error codes) — without requiring both to be running simultaneously. Each side tests against a shared contract file. Tool: Pact. Integration testing requires both services running and tests real communication. Senior insight: contract testing is essential in microservices — it catches breaking changes (e.g., renaming a JSON field) before deployment, without the overhead of a full E2E environment. Run contract tests in CI for every consumer and provider.
23→23→What is test isolation and why is it the most important property of a unit test?	Test isolation means a test's result depends ONLY on the code under test — not on other tests, the file system, the database, the clock, or random values. Without isolation: tests interfere with each other (order-dependent failures), tests are slow (real I/O), failures are hard to diagnose (too many moving parts). How to achieve: mock/stub all external dependencies, use in-memory fakes for state, control time via an IClock abstraction, use random seeds. Senior insight: the fastest way to debug a test suite is to make every test runnable in complete isolation.
24→24→What is the Arrange-Act-Assert (AAA) pattern?	AAA is the standard structure for a test: Arrange: set up the system under test and its dependencies (create objects, configure mocks, seed data). Act: invoke the behavior being tested (one action only). Assert: verify the outcome. Example: // Arrange var sut = new Calculator(); // Act var result = sut.Add(2, 3); // Assert Assert.Equal(5, result); Senior insight: a test with multiple Act sections is testing multiple behaviors — split it. The AAA pattern makes test intent immediately readable to any developer.
25→25→What is test coverage and when is it misleading?	Test coverage measures what percentage of code lines/branches are executed during tests. It is useful as a minimum floor (e.g., enforce 80% line coverage in CI). It is misleading when: 1) tests execute code without asserting anything (coverage without verification), 2) 100% coverage doesn't mean all important scenarios are tested, 3) it incentivizes trivial tests to hit numbers. Senior insight: coverage is a necessary but not sufficient condition for test quality. Mutation testing (e.g., Stryker for C#) is a stronger signal — it checks if your tests actually catch injected bugs.
26→26→What is mutation testing and how does it measure test suite strength?	Mutation testing automatically introduces small bugs (mutations) into your source code — e.g., changing > to >=, flipping a boolean — then runs your test suite. If a test fails: the mutation is "killed" (good, tests caught it). If all tests pass: the mutation "survived" (bad, tests missed a real bug). Tool for C#: Stryker.NET. Mutation score = killed mutations / total mutations. Senior insight: a 90% mutation score is far more meaningful than 90% line coverage. Use it on critical domain logic where correctness matters most.
27→27→What is a smoke test in the context of a production deployment?	A smoke test is a minimal, fast test executed immediately after deployment to verify the system is basically functional — not correct in every detail, just alive. Examples: health check endpoint returns 200, login flow completes, critical API endpoint responds. Named after hardware testing: power on a new circuit board and see if it smokes. Senior insight: smoke tests must be fast (under 2 minutes), environment-specific (use prod or staging URLs), and automated in the deployment pipeline. Failed smoke tests should automatically trigger a rollback.
28→28→What is a canary deployment and how does testing relate to it?	A canary deployment routes a small percentage of traffic (e.g., 5%) to the new version before full rollout. Tests in this context are monitoring + telemetry: error rate, latency, and business metrics are compared between canary and baseline. If metrics degrade, the canary is rolled back automatically. Senior insight: canary deployments are a form of production testing under real load. Combine with feature flags and automated rollback (e.g., Azure Deployment Slots, Argo Rollouts). This is risk mitigation, not quality assurance — it catches what pre-prod testing missed.
29→29→What is the difference between end-to-end tests and functional tests?	These terms are often used interchangeably, but there is a distinction: Functional tests verify a specific function or user story works correctly (can be UI or API level). End-to-end (E2E) tests simulate a complete user journey across multiple systems from start to finish (e.g., user registers → logs in → makes a purchase → receives email confirmation). E2E implies crossing system boundaries (UI, API, DB, email service). Senior insight: E2E tests are the most expensive to write and maintain — they break on unrelated infrastructure changes. Keep the E2E suite small and focused on critical user journeys only.
30→30→What is test data management and why is it a senior concern?	Test data management is the strategy for creating, maintaining, and cleaning up the data your tests need. Poor strategies: hard-coded IDs (breaks when DB is reset), shared test data (tests interfere), production data in tests (compliance violation). Good strategies: test data builders (Builder pattern to construct domain objects), database seeding per test (EF Core InMemory or real DB with transactions rolled back after each test), and synthetic data generation (Bogus library for C#). Senior insight: bad test data is the #1 cause of intermittent integration test failures.
31→31→What is the Builder pattern in the context of test data?	The Builder pattern creates complex test objects with sensible defaults and fluent customization. Example: var user = new UserBuilder().WithEmail("test@example.com").WithRole(Role.Admin).Build(); Defaults handle all other fields so each test only sets what it cares about. This prevents: 1) brittle tests that break when a new required field is added, 2) verbose test setup that obscures test intent. Senior insight: create a Builders/ folder in your test project and invest in it early — it pays dividends as the domain model grows.
32→32→How does dependency injection improve testability in C#?	Dependency injection (DI) decouples classes from their concrete dependencies, allowing tests to inject mocks or fakes instead. Without DI: public class OrderService { private readonly SqlDb _db = new SqlDb(); } — impossible to unit test without a real database. With DI: public class OrderService(IOrderRepository repo) — tests inject a mock IOrderRepository. Senior insight: if a class is hard to unit test, it usually means the dependencies are not injected. Testability is a forcing function for good architecture. Classes that use new() for infrastructure dependencies are an anti-pattern.
33→33→What is test doubles terminology: dummy, stub, spy, mock, fake?	Dummy: passed around but never used (fills a parameter). Stub: returns pre-programmed answers to calls. Spy: a real object that records calls for later assertion (like a stub that also tracks interactions). Mock: pre-programmed with expectations and verifies them. Fake: a simplified working implementation (in-memory DB). From Gerard Meszaros' "xUnit Test Patterns." Senior insight: most developers use "mock" to mean all of these — which is fine in conversation, but knowing the distinctions helps you choose the right tool: fakes for stateful infrastructure, mocks for verifying side effects, stubs for controlling return values.
34→34→What is the testing approach for message-driven / event-driven architectures?	Tests need to account for asynchrony. Strategies: 1) Unit test the message handler in isolation (pass the message object directly, no broker needed). 2) Integration test with an in-memory broker (e.g., MassTransit's InMemoryTestHarness). 3) Contract test the message schema (ensure producer and consumer agree on event structure). 4) E2E test with a real broker (Kafka, RabbitMQ) in a Docker Compose environment. Senior insight: never rely solely on E2E tests for event-driven systems — they are too slow and flaky. The handler's logic should be testable without a broker.
35→35→What is MassTransit's InMemoryTestHarness and when do you use it?	InMemoryTestHarness is a test utility from the MassTransit library that runs message consumers in-process without a real message broker. You publish a message and then assert that the consumer received and processed it. Example: await harness.InputQueueSendEndpoint.Send(new OrderPlaced { OrderId = 1 }); Assert.True(await harness.Consumed.Any<OrderPlaced>()); Senior insight: use the harness for integration tests of message consumers. It catches routing, serialization, and consumer logic bugs without the overhead of a real broker.
36→36→How do you test time-dependent code (e.g., expiry, scheduling) in C#?	Inject an IClock or IDateTimeProvider abstraction instead of using DateTime.UtcNow directly. In tests, inject a FakeClock that you control. Example: public interface IClock { DateTime UtcNow { get; } } In test: var clock = new FakeClock(new DateTime(2025, 1, 1)); sut = new TokenService(clock); clock.AdvanceBy(TimeSpan.FromHours(2)); Assert.True(sut.IsExpired(token)); Senior insight: code that calls DateTime.Now directly is impossible to unit test for time-based behavior. The clock abstraction is a canonical example of making code testable through DI.
37→37→What is property-based testing and how does it differ from example-based testing?	Example-based testing (normal unit tests): you choose specific inputs and assert specific outputs. Property-based testing: you define a property (a rule that should always hold) and a library generates hundreds of random inputs to try to falsify it. Example: "sorting a list should never change its length" — the library generates random lists to test this. Tool for C#: FsCheck or CsCheck. Senior insight: property-based testing excels at finding edge cases you would never think to write manually — boundary values, empty inputs, unicode strings, negative numbers.
38→38→What is approval testing (snapshot testing) and when is it useful?	Approval testing captures the output of a function (text, HTML, JSON, image) and saves it as an "approved" baseline file. Future test runs compare against the baseline — any change is flagged. Tool for C#: ApprovalTests. Useful for: testing complex report generation, API response shapes, serialization output, or any output too large to assert field-by-field. Senior insight: great for detecting unintended regressions in output format. The approved file becomes living documentation. Drawback: every intentional output change requires updating the baseline (can be tedious).
39→39→What is the difference between black-box and white-box testing?	Black-box testing: tests the system from the outside — inputs and outputs only, no knowledge of internal implementation. Tests are written against the public API or UI. White-box testing: tests with full knowledge of internal structure — tests may call private methods, check internal state, or verify specific code paths. Senior insight: prefer black-box tests for long-term maintainability — they don't break when you refactor internals. White-box tests are brittle. Exception: testing complex internal algorithms where the internal structure IS the interesting behavior (e.g., a parser, a compiler pass).
40→40→What is the shift-left testing principle?	Shift-left means moving testing earlier in the development lifecycle — closer to the moment code is written, rather than waiting for a dedicated QA phase. Methods: unit tests written alongside code (TDD), static analysis in the IDE (Roslyn analyzers), linting in pre-commit hooks, security scanning in the PR pipeline. Senior insight: the ROI of shift-left is enormous — bugs found by a developer in their IDE cost almost nothing to fix. Bugs found in a QA environment two weeks later are 10-100x more expensive. Shift-left is a systemic quality strategy, not just a testing tactic.
41→41→What is Test-Driven Development (TDD) and what are its benefits and trade-offs?	TDD: write a failing test first, then write the minimum code to make it pass, then refactor (Red → Green → Refactor). Benefits: forces clear specification before coding, ensures 100% test coverage of new code, produces naturally testable (decoupled) designs, gives confidence when refactoring. Trade-offs: slower upfront speed, requires discipline, works best for well-understood domains. Senior insight: TDD is most valuable for complex domain logic. Applying strict TDD to exploratory/prototype code is often counterproductive. Use it as a tool, not a religion.
42→42→What is Behavior-Driven Development (BDD) and how does it relate to testing?	BDD extends TDD by writing tests in natural language that stakeholders can read. Format: Given [context] / When [action] / Then [outcome]. Tools for C#: SpecFlow (Gherkin syntax), Reqnroll. The Gherkin scenarios live in .feature files and bind to step definition methods. Senior insight: BDD's value is communication, not the tooling. The real benefit is forcing developers and product owners to agree on behavior in concrete examples before coding starts. Avoid BDD for internal/technical tests — the ceremony is not worth it there.
43→43→How should a Senior Architect design a test automation framework from scratch?	Key decisions: 1) Layered structure: unit tests → integration tests → E2E tests in separate projects. 2) Shared test utilities project: builders, fakes, helpers. 3) CI integration: tests run on every PR, broken build blocks merge. 4) Test data strategy: isolated per test, cleaned up reliably. 5) Flakiness policy: track and quarantine flaky tests. 6) Coverage gate: minimum threshold enforced in CI. 7) Test naming convention: MethodName_Scenario_ExpectedResult. Senior insight: a framework is only as good as its adoption. Invest in developer experience — fast test runs, clear failure messages, easy local debugging.
44→44→What naming convention should tests follow and why does it matter?	Most common convention: MethodName_Scenario_ExpectedResult. Example: PlaceOrder_WhenInventoryIsZero_ThrowsOutOfStockException. Benefits: test name describes exactly what is tested, failed test names in CI are self-explaining without reading code, test names serve as living documentation. Alternative (BDD style): Should_ThrowException_When_InventoryIsZero. Senior insight: consistent naming is a team contract. Add it to your CONTRIBUTING.md. When a CI build fails at 3am, a well-named test saves 20 minutes of debugging.
45→45→What is the role of CI/CD in a test automation strategy?	CI (Continuous Integration) runs automated tests on every push/PR and blocks merges on failure. CD (Continuous Delivery/Deployment) gates releases on test results. A mature pipeline: 1) PR triggers unit + fast integration tests (< 5 min). 2) Merge to main triggers full regression suite. 3) Deployment to staging triggers smoke tests. 4) Deployment to production triggers canary monitoring. Senior insight: if tests aren't in the CI pipeline, they effectively don't exist — developers won't run them manually. The pipeline is the enforcer of quality standards.
46→46→How do you handle database state in integration tests?	Strategies: 1) Rollback transactions after each test (fastest, but doesn't work if the test spawns background threads). 2) Respawn (delete all data and re-seed before each test — tool: Respawn NuGet). 3) In-memory database (EF Core InMemory — fast but misses DB-specific behavior like constraints, JSON columns). 4) Testcontainers (spin up a real PostgreSQL/SQL Server in Docker per test run — slowest but most realistic). Senior insight: match the strategy to the risk. For critical financial logic, use Testcontainers with a real DB. For CRUD APIs, Respawn is usually sufficient.
47→47→What are Testcontainers and why are they valuable for integration testing?	Testcontainers is a library that programmatically starts Docker containers in tests — e.g., a real PostgreSQL, Redis, or Kafka instance. The container starts before the test suite, tests run against it, and it is disposed afterward. Available for C# (Testcontainers-dotnet), Java, Python, Go. Value: tests run against the exact same DB engine and version as production — catches collation issues, constraint behavior, JSON query differences that in-memory DBs miss. Senior insight: the tradeoff is speed (30-60s startup). Use Testcontainers for integration test suites, not unit test suites.
48→48→What is parallel test execution and what are its risks?	Running tests in parallel reduces total test suite runtime (important when suites exceed 10 minutes). xUnit supports parallelism within and across assemblies. Risks: 1) shared state (static fields, shared database rows) causes race conditions. 2) port conflicts (multiple tests starting HTTP servers on the same port). 3) flakiness that only appears under parallel load. Mitigation: design tests for isolation from the start (separate DB schemas or databases per test class, ephemeral ports). Senior insight: parallel testing is a forcing function for test isolation — poorly isolated tests will fail loudly under parallel execution.
49→49→How do you measure and improve test suite performance?	Measure: record test execution time per test and per category in CI. Identify the slowest 20% (usually integration/E2E tests). Improve: 1) Move slow tests to a separate "slow" suite run less frequently. 2) Use in-memory fakes instead of real infrastructure where confidence is equivalent. 3) Parallelize test execution. 4) Share expensive fixtures (shared DB connection) across tests in the same class using IClassFixture. Senior insight: a test suite that takes 30+ minutes will be skipped or run less frequently. Suite performance is a product quality issue.
50→50→How does HttpClient testing differ between unit tests and integration tests?	Unit test: mock IHttpClientFactory or use a custom HttpMessageHandler that returns pre-configured HttpResponseMessage — no real network call. Integration test: use WebApplicationFactory (for your own APIs) or a real/test HTTP server with Wiremock.Net (for external dependencies) to exercise the full HTTP pipeline. Common mistake: creating new HttpClient() in tests directly — this leads to socket exhaustion and DNS caching issues. Senior insight: HttpClient should always come from IHttpClientFactory in production code, which also makes it straightforward to substitute in tests.
51→51→How do you mock HttpClient in C# unit tests?	HttpClient is not easily mockable directly (it's a concrete class, not an interface). The correct approach is to mock the HttpMessageHandler: var handler = new Mock<HttpMessageHandler>(); handler.Protected().Setup<Task<HttpResponseMessage>>("SendAsync", ...).ReturnsAsync(new HttpResponseMessage(HttpStatusCode.OK) { Content = new StringContent("{\"id\":1}") }); var client = new HttpClient(handler.Object); Senior insight: consider wrapping HttpClient behind an interface (IApiClient) in your application code — this makes the unit test stub trivial and decouples your domain from HTTP concerns entirely.
52→52→What is WireMock.Net and when should you use it in testing?	WireMock.Net is a library that starts a local HTTP stub server that returns pre-configured responses for specific request patterns. Use it when: 1) you are testing code that calls external APIs (payment gateways, third-party services) in integration tests. 2) you want to simulate error responses (500, 429, timeouts) that are hard to reproduce with a real API. Example: server.Given(Request.Create().WithPath("/api/payment").UsingPost()).RespondWith(Response.Create().WithStatusCode(200).WithBodyAsJson(new { status = "approved" })); Senior insight: WireMock is better than mocking HttpClient in integration tests because it exercises the full HTTP client stack (serialization, headers, retry logic).
53→53→How do you test gRPC proto contracts to prevent breaking changes?	1) Generate the client from the .proto file in your test project — if the server breaks the contract, the generated code won't compile or calls will fail. 2) Use contract testing tools (e.g., Pact for gRPC). 3) Version your proto files with semantic versioning and treat field removal or type changes as breaking. 4) Add integration tests that call every RPC method and assert the response shape. Senior insight: proto backward compatibility rules: never remove or renumber a field; only add new optional fields. Teach your team these rules — a field renaming will silently break all consumers because proto uses field numbers, not names, for serialization.
54→54→What are common anti-patterns in test automation that Senior Architects must avoid?	1) Testing implementation details instead of behavior (tests break on every refactor). 2) Shared mutable state between tests (order-dependent failures). 3) Testing frameworks instead of your code (testing that Moq works). 4) Asserting too many things in one test (hard to diagnose failures). 5) Slow unit tests (unit tests with real I/O). 6) Commenting out failing tests instead of fixing them. 7) 100% coverage as the only quality metric. Senior insight: anti-patterns accumulate silently. Schedule regular test suite health reviews — check flakiness rates, slowest tests, coverage gaps, and duplicated test scenarios.
55→55→How do you test for performance (load and stress) and how does it differ from functional testing?	Functional testing verifies correctness (does it return the right answer?). Performance testing verifies behavior under load (how fast? how many concurrent users? does it degrade gracefully?). Types: Load test (normal expected load), stress test (beyond normal load to find breaking point), soak test (sustained load over time to find memory leaks). Tools for C#: k6, NBomber, Artillery, Azure Load Testing. Senior insight: performance tests require a production-like environment — running them against a dev machine gives meaningless results. Integrate performance baselines into CI to catch regressions before they reach production.
