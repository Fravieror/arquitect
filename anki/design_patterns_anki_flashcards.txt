What are Design Patterns in software engineering?	Design Patterns are reusable solutions to common problems in software design. Think of them like blueprints — not finished code, but proven templates for solving recurring challenges. They were popularized by the "Gang of Four" (GoF) book in 1994. Senior insight: patterns are a communication tool first. Saying "use a Strategy here" is faster than explaining the whole approach. But don't force patterns where simple code works — that's called "pattern fever."
What are the three main categories of GoF Design Patterns?	1) Creational — how objects are created. 2) Structural — how objects are composed/connected. 3) Behavioral — how objects communicate and share responsibilities. Analogy: Creational = how you build LEGO pieces, Structural = how you snap them together, Behavioral = what the assembled structure does.
What is the difference between a Design Pattern and an Anti-Pattern?	A Design Pattern solves a recurring problem well. An Anti-Pattern is a common "solution" that actually makes things worse. Example anti-pattern: God Object (one class that does everything). Senior insight: many anti-patterns start as patterns applied in the wrong context. Singleton is a pattern, but overusing it becomes the "Singletonitis" anti-pattern.
What is the Singleton pattern?	Singleton ensures a class has only ONE instance and provides a global access point to it. Analogy: there's only one president of a country at a time — you don't create a new one every time someone needs to talk to the government. Use case: database connection pools, configuration managers, logging services.
What are the dangers of the Singleton pattern?	1) Hidden dependencies — code secretly depends on global state. 2) Hard to test — you can't easily swap it with a mock. 3) Thread-safety issues — two threads might create two instances simultaneously. 4) Violates Single Responsibility Principle — manages its own lifecycle AND its business logic. Senior insight: in modern codebases, prefer Dependency Injection (DI) over Singleton. DI containers can manage single-instance lifetime without the downsides.
How do you implement a thread-safe Singleton in C#?	Use the Lazy<T> approach: private static readonly Lazy<MyClass> _instance = new Lazy<MyClass>(() => new MyClass()); public static MyClass Instance => _instance.Value; This is thread-safe, lazy (created on first access), and clean. Avoid the double-checked locking pattern — it's error-prone and unnecessary in modern C#.
How does Singleton differ in Python vs C# vs JavaScript?	Python: use a module-level variable (modules are natural singletons) or __new__ override. C#: use Lazy<T> or static readonly field. JavaScript: use ES6 module exports (modules are cached after first import, making them natural singletons). Senior insight: in Python and JS, the language itself provides singleton-like behavior through modules, so explicit Singleton classes are rarely needed.
What is the Factory Method pattern?	Factory Method defines an interface for creating objects but lets subclasses decide which class to instantiate. Analogy: a restaurant menu (the factory interface) says "we serve burgers" but each branch (subclass) decides if it's a beef burger or veggie burger. It decouples the client from the concrete class being created.
When should you use Factory Method vs. calling 'new' directly?	Use Factory Method when: 1) you don't know the exact type at compile time, 2) you want subclasses to control creation, 3) creation logic is complex or varies by context. Stick with 'new' when: the type is known, creation is simple, and there's no variation. Common mistake: using a factory for simple objects with no variation — that's over-engineering.
What is the Abstract Factory pattern?	Abstract Factory provides an interface for creating families of related objects without specifying their concrete classes. Analogy: IKEA has different furniture "lines" (modern, classic, rustic). Each line is a factory that produces matching chairs, tables, and sofas. You pick a line, and everything comes out coordinated. Use case: cross-platform UI toolkits (Windows widgets vs Mac widgets).
What is the difference between Factory Method and Abstract Factory?	Factory Method creates ONE product through inheritance (subclass overrides a method). Abstract Factory creates FAMILIES of related products through composition (an object with multiple factory methods). Factory Method = one product, uses inheritance. Abstract Factory = multiple related products, uses composition. Senior insight: Abstract Factory often uses Factory Methods internally.
What is the Builder pattern?	Builder separates the construction of a complex object from its representation, allowing the same construction process to create different representations. Analogy: building a custom pizza — you add ingredients step by step (dough, sauce, cheese, toppings) and call "bake" at the end. The order and choices vary, but the process is the same. Use case: constructing complex objects like HTTP requests, SQL queries, or UI components.
What problem does the Builder pattern solve that constructors can't?	Constructors with many parameters (telescoping constructors) are unreadable: new Pizza(true, false, true, 3, "large", null). Builder makes it clear: Pizza.builder().withCheese().withPepperoni().large().build(). It also allows immutable objects to be built step-by-step. Senior insight: in C#, consider Builder when you have more than 3-4 constructor parameters. In Python, keyword arguments reduce the need for Builder.
What is the Prototype pattern?	Prototype creates new objects by cloning an existing instance instead of building from scratch. Analogy: photocopying a document — faster than rewriting it. Use case: when object creation is expensive (e.g., objects loaded from a database or requiring heavy computation). You clone the prototype and modify only what differs.
What is the difference between shallow copy and deep copy in the Prototype pattern?	Shallow copy duplicates the object but shares references to nested objects — change a nested object in the copy and it changes in the original too. Deep copy duplicates everything, including nested objects. Common mistake: using shallow copy when your object has mutable nested references, causing subtle bugs. In C#: ICloneable is shallow by default. In Python: use copy.deepcopy() for deep copies.
What is the Adapter pattern?	Adapter converts the interface of a class into another interface that clients expect. Analogy: a power plug adapter — your US laptop plug doesn't fit a European socket, so you use an adapter. The laptop and socket don't change, only the adapter sits between them. Use case: integrating legacy code or third-party libraries with incompatible interfaces.
What are the two types of Adapter implementations?	1) Object Adapter — uses composition: the adapter HOLDS an instance of the adaptee. 2) Class Adapter — uses inheritance: the adapter EXTENDS the adaptee (requires multiple inheritance, so not possible in C# or Java). Senior insight: prefer Object Adapter (composition over inheritance). It's more flexible and works in all languages.
What is the Bridge pattern?	Bridge separates an abstraction from its implementation so both can vary independently. Analogy: a TV remote (abstraction) and the TV (implementation). You can change the remote without changing the TV, and vice versa. Without Bridge, you'd need SonyRemote, SamsungRemote, SonyAdvancedRemote, SamsungAdvancedRemote — a combinatorial explosion.
When should you use Bridge vs Adapter?	Adapter fixes incompatible interfaces AFTER the code exists (retrofit). Bridge is designed UPFRONT to prevent coupling between abstraction and implementation. Adapter = "make these work together." Bridge = "keep these independent from the start." Senior insight: if you find yourself creating many Adapters for the same hierarchy, you probably needed a Bridge from the beginning.
What is the Composite pattern?	Composite lets you treat individual objects and compositions of objects uniformly through a tree structure. Analogy: a file system — a folder can contain files or other folders, but you can "delete" or "move" both the same way. Use case: UI component trees, organizational hierarchies, menus with submenus.
What is a common mistake when implementing the Composite pattern?	Putting operations on the base component that only make sense for composites (like addChild on a leaf node). This violates the Interface Segregation Principle. Some implementations throw exceptions for unsupported operations on leaves, but that's a runtime error instead of a compile-time check. Senior insight: decide if you value transparency (uniform interface, less type-safe) or safety (separate interfaces, more type-safe). Most modern codebases prefer safety.
What is the Decorator pattern?	Decorator dynamically adds behavior to an object without modifying its class. It wraps the original object, delegating calls to it while adding extra logic. Analogy: adding toppings to ice cream — the base is still ice cream, each topping adds flavor without changing the original scoop. Use case: streams in C# and Java (BufferedStream wraps FileStream), middleware in web frameworks.
What is the difference between Decorator and Inheritance for extending behavior?	Inheritance is static (fixed at compile time) and creates tightly coupled hierarchies. Decorator is dynamic (applied at runtime) and uses composition. With inheritance you get class explosion: CoffeeWithMilk, CoffeeWithSugar, CoffeeWithMilkAndSugar... With Decorator: new Sugar(new Milk(new Coffee())). Senior insight: Decorator follows the Open/Closed Principle — open for extension, closed for modification.
What is the Facade pattern?	Facade provides a simplified interface to a complex subsystem. Analogy: a hotel concierge — instead of calling the restaurant, spa, and taxi yourself, you tell the concierge what you need and they handle the complexity. Use case: wrapping complex library APIs, simplifying multi-step processes like "place an order" (check inventory + charge payment + send confirmation).
What is the difference between Facade and Adapter?	Facade simplifies a complex interface into an easier one (changes complexity level). Adapter converts one interface to match another (changes interface shape). Facade = "make this simpler." Adapter = "make this compatible." A Facade often wraps multiple objects; an Adapter typically wraps one.
What is the Flyweight pattern?	Flyweight reduces memory usage by sharing common data across many similar objects. It separates intrinsic state (shared, immutable) from extrinsic state (unique per instance, passed in). Analogy: in a word processor, each character on screen doesn't store its own font data. The font data (intrinsic) is shared; the position (extrinsic) is unique. Use case: rendering thousands of trees in a game where they share the same texture/mesh.
When is Flyweight worth the added complexity?	Only when you have a LARGE number of similar objects (thousands+) consuming significant memory. Profile first — don't optimize prematurely. Flyweight adds complexity through the separation of intrinsic/extrinsic state and requires a factory to manage shared instances. Common mistake: applying Flyweight to save kilobytes when your app uses gigabytes elsewhere. Senior insight: string interning in C# and Java is a built-in Flyweight for string literals.
What is the Proxy pattern?	Proxy provides a surrogate or placeholder for another object to control access to it. Analogy: a security guard at a building entrance — they check credentials before letting you in, but once inside you interact with the building normally. Types: Virtual Proxy (lazy loading), Protection Proxy (access control), Remote Proxy (network communication), Caching Proxy.
What is the difference between Proxy and Decorator?	Both wrap an object, but their intent differs. Proxy controls ACCESS to the object (security, lazy loading, remote access). Decorator adds BEHAVIOR to the object. Proxy often creates or manages the wrapped object's lifecycle. Decorator receives it from outside. Senior insight: if you're adding functionality → Decorator. If you're controlling when/how the object is accessed → Proxy.
What is the Strategy pattern?	Strategy defines a family of algorithms, encapsulates each one, and makes them interchangeable at runtime. Analogy: GPS navigation — you choose a strategy (fastest route, shortest route, avoid tolls) and the GPS uses that algorithm without changing its core logic. Use case: sorting algorithms, payment processing methods, compression strategies.
Why is Strategy one of the most important patterns for senior architects?	It's the foundation of the Open/Closed Principle. Instead of giant if/else or switch blocks that grow every time you add a case, you add a new Strategy class. It enables runtime flexibility, testability (mock strategies easily), and clean separation of concerns. Common mistake: using Strategy for 2 simple options — an if/else is fine. Strategy shines when you have 4+ interchangeable behaviors or when the behavior changes at runtime.
How do you implement Strategy in Python vs C# vs JavaScript?	Python: pass functions directly (first-class functions make Strategy trivial) — sort(key=lambda x: x.age). C#: use interfaces or delegates/Func<T>. JavaScript: pass functions or objects with a common method. Senior insight: in languages with first-class functions (Python, JS), you often don't need formal Strategy classes. A function parameter IS a strategy. The pattern is implicit.
What is the Observer pattern?	Observer defines a one-to-many relationship where when one object (Subject) changes state, all its dependents (Observers) are notified automatically. Analogy: a YouTube subscription — when a channel uploads a video, all subscribers get notified. Use case: event systems, UI data binding, pub/sub messaging.
What are the pitfalls of the Observer pattern?	1) Memory leaks — if observers aren't unsubscribed, they can't be garbage collected (lapsed listener problem). 2) Unexpected cascades — one change triggers observers that trigger more observers. 3) Order dependency — observers may execute in unpredictable order. 4) Threading issues in concurrent systems. Senior insight: in modern systems, prefer event buses, reactive streams (RxJS, Reactor), or message brokers over raw Observer — they handle these issues better.
What is the Command pattern?	Command encapsulates a request as an object, letting you parameterize operations, queue them, log them, or support undo. Analogy: a restaurant order ticket — the waiter writes down your order (command), passes it to the kitchen (invoker), which executes it. The ticket can be saved, replayed, or canceled. Use case: undo/redo, task queues, macro recording, transaction logging.
How does the Command pattern enable undo/redo?	Each Command object stores the information needed to both execute AND reverse the action. You maintain a history stack. Undo = pop the last command and call its undo() method. Redo = push it back and call execute() again. Common mistake: not storing enough state to fully reverse the operation. Senior insight: Command + Memento together create robust undo systems. Event Sourcing in distributed systems is essentially Command pattern at scale.
What is the Template Method pattern?	Template Method defines the skeleton of an algorithm in a base class, letting subclasses override specific steps without changing the overall structure. Analogy: a recipe template — "1) prep ingredients, 2) cook, 3) plate." Each chef (subclass) does these steps differently, but the order is fixed. Use case: frameworks (e.g., ASP.NET page lifecycle), data processing pipelines.
What is the difference between Template Method and Strategy?	Template Method uses INHERITANCE — the base class controls the flow and subclasses override hooks. Strategy uses COMPOSITION — the client picks which algorithm object to use. Template Method = "subclass overrides steps." Strategy = "client chooses the whole algorithm." Senior insight: prefer Strategy (composition) over Template Method (inheritance) when possible. Template Method creates tight coupling to the base class and is harder to test.
What is the State pattern?	State allows an object to change its behavior when its internal state changes, appearing to change its class. Analogy: a vending machine — it behaves differently when it has no coins (waiting), has coins (ready to dispense), or is out of stock (rejecting input). Each state is a separate class with its own logic. Use case: workflow engines, game character states, UI component states.
What is the difference between State pattern and if/else state management?	If/else scatters state logic throughout the code and grows messy with each new state. State pattern encapsulates each state's behavior in its own class. Adding a new state = adding a new class, no existing code changes. Common mistake: using State pattern for simple two-state toggles (just use a boolean). Senior insight: State pattern + State Machines are essential for complex workflows in production. Libraries like Stateless (C#) or XState (JS) formalize this.
What is the Chain of Responsibility pattern?	Chain of Responsibility passes a request along a chain of handlers until one handles it. Each handler decides to process the request OR pass it to the next handler. Analogy: customer support escalation — first tier tries, if they can't help, it goes to tier 2, then tier 3. Use case: middleware pipelines (Express.js, ASP.NET Core), logging systems, validation chains.
How is Chain of Responsibility used in modern web frameworks?	Middleware in Express.js and ASP.NET Core IS Chain of Responsibility. Each middleware function processes the request (authentication, logging, CORS, etc.) and calls next() to pass it along. This is one of the most commonly used patterns in production web development. Senior insight: the key design decision is whether a handler can STOP the chain (short-circuit) or if every handler always runs. Both designs are valid for different use cases.
What is the Iterator pattern?	Iterator provides a way to access elements of a collection sequentially without exposing the underlying representation. Analogy: a TV remote's "next channel" button — you don't need to know how channels are stored, you just press next. Use case: built into virtually every modern language (for...of in JS, foreach in C#, for...in in Python).
Why is Iterator still relevant when languages have built-in iteration?	You rarely implement Iterator from scratch anymore, but understanding it matters because: 1) Custom iterators for complex data structures (trees, graphs). 2) Lazy evaluation — yield in C# and Python generates items on-demand, saving memory. 3) Async iterators (IAsyncEnumerable in C#, async generators in JS) are critical for streaming data. Senior insight: yield/generators are the modern Iterator pattern and are essential for handling large datasets efficiently.
What is the Mediator pattern?	Mediator reduces chaotic dependencies between objects by having them communicate through a central mediator instead of directly. Analogy: an air traffic control tower — planes don't talk to each other, they talk to the tower which coordinates everything. Use case: chat rooms, form validation (fields interact through a mediator), CQRS libraries like MediatR in C#.
What is the difference between Mediator and Observer?	Observer is a one-to-many broadcast — the subject doesn't know what observers do. Mediator is a hub that actively coordinates interactions between specific objects and contains logic about who communicates with whom. Observer = "I changed, whoever cares can react." Mediator = "I'll coordinate how A and B interact." Senior insight: MediatR in C# uses both patterns — it mediates between request handlers while observers (notification handlers) react to events.
What is the Visitor pattern?	Visitor lets you add new operations to existing object structures without modifying those objects. The objects "accept" a visitor that performs the operation. Analogy: a building inspector — they visit each room (accept) and perform their specific inspection. Adding a fire inspector doesn't require changing the rooms. Use case: compilers (AST traversal), document exporters (export to PDF, HTML, JSON).
Why is the Visitor pattern controversial?	It's powerful but has serious trade-offs: 1) Adding a new element type requires changing ALL visitors (violates Open/Closed for types). 2) It breaks encapsulation — visitors access object internals. 3) Double dispatch mechanism is confusing for many developers. 4) In languages with pattern matching (C# switch expressions, Python match), Visitor is often unnecessary. Senior insight: use Visitor when you have a stable set of types but frequently add new operations. If types change often, Visitor is the wrong choice.
What is the Memento pattern?	Memento captures an object's internal state so it can be restored later, without exposing its internal structure. Analogy: saving a game — the save file (memento) captures your progress, and you can restore to that point anytime. The game's internal data structures stay private. Use case: undo functionality, checkpointing, transaction rollback.
What is the difference between Memento and Command for undo?	Command stores the operation and reverses it. Memento stores the entire state snapshot. Command is more memory-efficient (stores just the delta). Memento is simpler but uses more memory (stores full state). Senior insight: use Command for fine-grained undo where operations are reversible. Use Memento when operations are too complex to reverse algorithmically — just restore the snapshot. In practice, many systems combine both.
What is the Interpreter pattern?	Interpreter defines a grammar for a language and an interpreter to process sentences in that language. Each grammar rule becomes a class. Analogy: a calculator that understands math expressions — each operator and number is a class that knows how to evaluate itself. Use case: SQL parsers, regular expression engines, configuration DSLs, rule engines.
Why is the Interpreter pattern rarely used directly in production?	It's too slow and inflexible for real-world parsing. Production parsers use parser generators (ANTLR), recursive descent parsers, or parser combinator libraries instead. However, the CONCEPT is everywhere: every DSL, query language, and expression evaluator uses interpreter principles. Senior insight: understanding Interpreter helps you design DSLs and config languages, even if you use a parser library instead of implementing the pattern directly.
What are Concurrency Patterns?	Concurrency Patterns solve problems that arise when multiple threads or processes execute simultaneously. They deal with coordination, synchronization, data sharing, and avoiding issues like deadlocks, race conditions, and starvation. They're critical for senior architects because almost all production systems handle concurrent operations.
What is the Producer-Consumer pattern?	Producer-Consumer decouples data production from data consumption using a shared buffer (queue). Producers add items; consumers remove and process them independently. Analogy: a factory assembly line — workers at station 1 produce parts and place them on a conveyor belt (queue), workers at station 2 pick them up and assemble. Use case: message queues, task processing pipelines, log aggregation.
What problems does Producer-Consumer solve?	1) Speed mismatch — producer and consumer run at different speeds. 2) Decoupling — they don't need to know about each other. 3) Load buffering — the queue absorbs bursts. Key decisions: bounded vs unbounded queue (bounded prevents memory issues), blocking vs non-blocking (blocking is simpler, non-blocking scales better). Common mistake: unbounded queues that grow until the system runs out of memory under load.
What is the Reader-Writer Lock pattern?	Reader-Writer Lock allows multiple concurrent readers but exclusive access for writers. Analogy: a library book — many people can read it simultaneously, but if someone wants to edit it, everyone else must wait. Use case: caches, configuration data, shared resources that are read far more often than written. In C#: ReaderWriterLockSlim. In Python: threading.RLock or asyncio locks.
What is the Thread Pool pattern?	Thread Pool maintains a pool of pre-created worker threads that process incoming tasks from a queue. Instead of creating/destroying threads for each task (expensive), you reuse them. Analogy: a taxi stand — taxis wait for passengers instead of building a new taxi for each rider. Use case: web servers handling HTTP requests, database connection pools. Senior insight: almost all modern runtimes provide built-in pools (C# ThreadPool/.NET Task Scheduler, Java ExecutorService, Python concurrent.futures).
What is the Monitor Object pattern?	Monitor Object synchronizes method execution to ensure only one thread executes any method on the object at a time. It combines a lock with the object it protects. Analogy: a single-occupancy bathroom — only one person at a time, with a lock on the door. In C#: the lock keyword (Monitor.Enter/Exit). In Java: synchronized keyword. In Python: threading.Lock in a with statement. Common mistake: locking too broadly (entire method) when only a small section needs protection.
What is the Active Object pattern?	Active Object decouples method invocation from method execution by giving each object its own thread and message queue. Callers enqueue requests and get a Future/Promise back. Analogy: sending a letter to a government office — you drop off your request (enqueue), they process it on their schedule, and you check your mailbox for the response (Future). Use case: actor systems (Akka, Orleans), UI thread models (WPF Dispatcher).
What is the difference between Active Object and Actor Model?	They're closely related. Active Object is the GoF-style pattern with explicit Futures and method proxy objects. Actor Model (Erlang, Akka) is a broader concurrency model where actors communicate only through messages — no shared state at all. Actor Model is more restrictive but eliminates shared-state bugs by design. Senior insight: Actor Model scales better for distributed systems; Active Object works better within a single process.
What is the Future/Promise pattern?	Future/Promise represents a value that will be available at some point in the future. It decouples "starting" an operation from "getting the result." Analogy: ordering at a fast food counter — you get a receipt number (Future), do other things, and come back when your number is called. In C#: Task<T> / async-await. In JS: Promise / async-await. In Python: asyncio.Future / await.
What are the common pitfalls with Futures/Promises?	1) Unhandled rejections — forgetting to catch errors on Promises (JS) leads to silent failures. 2) Async deadlocks — in C#, calling .Result or .Wait() on a Task inside a sync context causes deadlock. Use async all the way. 3) Fire-and-forget — starting an async operation without awaiting it loses errors and ordering guarantees. 4) Callback hell / promise chains — use async/await syntax instead. Senior insight: "async all the way" is not optional. Mixing sync and async is the #1 source of production deadlocks in C# and Python.
What is the Semaphore pattern?	Semaphore limits the number of threads that can access a resource simultaneously. Unlike a lock (binary: 0 or 1), a semaphore has a count N. Analogy: a parking lot with N spaces — cars enter if spaces are available, wait if full. Use case: limiting concurrent database connections, API rate limiting, controlling resource consumption. In C#: SemaphoreSlim. In Python: asyncio.Semaphore.
What is the Double-Checked Locking pattern?	Double-Checked Locking reduces locking overhead for lazy initialization by checking the condition before AND after acquiring the lock. First check avoids the lock in the common case (already initialized). Second check ensures only one thread initializes. Warning: this pattern is broken in many implementations due to memory ordering issues. In C# and Java, you need volatile keyword. Senior insight: use language-provided lazy initialization instead (Lazy<T> in C#, enum singleton in Java). Don't hand-roll double-checked locking.
What is the Reactor pattern?	Reactor handles service requests delivered concurrently to an application by one or more inputs. It demultiplexes events to event handlers synchronously. Analogy: a restaurant host who seats guests — they handle one arrival at a time but manage many tables. Use case: Node.js event loop, Nginx, Redis — single-threaded servers handling thousands of connections. Senior insight: Reactor is why Node.js can handle massive concurrency with one thread. It's the foundation of event-driven architecture.
What is the Proactor pattern and how does it differ from Reactor?	Reactor handles I/O synchronously (you're notified when data IS ready, then you read it). Proactor handles I/O asynchronously (you initiate the read, the OS notifies you when it's DONE). Analogy: Reactor = "the food is ready, come pick it up." Proactor = "I'll bring the food to your table when it's ready." Proactor is used in Windows IOCP, Boost.Asio. Senior insight: Linux traditionally uses Reactor (epoll), Windows uses Proactor (IOCP). io_uring in modern Linux brings Proactor-style to Linux.
Which design patterns are most commonly used in production systems?	1) Strategy — everywhere (sorting, validation, business rules). 2) Observer/Pub-Sub — event systems, reactive UIs, message brokers. 3) Factory Method/Abstract Factory — dependency injection, plugin systems. 4) Decorator — middleware, streams, logging wrappers. 5) Chain of Responsibility — HTTP middleware pipelines. 6) Builder — constructing complex configuration/request objects. 7) Proxy — lazy loading, caching, API gateways. 8) Singleton — (via DI containers) shared services. Senior insight: in modern code, many patterns are invisible because frameworks implement them for you. Understanding the patterns helps you USE frameworks correctly.
How does Dependency Injection relate to design patterns?	DI is the practical evolution of Factory + Strategy + Singleton patterns. Instead of objects creating their own dependencies (new DatabaseService()), dependencies are provided (injected) from outside. This enables testability, loose coupling, and configuration flexibility. DI containers (ASP.NET DI, Spring, Dagger) manage object lifetimes (singleton, scoped, transient). Senior insight: DI is the single most impactful architectural pattern for production code maintainability.
What is the Repository pattern and why do architects care about it?	Repository abstracts data access behind a collection-like interface. The business layer works with repositories (GetUserById, SaveOrder) without knowing if data comes from SQL, NoSQL, an API, or memory. It's a specialized Facade for data access. Senior insight: Repository shines when combined with Unit of Work for transactions. Common mistake: "generic repositories" (Repository<T>) that leak SQL abstractions or become thin wrappers around ORMs — adding complexity without value.
What is the CQRS pattern?	Command Query Responsibility Segregation separates read and write operations into different models. Write model (commands) handles business logic and validation. Read model (queries) is optimized for data retrieval. Analogy: a restaurant has a kitchen (writes/commands) and a menu board (reads/queries) — they're separate systems optimized for different purposes. Senior insight: CQRS shines at scale but is overkill for CRUD apps. Combine with Event Sourcing for full audit trails.
What is the Event Sourcing pattern?	Instead of storing current state, Event Sourcing stores a sequence of state-changing events. Current state is derived by replaying events. Analogy: a bank account — instead of storing "balance = $500," you store every deposit and withdrawal. You can replay to get the balance at any point in time. Senior insight: powerful for audit trails and temporal queries, but adds complexity in querying (need projections), versioning events, and handling schema evolution. Not a default choice — use when you genuinely need the full history.
What is the Circuit Breaker pattern?	Circuit Breaker prevents cascading failures by wrapping calls to external services. States: Closed (normal, requests pass through), Open (failures exceeded threshold, requests fail immediately), Half-Open (testing if service recovered). Analogy: an electrical circuit breaker — it trips to prevent damage and resets once safe. Use case: microservice communication, external API calls. Libraries: Polly (C#), Resilience4j (Java), opossum (Node.js).
What are the most common mistakes junior developers make with design patterns?	1) Using patterns where simple code would work ("golden hammer"). 2) Applying patterns without understanding the problem they solve. 3) Implementing patterns exactly as in the GoF book without adapting to modern languages. 4) Not recognizing patterns already built into frameworks. 5) Over-abstracting small codebases. Senior insight: the right time to introduce a pattern is when you feel the PAIN of not having it. If the code is simple and working, don't add a pattern "just in case."
How should a senior architect decide which pattern to use?	1) Identify the specific problem (not "I need a pattern" but "I need to decouple X from Y"). 2) Consider the simplest solution first — maybe an if/else is fine. 3) Evaluate trade-offs: added complexity vs flexibility gained. 4) Consider your team — can they understand and maintain it? 5) Consider your language — many patterns are unnecessary in Python/JS that are essential in Java/C#. Senior insight: the best architects use patterns invisibly. The code is clean and flexible, and you only notice the pattern if you look for it.
What is the Null Object pattern?	Null Object replaces null checks with a "do nothing" implementation of an interface. Instead of checking if (logger != null) logger.Log(msg) everywhere, you provide a NullLogger that implements Log() as an empty method. Analogy: a /dev/null device — it accepts input and does nothing. Use case: default implementations, testing, optional behaviors. Senior insight: this eliminates an entire class of NullReferenceExceptions and simplifies code significantly.
What is the Specification pattern?	Specification encapsulates a business rule into a reusable, combinable object. Each specification has an IsSatisfiedBy(candidate) method. You can combine them with And(), Or(), Not(). Analogy: a job listing filter — "5+ years experience AND knows Python OR knows C#". Use case: complex business rule validation, query filtering, authorization rules. Senior insight: specifications compose well and keep business rules testable and out of repository/service code.
How do patterns differ in microservices vs monolith architectures?	Monolith: in-process patterns (Strategy, Observer, Factory) dominate. Object references are cheap. Microservices: distributed patterns matter more — Circuit Breaker, Saga, API Gateway (Facade), Event Sourcing, CQRS, Sidecar (Decorator for infrastructure). Communication patterns (sync vs async, request-reply vs event-driven) become critical. Senior insight: microservices don't eliminate GoF patterns — they push them to the network level where failures are more likely and latency matters.
What is the Saga pattern for distributed transactions?	Saga manages transactions spanning multiple microservices by breaking them into a sequence of local transactions, each with a compensating action (rollback). Analogy: booking a trip — you book flight, then hotel, then car. If the car booking fails, you cancel the hotel, then cancel the flight (compensating actions in reverse order). Two types: Choreography (events, no coordinator) and Orchestration (central coordinator). Senior insight: Orchestration is easier to understand and debug; Choreography scales better but is harder to trace.
What is the Sidecar pattern?	Sidecar attaches a helper container alongside the main application container, providing supporting features like logging, monitoring, proxying, or configuration. Analogy: a motorcycle sidecar — it adds functionality without modifying the motorcycle itself. It's essentially the Decorator pattern for infrastructure. Use case: service meshes (Istio/Envoy), log collectors, security proxies. Senior insight: Sidecar is the foundation of the service mesh architecture — understanding it is essential for modern cloud-native systems.
What is the Strangler Fig pattern?	Strangler Fig is a migration pattern for gradually replacing a legacy system with a new one. New features are built in the new system; existing features are migrated incrementally. The old system "strangles" away over time. Analogy: like the strangler fig tree that grows around a host tree and eventually replaces it. Use case: monolith-to-microservices migration. Senior insight: this is the safest way to replace legacy systems. Big-bang rewrites fail over 70% of the time. Strangler Fig lets you deliver value continuously during migration.
What is the difference between Composition and Inheritance when applying design patterns?	Inheritance = "is-a" relationship, defined at compile time, creates tight coupling. Composition = "has-a" relationship, defined at runtime, creates loose coupling. GoF principle: "Favor composition over inheritance." Most modern patterns use composition (Strategy, Decorator, Adapter via objects). Senior insight: inheritance is not evil, but it should be shallow (1-2 levels) and used for genuine type hierarchies, not for code reuse. If you're inheriting for reuse, you probably want composition.
What are the SOLID principles and how do they relate to design patterns?	S = Single Responsibility (one reason to change). O = Open/Closed (extend behavior without modifying existing code — Strategy, Decorator). L = Liskov Substitution (subtypes must be substitutable — all patterns using polymorphism). I = Interface Segregation (small, focused interfaces — Adapter, Facade). D = Dependency Inversion (depend on abstractions — Factory, Strategy, DI). Senior insight: SOLID principles are the WHY; design patterns are the HOW. Learn SOLID first, and patterns become intuitive answers.
