What is Authentication vs Authorization — and why does the distinction matter?	<div style="text-align:left"><b>Authentication (AuthN)</b> = "Who are you?" — verifying identity.<br><b>Authorization (AuthZ)</b> = "What can you do?" — verifying permissions.<br><br><b>Analogy:</b> At a concert, showing your ID is authentication. The ticket type (GA vs VIP) is authorization.<br><br><b>Flow:</b><br>1. User logs in → system authenticates (confirms identity)<br>2. User tries to access a resource → system authorizes (checks permission)<br><br><b>Common mistake:</b> Conflating the two. Many bugs stem from checking "is this user logged in?" but not "does this user have permission to do this specific thing?"<br><br><b>Senior insight:</b> Always separate authN and authZ into distinct layers. Systems that mix them are hard to audit, scale, or change.</div>

What is the difference between session-based and token-based authentication?	<div style="text-align:left"><b>Session-based:</b> After login, the server stores a session in memory (or DB). The client holds only a session ID (in a cookie).<br><br><b>Token-based (JWT):</b> After login, the server issues a signed token. The client stores it and sends it with each request. The server validates the signature — no server-side state needed.<br><br><b>Session-based pros/cons:</b><br>- Easy to revoke (delete session from DB)<br>- Requires shared state across servers (sticky sessions or Redis)<br><br><b>Token-based pros/cons:</b><br>- Stateless — scales horizontally with ease<br>- Hard to revoke before expiry (requires a token blacklist)<br><br><b>Senior insight:</b> Token-based auth is popular in microservices. But if you need instant logout or token revocation, you must add a revocation layer — which partially defeats the stateless benefit.</div>

What is a JWT (JSON Web Token) and how does it work?	<div style="text-align:left">A <b>JWT</b> is a compact, self-contained token with three base64-encoded parts separated by dots:<br><br><code>header.payload.signature</code><br><br><b>Header:</b> algorithm used (e.g., HS256, RS256)<br><b>Payload:</b> claims — who the user is, roles, expiry<br><b>Signature:</b> cryptographic proof the token wasn't tampered with<br><br><b>Example payload:</b><br><code>{ "sub": "user123", "role": "admin", "exp": 1712345678 }</code><br><br><b>Common mistakes:</b><br>- Storing sensitive data in the payload (it's base64, NOT encrypted — anyone can decode it)<br>- Not validating the <code>exp</code> claim<br>- Using <code>alg: none</code> (a well-known attack vector)<br><br><b>Senior insight:</b> JWTs are opaque to users but transparent to servers. Use short expiry (15–60 min) + refresh tokens. Never store a JWT in localStorage if XSS is a risk — prefer HttpOnly cookies.</div>

What is OAuth 2.0 and what problem does it solve?	<div style="text-align:left"><b>OAuth 2.0</b> is an authorization framework that lets a third-party application access resources on behalf of a user — without exposing the user's credentials.<br><br><b>Example:</b> "Login with Google" — your app gets a token to read the user's profile. Google never shares the user's password with you.<br><br><b>Core roles:</b><br>- <b>Resource Owner</b> — the user<br>- <b>Client</b> — your app<br>- <b>Authorization Server</b> — issues tokens (e.g., Google, Auth0)<br>- <b>Resource Server</b> — hosts the protected data (e.g., Google API)<br><br><b>Common flows:</b><br>- <b>Authorization Code</b> — for server-side web apps (most secure)<br>- <b>PKCE</b> — for SPAs and mobile (replaces Implicit flow)<br>- <b>Client Credentials</b> — machine-to-machine<br><br><b>Common mistake:</b> Confusing OAuth 2.0 with authentication — it's an authorization protocol. Use <b>OpenID Connect (OIDC)</b> on top of OAuth 2.0 to handle identity.</div>

What is OpenID Connect (OIDC) and how does it extend OAuth 2.0?	<div style="text-align:left"><b>OpenID Connect (OIDC)</b> is an identity layer built on top of OAuth 2.0. While OAuth 2.0 handles authorization (access to resources), OIDC handles <b>authentication</b> (who the user is).<br><br><b>OIDC adds:</b><br>- An <b>ID Token</b> (a JWT with user identity claims: name, email, etc.)<br>- A <code>/userinfo</code> endpoint to fetch profile data<br>- Standardized claims (<code>sub</code>, <code>iss</code>, <code>aud</code>, <code>iat</code>, <code>exp</code>)<br><br><b>Flow summary:</b><br>1. User logs in via the identity provider (Google, Azure AD)<br>2. App receives an access token (OAuth) + an ID token (OIDC)<br>3. App uses the ID token to establish who the user is<br><br><b>Senior insight:</b> OIDC is the de facto standard for federated identity. Most enterprise SSO systems (Azure AD, Okta, Keycloak) implement it. Always prefer OIDC + OAuth 2.0 over rolling your own auth.</div>

What is SAML and when is it used instead of OIDC?	<div style="text-align:left"><b>SAML (Security Assertion Markup Language)</b> is an XML-based standard for exchanging authentication and authorization data — primarily for enterprise SSO.<br><br><b>Flow:</b><br>1. User tries to access an app (Service Provider)<br>2. App redirects to Identity Provider (e.g., Active Directory)<br>3. IdP authenticates user, sends back an XML assertion<br>4. App grants access<br><br><b>SAML vs OIDC:</b><br>- SAML: XML-based, older, enterprise-focused, complex to implement<br>- OIDC: JSON/JWT-based, modern, developer-friendly, works well on mobile/SPA<br><br><b>When to use SAML:</b> When integrating with enterprise systems (Salesforce, Workday, legacy IdPs) that support SAML but not OIDC.<br><br><b>Senior insight:</b> In greenfield projects, choose OIDC. In enterprise integrations, you'll likely encounter SAML and need to support both. Libraries like <code>passport-saml</code> (Node) or Microsoft's SAML stack (.NET) help.</div>

What is Multi-Factor Authentication (MFA) and why should it be enforced?	<div style="text-align:left"><b>MFA</b> requires users to prove identity using two or more factors:<br><br>1. <b>Something you know</b> — password, PIN<br>2. <b>Something you have</b> — phone (TOTP app, SMS), hardware key (YubiKey)<br>3. <b>Something you are</b> — fingerprint, face scan<br><br><b>Why it matters:</b> Even if a password is stolen, an attacker still can't log in without the second factor.<br><br><b>Common MFA methods:</b><br>- <b>TOTP</b> (Time-based One-Time Password) — Google Authenticator, Authy<br>- <b>SMS OTP</b> — convenient but weakest (SIM swap attacks)<br>- <b>FIDO2 / WebAuthn</b> — phishing-resistant hardware keys<br><br><b>Common mistake:</b> Treating SMS as "secure" MFA — it's better than nothing, but vulnerable to SIM swapping and SS7 attacks.<br><br><b>Senior insight:</b> Enforce MFA for admin accounts and sensitive operations. For highest security, mandate FIDO2/WebAuthn (phishing-proof by design).</div>

What is Role-Based Access Control (RBAC) and how does it work?	<div style="text-align:left"><b>RBAC</b> assigns permissions to <b>roles</b>, and roles to <b>users</b> — rather than assigning permissions directly to users.<br><br><b>Example:</b><br><code>User: Alice → Role: Admin → Permissions: read, write, delete</code><br><code>User: Bob  → Role: Viewer → Permissions: read</code><br><br><b>Why roles:</b> If you have 1000 users and want to change what "Managers" can do, you change one role — not 1000 user records.<br><br><b>Common mistake:</b> Role explosion — creating too many fine-grained roles that are hard to manage. Keep roles meaningful and business-aligned.<br><br><b>Senior insight:</b> RBAC is great for stable, well-defined permission sets. When permissions depend on data (e.g., "user can only edit their own records"), combine with ABAC or resource-level ownership checks.</div>

What is Attribute-Based Access Control (ABAC) and when does it shine over RBAC?	<div style="text-align:left"><b>ABAC</b> makes authorization decisions based on <b>attributes</b> of the user, resource, and environment — not just roles.<br><br><b>Example policy:</b><br><code>Allow access if:<br>  user.department == resource.department<br>  AND user.clearanceLevel >= resource.sensitivityLevel<br>  AND request.time is within business hours</code><br><br><b>ABAC vs RBAC:</b><br>- RBAC: simple, fast, good for static permissions<br>- ABAC: flexible, fine-grained, handles complex rules and context<br><br><b>Use ABAC when:</b><br>- Access depends on resource properties (owner, classification)<br>- You need time-based or location-based access<br>- RBAC would require hundreds of roles to cover all cases<br><br><b>Downside:</b> Harder to reason about and audit. Policy explosion is a real risk.<br><br><b>Senior insight:</b> Modern systems often combine RBAC and ABAC — roles as coarse-grained filters, attributes for fine-grained decisions. Tools like OPA (Open Policy Agent) implement this well.</div>

What is the Principle of Least Privilege and how do you apply it?	<div style="text-align:left"><b>Least Privilege</b>: every user, service, or process should have only the minimum permissions needed to do its job — nothing more.<br><br><b>Analogy:</b> A delivery driver gets a key to the mailroom, not to the entire building.<br><br><b>How to apply it:</b><br>- DB users: read-only user for read operations; separate user with write access only where needed<br>- API keys: scope them to specific resources/actions<br>- IAM roles (cloud): avoid wildcard permissions (<code>*</code>)<br>- Services: don't run as root; use service accounts with scoped permissions<br><br><b>Common mistake:</b> Giving admin access "just in case" — this is called permission creep and is a major breach multiplier.<br><br><b>Senior insight:</b> Pair least privilege with <b>just-in-time access</b> (e.g., AWS IAM roles, Vault) so elevated permissions are granted temporarily and automatically revoked.</div>

What is the Zero Trust security model?	<div style="text-align:left"><b>Zero Trust</b>: "Never trust, always verify." No user, device, or service is trusted by default — even inside the network perimeter.<br><br><b>Traditional model:</b> Trust everything inside the corporate network (castle-and-moat). If an attacker gets in, they roam freely.<br><br><b>Zero Trust model:</b><br>- Authenticate and authorize every request, every time<br>- Assume breach — minimize blast radius<br>- Micro-segment networks so lateral movement is limited<br>- Enforce least privilege at every layer<br><br><b>Key pillars:</b><br>1. Verify identity (MFA, device health)<br>2. Limit access scope (least privilege)<br>3. Inspect and log all traffic<br><br><b>Senior insight:</b> Zero Trust is not a product — it's a mindset and architecture. It's especially critical in cloud-native and remote-work environments where the network perimeter has dissolved. Tools: BeyondCorp, Cloudflare Access, Azure AD Conditional Access.</div>

What is the Defense in Depth security pattern?	<div style="text-align:left"><b>Defense in Depth</b>: layer multiple security controls so that if one layer fails, others still protect the system.<br><br><b>Analogy:</b> A medieval castle — moat, outer wall, inner wall, keep. Each layer slows down and detects attackers.<br><br><b>Layers in a modern app:</b><br>1. <b>Perimeter:</b> Firewall, WAF, DDoS protection<br>2. <b>Network:</b> VPC, private subnets, security groups<br>3. <b>Application:</b> AuthN/AuthZ, input validation, rate limiting<br>4. <b>Data:</b> Encryption at rest and in transit, field-level encryption<br>5. <b>Monitoring:</b> SIEM, alerting, anomaly detection<br><br><b>Common mistake:</b> Relying on a single layer (e.g., "we have a firewall, so we're fine"). Any single control can be misconfigured, bypassed, or defeated.<br><br><b>Senior insight:</b> Design with the assumption that any individual control will eventually fail. The goal is detection and containment, not just prevention.</div>

What is the Strangler Fig pattern applied to security hardening?	<div style="text-align:left">The <b>Strangler Fig</b> pattern (originally for migrating legacy systems) applies to security hardening: incrementally replace insecure legacy behavior while keeping the system running.<br><br><b>Example — migrating from HTTP Basic Auth to OAuth 2.0:</b><br>1. Add OAuth 2.0 support alongside existing Basic Auth<br>2. Migrate clients one by one to OAuth<br>3. Monitor usage of old auth method<br>4. Deprecate and remove Basic Auth once traffic drops to zero<br><br><b>Why it matters:</b> Big-bang security migrations are risky — they break systems and often stall. Strangler Fig lets you harden incrementally without downtime.<br><br><b>Senior insight:</b> The same pattern applies to TLS version upgrades (TLS 1.0 → 1.3), hash algorithm migrations (MD5 → bcrypt), and API authentication changes. Use feature flags and metrics to track adoption.</div>

What is a Security Proxy (API Gateway) pattern and why is it used?	<div style="text-align:left">A <b>Security Proxy</b> sits in front of your services and centralizes cross-cutting security concerns: authentication, authorization, rate limiting, logging, and TLS termination.<br><br><b>Example — API Gateway:</b><br><code>Client → [API Gateway: authN, authZ, rate limit, WAF] → Microservices</code><br><br><b>Benefits:</b><br>- Services don't each need to implement auth logic<br>- Consistent policy enforcement<br>- Single point for audit logging<br>- Easier to rotate credentials/tokens<br><br><b>Common mistake:</b> Assuming the gateway handles all security — services still need to validate tokens and enforce their own authorization rules. Never trust the internal network blindly (Zero Trust).<br><br><b>Senior insight:</b> Tools: AWS API Gateway, Kong, Nginx, Envoy. In Kubernetes, an Ingress controller or service mesh (Istio) serves a similar role with mTLS between services.</div>

What is the Valet Key pattern in cloud security?	<div style="text-align:left">The <b>Valet Key</b> pattern provides a client with a limited, time-bound credential to access a specific resource directly — without routing through your servers.<br><br><b>Analogy:</b> A valet gets a key that only starts the car — it doesn't open the trunk or the house.<br><br><b>Example — S3 Pre-signed URLs:</b><br><code># Python (boto3)<br>url = s3.generate_presigned_url(<br>  'get_object',<br>  Params={'Bucket': 'my-bucket', 'Key': 'file.pdf'},<br>  ExpiresIn=3600  # valid for 1 hour<br>)</code><br><br>The client downloads directly from S3. Your server never touches the file bytes.<br><br><b>Benefits:</b><br>- Offloads bandwidth from your servers<br>- Credential expires automatically<br>- Scoped to one specific operation<br><br><b>Use cases:</b> File uploads/downloads, Azure SAS tokens, GCS signed URLs, temporary DB credentials via Vault.</div>

What is the Federated Identity pattern?	<div style="text-align:left"><b>Federated Identity</b>: instead of managing user credentials yourself, you delegate identity verification to an external, trusted <b>Identity Provider (IdP)</b>.<br><br><b>Example:</b> "Login with Google" — Google verifies the user, sends your app a token. You trust Google's assertion of identity.<br><br><b>Benefits:</b><br>- You don't store passwords (massive security liability removed)<br>- Users get SSO across multiple apps<br>- IdP handles MFA, password reset, account recovery<br><br><b>How it works:</b><br>1. User authenticates with the IdP<br>2. IdP issues a token (OIDC ID Token or SAML assertion)<br>3. Your app validates the token signature<br>4. Your app creates a local session<br><br><b>Common mistake:</b> Not validating the token signature, audience (<code>aud</code>), or issuer (<code>iss</code>) — opens the door to token substitution attacks.<br><br><b>Senior insight:</b> Federated identity shifts responsibility to the IdP for identity assurance. Choose IdPs that support MFA enforcement and have strong security postures (SOC2, ISO 27001).</div>

What is a Claims-Based Identity model?	<div style="text-align:left">In <b>Claims-Based Identity</b>, a user's identity is expressed as a set of <b>claims</b> — key-value assertions made by a trusted authority.<br><br><b>Example JWT payload (claims):</b><br><code>{<br>  "sub": "user-456",<br>  "email": "alice@example.com",<br>  "roles": ["editor", "viewer"],<br>  "department": "engineering",<br>  "exp": 1712345678<br>}</code><br><br>Your application reads these claims to make authorization decisions — without querying a database on every request.<br><br><b>Benefits:</b><br>- Decouples identity from authorization logic<br>- Works across systems and organizations<br>- Self-contained — no extra DB lookup needed<br><br><b>Common mistake:</b> Trusting claims without verifying the token signature. Also: putting too many claims in the token makes it large and leaks data.<br><br><b>Senior insight:</b> Design authorization around claims, not around user IDs. This makes multi-tenant, federated, and service-to-service scenarios much cleaner.</div>

What is mTLS (Mutual TLS) and how does it differ from standard TLS?	<div style="text-align:left"><b>Standard TLS:</b> the client verifies the server's certificate. One-way trust.<br><br><b>mTLS (Mutual TLS):</b> both sides present and verify certificates. Two-way trust.<br><br><b>Flow:</b><br>1. Server presents its certificate → client verifies it<br>2. Client presents its certificate → server verifies it<br>3. Only if both pass → connection established<br><br><b>Why it matters:</b> In microservices, mTLS ensures:<br>- Service-to-service calls are authenticated<br>- A compromised service can't impersonate another<br>- Traffic is encrypted even inside the internal network<br><br><b>Where it's used:</b><br>- Service mesh (Istio, Linkerd enforce mTLS automatically)<br>- B2B APIs with strict client authentication requirements<br><br><b>Common mistake:</b> Skipping mTLS inside "trusted" networks. Zero Trust says: never trust the internal network.<br><br><b>Senior insight:</b> Istio and Linkerd handle mTLS transparently — apps don't need to manage certificates. Use them rather than rolling your own certificate management.</div>

What is HTTPS and why is TLS critical for all traffic, even internal?	<div style="text-align:left"><b>HTTPS</b> = HTTP + <b>TLS (Transport Layer Security)</b>. TLS provides:<br><br>1. <b>Encryption</b> — data is unreadable in transit<br>2. <b>Integrity</b> — data can't be tampered with undetected<br>3. <b>Authentication</b> — client verifies it's talking to the real server<br><br><b>Why even for internal traffic:</b><br>- Insiders and compromised services can intercept unencrypted traffic<br>- Compliance standards (PCI-DSS, HIPAA) often require it<br>- Zero Trust mandates encryption regardless of network location<br><br><b>Common mistakes:</b><br>- Using self-signed certs without proper CA chain validation<br>- Disabling certificate validation in internal services (<code>verify=False</code> in Python requests)<br>- Using outdated TLS versions (TLS 1.0/1.1 are deprecated and broken)<br><br><b>Senior insight:</b> Use TLS 1.2 minimum; prefer TLS 1.3 (faster handshake, stronger ciphers). Tools like Let's Encrypt, AWS ACM, or cert-manager (Kubernetes) automate certificate management.</div>

What is input validation and why is it a security boundary?	<div style="text-align:left"><b>Input validation</b> ensures that data from external sources (users, APIs, files) conforms to expected format, type, length, and range before being processed.<br><br><b>Security boundary rule:</b> Never trust input that crosses a trust boundary — user input, external API responses, file contents, query parameters.<br><br><b>Types of validation:</b><br>- <b>Syntactic:</b> Is the format correct? (valid email, valid UUID)<br>- <b>Semantic:</b> Does it make sense? (end date after start date)<br>- <b>Business rule:</b> Is it allowed? (user has permission to submit this value)<br><br><b>Common mistake:</b> Validating only on the frontend. Client-side validation is UX, not security — it can be bypassed. Always validate server-side.<br><br><b>Senior insight:</b> Use an allow-list (whitelist) approach: define what IS valid, reject everything else. Never use a deny-list alone — attackers find variations you didn't think of.<br><br><b>In C#:</b> FluentValidation<br><b>In Python:</b> Pydantic<br><b>In JS/TS:</b> Zod, Joi</div>

What is SQL Injection and how do you prevent it?	<div style="text-align:left"><b>SQL Injection</b> occurs when user input is concatenated directly into a SQL query, letting attackers manipulate the query logic.<br><br><b>Vulnerable code:</b><br><code>// NEVER do this<br>query = "SELECT * FROM users WHERE name = '" + userName + "'";<br>// If userName = "'; DROP TABLE users; --"<br>// → database table gets deleted</code><br><br><b>Prevention — use parameterized queries / prepared statements:</b><br><code>-- SQL parameterized (C# / Dapper)<br>connection.Query("SELECT * FROM users WHERE name = @name",<br>  new { name = userName });</code><br><br><code># Python (SQLAlchemy)<br>session.execute(text("SELECT * FROM users WHERE name = :name"),<br>  {"name": user_name})</code><br><br><b>Also helps:</b> ORMs (EF Core, Hibernate, SQLAlchemy) use parameterized queries by default.<br><br><b>Common mistake:</b> String interpolation in raw queries. Also: stored procedures don't automatically prevent injection if they concatenate strings internally.<br><br><b>Senior insight:</b> SQLi is in the OWASP Top 10 every year. It's completely preventable — there's no excuse for it in new code.</div>

What is XSS (Cross-Site Scripting) and how do you prevent it?	<div style="text-align:left"><b>XSS</b> occurs when an attacker injects malicious scripts into web pages viewed by other users. The browser executes the script in the victim's context.<br><br><b>Example:</b> A comment field stores <code>&lt;script&gt;document.cookie&lt;/script&gt;</code>. Every user who sees that comment runs the script — leaking their session cookie.<br><br><b>Types:</b><br>- <b>Stored XSS</b> — injected content saved to DB, served to all users<br>- <b>Reflected XSS</b> — script in URL, reflected in response<br>- <b>DOM-based XSS</b> — client-side JS writes attacker input to the DOM<br><br><b>Prevention:</b><br>1. <b>Output encoding</b> — encode user content before rendering (HTML entities)<br>2. <b>CSP (Content Security Policy)</b> — browser-enforced restriction on scripts<br>3. Use frameworks that auto-escape (React, Angular, Blazor do this by default)<br>4. Avoid <code>innerHTML</code>, <code>eval()</code>, <code>dangerouslySetInnerHTML</code><br><br><b>Senior insight:</b> CSP is your last line of defense — even if injection happens, CSP can block script execution. Always set a restrictive CSP header.</div>

What is CSRF (Cross-Site Request Forgery) and how is it prevented?	<div style="text-align:left"><b>CSRF</b> tricks a logged-in user's browser into sending an unintended request to your app — using their existing session cookie.<br><br><b>Example:</b> User is logged into bank.com. They visit evil.com which has:<br><code>&lt;img src="https://bank.com/transfer?to=attacker&amount=1000"&gt;</code><br>The browser sends the request with the user's cookie — triggering a transfer.<br><br><b>Prevention:</b><br>1. <b>CSRF tokens</b> — a unique, secret token in forms; server validates it<br>2. <b>SameSite cookie attribute</b> — <code>SameSite=Strict</code> or <code>Lax</code> prevents cookies from being sent cross-site<br>3. <b>Double Submit Cookie</b> — compare cookie value to request header value<br>4. <b>Origin/Referer header validation</b> — check where the request came from<br><br><b>Common mistake:</b> CSRF doesn't affect stateless token (Bearer JWT) auth — the token isn't sent automatically by browsers. It mainly affects cookie-based sessions.<br><br><b>Senior insight:</b> Modern frameworks (ASP.NET Core, Django, Rails) include CSRF protection by default. Know how to enable and configure it — don't disable it for convenience.</div>

What is the OWASP Top 10 and why should every architect know it?	<div style="text-align:left">The <b>OWASP Top 10</b> is a standard list of the most critical web application security risks, updated every few years by the Open Web Application Security Project.<br><br><b>Current Top 10 (2021):</b><br>1. Broken Access Control<br>2. Cryptographic Failures<br>3. Injection (SQL, NoSQL, OS, LDAP)<br>4. Insecure Design<br>5. Security Misconfiguration<br>6. Vulnerable and Outdated Components<br>7. Identification and Authentication Failures<br>8. Software and Data Integrity Failures<br>9. Security Logging and Monitoring Failures<br>10. Server-Side Request Forgery (SSRF)<br><br><b>Why it matters:</b> These are the most exploited, most impactful, and most preventable vulnerabilities. Attackers have automated tools to find them.<br><br><b>Senior insight:</b> Use OWASP Top 10 as a checklist during threat modeling and code reviews. Broken Access Control (#1) is often overlooked — it's not just about authentication; it's about what authenticated users can access that they shouldn't.</div>

What is a threat model and how do you build one?	<div style="text-align:left">A <b>threat model</b> is a structured analysis of your system to identify potential threats, their impact, and mitigations — before attackers find them.<br><br><b>Common framework — STRIDE:</b><br>- <b>S</b>poofing — impersonating a user or service<br>- <b>T</b>ampering — modifying data in transit or at rest<br>- <b>R</b>epudiation — denying an action occurred<br>- <b>I</b>nformation Disclosure — exposing sensitive data<br>- <b>D</b>enial of Service — making the system unavailable<br>- <b>E</b>levation of Privilege — gaining more access than authorized<br><br><b>Steps:</b><br>1. Draw a data flow diagram (DFD) — identify trust boundaries<br>2. Apply STRIDE to each component and data flow<br>3. Rate risk (DREAD or CVSS scores)<br>4. Define mitigations<br><br><b>Senior insight:</b> Threat modeling is most valuable done early (design phase), not after the fact. Even a 2-hour whiteboard session per feature dramatically reduces security debt. Tools: OWASP Threat Dragon, Microsoft Threat Modeling Tool.</div>

What is the Secure by Design principle?	<div style="text-align:left"><b>Secure by Design</b> means security is built into a system from the start — not added as an afterthought or patch.<br><br><b>Core principles:</b><br>1. <b>Fail securely</b> — errors should deny access, not grant it<br>2. <b>Least privilege by default</b> — start with no access, grant explicitly<br>3. <b>Minimal attack surface</b> — expose only what's needed<br>4. <b>Separation of concerns</b> — security controls shouldn't depend on application logic<br>5. <b>Defense in depth</b> — multiple layered controls<br><br><b>Example of failing securely:</b><br><code>// WRONG — failure grants access<br>if (isAuthError) return true;<br><br>// RIGHT — failure denies access<br>if (!isAuthenticated) return Forbidden();</code><br><br><b>Common mistake:</b> "We'll add security later" — security retrofitting is 6–100x more expensive than building it in from day one (NIST study).<br><br><b>Senior insight:</b> Shift security left — integrate security reviews, static analysis (SAST), and dependency scanning into the CI/CD pipeline, not just at release time.</div>

What is encryption at rest vs encryption in transit?	<div style="text-align:left"><b>Encryption in transit</b>: data is encrypted while moving between systems.<br>→ Implemented via TLS/HTTPS. Protects against network eavesdropping (MITM attacks).<br><br><b>Encryption at rest</b>: data is encrypted while stored on disk, in databases, or backups.<br>→ Protects against physical theft, unauthorized storage access, or breached storage.<br><br><b>Tools for encryption at rest:</b><br>- Database: Transparent Data Encryption (TDE) in SQL Server, PostgreSQL pgcrypto<br>- Cloud: AWS S3 SSE, Azure Storage Encryption, GCP CMEK<br>- Disk: BitLocker, LUKS<br><br><b>Common mistake:</b> Encrypting at rest but storing encryption keys next to the data — that's like locking a safe and leaving the combination on top of it.<br><br><b>Senior insight:</b> Use a dedicated Key Management Service (AWS KMS, Azure Key Vault, HashiCorp Vault) for key management. Implement key rotation policies. Field-level encryption adds an extra layer for highly sensitive data (SSN, PAN).</div>

What is the difference between hashing, encryption, and encoding?	<div style="text-align:left">These are frequently confused — each serves a different purpose:<br><br><b>Encoding</b> — transforms data for compatibility. Reversible. NOT security.<br><code>base64.encode("hello") → "aGVsbG8="  # anyone can decode this</code><br><br><b>Encryption</b> — transforms data so only authorized parties can read it. Reversible with the correct key.<br><code>AES-256-GCM encrypt("hello", key) → cipher  # decryptable with key</code><br><br><b>Hashing</b> — one-way transformation. Cannot be reversed.<br><code>SHA-256("hello") → "2cf24..."  # you can never get "hello" back</code><br><br><b>For passwords, always use:</b> slow hashing algorithms designed for this purpose:<br><code>bcrypt, Argon2id, scrypt — NOT SHA-256 or MD5</code><br><br><b>Common mistake:</b> Storing passwords hashed with MD5 or SHA-1 (fast hashes, crackable with GPU rainbow tables in seconds).<br><br><b>Senior insight:</b> SHA-256 is great for data integrity (checksums), terrible for passwords. Argon2id is the current gold standard for password hashing.</div>

What is the difference between symmetric and asymmetric encryption?	<div style="text-align:left"><b>Symmetric encryption:</b> the same key is used to encrypt and decrypt.<br><code>AES-256-GCM: encrypt(data, key) → cipher; decrypt(cipher, key) → data</code><br>- Fast — suitable for large data volumes<br>- Problem: how do you securely share the key?<br><br><b>Asymmetric encryption:</b> uses a key pair — public key (share freely) and private key (keep secret).<br><code>RSA: encrypt(data, publicKey) → cipher; decrypt(cipher, privateKey) → data</code><br>- Solves key distribution problem<br>- Slower — used for small data or key exchange<br><br><b>In practice — TLS uses both:</b><br>1. Asymmetric (RSA/ECDH) for key exchange → establishes a shared secret<br>2. Symmetric (AES) for actual data encryption → fast bulk encryption<br><br><b>Common mistake:</b> Using RSA to encrypt large files (slow and limited by key size). Encrypt the file with AES, encrypt the AES key with RSA.<br><br><b>Senior insight:</b> Prefer ECDH over RSA for key exchange — same security with much smaller key sizes (256-bit ECDH ≈ 3072-bit RSA).</div>

What is a Secret Management strategy and why is it critical?	<div style="text-align:left"><b>Secret management</b> is how you securely store, distribute, rotate, and audit access to secrets: API keys, passwords, certificates, tokens.<br><br><b>The problem:</b> Secrets in code, config files, or environment variables get leaked — committed to Git, logged, or exposed via errors.<br><br><b>Best practices:</b><br>1. <b>Never hardcode secrets</b> in source code<br>2. <b>Never commit secrets</b> to Git (use .gitignore + pre-commit hooks)<br>3. Use a dedicated secrets manager:<br>   - AWS Secrets Manager / Parameter Store<br>   - Azure Key Vault<br>   - HashiCorp Vault<br>   - Kubernetes Secrets (with encryption at rest enabled)<br>4. Rotate secrets regularly and on suspected compromise<br>5. Audit all access to secrets<br><br><b>Common mistake:</b> Storing secrets in environment variables without encryption — they appear in process lists and crash dumps.<br><br><b>Senior insight:</b> Use Vault's dynamic secrets feature — issue temporary DB credentials per-request, automatically revoked. This eliminates long-lived credentials entirely.</div>

What is Rate Limiting and how does it protect APIs?	<div style="text-align:left"><b>Rate limiting</b> restricts how many requests a client can make in a given time window — protecting against brute force, DoS, and API abuse.<br><br><b>Common algorithms:</b><br>- <b>Fixed Window:</b> 100 requests per minute, reset at clock boundary<br>- <b>Sliding Window:</b> 100 requests in any rolling 60-second window (smoother)<br>- <b>Token Bucket:</b> bucket fills at a fixed rate; each request costs a token (allows bursts)<br>- <b>Leaky Bucket:</b> requests process at a fixed rate; excess queued or dropped<br><br><b>Implementation levels:</b><br>- API Gateway (Kong, AWS API GW, Nginx)<br>- Application middleware<br>- Redis-based (store counters with TTL)<br><br><code># Example: Redis sliding window<br>key = f"ratelimit:{user_id}"<br>redis.zadd(key, {now: now})<br>redis.zremrangebyscore(key, 0, now - 60)<br>count = redis.zcard(key)</code><br><br><b>Senior insight:</b> Rate limit by IP, user ID, and API key separately. Also implement exponential backoff for login attempts — after 5 failures, wait 1s, 2s, 4s, 8s. Combine with account lockout policies.</div>

What is CORS (Cross-Origin Resource Sharing) and how should it be configured?	<div style="text-align:left"><b>CORS</b> is a browser mechanism that controls which origins can access your API from JavaScript. By default, browsers block cross-origin requests (same-origin policy).<br><br><b>Example:</b> <code>https://app.com</code> (frontend) → <code>https://api.example.com</code> (backend) — blocked by default. CORS headers tell the browser whether to allow it.<br><br><b>Key headers:</b><br><code>Access-Control-Allow-Origin: https://app.com<br>Access-Control-Allow-Methods: GET, POST<br>Access-Control-Allow-Headers: Authorization, Content-Type</code><br><br><b>Common mistakes:</b><br>- Setting <code>Access-Control-Allow-Origin: *</code> on authenticated endpoints — this negates CSRF protections<br>- Allowing all origins in production<br>- Reflecting the request's Origin header without validation<br><br><b>Senior insight:</b> CORS is a browser-only control — it doesn't protect your API from non-browser clients (curl, Postman, other services). Don't use CORS as your sole security measure. It's access control for the browser, not a replacement for authZ.</div>

What is Content Security Policy (CSP) and why is it important?	<div style="text-align:left"><b>CSP</b> is an HTTP response header that tells browsers which content sources are trusted — blocking inline scripts, unauthorized domains, and other injection vectors.<br><br><b>Example header:</b><br><code>Content-Security-Policy:<br>  default-src 'self';<br>  script-src 'self' https://cdn.trusted.com;<br>  style-src 'self' 'unsafe-inline';<br>  img-src *;<br>  frame-ancestors 'none';</code><br><br><b>What it prevents:</b><br>- XSS — even if an attacker injects a script tag, the browser won't execute it if the origin isn't whitelisted<br>- Clickjacking — <code>frame-ancestors 'none'</code> prevents embedding in iframes<br>- Data exfiltration — <code>connect-src</code> limits where JS can send data<br><br><b>Common mistake:</b> Setting <code>unsafe-inline</code> and <code>unsafe-eval</code> — this defeats most XSS protection. Refactor inline scripts to external files.<br><br><b>Senior insight:</b> Use CSP in report-only mode first (<code>Content-Security-Policy-Report-Only</code>) to audit violations before enforcing. Use a nonce-based CSP for dynamic inline scripts.</div>

What is HTTPS Strict Transport Security (HSTS)?	<div style="text-align:left"><b>HSTS</b> is an HTTP response header that tells browsers: "Always use HTTPS for this domain — never HTTP, even if the user types http://".<br><br><b>Header:</b><br><code>Strict-Transport-Security: max-age=31536000; includeSubDomains; preload</code><br><br><b>What it prevents:</b><br>- <b>SSL stripping attacks</b> — attacker downgrades HTTPS to HTTP to intercept traffic<br>- Accidental HTTP use even when a redirect is in place<br><br><b>How it works:</b><br>1. User visits <code>https://example.com</code> for the first time<br>2. Browser stores the HSTS policy<br>3. All future requests, even typed as http://, go directly as HTTPS<br>4. The browser refuses to connect if TLS fails (no "click through" warning)<br><br><b>preload:</b> Submit your domain to the HSTS preload list — browsers will enforce HTTPS before the first visit ever happens.<br><br><b>Senior insight:</b> HSTS is a zero-cost, high-value security header. Enable it on all production services. The <code>includeSubDomains</code> flag ensures subdomains are protected too.</div>

What is OWASP's Security Misconfiguration risk and how do you prevent it?	<div style="text-align:left"><b>Security Misconfiguration</b> is OWASP #5 — when systems are deployed with insecure default settings, unnecessary features enabled, or missing hardening.<br><br><b>Common examples:</b><br>- Default admin credentials not changed (admin/admin)<br>- Debug mode enabled in production (Django <code>DEBUG=True</code>)<br>- Directory listing enabled on web servers<br>- Verbose error messages exposing stack traces to users<br>- Open cloud storage buckets (S3 public read)<br>- Unused ports and services exposed<br><br><b>Prevention:</b><br>- Infrastructure as Code (IaC) with security baselines baked in<br>- CIS Benchmarks for OS and service hardening<br>- Automated scanning (AWS Config, Azure Security Center, Trivy, Prowler)<br>- Disable all non-essential features by default<br>- Separate configurations per environment — never use dev config in prod<br><br><b>Senior insight:</b> Treat your infrastructure config as code. Use policy-as-code tools (OPA, AWS SCPs) to prevent misconfiguration from being deployed in the first place, not just detected after.</div>

What is PCI-DSS and what does it require from architects?	<div style="text-align:left"><b>PCI-DSS (Payment Card Industry Data Security Standard)</b> is a mandatory security standard for any organization that stores, processes, or transmits cardholder data (credit/debit card numbers).<br><br><b>12 core requirements (high level):</b><br>1. Install and maintain a firewall<br>2. Don't use vendor-supplied defaults (passwords, settings)<br>3. Protect stored cardholder data<br>4. Encrypt transmission of cardholder data<br>5. Use and update anti-malware<br>6. Develop and maintain secure systems<br>7. Restrict access by business need-to-know<br>8. Identify and authenticate all users<br>9. Restrict physical access<br>10. Track and monitor all access<br>11. Regularly test security systems<br>12. Maintain an information security policy<br><br><b>Architect implications:</b><br>- Segment the cardholder data environment (CDE) — minimize what's in scope<br>- Never store PAN (primary account number) in plaintext<br>- Tokenize card data — use a payment processor (Stripe, Braintree) to keep raw card data out of your systems<br><br><b>Senior insight:</b> The cheapest PCI compliance strategy is scope reduction — if raw card data never touches your servers, most requirements don't apply. Use Stripe.js or equivalent tokenization.</div>

What is GDPR and what architectural decisions does it drive?	<div style="text-align:left"><b>GDPR (General Data Protection Regulation)</b> is an EU regulation governing how personal data of EU residents is collected, stored, processed, and deleted.<br><br><b>Key rights that affect architecture:</b><br>- <b>Right to Access</b> — users can request all their personal data<br>- <b>Right to Erasure (Right to be Forgotten)</b> — users can request deletion<br>- <b>Data Portability</b> — export data in machine-readable format<br>- <b>Consent</b> — must be explicit, informed, and withdrawable<br><br><b>Architectural implications:</b><br>1. <b>Data inventory</b> — know where PII lives across all systems<br>2. <b>Data residency</b> — EU data may not leave EU without adequate protections<br>3. <b>Erasure pipelines</b> — must be able to delete PII across all datastores (including backups)<br>4. <b>Audit logging</b> — who accessed what data and when<br>5. <b>Pseudonymization</b> — replace PII with tokens in analytics and logging<br><br><b>Senior insight:</b> "Right to be Forgotten" is architecturally hard — PII can be in relational DBs, event logs, caches, backups, and analytics warehouses. Design erasure flows from day one, not after.</div>

What is SOC 2 compliance and why do B2B SaaS companies pursue it?	<div style="text-align:left"><b>SOC 2 (System and Organization Controls 2)</b> is an auditing framework for service organizations that defines how they should manage customer data based on five Trust Services Criteria:<br><br>1. <b>Security</b> — protection against unauthorized access (mandatory)<br>2. <b>Availability</b> — system uptime and performance commitments<br>3. <b>Processing Integrity</b> — complete, accurate, timely processing<br>4. <b>Confidentiality</b> — protection of confidential information<br>5. <b>Privacy</b> — PII collected and used per stated policies<br><br><b>Type I vs Type II:</b><br>- <b>Type I</b> — snapshot: controls are designed correctly at a point in time<br>- <b>Type II</b> — ongoing: controls worked effectively over 6–12 months<br><br><b>Why it matters:</b> Enterprise customers require SOC 2 before signing contracts. Without it, you can't close enterprise deals.<br><br><b>Architect implications:</b> Evidence collection, audit logging, access reviews, vendor risk management, incident response plans — all must be documented and demonstrable.</div>

What is HIPAA and what security controls does it require?	<div style="text-align:left"><b>HIPAA (Health Insurance Portability and Accountability Act)</b> is a US law that protects <b>PHI (Protected Health Information)</b> — any data that identifies a patient and relates to their health or treatment.<br><br><b>Key rules:</b><br>- <b>Privacy Rule</b> — limits who can access and share PHI<br>- <b>Security Rule</b> — technical safeguards for electronic PHI (ePHI)<br>- <b>Breach Notification Rule</b> — must notify patients within 60 days of a breach<br><br><b>Technical safeguards required:</b><br>- Access controls (unique user IDs, automatic logoff)<br>- Audit logging (who accessed ePHI and when)<br>- Integrity controls (ensure ePHI isn't altered)<br>- Transmission security (encryption in transit)<br><br><b>Architectural decisions:</b><br>- HIPAA-eligible cloud services (AWS HIPAA BAA, Azure, GCP)<br>- Encryption at rest and in transit mandatory<br>- Strict access controls and audit trails<br>- Data minimization — only collect necessary PHI<br><br><b>Senior insight:</b> HIPAA violations can result in fines up to $1.9M per violation category per year. More importantly, a breach erodes patient trust permanently.</div>

What is ISO 27001 and how does it differ from SOC 2?	<div style="text-align:left"><b>ISO 27001</b> is an international standard for establishing, implementing, and managing an <b>Information Security Management System (ISMS)</b>.<br><br><b>Key difference from SOC 2:</b><br>- <b>ISO 27001</b> — international, prescribes a management system (process-focused, certifiable worldwide)<br>- <b>SOC 2</b> — US-focused, attestation by a CPA firm (evidence-focused)<br><br><b>What ISO 27001 covers:</b><br>- Risk assessment and treatment<br>- Asset management<br>- Access control policies<br>- Cryptography policies<br>- Physical security<br>- Supplier relationships<br>- Incident management<br>- Business continuity<br><br><b>Certification process:</b><br>1. Define ISMS scope<br>2. Conduct risk assessment<br>3. Implement controls (Annex A has 93 controls in ISO 27001:2022)<br>4. Internal audit<br>5. External certification audit (every 3 years)<br><br><b>Senior insight:</b> Large enterprises and government contracts often require ISO 27001. Some companies pursue both ISO 27001 and SOC 2 to satisfy different customer requirements. They share significant overlap — pursuing one helps with the other.</div>

What is the NIST Cybersecurity Framework (CSF)?	<div style="text-align:left">The <b>NIST Cybersecurity Framework</b> is a voluntary framework providing guidelines and best practices for managing cybersecurity risk, organized around 5 core functions:<br><br>1. <b>Identify</b> — understand your assets, risks, and environment<br>   (asset inventory, risk assessment, threat intelligence)<br><br>2. <b>Protect</b> — implement safeguards<br>   (access controls, training, data security, maintenance)<br><br>3. <b>Detect</b> — identify when an incident occurs<br>   (monitoring, anomaly detection, SIEM)<br><br>4. <b>Respond</b> — act on detected incidents<br>   (incident response plan, communication, analysis)<br><br>5. <b>Recover</b> — restore capabilities after an incident<br>   (recovery planning, improvements, communications)<br><br><b>Why it matters for architects:</b> The NIST CSF provides a common language for security discussions across business and technical teams. It's widely referenced in government and enterprise security programs.<br><br><b>Senior insight:</b> Many organizations skip "Detect" and "Respond" — they invest heavily in prevention but have no monitoring or incident response. An undetected breach is often worse than a detected and contained one.</div>

What is a WAF (Web Application Firewall) and what can it protect against?	<div style="text-align:left">A <b>WAF</b> inspects HTTP/HTTPS traffic between clients and your web application, blocking malicious requests based on rules.<br><br><b>What WAFs protect against:</b><br>- SQL Injection<br>- XSS (Cross-Site Scripting)<br>- CSRF<br>- Path traversal (<code>../../etc/passwd</code>)<br>- SSRF (Server-Side Request Forgery)<br>- Known CVE exploits (virtual patching)<br>- Bot traffic and scrapers<br>- DDoS Layer 7 attacks<br><br><b>Deployment modes:</b><br>- Cloud: AWS WAF, Cloudflare WAF, Azure Front Door WAF<br>- On-prem: ModSecurity (open source)<br>- Inline with API Gateway or CDN<br><br><b>Limitations:</b><br>- WAFs are not a replacement for secure coding — they can be bypassed with obfuscation<br>- High false-positive rates if rules are too aggressive<br>- Cannot protect against logic flaws in business logic<br><br><b>Senior insight:</b> Use a WAF as one layer of defense — not the only one. Enable OWASP Core Rule Set as a baseline, then tune false positives. Use AWS WAF + Shield for DDoS protection in AWS environments.</div>

What is SSRF (Server-Side Request Forgery)?	<div style="text-align:left"><b>SSRF</b> occurs when an attacker tricks your server into making an HTTP request to an unintended location — often internal services or cloud metadata endpoints.<br><br><b>Example:</b> Your app fetches a user-supplied URL to preview a webpage:<br><code>GET /preview?url=http://169.254.169.254/latest/meta-data/iam/credentials</code><br>On AWS, this returns IAM credentials — game over.<br><br><b>Impact:</b><br>- Access internal APIs not exposed to the internet<br>- Read cloud metadata (AWS, Azure, GCP instance metadata APIs)<br>- Scan internal network<br>- Read files via <code>file://</code> protocol<br><br><b>Prevention:</b><br>1. Validate and allowlist URLs before fetching<br>2. Block requests to private IP ranges (127.0.0.0/8, 10.0.0.0/8, 169.254.x.x)<br>3. Disable unused URL schemes (file://, ftp://)<br>4. Use network-level controls — egress filtering<br>5. On AWS: use IMDSv2 (requires session token, resists SSRF)<br><br><b>Senior insight:</b> SSRF was #10 in OWASP 2021. The Capital One breach (2019) exploited SSRF to access cloud metadata credentials, compromising 100M+ records.</div>

What is a Penetration Test and how does it differ from a vulnerability scan?	<div style="text-align:left"><b>Vulnerability Scan:</b> automated tool runs against your system to identify known vulnerabilities (CVEs, misconfigurations). Fast, broad, low depth.<br><code>Tools: Nessus, OpenVAS, Qualys, Trivy (containers)</code><br><br><b>Penetration Test (Pentest):</b> a skilled human (or team) attempts to actively exploit vulnerabilities — simulating a real attacker. Slow, targeted, deep.<br><br><b>Types of pentests:</b><br>- <b>Black box</b> — tester has no prior knowledge (simulates external attacker)<br>- <b>White box</b> — full code and architecture access (most thorough)<br>- <b>Gray box</b> — partial knowledge (simulates insider or authenticated attacker)<br><br><b>What a pentest finds that a scan misses:</b><br>- Business logic flaws<br>- Chained vulnerabilities (A + B together = critical)<br>- Human-factor exploits (social engineering)<br>- Application-specific logic errors<br><br><b>Senior insight:</b> Run automated scans continuously in CI/CD. Run pentests annually or before major releases. Combine with bug bounty programs (HackerOne, Bugcrowd) for continuous coverage.</div>

What is SAST and DAST in application security?	<div style="text-align:left"><b>SAST (Static Application Security Testing)</b> — analyzes source code, bytecode, or binaries for vulnerabilities without running the application.<br>→ "White box" — checks the code itself<br>→ Run in CI/CD at build time<br><br><code>Tools: SonarQube, Semgrep, Checkmarx, Coverity, Bandit (Python), Roslyn Analyzers (C#)</code><br><br><b>DAST (Dynamic Application Security Testing)</b> — tests the running application by sending malicious inputs and observing responses. No source code needed.<br>→ "Black box" — tests the live app<br>→ Run against staging environments<br><br><code>Tools: OWASP ZAP, Burp Suite, Nuclei</code><br><br><b>SCA (Software Composition Analysis)</b> — checks third-party dependencies for known CVEs.<br><code>Tools: Snyk, Dependabot, OWASP Dependency-Check</code><br><br><b>Senior insight:</b> No single tool catches everything. Layer SAST + DAST + SCA in your pipeline. SAST has false positives — train developers to triage them. DAST finds issues SAST misses (runtime behavior, config issues).</div>

What is the principle of Secure Defaults?	<div style="text-align:left"><b>Secure Defaults</b> means a system is in its most secure configuration out of the box — security doesn't require extra effort; insecurity does.<br><br><b>Examples of insecure defaults (wrong):</b><br>- Default password: admin/admin<br>- Debug mode on<br>- All users can create admin accounts<br>- Verbose error messages enabled<br>- CORS set to <code>*</code><br><br><b>Examples of secure defaults (right):</b><br>- Force password change on first login<br>- Debug off in production<br>- Minimal permissions until explicitly granted<br>- Generic error messages (log details server-side only)<br>- CORS restricted to known origins<br><br><b>Language/framework examples:</b><br>- Django: <code>DEBUG=False</code> disables debug info; CSRF protection on by default<br>- ASP.NET Core: HTTPS redirect middleware; anti-forgery tokens by default<br>- React: HTML encoding in JSX by default (prevents XSS)<br><br><b>Senior insight:</b> Design libraries and internal frameworks with secure defaults. A developer who takes the path of least resistance should end up with a secure result, not a vulnerable one.</div>

What is an API Security checklist that every architect should know?	<div style="text-align:left">A practical security checklist for APIs (based on OWASP API Security Top 10):<br><br><b>Authentication:</b><br>- Use OAuth 2.0 + OIDC, not custom auth<br>- Validate JWT: signature, expiry, audience, issuer<br>- Enforce MFA for sensitive operations<br><br><b>Authorization:</b><br>- Check authorization on every endpoint (don't rely on route obscurity)<br>- Use object-level authorization (IDOR prevention): verify the user owns the resource<br>- Avoid exposing internal IDs — use UUIDs or opaque tokens<br><br><b>Input:</b><br>- Validate and sanitize all inputs server-side<br>- Restrict payload size<br>- Use strict content-type enforcement<br><br><b>Transport:</b><br>- HTTPS only, TLS 1.2+, HSTS<br>- No sensitive data in URLs (logged by proxies)<br><br><b>Rate limiting &amp; monitoring:</b><br>- Rate limit all endpoints<br>- Log all auth events<br>- Alert on anomalies<br><br><b>Senior insight:</b> OWASP API Security Top 10 highlights that broken object-level authorization (BOLA/IDOR) is the #1 API risk — it's not caught by scanners, only by testing every endpoint with different user identities.</div>

What is IDOR (Insecure Direct Object Reference) and how is it prevented?	<div style="text-align:left"><b>IDOR</b> occurs when an API exposes internal object identifiers and doesn't verify that the requesting user is authorized to access that specific object.<br><br><b>Example:</b><br><code>GET /api/invoices/1042  # attacker changes to 1043 → sees another user's invoice</code><br><br>The server returns the invoice without checking if the user owns invoice 1043.<br><br><b>Why it's so common:</b> Authentication checks "are you logged in?" but IDOR is about authorization — "are you allowed to access THIS object?"<br><br><b>Prevention:</b><br>1. Check ownership on every object access:<br><code># Python example<br>invoice = Invoice.get(id=invoice_id)<br>if invoice.owner_id != current_user.id:<br>    raise Forbidden()</code><br>2. Use indirect references — UUIDs instead of sequential IDs<br>3. Use access control lists (ACLs) or resource-based policies<br><br><b>Senior insight:</b> IDOR is the #1 API vulnerability (OWASP API Top 10). It's not caught by static analysis or dependency scanners — only by business-logic-aware testing. Include IDOR tests in every API security review.</div>

What is the security implication of dependency management and supply chain attacks?	<div style="text-align:left">A <b>supply chain attack</b> targets the software dependencies and build pipeline of an application — inserting malicious code into trusted libraries or build tools.<br><br><b>Famous examples:</b><br>- <b>SolarWinds (2020)</b> — malicious code injected into a software update, compromising 18,000+ organizations<br>- <b>event-stream (npm, 2018)</b> — malicious package injected into a popular npm library<br>- <b>Log4Shell (2021)</b> — critical RCE in Log4j, embedded in thousands of systems<br><br><b>Mitigation strategies:</b><br>1. <b>SCA scanning</b> — scan dependencies for known CVEs (Snyk, Dependabot, OWASP Dependency-Check)<br>2. <b>Lock files</b> — pin exact dependency versions (package-lock.json, poetry.lock, packages.lock.json)<br>3. <b>SBOM (Software Bill of Materials)</b> — document all components in your build<br>4. <b>Verified signatures</b> — check package signatures where supported (npm provenance, NuGet signing)<br>5. <b>Private artifact registry</b> — proxy public packages through internal registry (Artifactory, Nexus)<br>6. <b>Minimal dependencies</b> — fewer dependencies = smaller attack surface<br><br><b>Senior insight:</b> Supply chain attacks are asymmetric — one compromised package can affect millions of apps. Treat your dependency tree as part of your threat model.</div>

What is Security Information and Event Management (SIEM) and why is it critical?	<div style="text-align:left">A <b>SIEM</b> aggregates, correlates, and analyzes security logs and events from across your infrastructure in real time — enabling threat detection and incident response.<br><br><b>What it ingests:</b><br>- Auth logs (login attempts, MFA failures)<br>- API access logs<br>- Firewall and network logs<br>- Application error logs<br>- Cloud audit trails (AWS CloudTrail, Azure Activity Log)<br><br><b>What it does:</b><br>- Correlates events across systems (e.g., 5 failed logins + 1 success from a new IP = alert)<br>- Detects known attack patterns (rules)<br>- Anomaly detection (behavior baselines)<br>- Generates alerts for SOC team<br><br><b>Tools:</b> Splunk, Microsoft Sentinel, Elastic SIEM, Datadog Security, AWS Security Hub + GuardDuty<br><br><b>Common mistake:</b> Logging everything but alerting on nothing — alert fatigue is as dangerous as no alerts.<br><br><b>Senior insight:</b> GDPR, SOC 2, PCI-DSS, and HIPAA all require audit logging and monitoring. Design your logging strategy early — retrofitting it is painful. Log security-relevant events explicitly; don't rely on debugging logs alone.</div>

What is the difference between authentication tokens: Access Token, Refresh Token, and ID Token?	<div style="text-align:left">In OAuth 2.0 + OIDC, three types of tokens serve different purposes:<br><br><b>Access Token:</b><br>- Authorizes access to a resource (API)<br>- Short-lived: 5–60 minutes<br>- Sent in every API request: <code>Authorization: Bearer &lt;token&gt;</code><br>- May be a JWT or opaque string<br><br><b>Refresh Token:</b><br>- Used to get a new access token without re-login<br>- Long-lived: days to months<br>- Stored securely server-side (or HttpOnly cookie)<br>- Never sent to resource servers — only to the auth server<br><br><b>ID Token:</b><br>- Proves who the user is (OIDC only)<br>- Always a JWT with identity claims (sub, email, name)<br>- Consumed by the client app, not by APIs<br>- Not for authorization — for identity<br><br><b>Common mistake:</b> Sending the ID token to APIs for authorization — it's an identity token, not an access token. APIs should only accept access tokens.<br><br><b>Senior insight:</b> Refresh token rotation — issue a new refresh token each time it's used and invalidate the old one. This detects refresh token theft (if old token is reused, both sessions are killed).</div>

What is the Cryptographic Agility principle in security architecture?	<div style="text-align:left"><b>Cryptographic Agility</b> means designing systems so that cryptographic algorithms can be swapped out without requiring major code rewrites — because today's strong algorithm may be tomorrow's broken one.<br><br><b>Why it matters:</b><br>- MD5 was once strong — now cracked in seconds<br>- SHA-1 deprecated after collision attacks<br>- RSA and ECC are threatened by quantum computing (Post-Quantum Cryptography is coming)<br>- NIST finalized post-quantum standards in 2024 (CRYSTALS-Kyber, CRYSTALS-Dilithium)<br><br><b>How to achieve it:</b><br>1. Abstract cryptographic operations behind interfaces<br>2. Store the algorithm identifier alongside encrypted data:<br><code>{ "alg": "AES-256-GCM", "ciphertext": "...", "iv": "..." }</code><br>3. Use a KMS or Vault that supports algorithm configuration<br>4. Avoid hardcoding algorithm names — make them configurable<br><br><b>Senior insight:</b> Start planning post-quantum migration now for systems handling data that must remain secret for 10+ years. Harvest now, decrypt later attacks are real — adversaries store encrypted traffic today to decrypt it when quantum computers arrive.</div>
